# S4.B — Checkpoint & Undo

## Problem

No way to revert AI-made changes short of `git reset`. For a "transparency + user control" product, absence of undo is embarrassing. RooCode / Cline both have one-click turn revert.

## Tasks

- [x] Pre-mutation snapshot: on any write/create/delete/edit tool success, copy prior file state to `.locus/checkpoints/<session_id>/<turn_id>/<relative_path>`
- [x] `checkpoint.json` per turn: list of `(path, action, prev_hash)` for deterministic restore
- [x] `ILocusCore::undo_turn(session_id, turn_id)` — restore all files from that turn's snapshot, emit activity event
- [x] GC: keep last N=50 turns per session (configurable); prune on session close
- [x] Skip large files (> 1 MB) with a warning — defer to git for those
- [x] UI: "Undo last turn" button in chat footer; per-turn revert from activity panel
- [x] Catch2 tests: snapshot/restore round-trip, GC eviction, size-limit skip

## Implementation notes

- New subsystem: [src/agent/checkpoint_store.h/cpp](../../src/agent/checkpoint_store.h). On-disk layout is `<workspace>/.locus/checkpoints/<session_id>/<turn_id>/checkpoint.json` + `files/<rel_path>` byte-for-byte snapshots. The session id is a process-lifetime timestamp generated by `AgentCore` (not the same identity as a saved conversation session — see "Session identity" below). Turn ids are a monotonic counter bumped on every user message.
- The snapshot is taken *after* the approval gate but *before* the tool executes — so undo restores the bytes the user actually saw at approval time, even when they used "Modify" to tweak the args. Only `edit_file` / `write_file` / `delete_file` snapshot; everything else is a no-op.
- Atomicity: snapshots are written with `temp + rename`-style semantics through the OOXML/zip helpers we already use for atomic file writes. The manifest is rewritten in full on every entry append (read-modify-write under a `std::mutex`), which is fine because turns are small (a few mutations each).
- Hash is FNV-1a 64-bit, hex-encoded. It's used to identify the captured bytes in the manifest, not for any security purpose. SHA-1 / SHA-256 would have meant pulling in another dependency for no gain.
- "Per-turn revert" is exposed via `/undo [turn_id]` and via `AgentCore::list_checkpoints()` (the GUI's chat-footer "Undo" button calls `undo_turn(0)`, which resolves to "the most recent turn that has a checkpoint"). The activity panel doesn't render per-turn revert as separate clickable rows — building that needs a turn → activity-event correlation we don't have yet (the activity log doesn't carry turn ids). Tracked as a follow-up; the building blocks (`list_checkpoints` / `undo_turn(id)`) are ready.
- Idempotency: a second `snapshot_before` call for the same `(session, turn, path)` is a no-op so that a tool which touches a file twice in one turn doesn't lose the original bytes. Restore drops the manifest after applying so the same turn cannot be undone twice.
- GC fires at the *start* of every turn (`gc_session(session_id)`), pruning oldest turns above `max_turns_per_session = 50`. Empty turn dirs (no mutations) are dropped at end-of-turn so `/undo` doesn't see phantom no-op turns.
- Reset: `AgentCore::reset_conversation()` drops the entire current checkpoint session and starts a fresh one. Saved conversation sessions are deliberately separate — checkpoint history doesn't follow `save_session` / `load_session` because the per-process snapshot is meant to be short-lived ("undo what just happened"), not part of the persistent conversation record.
- Size limit: 1 MB hard-coded for now (`CheckpointStore::set_max_file_size_bytes`). Oversized files get a manifest entry with `skipped=true` and no body — restore reports them in `result.skipped` with an explanatory error. The expectation is users use `git stash` / `git checkout` for big binary files; checkpoints are for the source-code working set.
- Slash commands (`SlashCommandDispatcher`) bypass the dispatcher and call `tool->execute` directly, so a tool invoked through `/edit_file foo` does **not** snapshot. This is acceptable for now — a slash command is an explicit user action — but if it bites later, the fix is to route slash dispatch through `ToolDispatcher` too.

### Session identity

Two distinct "session" concepts coexist in the codebase:

| Concept | Owner | ID space | Lifetime |
|---|---|---|---|
| Saved conversation session | `SessionManager` | timestamped filename in `.locus/sessions/` | persistent across runs |
| Checkpoint session | `AgentCore` | timestamp generated at process start | one per AgentCore lifetime |

A saved session being loaded does not adopt the checkpoint session of the original run. This avoids the awkward state where loading a 3-day-old session would expose stale checkpoints (or worse, snapshots whose file paths drifted). The cost is that checkpoint history isn't preserved across restarts — which matches the "undo what just happened" UX, not "rewind history."

## Tests

`tests/test_checkpoint_store.cpp` (`[s4.b]`):
- `snapshot + restore round-trip (edit)` — happy path
- `deleting a created file undoes the create` — `existed=false` branch
- `delete-action snapshot restores the file` — symmetric to write
- `GC keeps last N turns per session` — eviction
- `oversized files are recorded but skipped` — size limit
- `re-snapshotting same path in a turn is a no-op` — idempotency
- `drop_turn / drop_session` — manual cleanup
- `empty session_id disables snapshotting` — defensive
