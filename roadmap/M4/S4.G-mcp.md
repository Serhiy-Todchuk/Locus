# S4.G -- MCP (Model Context Protocol) Client

## Problem

Every serious agent in 2026 speaks MCP: Claude Code, Cline, RooCode, OpenCode, Continue, Cursor. It's how users plug in Jira, Postgres, Playwright, Figma, internal tools. Locus has 12 hardcoded tools and zero plugin story (community-plugin item is deferred to post-M3 in the old plan -- that's wrong; MCP is the plugin story).

## Tasks

### Phase 1 -- Engine ✔

- [x] `McpClient` class: spawn MCP stdio server as child process; handshake via JSON-RPC
- [x] `.locus/mcp.json` + global `%APPDATA%/Locus/mcp.json`: list of servers with `command`, `args`, `env`
- [x] Discover tools via `tools/list`; register into `ToolRegistry` with namespaced names (`mcp:servername:toolname`)
- [x] Proxy tool calls: `execute()` on MCP tool -> JSON-RPC call -> result back to agent
- [x] Per-server approval policy default: `ask` (MCP tools untrusted until user opts in via the existing per-tool approval map)
- [x] Graceful degradation: server crash -> `available()=false` so the LLM stops seeing the tools, with a warning event for the activity log
- [x] Catch2 tests with a bundled mock MCP server (`locus_mock_mcp_server`)
- [x] Integration test (`tests/integration/test_int_mcp.cpp`, tag `[integration][llm][mcp]`) that exercises the end-to-end agent loop: live LLM picks the namespaced `mcp:mock:echo` tool, call routes through `ToolDispatcher -> McpTool -> McpClient -> stdio -> mock server`, echoed text comes back in the result. Soft-skips the round-trip assertions when small models fail to pick the MCP tool, matching the `[plan]` convention.

### Phase 2 -- engine debt + GUI (deferred follow-up)

**Engine prerequisites (block the GUI work):**

- [ ] `IToolRegistry::unregister_tool(name)` -- removes the hot-restart skip-on-duplicate workaround in `McpManager::register_tools_locked`. A server that restarts with a different tool set must reflect that exactly, not leak the old entries. ~10 LOC plus a Catch2 case.
- [ ] Explicit child-exit detection. Today `McpClient::handle_close` infers "crashed" from "pipe closed", which conflates a graceful exit, a stdout close, and a real crash. Either start a small monitor thread on `process_` or call `GetExitCodeProcess` in the reader-loop tail, and surface the exit code on `ServerStatus`. The settings panel needs the real signal.
- [ ] Cap MCP tool-result size at 1 MB to match the checkpoint per-file cap. A chatty or buggy server returning a 10 MB blob would bloat `ConversationHistory` before any compaction triggers. Implement in `McpTool::execute` after `flatten_content`: if `body.size() > cap`, truncate with a `[truncated N MB -- raise cap or fix server]` suffix and log a warning.

**GUI:**

- [ ] MCP server panel in settings (running / stopped / failed / crashed, exit code on failure, tool count, last-error tail, **Restart** button, **Open mcp.json** shortcut).
- [ ] Per-server "trust" toggle that applies `auto_approve` to all of `mcp:<name>:*`. Needs `ToolDispatcher` prefix-match support in the per-tool override lookup -- trivial extension.
- [ ] `tail -f` style view over each server's recent log lines so users can debug a flaky server without an external terminal.

**Documentation:**

- [ ] Document the `mcp.json` schema outside of code -- a new section in [CONTRIBUTING.md](../../CONTRIBUTING.md) or a dedicated `architecture/mcp.md` listing every accepted key (`command`, `args`, `env`, `cwd`, `enabled`, `init_timeout_ms`), the workspace-overrides-global precedence rule, and a worked example. Currently users have to read `src/mcp/mcp_config.cpp` to know what's valid.

**Phase 2 exit criterion:**

- [ ] Validate end-to-end against a real shipping MCP server (e.g. `@modelcontextprotocol/server-filesystem` via `npx`). The mock server only exercises the protocol surface we designed for ourselves; capability negotiation, multi-content responses, and progress notifications need a real implementation to catch corner cases the spec allows but our mock skips.

### Out of scope for S4.G (parked)

- [ ] Streamable-HTTP MCP transport -- stdio works for every server we care about today; HTTP is its own stage if/when needed.
- [ ] POSIX `stdio_transport` -- belongs to the broader Locus cross-platform effort, not S4.G specifically.

## Implementation notes

- **Layout**: new `src/mcp/` directory. Six pairs land in `locus_core`:
  - [src/mcp/mcp_types.h](../../src/mcp/mcp_types.h) -- `McpServerConfig`, `McpToolDefinition`
  - [src/mcp/mcp_config.h/cpp](../../src/mcp/mcp_config.h) -- merged loader (workspace overrides global)
  - [src/mcp/json_rpc.h/cpp](../../src/mcp/json_rpc.h) -- ~100 LOC JSON-RPC 2.0 helpers + tolerant parser
  - [src/mcp/stdio_transport.h/cpp](../../src/mcp/stdio_transport.h) -- Win32 child-process spawn (KILL_ON_JOB_CLOSE, env-block merge, line-oriented reader thread)
  - [src/mcp/mcp_client.h/cpp](../../src/mcp/mcp_client.h) -- handshake, sync request/response via promise/future, status state machine
  - [src/mcp/mcp_tool.h/cpp](../../src/mcp/mcp_tool.h) -- `ITool` wrapper, namespaced names, JSON Schema passthrough
  - [src/mcp/mcp_manager.h/cpp](../../src/mcp/mcp_manager.h) -- per-workspace pool, registers tools into `IToolRegistry`
- **JSON-RPC framing**: newline-delimited JSON messages over stdin/stdout. (The MCP spec also allows LSP-style Content-Length framing, but every shipping stdio MCP server uses the newline form.)
- **JSON Schema passthrough**: new optional virtual `ITool::parameters_schema()` returns the raw JSON Schema for the LLM-facing tool manifest. Built-in tools leave the default empty implementation, in which case `ToolRegistry::build_schema_json` derives the schema from the flat `params()` list. MCP tools override `parameters_schema()` to ship the full server-supplied `inputSchema` -- nested objects, enums, `oneOf`, anything the server advertises -- without lossy round-tripping.
- **Approval defaults**: `McpTool::approval_policy()` returns `ToolApprovalPolicy::ask`. Users opt in per-tool via the existing `tool_approvals` map in `.locus/config.json` (key = `mcp:<server>:<tool>`). No new code path -- `ToolDispatcher`'s existing per-tool override lookup just works for the namespaced names.
- **Lifecycle in `LocusSession`**: declared between `llm_` and `tools_` but assigned AFTER `tools_` (built-ins first, then `mcp_->start_all()`). `unique_ptr` lets construction order diverge from declaration order; declaration order alone determines destruction. The result: agent thread joins -> `mcp_` destroys (terminates clients) -> `tools_` destroys (McpTools see dangling but unused `client_` pointers; their dtors don't dereference, so it's safe).
- **Crash handling**: `McpClient::handle_close()` runs on the StdioTransport reader thread. Status flips to `crashed`, every pending request fails with an exception, and the optional `on_crash` callback fires. `McpManager` registers a callback that emits a status event so the GUI (Phase 2) can refresh; `McpTool::available()` returns false after a crash so the LLM stops seeing the tools without us having to remove them from the registry (`ToolRegistry` doesn't currently support unregistration).
- **Mock server for tests**: tiny [tests/mock_mcp_server/main.cpp](../../tests/mock_mcp_server/main.cpp) speaks the bare subset of MCP we exercise (`initialize`, `tools/list`, `tools/call` with two tools: `echo`, `fail_once`). Built as a separate `locus_mock_mcp_server` exe target placed next to `locus_tests.exe` so the tests resolve it via `GetModuleFileName`. Crash simulation via `LOCUS_MOCK_MCP_CRASH_AFTER=<n>` env.
- **Tests** ([tests/test_mcp.cpp](../../tests/test_mcp.cpp)): 13 cases covering JSON-RPC parser, `mcp.json` loading + merge, stdio handshake, tools/list + tools/call round trip, `isError` propagation, server crash lifecycle, manager registration in `ToolRegistry`, and graceful no-op when a configured command can't be spawned. All tagged `[s4.g][mcp]`.
- **Schema injection cost**: each registered MCP tool adds its description + `inputSchema` to the per-turn manifest. The existing S3.L `tool_manifest_warn_tokens` guardrail (4000 default) still fires; users with chatty MCP servers will see warnings in the log and can disable individual servers in `mcp.json`.

## Phase 2 notes

Concrete task list is in the **Phase 2 -- engine debt + GUI** section above. The engine items (unregister hook, explicit exit detection, result-size cap) are sequenced first because the GUI surfaces depend on them: the restart button can't behave correctly without `unregister_tool`, the status panel wants real exit codes, and the log-tail view assumes the result-size cap is keeping `ConversationHistory` clean. Real-server validation gates the whole phase -- the mock server alone is not a sufficient signal that we speak MCP correctly.

Streamable-HTTP transport remains parked: every MCP server in the wild today ships a stdio mode, so the engine landed here covers the protocol's installed base.
