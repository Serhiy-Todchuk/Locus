# S4.T — File-Change Awareness Between Turns ✔

## Problem

If user edits a file between LLM turns, agent has no idea. Index updates silently; agent's mental model is stale next turn.

## Tasks

- [x] Track file mtimes between turns; compute "files changed since last assistant turn"
- [x] Auto-inject as ephemeral system note on next user turn: `[Files changed since last turn: foo.cpp, bar.h]`
- [x] Configurable: `agent.notify_external_changes` (default on)
- [x] Exclude files the agent itself changed (don't echo its own edits back)

## Implementation notes

- New thread-safe [`FileChangeTracker`](../../src/agent/file_change_tracker.h) holds
  a `path -> mtime` snapshot plus an "agent touched this turn" suppression set.
  `snapshot()` and `diff_since_snapshot()` both source from
  `IndexQuery::list_directory("", huge_depth)` -- the same enumeration the S4.P
  regex tool uses, so the indexer's exclude patterns are respected for free
  and there's no second filesystem walk.
- `AgentCore` owns the tracker as a `unique_ptr` member. Lifecycle:
  - **Construction**: take the initial snapshot so the first user turn diffs
    against "the world as it was when we started", not against an empty set
    (which would otherwise flag every indexed file as new).
  - **Top of each user turn** (after `++turn_id_`, before `history_.add(user)`):
    diff against the snapshot, prepend `[Files changed since last turn: ...]\n\n`
    to the user's message content if non-empty, then `clear_agent_touched()`.
  - **End of each turn**: `snapshot()` again so the next turn's diff is
    relative to the post-turn state (the touched set is cleared next turn so
    we don't suppress forever).
  - **Reset / load_session**: re-snapshot + clear touched set.
- `ToolDispatcher::set_change_tracker()` lets `AgentCore` hand the tracker to
  the dispatcher. The existing `maybe_snapshot()` (which already discriminates
  `edit_file` / `write_file` / `delete_file` for S4.B checkpoints) gained a
  second sink: it calls `change_tracker_->mark_agent_touched(rel_path)` for
  the same set of mutating tools, using the same path resolution. Slash
  commands deliberately bypass (they don't go through the dispatcher) -- they
  are explicit user actions and don't need to be filtered out.
- The note is prepended to the user message (rather than emitted as a
  separate system message) so it travels naturally with the turn that
  consumed it, persists in saved sessions for replay, and survives
  compaction in the same way as the user's own text. This is the same
  pattern RooCode / Cline use.
- Each diff bucket caps at 50 entries by default; if the user touches a
  thousand files between turns the LLM gets a representative slice rather
  than a wall of paths flooding the prompt.
- Watcher latency caveat: the index sees external edits only after the
  `WatcherPump` debounce (1.5 s quiet period). If the user edits and asks
  the agent within that window, the change shows up on the *next* turn
  rather than this one. Acceptable trade-off -- fixing it would mean a
  per-turn synchronous `flush_now()` from `AgentCore`, which would block
  the agent thread on indexer I/O for every prompt.

## Tests

- [tests/test_file_change_tracker.cpp](../../tests/test_file_change_tracker.cpp)
  -- 9 cases tagged `[s4.t][file-change-tracker]`. Bypass the async file
  watcher by feeding synthetic `FileEvent`s into `Indexer::process_events`
  directly: covers initial snapshot, no-change, modified, added, deleted,
  agent-touched suppression, suppression-clear, snapshot rebasing, and the
  per-bucket cap.
