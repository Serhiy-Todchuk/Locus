[ ] **Force approval + warn on outside-workspace paths in `run_command` / `run_command_bg`.** Even when the tool is set to "always approve in this workspace" (bullet above), individual shell calls whose command string references paths *outside* the current workspace re-trigger the approval modal with an explicit warning. Reason: a workspace-level trust decision says "I trust the agent to run shell commands inside this folder"; it does NOT say "I trust the agent to delete `C:\Users\me\Documents\notes`." Implementation: heuristic path scanner runs against the command string (tokenize on whitespace + shell quotes, look for tokens that are absolute paths, parent-relative paths with `..`, or expanded environment variables like `%USERPROFILE%` / `$HOME` / `~`). For each candidate, canonicalize against the workspace root; if the canonical path is not under the root, flag the call. `ToolDispatcher` upgrades the effective approval policy from `always` -> `require_approval` for flagged calls, attaches the offending paths as call metadata. The approval panel renders a yellow/red banner ("âš  This command references paths outside the workspace: `C:\Users\me\Documents`, `..\..\other-repo`") and highlights the offending tokens in the args display in red. The user can still approve as normal -- this is a safety prompt, not a refusal. Doesn't apply to file tools (`read_file` / `write_file` / `edit_file` / `delete_file`), which already refuse outside-workspace paths at the `resolve_path` layer in [src/tools/shared.h](../../src/tools/shared.h). Heuristic limits documented (e.g., we don't parse `cmd /c "cd D:\foo && ..."` deeply -- shell-language reasoning is out of scope; if the user wants deep auditing they keep "always approve" off).
