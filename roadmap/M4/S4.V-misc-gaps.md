[x] **Force approval + warn on outside-workspace paths in `run_command` / `run_command_bg`.** Even when the tool is set to "always approve in this workspace" (bullet above), individual shell calls whose command string references paths *outside* the current workspace re-trigger the approval modal with an explicit warning. Reason: a workspace-level trust decision says "I trust the agent to run shell commands inside this folder"; it does NOT say "I trust the agent to delete `C:\Users\me\Documents\notes`." Implementation: heuristic path scanner runs against the command string (tokenize on whitespace + shell quotes, look for tokens that are absolute paths, parent-relative paths with `..`, or expanded environment variables like `%USERPROFILE%` / `$HOME` / `~`). For each candidate, canonicalize against the workspace root; if the canonical path is not under the root, flag the call. `ToolDispatcher` upgrades the effective approval policy from `always` -> `require_approval` for flagged calls, attaches the offending paths as call metadata. The approval panel renders a yellow/red banner ("⚠ This command references paths outside the workspace: `C:\Users\me\Documents`, `..\..\other-repo`") and highlights the offending tokens in the args display in red. The user can still approve as normal -- this is a safety prompt, not a refusal. Doesn't apply to file tools (`read_file` / `write_file` / `edit_file` / `delete_file`), which already refuse outside-workspace paths at the `resolve_path` layer in [src/tools/shared.h](../../src/tools/shared.h). Heuristic limits documented (e.g., we don't parse `cmd /c "cd D:\foo && ..."` deeply -- shell-language reasoning is out of scope; if the user wants deep auditing they keep "always approve" off). **Done:** new pure scanner in [src/tools/command_path_scanner.h/cpp](../../src/tools/command_path_scanner.h) -- `scan_outside_workspace_paths(cmd, root)` tokenizes on whitespace respecting `"..."` / `'...'` quotes and flags tokens matching any of five shapes: absolute Windows path (`C:\...`), UNC path (`\\server\share`), Unix-absolute (`/...`), parent-traversal segment (`..\...` / `../...`), env-var or `~` expansion (`%FOO%` / `$BAR` / `~`). Categories 1-4 are canonicalized against the workspace root via `weakly_canonical` and only flagged when the result is outside; env-var tokens are flagged unconditionally (we don't expand). `ToolDispatcher::dispatch` runs the scanner for `run_command` / `run_command_bg`; non-empty results downgrade `auto_approve` -> `ask` and flow as `safety_warnings` through `IFrontend::on_tool_call_pending`'s new fourth parameter. `ToolApprovalPanel::show_tool_call` now accepts the warning list and renders a bold amber banner above the JSON args ("WARNING: command references paths outside the workspace:" plus one indented line per token); `CliFrontend::on_tool_call_pending` prints the same content above the y/n/e prompt. Highlighting individual tokens *inside* the JSON args view in red was scoped out -- the dedicated banner above the args is the user-facing signal. The IFrontend signature change ripples through `WxFrontend`, `CliFrontend`, `HarnessFrontend`, and the three test stub frontends. 13 new `[s4.v][path-scanner]` Catch2 cases / 27 assertions in [tests/test_command_path_scanner.cpp](../../tests/test_command_path_scanner.cpp); manual test plan at [tests/manual/workspace-trust.md](../../tests/manual/workspace-trust.md).
