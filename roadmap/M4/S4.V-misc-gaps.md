- [ ] `@`-mention file references in chat input: type `@path/to/file` -> auto-attach to context. Builds on the existing S2.4 attach surface; UX parity with Cursor / Continue / RooCode.
- [ ] "Always approve this tool in this workspace" -- persistent per-workspace trust decision, stored in `.locus/config.json`. Same shape as the per-server MCP approval policy ([S4.G](S4.G-mcp.md) defaults `always`), generalised to built-in tools. Transparency thesis preserved: the activity log still records every call, undo via [S4.B](S4.B-checkpoint-undo.md) still works, the entry is visible/deletable in config. Implementation note: for mutating tools (`edit_file`, `write_file`, `delete_file`, `run_command`) show a one-time friction confirm on the first always-approve action ("this skips approval for all future calls in this workspace; activity log still records them"). One speed bump per tool per workspace. Revoke via Settings or by removing the config entry.
- [ ] **Force approval + warn on outside-workspace paths in `run_command` / `run_command_bg`.** Even when the tool is set to "always approve in this workspace" (bullet above), individual shell calls whose command string references paths *outside* the current workspace re-trigger the approval modal with an explicit warning. Reason: a workspace-level trust decision says "I trust the agent to run shell commands inside this folder"; it does NOT say "I trust the agent to delete `C:\Users\me\Documents\notes`." Implementation: heuristic path scanner runs against the command string (tokenize on whitespace + shell quotes, look for tokens that are absolute paths, parent-relative paths with `..`, or expanded environment variables like `%USERPROFILE%` / `$HOME` / `~`). For each candidate, canonicalize against the workspace root; if the canonical path is not under the root, flag the call. `ToolDispatcher` upgrades the effective approval policy from `always` -> `require_approval` for flagged calls, attaches the offending paths as call metadata. The approval panel renders a yellow/red banner ("⚠ This command references paths outside the workspace: `C:\Users\me\Documents`, `..\..\other-repo`") and highlights the offending tokens in the args display in red. The user can still approve as normal -- this is a safety prompt, not a refusal. Doesn't apply to file tools (`read_file` / `write_file` / `edit_file` / `delete_file`), which already refuse outside-workspace paths at the `resolve_path` layer in [src/tools/shared.h](../../src/tools/shared.h). Heuristic limits documented (e.g., we don't parse `cmd /c "cd D:\foo && ..."` deeply -- shell-language reasoning is out of scope; if the user wants deep auditing they keep "always approve" off).
- [ ] Model-card presets: LM Studio / Ollama / llama-server + known-good params per model family (temperature, sampler defaults, `tool_format` hint). "Load preset" button in Settings; presets are static metadata, not auto-detected. S4.N's `ToolFormat::Auto` already covers the most common pain point, so this is ergonomics, not load-bearing.
- [ ] LLM sampler controls exposed in Settings: `top_p`, `top_k`, `min_p`, repetition penalty. Already passed through `LLMConfig`; UI-only addition. Power-user surface; defaults stay as-is.

On a 26B local model, generation tokens are far slower per-token than prompt tokens, so the split is real diagnostic information for "why was that turn slow?".
