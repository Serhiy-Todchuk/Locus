# S4.M -- Tree-Sitter Query Tool (Structural Grep)

## Problem

`search_symbols` finds names. Tree-sitter `.scm` queries find *shapes*: all `call_expression` nodes where function is `malloc`, all `if_statement` with empty body, all classes inheriting `ITool`. `ast-grep` built a product on this -- killer for refactoring.

## Implementation notes

### TreeSitterRegistry sharing

Each `search_ast` call constructs its own short-lived `TreeSitterRegistry`. The class is cheap to instantiate (`ts_parser_new()` + a 9-entry language map of static globals); creating one per call sidesteps any locking against the indexer's parser, which is owned by `Indexer` and serialised under `process_mutex_`. Long-running parser pools can be added later if profiling shows it matters.

### Predicate evaluation

Tree-sitter's `ts_query_cursor_next_match` does NOT enforce `#eq?` / `#match?` / `#any-of?` predicates -- it only parses them and exposes the steps via `ts_query_predicates_for_pattern`. The first iteration of S4.M skipped this and returned every structural match, which made `(#eq? @fn "malloc")` a no-op (test caught it). The fix: a `match_passes_predicates()` helper that walks the predicate steps, resolves capture text + string literals, and applies the common subset:

- `eq?` / `not-eq?` -- string equality between a capture and a literal or another capture.
- `match?` / `not-match?` -- ECMAScript regex against a capture.
- `any-of?` / `not-any-of?` -- capture text against a list of literals.

Unknown predicates accept the match (forward-compatible with future Tree-sitter additions). Bad regex inputs are treated as no-ops rather than dropping the match silently.

### File enumeration

Same pattern as `mode=regex` (S4.P): `IndexQuery::list_directory("", huge_depth)` gives every indexed, non-binary file with `language` already populated by the indexer's extension map. We filter on `fe.language == requested_language` so a `language=cpp` query only parses `.cpp/.cc/.cxx/.h/.hpp`, not the whole tree. `path_glob` is applied with the same `glob_match` helper used by the regex tool. 1 MB per-file read cap mirrors the regex tool's pathological-file guard.

### Tool catalog impact

Tool count unchanged at 13 -- `SearchAstTool` is registered as a peer ITool for direct test access but is NOT in `register_builtin_tools()`. The LLM only sees the unified `search` tool, with one extra mode value in its description and three extra params (`language`, `capture`, `path_glob`). Manifest growth: a few dozen tokens, well under the warn threshold.

### What's deferred

Replacing the rule-based symbol extractors (`src/index/symbol_extractors/*.cpp`) with `.scm` queries was listed in the original task list. The work is non-trivial -- nine languages, each currently using a small `SymbolRule` table that maps node types to symbol kinds and reads names via field-name walks. The rule-based path was just refactored in S3.D; a `.scm` rewrite would be a parallel implementation, not a simplification. Punted to a future stage so S4.M ships the user-visible feature (structural search) without dragging the indexer along.
