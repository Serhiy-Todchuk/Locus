# S4.F -- KV / Prompt Cache Preservation

## Problem

llama.cpp / LM Studio support prefix caching, but only when the system-prompt prefix is byte-stable across turns. Any per-turn change in the system message invalidates the cache and forces re-processing of the full prefix (2000+ tokens; 10-30 s on a 26B model).

## Audit findings (2026-05)

Walked `SystemPromptBuilder::build()` and `AgentCore::compose_system_prompt()`. The phantoms originally listed here (timestamps, session IDs, last-indexed times, live file counts) are **not** in the prompt today:

- `base_system_prompt_` is built **once** in `AgentCore::AgentCore()` ([src/agent/agent_core.cpp:51](../../src/agent/agent_core.cpp)) from the base instructions, workspace metadata snapshot, LOCUS.md, tool manifest, and tool-format addendum. It is never recomputed within a session.
- Workspace metadata (`file_count`, `symbol_count`, `heading_count`) is captured at construction. It does NOT refresh as the indexer updates -- a deliberate cache-friendly choice (the model rarely needs live counts; the tradeoff is acceptable staleness).
- File-change awareness (S4.T) prepends `[Files changed since last turn: ...]` to the **user** message, not the system prompt -- correct, lives in the volatile tail.

The one real cache-buster is **attached-file context**. `set_attached_context()` calls `refresh_system_prompt()`, which appends the attached-file outline + preview to the base prompt and rewrites the in-history system message via `replace_system_prompt()` ([src/agent/agent_core.cpp:163-169](../../src/agent/agent_core.cpp)). Each attach / detach invalidates the cache from the system message onward.

## Tasks

- [ ] Move the attached-file block out of the system prompt and onto the latest **user** message (same shape S4.T uses for file-change awareness). The system prompt stays byte-stable; attach-state lives in the volatile tail where invalidation is already expected. Matches Cline / RooCode behavior.
- [ ] Document the invariant in `system_prompt.h`: *"`SystemPromptBuilder::build()` output must be byte-stable across turns within a session. Anything that changes per turn (counts, timestamps, attached context, file-change deltas) belongs on a user message, not here."*
- [ ] Add a debug assertion in `AgentCore` (debug builds only): on every LLM round trip, hash the system message and warn if it changed without an explicit attach/detach event. Cheap canary against future regressions.
- [ ] Verify with LM Studio verbose logging / llama.cpp `-v` that prefix-cache hits actually fire after the move. Document the log lines that prove a cache hit in the stage doc so future debugging has a known target.
- [ ] Add a `prompt_prefix_stable` boolean (or `kv_cache_hit_est` heuristic) to the metrics panel, derived from per-turn system-message byte equality. Cheap to compute, gives the user a visible signal that local inference is fast for the right reason.

## Implementation notes

- The attached-file block format itself need not change -- only its location in the message list. The tool-call format and outline rendering stay as today.
- LOCUS.md and tool manifest are intentionally allowed to change *between sessions* (workspace reload, tool registry edits) -- those are full-prompt-rebuild events and the cache miss is appropriate. The invariant is per-session.
- The debug-only hash check should be cheap (`std::hash<std::string>` over the system message). Not a release-build cost.
- Workspace-metadata staleness is a known tradeoff: refreshing live counts would torch the cache on every index event. If a future stage needs fresh counts, expose them via a tool (`get_workspace_stats`) rather than the system prompt.
