# S4.F -- KV / Prompt Cache Preservation

## Problem

llama.cpp / LM Studio support prefix caching, but only when the system-prompt prefix is byte-stable across turns. Any per-turn change in the system message invalidates the cache and forces re-processing of the full prefix (2000+ tokens; 10-30 s on a 26B model).

## Audit findings (2026-05)

Walked `SystemPromptBuilder::build()` and `AgentCore::compose_system_prompt()`. The phantoms originally listed here (timestamps, session IDs, last-indexed times, live file counts) are **not** in the prompt today:

- `base_system_prompt_` is built **once** in `AgentCore::AgentCore()` ([src/agent/agent_core.cpp:51](../../src/agent/agent_core.cpp)) from the base instructions, workspace metadata snapshot, LOCUS.md, tool manifest, and tool-format addendum. It is never recomputed within a session.
- Workspace metadata (`file_count`, `symbol_count`, `heading_count`) is captured at construction. It does NOT refresh as the indexer updates -- a deliberate cache-friendly choice (the model rarely needs live counts; the tradeoff is acceptable staleness).
- File-change awareness (S4.T) prepends `[Files changed since last turn: ...]` to the **user** message, not the system prompt -- correct, lives in the volatile tail.

The one real cache-buster is **attached-file context**. `set_attached_context()` calls `refresh_system_prompt()`, which appends the attached-file outline + preview to the base prompt and rewrites the in-history system message via `replace_system_prompt()` ([src/agent/agent_core.cpp:163-169](../../src/agent/agent_core.cpp)). Each attach / detach invalidates the cache from the system message onward.

## Field findings (2026-05-08, qwen3.6-35b-a3b on LM Studio)

Real session log: `D:/Projects/TestLocalVibe4/.locus/locus.log` -- one user prompt ("Create cornell-box renderer with WebGPU, multiple files"), 19 LLM rounds, ~75 minutes wall time on a consumer CPU/GPU.

**Every round reports `cached=0`.** Sample lines from the log (decoded via S4.U's `usage.prompt_tokens_details.cached_tokens` plumbing):

```
R1:  prompt=4164  (cached=0), completion=802
R2:  prompt=4695  (cached=0), completion=598
R5:  prompt=8557  (cached=0), completion=3649
R10: prompt=15893 (cached=0), completion=3470
R14: prompt=22683 (cached=0), completion=464
R18: prompt=26905 (cached=0), completion=76
```

The user's machine has slow generation throughput on this model, so re-prefilling 4K → 27K growing prefix on every round dominates wall time. **No** turn paid less prefill than the previous turn -- caching is doing nothing.

Two stalls of 600s tripped the stream watchdog (lines [414](../../D:/Projects/TestLocalVibe4/.locus/locus.log#L414), [460](../../D:/Projects/TestLocalVibe4/.locus/locus.log#L460)) -- consistent with cold prefill of 12K-16K tokens taking 10+ minutes before the first generated token reaches the wire. With a working prefix cache those rounds would have started generating in seconds.

### Why caching is dead in this session

This requires investigation, but the prime suspects are:

1. **Attached context churning the system prompt** -- the audited cache-buster above. If the user attached a file mid-session (or LOCUS.md was reloaded), `replace_system_prompt()` torched the cache. **Verify by**: hashing the system message on every `LLM POST` and logging at `info` when it changes; correlate with `cached=0` rounds.
2. **LM Studio's qwen template re-encoding the prefix** -- some templates inject the *current* tool list / role tokens differently per turn even when our outgoing JSON `messages[0].content` is byte-identical. **Verify by**: enabling LM Studio's verbose logging (`Developer → Logs → Raw Prompts`) and diffing the actual rendered prompt sent to llama.cpp across two consecutive turns. Look for chat-template re-tokenization of the system prefix.
3. **`/v1/chat/completions` cache scope** -- LM Studio caches per-conversation hash; if our request body's `messages` array order shifts even one byte (e.g. tool_calls JSON serialization order), the cache key changes. **Verify by**: dumping the exact request body for two consecutive turns and `diff`'ing.
4. **`prompt_tokens_details.cached_tokens` not populated by this LM Studio version** -- some servers don't include this field even when caching IS firing. **Verify by**: comparing wall-clock prefill time between turn N and turn N+1 with identical prefix; if turn N+1 is much faster, the cache works but the field is unset (and we should probe by timing instead).
5. **Per-tool-call argument escaping** -- when the model emits a tool_call, our serialization of `arguments` (a JSON-encoded string) might be byte-stable but LM Studio's chat-template renders it differently than its own emission. Crosses tool-call boundaries are common cache-miss points.

The investigation order should be 1 → 4 → 2 → 3 → 5 (cheapest signal first). Adding the per-turn system-prompt-hash log line is a 5-line change and instantly tells us whether (1) is happening.

## Implementation notes

- The attached-file block format itself need not change -- only its location in the message list. The tool-call format and outline rendering stay as today.
- LOCUS.md and tool manifest are intentionally allowed to change *between sessions* (workspace reload, tool registry edits) -- those are full-prompt-rebuild events and the cache miss is appropriate. The invariant is per-session.
- The debug-only hash check should be cheap (`std::hash<std::string>` over the system message). Not a release-build cost.
- Workspace-metadata staleness is a known tradeoff: refreshing live counts would torch the cache on every index event. If a future stage needs fresh counts, expose them via a tool (`get_workspace_stats`) rather than the system prompt.
