# S4.P — Grep Tool (Regex over Raw Content) ✔

## Problem

`search_text` uses FTS5 tokenization — loses punctuation, case, exact identifiers. For `->m_cache`, `TODO(XXX)`, regex patterns, FTS5 is the wrong tool.

Added `pattern`, `path_glob`, `case_sensitive` to `SearchTool::params()`; dispatch in `SearchTool::execute` next to the existing text/symbols/semantic/hybrid branches. (S3.L collapsed the search family; see [roadmap/M3/S3.L-tool-catalog-hygiene.md](../M3/S3.L-tool-catalog-hygiene.md).)

- [x] Decision: keep in-process (no external ripgrep dep) — `std::regex` is good enough at Locus scale
- [x] Line-numbered output, 2-line context snippet (±1 line around match)
- [x] Catch2 tests in [tests/test_search_regex.cpp](../../tests/test_search_regex.cpp): special chars, multi-line, case sensitivity, path_glob, max_results truncation, binary skip, schema

## Implementation notes

- New `SearchRegexTool` peer class in [src/tools/search_tools.h](../../src/tools/search_tools.h) / [.cpp](../../src/tools/search_tools.cpp), following the per-mode-class pattern used by `SearchTextTool` / `SearchSymbolsTool` / etc. Not registered directly — `SearchTool::execute` delegates to it on `mode=regex`. Exposed as a direct `ITool` so tests and the slash-command surface can invoke it.
- File enumeration: reuses `IndexQuery::list_directory("", huge_depth)` to get every indexed file (binary files flagged and skipped by the tool). No new query on `IndexQuery`.
- Per-file content read is capped at 1 MB; 500 B cap per snippet. Zero-width-match infinite loops are not a risk — `std::sregex_iterator` auto-advances past empty matches.
- `query` on the unified `search` tool is no longer `required: true` in the schema — each mode enforces its own required field at runtime (`query` for text/semantic/hybrid, `name` for symbols, `pattern` for regex). The description now spells out which mode uses which field.
