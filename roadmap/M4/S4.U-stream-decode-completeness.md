# S4.U -- Stream Decode Completeness + max_tokens Discipline done

## Problem

A user asked qwen3.6-35b-a3b to scaffold a multi-file WebGPU project. Round 1
succeeded (`write_file index.html`), round 2 silently produced a malformed
second tool call -- `write_file` with `arguments=""` -- and the round after
that failed with **HTTP 500** when LM Studio's chat template tried to render
the empty arguments string back into a prompt. From the user's seat: chat
window stayed empty, the tool approval panel showed a `(0 bytes)` write,
and the only visible error was the cryptic 500 ~90 seconds later.

Three independent gaps lined up to cause it:

1. **`max_tokens` was hardcoded to `2048`.** A single big-file write_file
   payload eats most of that; with reasoning tokens and a second tool call
   queued, the model's stream got chopped mid-arguments.
2. **`finish_reason` was never decoded.** The OpenAI/LM Studio stream sends
   `"finish_reason": "length"` on the closing chunk when `max_tokens` is hit,
   but [openai_decoder.cpp](../../src/llm/stream_decoders/openai_decoder.cpp)
   parsed only `delta.{content,reasoning_content,tool_calls}` and top-level
   `usage`. The truncation signal was thrown on the floor.
3. **Truncated tool calls flowed straight into history.** The accumulator in
   `LMStudioClient::stream_completion` delivered every `tool_accum` entry
   regardless of whether `arguments` parsed as JSON. The malformed call
   reached the dispatcher (returned a `'path' parameter is required` error,
   which is fine) -- but it also landed in `ConversationHistory` with the
   broken `arguments=""` baked in, which is what made the next request to
   the chat template fail.

Pi.dev didn't suffer because it defaults `max_tokens` higher; we needed to
fix it on three layers so the same class of failure can't recur.

## Tasks

- [x] `max_tokens` plumbed through `WorkspaceConfig` (`llm.max_tokens` in
      `.locus/config.json`), Settings dialog, and `LocusSession::merge_workspace_defaults`.
- [x] Default bumped from `2048` to `8192` in both `LLMConfig::max_tokens` and
      `WorkspaceConfig::llm_max_tokens`.
- [x] `IStreamDecoder` extended: new `on_finish_reason` and `on_stream_error`
      callbacks on `StreamDecoderSink`. All four decoders (OpenAi, Qwen,
      Claude, Auto) forward them.
- [x] `OpenAiDecoder` decodes `choices[0].finish_reason`,
      `delta.refusal`, top-level `{"error":...}` chunks, and
      `usage.prompt_tokens_details.cached_tokens`.
- [x] `LMStudioClient::stream_completion` warns + fires `on_error` when
      `finish_reason == "length"` (and on any other non-trivial stop reason
      like `content_filter`). The warning includes the actual `max_tokens`
      and `completion_tokens` so the user knows the gap, plus a pointer to
      `llm.max_tokens` and the Settings dialog.
- [x] Tool-call validation gate: before delivering accumulated tool calls,
      each entry's `arguments` string is JSON-parsed; empty / whitespace /
      malformed entries are dropped with a `spdlog::warn` and a clear
      `on_error` message. When the drop coincides with `length` truncation,
      the error message says so explicitly.
- [x] `CompletionUsage::cached_tokens` field + decoder support, ready for
      future cache-rate telemetry without another schema bump.
- [x] Catch2 tests in [tests/test_stream_decoders.cpp](../../tests/test_stream_decoders.cpp)
      tagged `[s4.u]`: finish_reason decoded, null finish_reason ignored,
      refusal channel routed to text with prefix, top-level error chunk lands
      on `on_stream_error`, cached_tokens flows through, and each XML decoder
      forwards finish_reason/stream_error from its inner OpenAi decoder.

## Implementation notes

- `StreamDecoderSink` gained two `std::function` slots
  ([src/llm/stream_decoder.h](../../src/llm/stream_decoder.h)). Existing
  decoder implementations were a single-line update each -- copy the slots
  into their inner sink so the wrapped `OpenAiDecoder` can fire them. The
  XML decoders themselves never synthesize finish_reason or stream_error;
  they just pass through.
- The OpenAi-side change is additive. Mid-stream chunks normally carry
  `"finish_reason": null`; the decoder explicitly tests `is_string()` so a
  `null` value never fires the callback (the new test
  `OpenAiDecoder: null finish_reason is ignored` guards that).
- `delta.refusal` is rare (Claude / latest GPT emit it when declining); we
  route it through `on_text` with a `[refusal] ` prefix rather than a
  dedicated channel, on the theory that the user wants to see the refusal
  in the chat bubble, not in a side panel. Two extra lines of code, one
  test.
- `LMStudioClient::stream_completion` keeps a single `finish_reason` local
  (last writer wins -- defensive against backends that resend on the trailing
  chunk). The `length` branch is explicit; everything else non-trivial is
  bucketed into a generic "LLM stop reason: X" surface so unknown reasons
  still reach the user. `stop`, `tool_calls`, and `function_call` are the
  silent set.
- Tool-call validation runs after the stream ends and after the decoder's
  `finish_stream` flush. Drop criteria: empty `name`, empty (after trim)
  `arguments`, or `arguments` that doesn't parse as JSON. The XML decoders
  already pre-validate inside `XmlToolCallExtractor` (Qwen JSON-parses the
  body, Claude builds a `json::object` and `dump()`s it), so this gate
  primarily catches the OpenAI native-deltas truncation case. When at
  least one call is dropped because of `length` truncation, the error
  message is explicit about the cause.
- `cached_tokens` is decoded but not yet wired into `MetricsAggregator` --
  S4.S didn't track it; a follow-up can add a "cache hit rate" tile to the
  Metrics tab once we observe a server that actually reports it.
- Settings dialog adds a single "Max tokens (per response)" spin control
  next to "Context limit", with a one-line hint: "Max tokens caps a single
  response. Bump if multi-file edits get cut off." Range 256-1048576 (the
  spin can't be 0 -- the request body always sends a positive value).
- No ADR. The tool-call validation gate is a one-screen change inside one
  file; the decoder additions are pure-additive contract extensions; the
  `max_tokens` plumbing follows the exact pattern of `context_limit` /
  `tool_format` / `timeout_ms` already in `merge_workspace_defaults`.

## Validation

- Unit tests: 8 new `[s4.u]` cases pass alongside the 24 `[s4.n]` regression
  cases, no XML-extractor behaviour changed.
- Integration tests: full smoke + search + bg + slash + file-change-awareness
  run against qwen3.5-9b @ 16k.
- Manual reproduction: original "create cornel box with multiple files"
  prompt now either (a) completes both `write_file` calls within the new
  8192 cap, or (b) when forced to truncate by lowering `llm.max_tokens` in
  config, surfaces a clear "Response truncated: max_tokens limit reached
  (max_tokens=N, completion_tokens=N)" error in chat and a `spdlog::warn`
  in the log -- no more silent malformed call, no more HTTP 500 on the
  next round.
