# S4.E -- LSP Integration (via MCP)

## Problem

Tree-sitter gives a shallow symbol table. LSP gives semantic depth: cross-TU goto-def, find-references with overload resolution, rename-symbol, compiler diagnostics. For C++ (templates, namespaces) and TypeScript, `clangd` / `tsserver` are genuinely better than what Tree-sitter can provide.

But the original framing of this stage -- "build a custom C++ `LspClient` with five dedicated tools" -- doesn't survive a hard look. Three things changed since this stage was first drafted:

1. [S4.G -- MCP](S4.G-mcp.md) is in M4 and gives Locus a clean external-tool boundary.
2. [S4.C -- Auto-Verify](S4.C-auto-verify.md) covers the most common LSP need (compiler diagnostics) by parsing build output.
3. Mature LSP-as-MCP servers exist upstream (`mcp-language-server` and friends) and wrap clangd / tsserver / pylsp / gopls without us writing a line of LSP code.

The honest comparison: a custom C++ LSP client is months of ongoing maintenance (per-server quirks, crash handling, indexing-progress UX, `compile_commands.json` discovery for clangd, venv discovery for pylsp, ...) for capability that MCP delivers via a config snippet. The latency overhead of MCP-vs-native is invisible at agent loop speeds.

Locus stays out of the LSP business. We curate the wiring.

## Tasks

- [ ] Curated `mcp.json` snippets per OS for the common stack: clangd, tsserver / typescript-language-server, pylsp / pyright, gopls, rust-analyzer. Live under `architecture/lsp-presets/` so they aren't auto-installed -- the user opts in by copy-pasting into `.locus/mcp.json` (or via the UI shortcut below).
- [ ] Settings UI in [src/frontends/gui/settings_dialog.cpp](../../src/frontends/gui/settings_dialog.cpp): "Add language server..." button under the MCP section. Dropdown lists the curated presets; selecting one writes the corresponding entry to `.locus/mcp.json`. User still confirms an explicit approval policy on first activation -- matches every other MCP server.
- [ ] One known-good LSP-MCP server bundled as the default suggestion (most likely `mcp-language-server` in npm/pip form, pinned to a specific version). Document version + checksum in the preset file so we don't drift silently.
- [ ] Documentation: short page in `architecture/` explaining when to reach for LSP-via-MCP vs Tree-sitter `search_symbols` / `mode=ast` / S4.C diagnostics. The decision matrix from the Locus "AST vs LSP" comparison goes here verbatim. Goal: stop the agent (and the user) from reflexively spinning up clangd when textual or shape search would do.
- [ ] Acceptance test: with `mcp-language-server` + clangd configured against a small C++ fixture, the agent can answer "find all callers of `foo` (overloaded)" correctly. Manual / integration-test only -- not gated in CI.

## What was deliberately dropped from the original spec

The original task list -- custom `LspClient`, five dedicated tools (`goto_definition`, `find_references`, `get_diagnostics`, `rename_symbol`, `hover_info`), per-language server registry, lazy startup -- is not happening. Reasons captured here so a future stage author can revisit if conditions change:

- **Capability redundancy with S4.C and S4.G.** S4.C handles the diagnostics use case via build output; S4.G handles the tool-protocol use case via MCP. Native LSP would duplicate both.
- **Maintenance burden.** Each LSP server (clangd, tsserver, pylsp, gopls, rust-analyzer) has its own setup quirks, crash modes, indexing-progress semantics, and capability matrix. We do not have the headcount to maintain a custom client across all of them.
- **Memory cost.** clangd holds full ASTs in memory (1-4 GB on a real C++ codebase). Locus's "lean app" stance does not survive that as an in-process subsystem; it's better isolated in a child MCP process.
- **Locus thesis fit.** "User as architect" -- the user installing an MCP server is more transparent than a built-in subsystem auto-spawning a 2 GB child. It also matches how the user already manages tools.

If a future user demonstrates a workflow where MCP-wrapped LSP is meaningfully worse than native (latency-bound interactive use? large-batch reference lookups?), revisit then. Until that evidence exists, "MCP it is" is the right call.

## Implementation notes

- The presets file is checked into the repo, not generated. It's a small set of YAML/JSON snippets plus a one-line description per server. Treat it like a recipe collection.
- For clangd specifically, document the `compile_commands.json` requirement in the preset header. clangd silently degrades to crippled mode without it; users hit this and blame Locus.
- The "Add language server" UI does not start the server itself -- it writes the config and prompts the user to approve activation through the existing MCP settings flow. One trust path, not two.
- Bundling a vetted version of `mcp-language-server` is a deliberate trust handoff: we vouch for the default. Users can swap in their own MCP server in the same config slot, but the default is curated.
- This stage depends on [S4.G -- MCP](S4.G-mcp.md) being done first. Without MCP, this stage has nothing to wire into.
