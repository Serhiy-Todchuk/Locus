# S4.Y -- Tree-sitter Grammar Coverage Expansion ✔

## Problem

The current 9 grammars (`c`, `cpp`, `python`, `javascript`, `typescript`, `go`, `rust`, `java`, `csharp`) cover one slice of the modern stack and miss whole ecosystems. Ruby/PHP/Swift/Kotlin users see no symbols, no AST search, no language-aware indexing. Bash, SQL, JSON, YAML, and Markdown appear in *every* repo regardless of primary language and are currently invisible to structural search. For doc workspaces (WS3) this is even more acute: Markdown has only the regex-based extractor, no AST queries.

Adding a grammar is mechanically simple -- one entry in `TreeSitterRegistry`, one source dependency in CMake/vcpkg. The work is per-language: for code languages, write a `SymbolRule` table; for markup, decide whether "symbols" maps onto headings/sections/keys and what the extractor should emit. AST search comes for free once the grammar is in the registry.

## Scope

This stage adds 9 grammars in three tiers. Tier 1 is in scope; Tier 2 lands if effort permits; Tier 3 is explicitly deferred.

### Tier 1 -- core (must-land)

| Language | Symbol extractor | Notes |
|---|---|---|
| Ruby | yes (functions, classes, modules, methods) | Rails et al; large ecosystem |
| PHP | yes (functions, classes, traits, interfaces) | WordPress / Laravel |
| Bash | yes (function definitions only) | Almost every repo has shell scripts |
| SQL | no (statement-shaped, no symbols) | AST search is the win: "all DELETE without WHERE", "all SELECT *" |
| JSON | no (treat keys as headings via extractor) | AST search is the win: "find every `port` field across configs" |
| YAML | no (treat top-level keys as headings) | Same rationale as JSON |
| Markdown | already has text extractor; add Tree-sitter for AST queries | Headings already extracted via `MarkdownExtractor`; AST adds query-by-shape (e.g., "all code blocks", "all link targets") |

### Tier 2 -- mobile / JVM (land if effort permits)

| Language | Symbol extractor |
|---|---|
| Swift | yes (functions, classes, structs, protocols, extensions) |
| Kotlin | yes (functions, classes, objects, interfaces) |

### Tier 3 -- explicitly deferred

Scala, Dart, Lua, Zig, Elixir, Haskell, Clojure, OCaml, R, Perl, GraphQL, Vue/Svelte, HTML, CSS, TOML, Dockerfile. Add on demand; not blocking.

## Implementation notes (post-build)

### Recursive symbol-extractor walker

The original rule-based walker iterated three levels deep (root.children -> children -> grandchildren). That was enough for top-level functions and one level of nesting (class.method) in C-family languages, but breaks in **Ruby** (`module > body_statement > class > body_statement > method` = depth 5) and similar shapes in **PHP** (namespace + class + method), **Swift** (extension + class + method), and **Kotlin** (file + class + class_body + method). [`RuleBasedSymbolExtractor::extract`](../../../src/index/symbol_extractor.cpp) is now a depth-bounded recursion (`k_max_depth = 8`) that tracks `parent_name` through container kinds (`class` / `struct` / `module` / `namespace` / `interface` / `trait` / `object`). All existing language tests still pass; nested-symbol counts on WS1 increased 2149 -> 2251 (deeper C# namespaces being the biggest contributor).

### Identifier-fallback in `extract_name_from_node`

Kotlin's `class_declaration`, `function_declaration`, and `object_declaration` expose **no** `name` field at all -- the identifier is just the first identifier-shaped named child. Extended the helper in [symbol_extractor.cpp](../../../src/index/symbol_extractor.cpp) with a final fallback that walks direct named children looking for `identifier` / `type_identifier` / `field_identifier` / `simple_identifier`. Strictly additive: only fires when the field-name path and the "is the node itself an identifier" path both return empty. `simple_identifier` was also added to the identifier-type list for consistency.

### JSON / YAML "keys-as-headings" -- ITextExtractor route

[`JsonExtractor`](../../../src/extractors/json_extractor.cpp): single-pass byte walker. Tracks `brace_depth`, `array_depth`, string state, escape state, and `//` / `/* */` comments (so `.jsonc` and `settings.json` work). At `brace_depth == 1 && array_depth == 0 && !saw_key_this_line`, the next `"..."` followed by `:` records a heading. Nested keys never surface as top-level.

[`YamlExtractor`](../../../src/extractors/yaml_extractor.cpp): line-oriented. Column-0 lines whose first non-space char is printable (and not `-` / `#` / `!` / `%`), containing a `:` outside quotes followed by EOL or whitespace, emit one heading. Quoted-key support (single + double) and trailing whitespace trimming included. Nested keys (indented) are ignored by design.

Both register under `.json` / `.jsonc` and `.yaml` / `.yml` in [`Workspace`](../../../src/core/workspace.cpp). Existing index rows for these files won't pick up new headings until the file mtime changes -- that's intentional, matches the rest of the indexer's stat-skip behaviour.

### SQL deferral

DerekStride/tree-sitter-sql (the most maintained SQL grammar) gitignores `parser.c` and expects users to run `tree-sitter generate` at build time. The alternatives are all worse:

- **m-novikov/tree-sitter-sql**: ships `parser.c` but a C++ scanner.cc (would force `ts_grammars` to also compile C++; one-off complexity for one grammar).
- **Vendor a multi-MB generated `parser.c` into the repo**: makes regeneration awkward and bloats the source tree.
- **Add the Rust-based `tree-sitter` CLI as a build dependency**: real value, but a bigger toolchain decision than this stage.

Deferred to a follow-up stage that introduces the CLI dep cleanly. SQL is otherwise the cheapest of the Tier 1 grammars (AST-only, no symbol extractor), so it's a 30-LOC drop-in once the CLI is in place. Mark recorded in [roadmap/backlog/README.md](../backlog/README.md).

### Swift grammar tag

alex-pinkus/tree-sitter-swift's plain `0.7.x` tags omit the generated `parser.c` (gitignored on `main`). The pin uses `0.7.2-with-generated-files`, the variant tag that vendors `parser.c` + `parser_abi13.c` + `parser_abi14.c` + `scanner.c`. We compile `parser.c` (current ABI) only; the older ABI variants stay on disk but aren't built. One warning C4293 in `scanner.c` (third-party shift-count edge case) -- harmless.

### Tier 2 (Swift / Kotlin) landed

Both grammars are in `TreeSitterRegistry`; both have symbol extractors; both have query starter files. The doc's deferral threshold ("if Tier 1 takes more than the planned budget") wasn't hit -- Tier 1 was light enough that mobile / JVM coverage came for free in the same stage.

## Implementation notes

- **Source vendoring vs vcpkg**: most upstream Tree-sitter grammars publish parser.c (and sometimes scanner.c) directly. `FetchContent` against the official repo is cleanest -- avoids waiting on vcpkg ports for less-common languages. Pin to a commit, not a branch.
- **Markup-as-symbols question**: JSON/YAML "symbols" are not symbols in the code sense. Two design options. (a) New `IDocumentStructureExtractor` parallel to `ISymbolExtractor`, emitting heading-like entries from key paths. (b) Reuse `ISymbolExtractor` with a `SymbolRule` that maps `pair > key` to a heading-flavored extracted symbol. Pick before coding -- the answer affects the index schema. Lean toward (b) for surface-area reasons; the index already understands headings, so reusing that path means `get_file_outline` works on configs without a new code path.
- **SQL dialect**: tree-sitter-sql has multiple flavors. PostgreSQL has the largest grammar; the SQLite Tree-sitter parser is closer to "subset of standard SQL" and handles less. Default to `tree-sitter-sql` (the one tagged for ANSI/PostgreSQL) and accept some MS-T-SQL parse errors as a known limitation. Document in `queries/sql/README.md`.
- **Bash gotcha**: function definitions in Bash come in two syntaxes (`function foo() { ... }` and `foo() { ... }`). Make sure both `function_definition` rules in the symbol extractor handle them; mirror what `tree-sitter-bash` upstream uses in their highlight queries for guidance.
- **Tier 2 (Swift / Kotlin) deferral threshold**: if Tier 1 takes more than the planned budget, defer Swift and Kotlin to a follow-up stage rather than rushing them. Mobile dev workspaces are a strong use case but not in WS1-WS3.
- **Tier 3 stays deferred indefinitely** -- the same `vcpkg + registry + extractor` template means any user with a real need for one of these languages can land it as a one-stage contribution without rearchitecting anything. The roadmap doesn't need to enumerate every grammar Tree-sitter supports.
