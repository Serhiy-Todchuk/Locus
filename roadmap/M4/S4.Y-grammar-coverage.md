# S4.Y -- Tree-sitter Grammar Coverage Expansion

## Problem

The current 9 grammars (`c`, `cpp`, `python`, `javascript`, `typescript`, `go`, `rust`, `java`, `csharp`) cover one slice of the modern stack and miss whole ecosystems. Ruby/PHP/Swift/Kotlin users see no symbols, no AST search, no language-aware indexing. Bash, SQL, JSON, YAML, and Markdown appear in *every* repo regardless of primary language and are currently invisible to structural search. For doc workspaces (WS3) this is even more acute: Markdown has only the regex-based extractor, no AST queries.

Adding a grammar is mechanically simple -- one entry in `TreeSitterRegistry`, one source dependency in CMake/vcpkg. The work is per-language: for code languages, write a `SymbolRule` table; for markup, decide whether "symbols" maps onto headings/sections/keys and what the extractor should emit. AST search comes for free once the grammar is in the registry.

## Scope

This stage adds 9 grammars in three tiers. Tier 1 is in scope; Tier 2 lands if effort permits; Tier 3 is explicitly deferred.

### Tier 1 -- core (must-land)

| Language | Symbol extractor | Notes |
|---|---|---|
| Ruby | yes (functions, classes, modules, methods) | Rails et al; large ecosystem |
| PHP | yes (functions, classes, traits, interfaces) | WordPress / Laravel |
| Bash | yes (function definitions only) | Almost every repo has shell scripts |
| SQL | no (statement-shaped, no symbols) | AST search is the win: "all DELETE without WHERE", "all SELECT *" |
| JSON | no (treat keys as headings via extractor) | AST search is the win: "find every `port` field across configs" |
| YAML | no (treat top-level keys as headings) | Same rationale as JSON |
| Markdown | already has text extractor; add Tree-sitter for AST queries | Headings already extracted via `MarkdownExtractor`; AST adds query-by-shape (e.g., "all code blocks", "all link targets") |

### Tier 2 -- mobile / JVM (land if effort permits)

| Language | Symbol extractor |
|---|---|
| Swift | yes (functions, classes, structs, protocols, extensions) |
| Kotlin | yes (functions, classes, objects, interfaces) |

### Tier 3 -- explicitly deferred

Scala, Dart, Lua, Zig, Elixir, Haskell, Clojure, OCaml, R, Perl, GraphQL, Vue/Svelte, HTML, CSS, TOML, Dockerfile. Add on demand; not blocking.

## Tasks

- [ ] CMake: add Tier 1 grammars as source dependencies (most have `tree-sitter-<lang>` repos under the official `tree-sitter` GitHub org; vendor via `FetchContent` or vcpkg if the port exists).
- [ ] `TreeSitterRegistry`: register each Tier 1 grammar in [src/index/tree_sitter_registry.cpp](../../src/index/tree_sitter_registry.cpp) with the right `extern "C"` declaration and `langs_["name"] = ...` entry.
- [ ] Per code language (ruby, php, bash, swift, kotlin): add `src/index/symbol_extractors/<lang>_extractor.cpp` with a `SymbolRule` table; register via `make_<lang>_symbol_extractor()` in `register_builtin_symbol_extractors`. Mirror the existing cpp/python/etc. structure exactly.
- [ ] For SQL: AST-only (no symbol extractor). Verify the grammar parses real-world dialect variation acceptably; SQL has many dialect grammars on the Tree-sitter side -- pick the one closest to ANSI/PostgreSQL and document the choice.
- [ ] For JSON / YAML: extend the symbol extractor pattern to emit "key headings" -- top-level keys (and one level deep) become heading-like entries so `search_symbols` and `get_file_outline` work usefully on configs. Decide before implementation whether this rides the existing extractor protocol or a parallel one.
- [ ] For Markdown: add Tree-sitter grammar to the registry so `mode=ast` works against `.md` files. Keep the existing `MarkdownExtractor` for text extraction -- they cover different needs. Markdown gets a `queries/markdown/` library with starters: "all code blocks", "all link targets", "all `[ ]`/`[x]` task list items".
- [ ] Per language with a symbol extractor: write 5-10 canonical AST queries under `queries/<lang>/` -- matches the existing convention from S4.M.
- [ ] File-extension mapping: ensure the indexer dispatches the right grammar for each new extension (`.rb`, `.php`, `.sh` / `.bash`, `.swift`, `.kt` / `.kts`, `.sql`, `.json` / `.jsonc`, `.yaml` / `.yml`, `.md` / `.markdown`).
- [ ] Update the `SearchAstTool` schema and `SearchTool`'s `query` description to list the new languages -- one extra clause in the existing prose, not a structural change.
- [ ] Catch2 tests:
  - parse-and-extract round trip per language (file in -> expected symbols out)
  - AST query round trip per language (one canonical query against a fixture file)
  - `.md` AST query separate from existing Markdown extractor tests
- [ ] Binary-size sanity check: track `locus_core.lib` size before/after Tier 1. Tree-sitter grammars compile to lookup tables; each adds 100-500 KB. The total is acceptable (~3-5 MB across Tier 1) but worth a single line in the stage retro.

## Implementation notes

- **Source vendoring vs vcpkg**: most upstream Tree-sitter grammars publish parser.c (and sometimes scanner.c) directly. `FetchContent` against the official repo is cleanest -- avoids waiting on vcpkg ports for less-common languages. Pin to a commit, not a branch.
- **Markup-as-symbols question**: JSON/YAML "symbols" are not symbols in the code sense. Two design options. (a) New `IDocumentStructureExtractor` parallel to `ISymbolExtractor`, emitting heading-like entries from key paths. (b) Reuse `ISymbolExtractor` with a `SymbolRule` that maps `pair > key` to a heading-flavored extracted symbol. Pick before coding -- the answer affects the index schema. Lean toward (b) for surface-area reasons; the index already understands headings, so reusing that path means `get_file_outline` works on configs without a new code path.
- **SQL dialect**: tree-sitter-sql has multiple flavors. PostgreSQL has the largest grammar; the SQLite Tree-sitter parser is closer to "subset of standard SQL" and handles less. Default to `tree-sitter-sql` (the one tagged for ANSI/PostgreSQL) and accept some MS-T-SQL parse errors as a known limitation. Document in `queries/sql/README.md`.
- **Bash gotcha**: function definitions in Bash come in two syntaxes (`function foo() { ... }` and `foo() { ... }`). Make sure both `function_definition` rules in the symbol extractor handle them; mirror what `tree-sitter-bash` upstream uses in their highlight queries for guidance.
- **Tier 2 (Swift / Kotlin) deferral threshold**: if Tier 1 takes more than the planned budget, defer Swift and Kotlin to a follow-up stage rather than rushing them. Mobile dev workspaces are a strong use case but not in WS1-WS3.
- **Tier 3 stays deferred indefinitely** -- the same `vcpkg + registry + extractor` template means any user with a real need for one of these languages can land it as a one-stage contribution without rearchitecting anything. The roadmap doesn't need to enumerate every grammar Tree-sitter supports.
