# S4.I — Background / Long-Running Commands ✔

## Problem

`run_command` blocks with 30s timeout. Cannot start dev server, run long builds (cmake 2min+), tail logs. Claude Code's `run_in_background` + `Monitor` solves this.

## Tasks

- [x] `run_command_bg(command)` → returns `process_id`, stays alive across turns
- [x] `read_process_output(pid, since_offset?)` → stream bytes since last read
- [x] `stop_process(pid)` → terminate, capture final exit code
- [x] `list_processes()` → active background processes `(pid, command, started_at, status)`
- [x] Process registry owned by Core (Workspace); terminated on workspace close
- [x] Auto-truncate output buffer per process (ring buffer, configurable cap)
- [ ] UI: background process panel (tray badge count, kill buttons) — deferred; tool calls and their results already surface through the existing activity log + chat panel. A dedicated panel + tray badge belongs with the broader activity-panel work.

## Implementation notes

- New [src/process_registry.h/cpp](../../src/process_registry.h) — `ProcessRegistry` owns a map of `BackgroundProcess` instances; each child is spawned under a Win32 Job Object with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` so killing the registry kills the whole process tree (including grandchildren spawned by `cmd /c …`). `CREATE_SUSPENDED` + post-spawn `AssignProcessToJobObject` matches the existing `RunCommandTool` so descendants can never escape job membership before assignment.
- One reader thread per process appends stdout+stderr into a `std::string` capped at `WorkspaceConfig::process_output_buffer_kb` (default 256 KB). `total_bytes_received_` is monotonic across the process lifetime; the registry tracks `bytes_dropped_` so callers reading from an old offset are told exactly how many bytes the ring evicted under them.
- `read_output(pid, since_offset?)` exposes two modes: with an explicit offset the LLM controls bookkeeping, without one the registry returns "everything new since this tool last read this pid". The latter is the default in the tool because LLMs are lousy at offset arithmetic.
- `IWorkspaceServices::processes()` was added as another optional subsystem (alongside `embedder()` / `reranker()`). The four new tools mark themselves `available()=false` when the workspace doesn't expose a registry — that's how the `FakeWorkspaceServices` unit tests stay 9-tool clean.
- `Workspace` owns the registry as the last subsystem to start and the first to tear down (right above the watcher pump in the dtor). Workspace close → registry dtor → terminate every still-running child → join reader threads → release handles. No process can outlive the workspace.
- System prompt now distinguishes synchronous `run_command` (≤30s) from `run_command_bg` (servers, long builds, watchers). Tool count rose from 9 to 13.

## Open follow-ups (not blocking)

- POSIX support: spawning currently throws `std::runtime_error("Background commands not implemented on this platform")` on non-Windows. Mirrors the pre-existing `RunCommandTool` situation; pick this up when CLI is brought up on macOS/Linux.
- Output streaming activity events: today the LLM sees process output only when it polls. A future enhancement could push deltas through the activity log so frontends can tail without an LLM round-trip.
