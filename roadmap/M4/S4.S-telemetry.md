# S4.S — Telemetry & Agent Performance Metrics ✔

## Problem

Activity panel is per-event. No aggregate metrics: no tokens/s, no avg turn time, no tool-call histogram, no retrieval hit rate. Can't see trends, can't spot regressions.

## Tasks

- [x] Aggregated metrics tab in activity panel: total tokens in/out, tokens/s (streaming rate), avg / p95 turn time, tool-call counts by name, retrieval top-K hit rate
- [x] Per-session metrics snapshot saved alongside session JSON
- [x] Metrics exporter: CSV / JSON for offline analysis
- [x] Compact chart widgets — no heavy charting library; spark bars only

## Implementation notes

New [src/agent/metrics.h/cpp](../../src/agent/metrics.h) hosts `MetricsAggregator` — thread-safe, owned by `AgentCore` alongside `ActivityLog` and `ContextBudget`. Per-turn samples (`TurnSample`), per-tool stats (`ToolStat`), and a single `RetrievalStat`; `aggregates()` returns a flattened POD the GUI consumes (tokens/s, avg / p95 / max turn time, tool histogram, retrieval hit rate, plus two spark-bar series).

Wiring follows the same dependency-injection pattern as `ActivityLog`: `AgentLoop` and `ToolDispatcher` take an optional `MetricsAggregator*` and record from the points where they already see the data — `AgentLoop` wraps `stream_completion` in `steady_clock` and feeds `(usage, stream_ms)` through `record_llm_step`; `ToolDispatcher` wraps `tool->execute` and records each call's name + ms + success + content size, with retrieval count parsed out of the search-tool result header (helper `parse_search_result_count` in metrics.cpp). `AgentCore::process_message` brackets the turn with `begin_turn(turn_id)` / `end_turn(had_error)` and clears state on `reset_conversation` / `load_session`.

Persistence: `SessionManager::save` now takes an optional `extras` JSON parameter merged into the top-level session object. `AgentCore::save_session` ships the metrics snapshot under the `"metrics"` key. The shape mirrors `to_json()`: `turns[]`, `tools{}`, `retrieval{}`, `totals{}`.

Exporter: `AgentCore::export_metrics(format)` writes `.locus/metrics/metrics_<timestamp>.{json,csv}` and returns the absolute path. Surfaced via two new built-in slash commands handled directly in `AgentCore::try_slash_command` (alongside `/undo`):
- `/metrics` — print a one-shot text summary (turns, tokens, throughput, turn-time stats, retrieval, tool histogram).
- `/export_metrics [json|csv]` — write the snapshot file. Default format is JSON.

GUI ([src/gui/metrics_view.h/cpp](../../src/gui/metrics_view.h)): `ActivityPanel` now wraps the existing log list in a `wxNotebook` with a second tab, "Metrics". `MetricsView` shows the textual aggregates plus two pure-`wxDC` spark bars (turn duration, total tokens) — no charting library, < 100 LOC of paint code. The tab updates on a 1 s timer and on every appended activity event so user input is reflected immediately. "Export JSON…" / "Export CSV…" buttons inside the tab pop a `wxFileDialog` so the user picks the destination — they don't have to hunt under `.locus/metrics/`.

Tests: [tests/test_metrics.cpp](../../tests/test_metrics.cpp), 12 cases, tag `[s4.s][metrics]`. Covers turn timing, LLM-step recording (50 tok/s sanity check), tool histogram ordering, retrieval hit rate (2/3), reset, JSON/CSV serialisation shape, and `parse_search_result_count` against every search-tool header format including the indexing-progress preamble that `search_semantic` prepends.

Tool count unchanged at 13. The two slash commands are not registered tools — they're built-in commands handled before `slash_->try_dispatch`, like `/undo`.
