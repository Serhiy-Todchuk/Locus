# S4.K — Retrieval Evaluation Harness ✔

## Problem

Zero way to measure if hybrid search gets better or worse when tweaked. Current retrieval ships on vibes.

## Tasks

- [x] [tests/retrieval_eval/queries.json](../../tests/retrieval_eval/queries.json): 80 gold queries against WS1 itself (`{query, expected_files[]}`; symbol-level gold dropped — `search_symbols` is exact-match and covered by unit tests)
- [x] Evaluator binary [`locus_retrieval_eval`](../../tests/retrieval_eval/main.cpp): runs each query through `search_text`, `search_semantic`, `search_hybrid`; computes recall@{1,3,5,10}, MRR, nDCG@10
- [x] Markdown report committed to [`tests/retrieval_eval/results.md`](../../tests/retrieval_eval/results.md) on each run (aggregate table + per-query R@3 hit map)
- [x] Baseline regression check: compares against [`tests/retrieval_eval/baseline.json`](../../tests/retrieval_eval/baseline.json), exits 1 on >5% drop (configurable via `--tolerance`); commit a fresh baseline with `--write-baseline`
- [x] Curate queries from real sessions: [`--curate-from <sessions_dir>`](../../tests/retrieval_eval/curate.h) walks `.locus/sessions/*.json`, extracts every `search` tool invocation, prints a `queries.json` template with empty `expected_files` arrays for hand-filling
- [x] Pure-math metric tests in [tests/test_retrieval_eval.cpp](../../tests/test_retrieval_eval.cpp) — recall edge cases, nDCG perfect/reversed, dedup-on-rank, aggregate

## Implementation notes

- **Not wired into `ctest` / CI.** The harness needs a fully indexed workspace + the bge-m3 GGUF (~600 MB) loaded + the embedding queue drained — too heavy for "every build". Manual-only target, like `locus_integration_tests`. The exit-code regression check is preserved so a future CI hook needs one line.
- **File-level granularity.** `expected_files` is the only ground truth — symbol-level recall would duplicate `search_symbols` unit tests (exact LIKE match) without measuring anything new about hybrid retrieval.
- **Tight `expected_files` per query** (1–3 paths). Listing 10 "related" files per query inflates recall and washes out signal.
- **FTS5 sanitisation.** Gold queries are natural-language ("BM25 full-text search FTS5"); FTS5 reads its input as a MATCH expression where `:`, `"`, `(`, `*` etc. are operators. We strip those before passing to `search_text` / `search_hybrid` so a query with punctuation doesn't fail at the parser.
- **Embedding queue wait.** Workspace ctor finishes the synchronous index but the embedding worker is async. Eval blocks up to 600s (or until `--no-wait`) so semantic recall reflects full coverage rather than whichever chunks happened to be embedded by query time.
