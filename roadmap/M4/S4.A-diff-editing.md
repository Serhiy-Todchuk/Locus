# S4.A ‚Äî Diff-Based Editing

## Problem

`WriteFileTool` overwrites whole files. On local LLMs (Gemma-27B, Qwen3, Llama3) this (a) corrupts files via drift / whitespace damage / silent function deletion, (b) costs 10‚Äì100√ó more tokens than needed, (c) feels 20√ó slower on mid-range GPUs. Every competitive agent uses diff-style edits (Aider `udiff`/`diff-fenced`, Cline/RooCode `replace_in_file`, Claude Code `Edit old_string/new_string`).

## Tasks

- [x] `edit_file` tool: `path` + `edits` array of `{old_string, new_string, replace_all?}` ‚Äî exact-match with uniqueness check (fail if `old_string` ambiguous), single-element array for the common case
- [x] Require prior `read_file` on the same path in the same session before editing (Claude Code pattern) ‚Äî reduces hallucinated edits
- [x] Batch edits: the same `edit_file` accepts N edits applied atomically (subsumes the originally-planned `multi_edit_file` ‚Äî kept one tool to keep the manifest slim for local LLMs)
- [ ] Evaluate alternative format: unified-diff (`apply_patch`) ‚Äî compare which local models produce cleaner output
- [x] Preview in approval panel: red/green diff via wxStyledTextCtrl markers, not raw text
- [x] Update system prompt guidance: edit > rewrite; `write_file` only for fresh files or full rewrites (and `create_file` merged into `write_file` ‚Äî `overwrite=true` opts in to replace)
- [x] Catch2 tests: uniqueness check, non-existent-match failure, indent preservation, batch atomicity, later-sees-earlier sequencing

## Implementation notes

- One tool, not two. `edit_file` takes `{path, edits: [{old_string, new_string, replace_all?}, ...]}`. A single edit is just an array of one. Local LLM manifests stay smaller than if we had shipped separate `edit_file` + `multi_edit_file` entries (Claude Code's pattern), at the cost of slightly more verbose single-edit syntax.
- Exact-match engine lives in [src/tools/file_tools.cpp](../../src/tools/file_tools.cpp); `apply_single_edit` / `replace_all_occurrences` run inside the batch loop. Writes go through a temp file + rename to stay atomic on crash. Any per-edit failure discards the in-memory buffer ‚Äî the file on disk is untouched.
- "Must read before edit" is enforced by a process-wide `ReadTracker` in [src/tools/shared.h](../../src/tools/shared.h) ‚Äî `ReadFileTool` populates it on success, `edit_file` fails loudly otherwise. Canonical paths, so workspace switches do not collide. Tests `clear()` between cases.
- Unified-diff (`apply_patch`) format is deferred ‚Äî revisit when M4.J/M4.K let us A/B local models on patch-style output.
- Approval UI renders edit tools as a colored unified diff inside the existing Scintilla pane (`render_diff_view()` in [src/gui/tool_approval_panel.cpp](../../src/gui/tool_approval_panel.cpp)); `Modify` flips back to raw JSON, then back to diff on cancel.
- `create_file` folded into `write_file` with an `overwrite` flag (default `false`). Same safety as before ‚Äî the default refuses to replace an existing file ‚Äî but one fewer entry in the tool manifest. The approval preview still distinguishes `Create new: path` from `Write (overwrite allowed): path`.
