# S4.R -- Memory Bank (Persistent + Searchable)

## Problem

The agent forgets within a session (re-learns the same fact at turn 15 that it figured out at turn 3) and forgets entirely between sessions (LOCUS.md is the only cross-session story today, and LOCUS.md is for **user-curated workspace conventions**, not facts the agent learns over time). Cline's memory bank, Claude Code's todo list, OpenCode's AGENTS.md, MemPalace's drawers, and Anthropic's Memory tool all close this gap; we don't.

The original framing of this stage was "within-session scratchpad only" -- intentionally small. After the MemPalace comparison, that ages poorly: the moment a user closes Locus and reopens, they expect facts to survive. Promoting this stage to **persistent + searchable workspace-scoped memory** with verbatim entries and hybrid retrieval. The within-session scratchpad is the degenerate case of cross-session memory with all entries pinned to the current session.

## Design principles

- **Verbatim storage.** No summarisation. Each memory is a small markdown note (one to a few sentences typically), kept exactly as written. Compaction loses content; this complements compaction by giving facts a home that survives it.
- **Workspace-scoped.** Memories live in `.locus/memory/` and never leak between workspaces. Cross-workspace sharing is explicitly out of scope; users who want shared facts use LOCUS.md or copy entries.
- **Two read paths**: always-in-context (small budget, recent + pinned) and on-demand (hybrid retrieval via tool). Most facts the agent needs are queried; a few load-bearing ones live in every prompt.
- **Two write paths**: user-driven (slash command + chat right-click) and agent-driven (tool, normal approval gate). Same approval discipline as every other mutating tool.
- **Flat with tags.** No hierarchy. No rooms or folders. Tags + full-text search + semantic search cover the organisation use cases without the navigation overhead.

## Storage layout

```
.locus/memory/
  index.json         -- top-level index (id -> path, tags, created_at, source, pinned)
  <id>.md            -- one verbatim entry per file; YAML frontmatter for metadata
```

Per-entry frontmatter:

```yaml
---
id: 2026-05-08-a3b1
created_at: 2026-05-08T14:23:11Z
source: user | agent | mined
tags: [build, conventions]
pinned: false
---

The build command for this project is `cmake --build build/release`,
not the default `make` -- there is no Makefile in the workspace root.
```

Vectors land in a new `memory_vectors` table in `.locus/vectors.db`, parallel to `chunk_vectors`. Same dim as the workspace embedder. FTS5 indexed in `.locus/index.db` for keyword retrieval -- new `memory_fts` virtual table.

## Always-in-context surface

A small slot in the system prompt (after LOCUS.md, before the tool manifest) auto-injects:

1. All `pinned: true` entries (no cap on count, but total-token cap enforced)
2. The N most-recently-used entries until the cap fills

Cap: `memory.in_context_budget_tokens` (default `500`). Workspace setting; surfaced in the [S5.D](../M5/S5.D-reserve-and-token-xray.md) stacked-bar context meter as a distinct slice so the user sees what memory is costing the prompt at a glance.

When the budget is full, the *least-recently-used unpinned* entries are dropped from the in-context slot but remain searchable via the on-demand tool.

## On-demand retrieval

Two tools added to the manifest:

- `add_memory(content, tags?, pinned?)` -- creates a new memory entry. Default `pinned=false`. Approval policy: `require_approval` (memories are durable; the user sees what's being committed before it lands).
- `search_memory(query, max_results=5, tags?)` -- hybrid retrieval over the memory store. Auto-approve.

Hybrid scoring (mirrors the workspace `search_hybrid` shape):
- FTS5 BM25 score
- Cosine similarity over `memory_vectors`
- Recency boost (entries from the last week ranked higher)
- Optional reranker pass when `reranker_enabled` is on

Default cap: `max_results=5`. Format: `path:id` header + verbatim content + tags. Total response capped at `memory.search_response_max_tokens` (default 1500).

## User-driven write paths

- **Slash command**: `/memorize <text>` -- creates a new entry with the text as content, `source=user`, no tags. `/memorize +tag1 +tag2 <text>` adds tags. `/memorize --pin <text>` creates pinned. Same parser as other slash commands ([S3.J](../M3/S3.J-slash-commands.md)).
- **Chat right-click on any message**: "Save as memory" -- pre-populates the input with the message content; user edits + confirms.
- **Settings -> Manage Memories** panel: list view with search, edit, delete, pin/unpin, tag operations. Sort by created_at / last_used / pinned-first. The single piece of UI that gives the user direct control over their memory bank.

## Lifecycle

- **GC**: keep `memory.max_entries` (default 200) most-recent unless pinned. Pinned entries survive GC unconditionally. Triggered on workspace open and after every `add_memory` call.
- **Manual delete**: from the Settings panel; one-click; archived first to `.locus/memory/.deleted/<id>.md` with a 30-day retention before permanent removal (cheap safety net for accidental deletion).
- **Edit**: from the Settings panel. Edits don't change `id` but bump `updated_at`. The vector + FTS entries re-index on save.
- **Validity windows**: NOT in v1 -- temporal "fact F was true between dates X and Y" semantics is real but adds a versioning layer. Tracked as an open question below for v2.

## What this is NOT (deliberate scope cuts)

- **No auto-mining from sessions.** End-of-session sweeps that summarise conversation into memory entries are powerful but bring privacy concerns ("the agent is summarising my chats into a permanent DB"), per-session LLM cost, and quality dependence on the model. Tracked as a backlog item with reactivation gated on real demand.
- **No validity windows in v1.** Temporal supersession (fact F was true between dates X and Y) is a real layer. v2 if a real workload surfaces the need.
- **No cross-workspace sharing.** Memories are per-workspace. Users who want a shared fact use LOCUS.md or copy the entry manually.
- **No hierarchy.** Flat with tags. No "rooms" or "folders." Tags + search cover organisation; navigating a tree adds friction.
- **No model-side memory tool.** The model gets `add_memory` and `search_memory`; it does not get tools to delete, edit, pin, or list entries. Those are user-only operations through the Settings panel. Asymmetric on purpose: the agent can grow memory, the user prunes it.

## Implementation notes

- The MemoryStore reuses the workspace's `Embedder` instance via [IWorkspaceServices](../../src/core/workspace_services.h). No new model loaded. Memory vectors share the workspace's embedder dim; switching the workspace embedder triggers the same dim-mismatch wipe-and-rebuild path the workspace `chunk_vectors` already has (so memory entries get re-embedded automatically).
- The in-context slot is owned by the system-prompt assembly. After [S5.J](../M5/S5.J-llm-context-refactor.md) lands, the assembly is immutable post-construction -- meaning the in-context memory list is captured *at session start*. Entries added during a session (via `add_memory`) become available *to the next session's in-context slot* but aren't auto-injected into the current session's prompt (would require regenerating `SystemPromptAssembly`, breaking [S4.F](S4.F-kv-cache.md)'s byte-stable invariant). Within-session, the agent reaches new entries via `search_memory`. Acceptable tradeoff: KV-cache wins out, on-demand retrieval is the supported path within a session.
- Manual `/memorize` and right-click "Save as memory" within a session are still useful even though they don't change the current in-context slot -- the entries persist and are searchable immediately. The in-context slot picks them up at the next session start (or workspace reopen).
- `add_memory` tool result includes the entry id and a confirmation tail showing what was stored. This gives the user (in the activity log) and the agent (in tool result) a verbatim record of what was committed.
- Reranker integration: when [S4.J](S4.J-embeddings-reranker.md)'s reranker is enabled, `search_memory` re-ranks the top-K candidates (default `reranker_top_k=20`) before trimming to `max_results`. Same pattern as `search_semantic` / `search_hybrid` in the workspace.
- Privacy: nothing leaves the local machine. All embedding runs in the workspace embedder (already local). All storage is local. The cloud-LLM use case (if [Online model backends](../backlog/README.md) ever lands from backlog) is the only path where memory content could be exposed -- and that exposure happens *only* when the in-context slot is sent as part of the prompt; same model already sees the conversation.

## Open questions for v2

- **Validity windows / temporal supersession.** "The build command is X" written in March, then "The build command is Y" written in May. Should the agent see both with timestamps, or only the latest? MemPalace's approach: validity intervals + invalidation. Adds a real complexity layer. Defer until needed.
- **Memory chunking.** Today an entry is one verbatim note. If users start writing 5-paragraph memories, retrieval gets noisy. Consider per-paragraph chunking like the file index does. Defer until real entries demand it.
- **Cross-session "active memories"** -- carrying recent in-context entries across session boundaries within the same workspace. v1 lets the next session's in-context slot pick up whatever's most recent + pinned; this is roughly the right behaviour. If users want explicit "carry these N entries to next session," add a pin-during-session knob.

## Manifest cost

**~250-300 prompt tokens** for two new tools (`add_memory` + `search_memory`). Plus the in-context slot budget when populated (default cap 500 tokens; configurable per workspace; surfaced in the [S5.D](../M5/S5.D-reserve-and-token-xray.md) X-ray view as its own slice). On a 32k context: ~2.5% combined when full. Acceptable for the value delivered.
