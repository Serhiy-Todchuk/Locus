# S4.L -- Git Integration (.gitignore + Auto-Commit)

## Problem

Two specific git-related gaps. They are different in nature; this stage covers both because they share the same target user (developers working in a git-tracked workspace) and the same subsystem (workspace policy + post-turn hook).

1. **`.gitignore` is not respected by the indexer.** The indexer uses `.locus/config.json` `exclude_patterns` only. Users routinely have `node_modules/`, `dist/`, `.venv/`, generated SDKs, etc. in `.gitignore` but not in the Locus exclude list -- the indexer happily walks them, blowing up the symbol table and the embedding queue. This is a concrete user-visible footgun.

2. **No per-turn audit trail in git.** [S4.B](S4.B-checkpoint-undo.md) gives us file-level undo via private snapshots. For users who want their workspace's *git history* to record what the agent did (Aider's signature feature), we have nothing. A `git.auto_commit` flag that creates a one-commit-per-agent-turn record on a side branch closes this gap.

## What this stage is NOT

The original draft of S4.L proposed four dedicated git tools (`git_status`, `git_diff`, `git_log`, `git_blame`) plus a libgit2 dependency. **That is dropped.** Reasons captured here so a future stage author can revisit if conditions change:

- The agent already has all four capabilities through `run_command "git status"` / `git diff` / `git log -n 10` / `git blame file.cpp -L 10,20`. Every comparable agent (Aider, Cline, RooCode, OpenCode, Claude Code, Pi) uses subprocess for git -- none ship dedicated git tools at the LLM-tool layer.
- Each dedicated tool costs ~150 prompt tokens of schema. Four tools = ~600 tokens of manifest tax for capability that already works through `run_command`. On 8k-context local LLMs that is 7-8% of the prompt budget gone for no new ability.
- The structured-output / no-shell-quoting / libgit2-perf arguments do not outweigh the manifest cost at our budget profile. They might, on a frontier-cloud model with 200k context, where 600 tokens is rounding error.
- Without dedicated tools, libgit2 is unjustified: `.gitignore` parsing is ~50 lines on top of [src/index/glob_match.h](../../src/index/glob_match.h); auto-commit shells out to `git`.

If a future workload demonstrates that subprocess overhead per `git ...` call is actually hurting (latency-sensitive batch reads, e.g. `git_blame` over hundreds of lines feeding a refactor), revisit then.

## Tasks

### .gitignore respect

- [ ] Indexer reads `.gitignore` files at workspace open and on `.gitignore` change (file-watcher already covers it). Patterns are appended to the in-memory exclude set used by [src/index/indexer.cpp](../../src/index/indexer.cpp) `should_exclude()`.
- [ ] Support nested `.gitignore` files (sub-directories can have their own). Pattern scope is tied to the directory the file lives in -- a `node_modules/` pattern in `frontend/.gitignore` excludes `frontend/node_modules/` only.
- [ ] Implement minimal gitignore-pattern parsing: `#` comments, blank lines, leading `!` negation (skip for v1 -- treat negations as no-op and document), trailing `/` (directory-only), leading `/` (anchor to .gitignore's directory). Reuse `glob_match()` for the actual matching after pattern normalization. **Do not pull in libgit2 for this.**
- [ ] Setting in `.locus/config.json`: `index.respect_gitignore` (default `true`). When false, indexer falls back to the explicit `exclude_patterns` only. Off-switch for users who deliberately want their `.gitignore`'d files indexed (rare, but possible -- e.g. archived generated docs).
- [ ] Watcher integration: editing `.gitignore` re-runs the workspace-level exclude rebuild. Files that newly match an exclude pattern are removed from the index on the next watcher pass; files no longer matching are re-indexed.
- [ ] Activity-log event when the index drops files due to a `.gitignore` change ("Excluded 1247 paths from index after .gitignore update").
- [ ] Catch2 tests:
  - root `.gitignore` patterns excluded from index
  - nested `.gitignore` scoped to its directory
  - directory-only (`build/`) vs file (`*.log`) patterns
  - watcher re-applies on `.gitignore` edit
  - `respect_gitignore=false` falls back to `exclude_patterns` only

### Auto-commit-per-turn (optional, off by default)

- [ ] Setting in `.locus/config.json`: `git.auto_commit` (default `false`), `git.commit_branch` (default empty -> commits to current branch), `git.commit_prefix` (default `[locus] `).
- [ ] Workspace ctor checks: workspace is a git repo (presence of `.git`); if not, the feature silently no-ops regardless of `auto_commit`.
- [ ] End-of-turn hook (same hook point as [S4.C](S4.C-auto-verify.md) -- after the agent yields and at least one file mutation occurred): if `auto_commit` is true, run `git add -A && git commit -m "<prefix><agent-summary>"` via `run_command` machinery. Commit message body sourced from the agent's final assistant message, truncated to ~200 chars.
- [ ] Failure handling: if the commit command fails (uncommitted merge conflict, hooks fail, etc.), surface a single warning in the activity log -- do not retry, do not block the agent. The user resolves manually.
- [ ] Optional dedicated branch: if `git.commit_branch` is set and exists, the auto-commit goes there (current branch position is preserved). If it doesn't exist, create it once and warn the user. Useful for users who want agent commits on `locus-agent` and rebase later.
- [ ] UI: footer chip shows the most recent auto-commit short SHA when active. Click expands to the activity panel entry.
- [ ] Catch2 tests behind a `[git]` tag: in-memory git fixtures (init repo, agent-mutation simulation, assert commit landed). Skip when `git` not on PATH.

## Implementation notes

- The `.gitignore` work and the auto-commit work share only the milestone; they are independently shippable. Land `.gitignore` first -- it has zero risk and concrete user value.
- Auto-commit interacts with [S4.B](S4.B-checkpoint-undo.md) checkpoints: both record agent mutations, but at different layers. Checkpoints are private (in `.locus/checkpoints/`) and per-turn-undoable. Auto-commit is public (in the user's git history) and intended as a record, not an undo path. They are complementary -- a user can have both on, with checkpoints for fast undo and git for archival.
- The "agent already does git via `run_command`" argument applies to *agent reads*. For *agent writes* (e.g. the agent deciding to `git commit -m "..."` itself), keep that as agent-driven `run_command` -- auto-commit is a workspace-level policy, not an agent decision.
- Negation patterns (`!foo`) in .gitignore are explicitly skipped in v1. The use case is rare and the parser stays simple. If needed later, add a second pass that walks negations after the main exclude pass.
- The `.gitignore` watcher integration must avoid the file-watcher feedback loop -- our own writes (none here, since indexer is read-only on `.gitignore`) shouldn't fire the re-read. The existing [S4.T](S4.T-file-change-awareness.md) `mark_agent_touched` discipline is the template if a future stage needs to write `.gitignore` programmatically.

## Manifest cost

**Zero new tools.** Both halves of this stage are workspace-level policy / hooks; the agent's tool surface is unchanged.
