# S4.D — Plan Mode

## Problem

OpenCode, Cline, RooCode, Claude Code all have "propose plan → user approves plan → agent executes plan" workflow. Matches Locus's "user as architect" thesis exactly. Current step-by-step approval is too granular for multi-step work — user sees trees, not forest.

## Phase 1 (engine) ✔

Engine-side wiring landed first; the GUI ships in Phase 2 once the back-end is verified end-to-end with a real LLM.

- [x] Agent mode enum (`AgentMode { chat, plan, execute }`) on `AgentCore`. Mutated only on the agent thread via queued requests (same pattern as `compact_context` after S3.I) — `set_mode` from a foreign thread sets a flag, the agent thread drains it under `ConversationOwnerScope` between turns or rounds.
- [x] `propose_plan` tool, plan-mode-only via `visible_in_mode(ToolMode::plan)`. Validates `steps[].description` is non-empty, normalises optional `tools_needed`, returns the plan as `{"ok": true, ...}` JSON. Auto-approve (the user approves the resulting *plan* via the chat panel, not the per-tool gate).
- [x] `mark_step_done` tool, execute-mode-only. Takes `{step: int(1-based), status: "done"|"failed", notes?: string}`. Validates type / range / status enum. Auto-approve.
- [x] `ToolMode::execute` added; default `visible_in_mode` updated so existing tools are visible in `agent` AND `execute` (full catalog), hidden from `plan` and `subagent`.
- [x] `AgentLoop::run_step(history, ToolMode)` -- mode passed in per-call so the API tool catalog is filtered correctly each round. `AgentCore` maps `AgentMode` -> `ToolMode` (chat→agent, plan→plan, execute→execute).
- [x] `IFrontend` callbacks (default no-op): `on_mode_changed`, `on_plan_proposed`, `on_plan_step_advanced`, `on_plan_completed`. `ILocusCore` API: `set_mode`, `mode`, `approve_plan`, `reject_plan`.
- [x] `AgentCore::observe_plan_tool_result` runs after each tool dispatch; parses the result JSON and updates plan state (sets `plan_awaiting_decision_` after `propose_plan`, advances step status after `mark_step_done`, fires `on_plan_completed` when every step is done/failed).
- [x] `approve_plan` switches mode to `execute`, builds a structured plan-recap message, and pushes it onto the message queue so the agent resumes automatically. `reject_plan` clears the plan and stays in plan mode.
- [x] `reset_conversation` resets mode to `chat` and clears plan state.
- [x] System prompt's text "Available Tools" section now filters by `visible_in_mode(ToolMode::agent)` so chat-mode model calls don't see the plan-only tools listed.
- [x] 21 Catch2 tests in [tests/test_plan_mode.cpp](../../tests/test_plan_mode.cpp) tagged `[s4.d]`: AgentMode round-trip, Plan/PlanStep helpers, every `propose_plan` validation path, every `mark_step_done` validation path, tool catalog filtering by mode (agent / plan / execute).

### Notable design choices

- **No system-prompt mode-addendum.** The tool descriptions for `propose_plan` and `mark_step_done` already encode their mode behavior; the API tool-list filtering by `visible_in_mode` is the actual enforcement. Avoiding a mode-aware addendum keeps the system prompt byte-stable across mode switches -- friendly to S4.F's KV-cache work.
- **Plan recap as a synthetic user message, not a system prompt mutation.** When the user approves, AgentCore emits a multi-line user message: `[Plan approved -- executing] Title: ... Steps: 1. ... 2. ... Use mark_step_done as you complete each step.` The model sees this and continues; the system prompt is untouched.
- **Plan rejection is reject-and-re-prompt only.** Inline plan editing is deferred (Phase 2-or-later); for MVP the user rejects, types a refinement, the model proposes a new plan.
- **Mid-turn break in plan mode.** Once `propose_plan` fires and lands a plan, `process_message` breaks out of the round loop -- the agent waits for the user's decision rather than firing another LLM round.

## Out of scope (revisit if needed)

- Per-step manual approval (the `(c)` option from the original design discussion). Reject-and-re-prompt + the existing per-tool approval gate cover the same ground.
- Plan persistence across saved sessions. The plan's `propose_plan` tool call is in `ConversationHistory` already; the structured `Plan` object is in-memory only. If a user reloads a session mid-plan, they'll need to re-ask.
- Re-plan on failure. If `mark_step_done(status=failed)` fires, AgentCore surfaces it via `on_plan_step_advanced` but doesn't auto-flip back to plan mode -- the user (or model) decides what to do next.
