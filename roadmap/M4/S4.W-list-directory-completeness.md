# S4.W -- list_directory: Surface Unindexed Files

## Status: done done

Filesystem-sourced enumeration with index annotation. Tool tested at WS1; 7 new
`[s4.w][list_directory]` Catch2 cases.

## Problem

`list_directory` currently goes straight through `IndexQuery::list_directory()`, so the model only sees files the indexer tracks. Anything skipped by the indexer is invisible: binaries (PNGs, PDFs without an extractor, ZIPs, executables), files over the indexer's size cap, and files whose extension nobody registered an extractor for. On code workspaces this is mostly harmless. On document / mixed workspaces it is a real footgun -- "is there a logo in `assets/`?" returns "no" when there is one. Pi sidesteps this entirely because its `bash ls` shows the real filesystem.

The fix: make `ListDirectoryTool` filesystem-sourced and use the index for annotation, not enumeration. Keep `IndexQuery::list_directory` pure (indexed state only) -- the tool layer composes the view shown to the model.

## Tasks

- [x] Filesystem enumeration in `ListDirectoryTool::execute` via `std::filesystem::directory_iterator`, walking `depth` levels.
- [x] Apply workspace exclude patterns at the tool layer (still hide `.git/`, `node_modules/`, `build/`, `.locus/`, etc. -- excludes are intentional user policy and stay enforced).
- [x] Per-call: build a hashmap of `IndexQuery::list_directory(path, depth)` results once, then look up each filesystem entry to merge in indexed metadata (size, language).
- [x] Annotation rules:
  - Indexed text file -> existing format: `path  (size)  [lang]`
  - On disk, recognized binary extension -> `path  (size)  [binary]`
  - On disk, indexed but content skipped due to size cap -> `path  (size)  [oversized]`
  - On disk, not in index, unknown extension -> `path  (size)  [unindexed]` (invites `read_file` if model suspects text)
- [x] Explicit binary extension list (header-internal): `.png .jpg .jpeg .gif .webp .ico .bmp .tif .tiff .pdf .zip .tar .gz .bz2 .xz .7z .rar .exe .dll .so .dylib .a .lib .o .obj .pdb .ttf .otf .woff .woff2 .mp3 .mp4 .wav .mov .mkv .avi .flac .ogg .bin`. Anything outside the list and not indexed falls through to `[unindexed]`.
- [x] `max_entries` cap on output (default 200) with `[N more entries truncated; refine path or use search]` footer. Replaces the current unbounded enumeration -- already a footgun even before this change (asset folders with thousands of files).
- [x] Tool schema: add optional `max_entries` param; document binary / oversized / unindexed annotations in the description.
- [x] Catch2 tests in `tests/test_list_directory_tool.cpp`:
  - Indexed file annotation matches existing output.
  - PNG / PDF / ZIP show up as `[binary]`.
  - Excluded dirs (`.git`, `node_modules`) stay hidden.
  - Files over indexer size cap show as `[oversized]` with real on-disk size.
  - Unknown-extension non-indexed file shows as `[unindexed]`.
  - Truncation footer fires at `max_entries`.
  - `depth=0` and `depth=N` both behave correctly.

## Implementation notes

- Don't touch `IndexQuery::list_directory` -- its contract (only indexed entries) is correct and used elsewhere (regex search, file-change tracker). The tool is the policy layer.
- Exclude pattern matching: reuse `glob_match.h` and the workspace's exclude list -- the indexer already loads it; expose via `IWorkspaceServices` if not already accessible.
- Distinguishing `[oversized]` requires knowing why the indexer skipped a file. Two options: (a) a sentinel row in the `files` table with a "skipped reason" column, or (b) re-derive at tool time by stat'ing and comparing against the indexer's size threshold. (b) is cheaper and avoids a schema change; pick (b) unless we already track skip reason.
- Symlink handling: follow `std::filesystem::directory_iterator`'s default (no follow). Do not chase symlinks into excluded territory.
- Cross-reference cost: one `IndexQuery::list_directory` call per tool invocation, plus one filesystem walk. Both are already O(entries); the merge is a hashmap lookup. No measurable overhead at workspace scale.
- The `[unindexed]` annotation is deliberately ambiguous -- it tells the model "I don't know what this is, try `read_file` if you care". Better than guessing wrong with `[binary]`.
