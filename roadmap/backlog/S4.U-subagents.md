# S4.U -- Subagent / Task Delegation

> **Status: Backlog (parked).** Reactivate by promoting back into [M4/README.md](../M4/README.md) -- but only if the constraints listed below stop applying.

## Why this is parked

Cribbed from Claude Code's `Task` and Cursor's agent mode. Both run against cloud APIs where parallel inference, separate workers, and abundant context make subagents feel free. None of those preconditions hold for Locus.

Concrete reasons:

1. **Permanent manifest tax for a corner case.** `spawn_subagent(task_description, tools_allowed[], max_turns)` is ~250 prompt tokens of schema sitting in every prompt forever, despite realistic per-session usage being zero. Most workflows that "feel like" they want a subagent are actually solved by `search_semantic` or [S4.R](../M4/S4.R-memory-bank.md) memory bank.
2. **Direct contradiction with our own thesis.** [DIFFERENTIATORS.md](../../DIFFERENTIATORS.md) #3 (user as architect, full transparency, every tool call visible before execution) and #4 (tools over context stuffing) both push the other way. A subagent is a black box that runs its own approval-gated agent loop, makes its own tool calls, and returns a summary. Either the parent's approval policy applies (UI nightmare: nested approvals) or it doesn't (silent autonomous loop -- exactly what we differentiate against).
3. **Single-GPU runtime gives no parallelism.** Cloud agents get free parallelism because each subagent hits a different cloud worker. On a single 24 GB consumer GPU running a 26 B model, a subagent runs *serially* on the same model after the parent yields. Net latency is no better than the parent doing the work in-line, but with extra ceremony.
4. **`tools_allowed[]` nests a hard problem inside itself.** Picking the right tool subset is a skill the agent already struggles with at the top level. Asking a 7-13 B local model to *also* pick which tools to grant a child is compounding the failure mode.
5. **Budget caps are vibes.** "max concurrent subagents, max total tokens per session" is a band-aid for an obvious failure mode (a small local model in a delegation loop). The cap papers over the real issue: this feature shouldn't be there to need capping.
6. **"Summary not transcript" is compaction with extra steps.** If summarizing in-progress work into the parent's context is valuable, that's [S4.R](../M4/S4.R-memory-bank.md) and the existing compaction story. We don't need a new agent runtime to deliver it.

## Reactivation triggers

Bring this back if **any** of the following becomes true:

- Locus gains parallel inference (multi-GPU, batched vLLM, cloud-API fallback) -- the runtime would actually benefit from concurrent agent loops.
- A real workload demonstrates that an inline `search_semantic` + `read_file` loop is materially worse than a subagent for the same task (with measurements, not vibes).
- The local-model ecosystem produces models that reliably emit batched tool calls + handle nested approval semantics cleanly. We are not close.

## Original spec (preserved for reference)

### Problem (original framing)

Claude Code's `Task`, Cursor's agent mode. Owner-profile (high-level thinker, AI-as-heavy-lifter) wants this -- delegate "research how X works" to a subagent, get a report back, keep main context clean.

### Tasks (original)

- [ ] `spawn_subagent` tool: `task_description`, `tools_allowed[]`, `max_turns`
- [ ] Subagent runs in a child `AgentCore` with isolated history + scoped tools
- [ ] On completion: summary (not full transcript) returned to parent as tool result
- [ ] UI: subagent activity visible in a collapsible group or separate panel
- [ ] Budget: max concurrent subagents, max total tokens per session
- [ ] Nice-to-have: typed subagent presets (researcher, codegen, reviewer) configured in `.locus/agents/`
