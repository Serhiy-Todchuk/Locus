# S4.Q -- Multi-Model Split (Weak + Strong)

> **Status: Backlog (parked).** The original framing is Aider-envy that doesn't translate to consumer GPUs. If this comes back, it should be rewritten around the **CPU-side weak model** story below, not the cloud cost-tier story.

## Why this is parked

Aider's `--weak-model` exists because cloud APIs charge by token tier -- `gpt-3.5` vs `gpt-4` is a real $0.0005 vs $0.06 per-call decision, so routing summarization/titling to the cheap tier is obviously correct. Locus doesn't run against cloud APIs. None of the cost-tier math applies.

Concrete reasons against the spec as written:

1. **Local weak/strong = two model files in finite VRAM.** A 26 B "strong" + 7 B "weak" on a 24 GB consumer GPU is a crash, not a config option. Realistic options shrink to (a) swap models per task -- tens of seconds of dead air the user didn't have before, or (b) run weak on CPU -- which the spec does not mention.
2. **The use-case list is comically thin.** "Compaction summary, session titling, commit message generation, activity-event summaries" -- five occasional ~500-token jobs per session. Dual-model infrastructure with a router for ~3 K weak-model tokens of total work in a multi-hour session is upside down.
3. **The skeleton already shipped.** [S3.B](../M3/S3.B-llm-client-split.md) landed `LlmRouter` with `Role::Strong | Role::Weak`. The remaining work in the original spec is "config field + Settings checkbox + UI lists both endpoints" -- a UI tweak, not a stage.
4. **"Same endpoint, different model names" is not a feature.** That's a user filling in `LLMConfig` twice. We don't need a router to expose it.
5. **The pain it claims to solve is mis-targeted.** Users who run 26 B locally already accept their hardware constraints. Their friction is "26 B inference is slow," not "I wish 1 % of my tokens went to a cheaper model." The right interventions for that pain are [S4.F](../M4/S4.F-kv-cache.md) (KV cache preservation) and [S5.A](../M5/S5.A-capability-toggles.md) (manifest pruning), not weak/strong split.

## The actually-good story (for a future rewrite)

The version of this stage that *would* land cleanly on local hardware is:

- **CPU-side weak model.** Keep the GPU-resident strong model warm. Run a tiny CPU-quantized model (1-3 B, Q4) for non-latency-critical jobs that the user won't notice -- titling, commit messages, activity summarisation. No VRAM contention; no model swap; no KV-cache eviction on the strong model.
- **Lifetime + cache discipline.** A weak inference must not evict the strong model's KV cache. Document the constraint and verify with logs.
- **Honest perf claims.** Measure wall-clock end-of-turn savings on real sessions before promoting; the realistic number is small.

That stage has a coherent reason to exist. The Aider-cribbed version does not.

## Reactivation triggers

Bring this back if **any** of the following becomes true:

- We pick up a real CPU-inference path (llama.cpp CPU build for the weak model) and there is concrete user demand for the offloaded jobs.
- The local-model ecosystem produces a generally-recognised "Q1-class summarisation model" that runs trivially on CPU and is good enough for our use cases.
- Multi-GPU or batched-inference setups become a Locus-targeted profile.

## Original spec (preserved for reference)

### Problem (original framing)

Aider's `--weak-model` uses a cheap model for summarization/naming, strong model for reasoning. On local GPUs (tight VRAM), this is the difference between feasible and infeasible.

### Tasks (original)

- [ ] `llm.weak_endpoint` + `llm.weak_model` config (fallback to main if unset)
- [ ] Weak-model use cases: compaction summary, session titling, commit message generation, activity-event summaries
- [ ] `ILLMClient` factory per role: `get_client(Role::Strong | Role::Weak)`
- [ ] UI: settings lists both; allow same endpoint with different model names
