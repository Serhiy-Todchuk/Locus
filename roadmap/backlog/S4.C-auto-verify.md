# S4.C -- Auto-Verify Feedback Loop

> **Parked.** Originally an M4 stage; demoted because the "agent self-corrects from build/test error tail" premise assumes the model can reliably interpret arbitrary verifier output, which 7B-26B local models do poorly -- especially on the C++ template / linker errors that are the most common case in this very repo. With `verify.max_retries=2` per turn, a stubborn failure burns tens of seconds of local-inference wall-clock on retries the user could resolve in one sentence. `run_command` already exists for explicit invocation, and the user-as-architect / transparency thesis means the user is in the loop for every turn anyway -- they see broken code and can call [S4.B](../M4/S4.B-checkpoint-undo.md) `/undo` or prompt a fix. Plan Mode ([S4.D](../M4/S4.D-plan-mode.md)) covers the "scope work before mutating" angle that auto-verify partially addresses.
>
> **Reactivate when**: (a) local models reliably interpret build/test failure output (telemetry shows them succeeding on `run_command`-driven retry attempts), OR (b) Locus gains a CPU-side weak model that can cheaply pre-process error tails into actionable summaries (aligns with the parked [S4.Q](S4.Q-multi-model.md) CPU-offload rewrite), OR (c) a non-C++ workspace surfaces where verifiers are fast (sub-second) and produce one-line errors that small models handle well -- in which case the original `on_edit` trigger that v1 dropped becomes interesting again.

## Problem

Agent writes broken code, reports success. `run_command` already exists but nothing triggers it automatically -- for a C++ project that means template / link errors reach the human instead of being caught by the agent's own loop.

The standard fix (Aider's `--lint-cmd` / `--test-cmd`, Cline's terminal-watching pattern): after the agent finishes its turn, run a configured verifier. On non-zero exit, inject the error tail into the *same turn* so the agent can react before yielding control to the user. Result: the agent self-corrects in one user-turn instead of three.

This is not novel mechanism. We are catching up to a pattern that has shipped in other agents for years.

## Trigger semantics: turn-end only (no per-edit)

The trigger fires **after the agent yields control** -- not after each individual file mutation. Reason: a multi-file refactor (e.g., changing a method signature in both `.h` and `.cpp`) goes through several `edit_file` / `write_file` calls within a single turn. A per-edit trigger would fire after the `.h` edit, see a linker error because `.cpp` hasn't been updated yet, and inject a misleading "verify_failed" mid-task. The agent either burns turns "fixing" non-existent breakage or learns to ignore the verifier -- both worse than no verify.

The original draft of this stage offered `trigger: on_edit | on_turn_end | manual`. **`on_edit` is dropped from v1.** Only `on_turn_end` (default) and `manual` (explicit `/verify` slash command) are exposed. If a future workflow demands per-edit feedback (single-file workflows with sub-second verifiers), reintroduce it then.

## Tasks

- [ ] `.locus/config.json` gains a `verify` block: `verify.cmd` (string, default empty -> stage dormant), `verify.trigger` (`on_turn_end` | `manual`, default `on_turn_end` when `cmd` is set), `verify.timeout_ms` (default 30000), `verify.tail_lines` (default 80).
- [ ] Hook the trigger into `AgentLoop`'s end-of-turn path: when the agent emits its final assistant message for the turn (no further tool calls pending) and at least one file-mutating tool fired in this turn, run `verify.cmd`. Non-mutating turns (pure search / question answering) skip verify.
- [ ] Non-zero exit: assemble a `<verify_failed exit_code=N>` message with the last `tail_lines` lines of combined stdout+stderr; inject as a tool-result-shaped message and re-enter the agent loop so the agent gets one more LLM round to react. Cap injected payload at ~500 tokens; if the tail is bigger, truncate with a `[... truncated; N more bytes ...]` marker.
- [ ] Zero exit: log only ("verify: ok in 4.2s"). No context injection. Footer chip flips green.
- [ ] Loop guard: cap verify-driven re-entries at `verify.max_retries` (default 2) per user turn. Without this, a stubborn build error could ping-pong indefinitely.
- [ ] UI: verify status chip in footer (idle / running spinner / green / red). Click expands to show last command + tail in the activity panel.
- [ ] Slash command: `/verify` triggers a manual run regardless of trigger mode; output goes to the activity panel and is NOT injected into the agent's context (manual is for the user, not the agent).
- [ ] Per-language preset commands surfaced in Settings (not auto-applied): `cmake --build build`, `tsc --noEmit`, `cargo check`, `pytest -q`, `go build ./...`, `mvn -q compile`. User picks one as a starting point and edits.
- [ ] Catch2 tests:
  - turn-end trigger fires only after a mutation occurred in the turn
  - zero-exit silence (no injection, log line present)
  - non-zero injection with correct tail size + truncation marker
  - retry cap honored
  - manual `/verify` does not feed the agent

## Implementation notes

- The hook lives where the existing post-mutation snapshot machinery lives ([src/agent/tool_dispatcher.cpp](../../src/agent/tool_dispatcher.cpp) -- `set_turn_context` already tracks file-mutation events for [S4.B](S4.B-checkpoint-undo.md) checkpoints). Verify reuses the same "did any mutation happen this turn?" flag rather than re-deriving it.
- The verify subprocess uses the existing process-execution machinery from [S4.I](S4.I-background-commands.md), but **synchronously** with the timeout cap, not as a tracked background process. The user does not need to see verify in `list_processes`.
- Pairs naturally with [S4.B](S4.B-checkpoint-undo.md): if verify fails repeatedly, the agent has the option to call `undo_turn` and retry from the snapshot. No new mechanism for that flow -- the tool already exists.
- KV cache: the verify failure injection happens on the volatile tail (latest tool-result-shaped message), not the system prompt. No interaction with [S4.F](S4.F-kv-cache.md)'s byte-stable prefix invariant.
- The footer chip is purely informational; it does not block the agent or the user. Closing a workspace mid-verify cancels the subprocess via the existing process-cancel path.
