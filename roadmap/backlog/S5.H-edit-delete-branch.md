# S5.H -- Per-Message Edit / Branch & Rewind (parked); Move-to-new-tab (parked)

**Status: PARKED (demoted from M5 on 2026-05-16).** Original spec preserved below for reactivation.

### Why parked

Cross-tool review found that most of S5.H's scope is either speculative or strictly dominated by simpler primitives:

- **Per-message delete is the only load-bearing operation here.** It covers the real use cases (secret/PII redaction, dropping a fat stale tool result, removing a wrong-direction reasoning blob). It landed instead as a hover-reveal X button per chat bubble in [S5.G](../M5/S5.G-chat-activity-restructure.md) -- single method on `ConversationHistory`, single confirm dialog, no archive infrastructure.
- **Edit is a footgun.** Editing assistant messages or tool results injects hallucinated "ground truth" the downstream conversation already assumed otherwise; the next turn's KV-cache state is already shaped by the unedited content, so editing the historical record creates a false sense of safety without actually unwinding what the model saw. For secret redaction, delete is strictly safer (no half-tampered state). For trimming verbose tool results, [S5.F](../M5/S5.F-compaction-v2.md) Layer 2 does it automatically and better. For fixing typo'd user messages, "delete the last few messages and retype" is one extra click and avoids the rewriting-history problem entirely.
- **Rewind / branch / restore-from-archive / Move-to-new-tab-from-here are all the same operation with different UX framings**, and the real workflow they targeted (back up a few turns, try a different direction) is covered by the simpler "delete the last few messages and retype" path. The user explicitly chose the simpler path: no archive chain, no "Rewind..." picker, no Settings "Recover history" menu, no per-archive metadata, and no tab-clone-from-message-X right-click action in [S5.I](../M5/S5.I-tabs-and-sessions.md).
- **The archive chain is expensive infrastructure for a problem nobody has yet.** Every edit / delete / rewind would write a `history.before-<op>-<N>.json`; restore-from-archive needs a Settings UI; the chain needs GC. None of this pays rent if delete-and-retype is the actual user workflow.

### What landed instead

- [S5.G](../M5/S5.G-chat-activity-restructure.md) gains a hover-reveal X button per chat bubble. Single click, confirm dialog, message dropped from `ConversationHistory`. No archive. No backend schema change. This is everything the user actually needs from S5.H for v1.

### Reactivate when

Real usage shows users wanting to edit (not delete-and-retype) past messages -- e.g., a long-running planning conversation where retyping costs more than fixing one word -- OR a parallel-exploration workflow surfaces that delete-and-retype + multi-tab ([S5.I](../M5/S5.I-tabs-and-sessions.md)) doesn't cover. The "Move to new tab from here" right-click in S5.I was parked alongside this stage for the same reason: if a real workflow demands it, reactivate both together.

## Problem

[S5.D](../M5/S5.D-reserve-and-token-xray.md) exposes the budget. [S5.F](../M5/S5.F-compaction-v2.md) gives a layered compaction toolkit with auto-preservation heuristics. [S5.G](../M5/S5.G-chat-activity-restructure.md) puts every LLM-visible message in chat. (Persistent per-message importance markers were originally [S5.E](S5.E-pin-and-tag.md); parked.)

What's still missing is **direct surgical control over individual messages and over the conversation as a whole**:

1. The user can't fix a misremembering, redact a secret, or trim verbosity in a past message.
2. The user can't delete a single message that's no longer useful (a stale tool result, a wrong-direction reasoning blob).
3. The user can't rewind to an earlier point and try a different direction without throwing away the entire conversation.

This stage delivers the surgical tools. It is the most ambitious M5 context-management stage and the right place to land last -- everything before this benefits from the foundations; this stage can sit on top of them safely.

## Tasks

### Per-message edit

- [ ] Right-click on any chat bubble -> "Edit". Opens an inline edit surface (existing chat input widget reused, pre-populated with the message body).
- [ ] Editing applies on save and triggers a re-render of the bubble. Edited messages get a small "edited" badge with a tooltip showing the original timestamp.
- [ ] Edits to **assistant or tool-call/result** messages: a confirmation prompt warns "this rewrites history -- subsequent turns in this session were generated against the original. Continue?" Accept rewrites; reject cancels.
- [ ] Edits to **user** messages: re-running from this point is offered ("regenerate from here?"). If accepted, every message after the edited user message is dropped (archived to history archive from [S5.F](../M5/S5.F-compaction-v2.md)) and the agent re-runs from the edited prompt.
- [ ] System-prompt bubble cannot be edited from chat (it's owned by `AgentCore`); attempts surface a hint "edit LOCUS.md or workspace metadata to change the system prompt."
- [ ] Persistence: edited messages round-trip in saved sessions; the `edited: true` + original-timestamp metadata is preserved.

### Per-message delete

- [ ] Right-click -> "Delete". Confirmation dialog spelling out what will happen: "this drops the message from history; subsequent turns will not see it. Original is preserved in this session's history archive."
- [ ] Deletion creates a history-archive snapshot (reusing [S5.F](../M5/S5.F-compaction-v2.md)'s `history.before-edit-<N>.json` analog -- new `before-delete-<N>` filename pattern).
- [ ] Pinned messages can be deleted -- pin is about compaction, not deletion. The confirmation dialog notes "this message is pinned; deleting still removes it permanently from active history (recoverable from archive)."
- [ ] Tool-call deletion: if the matching tool-result is still in history, surface the relationship in the confirmation ("this is the tool call for `read_file` whose result is at message #42; delete only the call, only the result, or both?"). Three buttons.

### Branch and rewind

- [ ] Right-click on any user message -> "Rewind to this turn". Drops every message after this user message (archived); agent ready to receive a new user input from this point. The dropped lineage is preserved in the archive chain so the user can compare or restore.
- [ ] **Branch** is the same operation under a different name and slightly different intent: rewind drops the lineage from active history but archives it; branch additionally tags the archive with a user-supplied label so the user can recognize it later. UX-wise: both options on the same right-click menu, separate buttons, same underlying mechanism.
- [ ] New chat-toolbar button: "Rewind...". Opens a small picker showing every user message in the current session with a snippet preview, click one to rewind to it.
- [ ] **Restore from archive** -- a new menu in Settings -> Sessions ("Recover history") lists every archive file for the current session with timestamp + a one-line preview. Clicking restores the archive into the active session, archiving the current state first (so restore is reversible).

### "Move to new tab from here" (also parked, originally a S5.I right-click)

- [ ] Right-click on any chat message in [S5.I](../M5/S5.I-tabs-and-sessions.md) tabs context menu -> "Move to new tab from here". Uses this stage's rewind+archive machinery: rewind the active tab's history to the message, archive the rewound lineage, then create a new tab seeded with the archived state up to the rewind point. Powerful but depends on this stage's archive chain landing. When this stage was demoted to backlog, S5.I dropped the right-click entry; reactivate both together if the workflow surfaces.

- [ ] Catch2 tests:
  - edit + persistence round-trip
  - delete preserves archive
  - rewind drops post-target messages, archive captures them
  - restore-from-archive replaces active state, current state archived
  - tool-call/result paired-delete options behave correctly

## Implementation notes

- All four operations (edit, delete, rewind, restore) write a `history.before-<op>-<N>.json` archive *before* mutating the active history. This is the same machinery [S5.F](../M5/S5.F-compaction-v2.md) uses; reuse the writer. GC respects [S5.F](../M5/S5.F-compaction-v2.md)'s `archive_keep_count` shared cap.
- "Edit + regenerate from here" on a user message is essentially a manual rewind followed by submission. Implement as such: archive -> truncate history -> submit edited content as new user input.
- The chat re-render after edit / delete / rewind is the only piece that touches the wxWebView state. Existing `setMsgHtml` / DOM manipulation suffices; a full chat refresh from `ConversationHistory` is the safe fallback.
- Edits on assistant or tool-result messages are powerful but dangerous: subsequent turns were generated assuming the original. The confirmation language is the safety net; the archive is the recovery.
- Branch labels are just a string field on the archive metadata. No fancy branching UI in v1 -- a list of labeled archives is enough. Tree-style branch visualization is a future stage if demand surfaces.
- The "Rewind..." picker UI is a `wxListBox` / lightweight picker; no need for a custom widget.
- KV-cache discipline ([S4.F](../M4/S4.F-kv-cache.md)) is unaffected -- these operations rewrite the volatile tail (history past the system prompt). System prompt stays byte-stable.

## Out of scope (v1)

- **Tree branching UI** (visualizing multiple parallel branches). One archive list with labels is enough; advanced branching can come later if real workflows demand it.
- **Cross-session restore** (importing an archive from session A into session B). Out of scope; archives are per-session.
- **Auto-archiving on every turn** as a defensive measure. Archives are explicit (created on edit / delete / rewind / compaction); not on every assistant response.
- **Diff view between two archive states.** Manual inspection of the JSON archives is enough for v1; a UI diff tool can come later.

## Manifest cost

**Zero new tools.** All operations are user-driven through the chat UI; the agent has no tools to edit / delete / rewind history.
