# S4.H -- Parallel Tool Calls

> **Status: Backlog (parked).** The optimisation targets a runtime profile we don't have. Reactivate if the preconditions below start to hold for local LLMs.

## Why this is parked

Cribbed from cloud-agent observation -- where models reliably emit `tool_calls: [...]` arrays in a single assistant message and per-tool latency is meaningful relative to inference time. Neither precondition holds for Locus's target stack (LM Studio + 7 B-26 B local model + local SQLite/FTS5 index).

Concrete reasons:

1. **The precondition rarely fires.** The "common case" the original spec assumes (model emits read + outline + search together) is observation-biased toward cloud-frontier models that are well-trained on parallel tool use. Local 7 B-26 B models typically emit one tool call per turn. We would build the entire batched-execution path for an event that fires in a small minority of turns.
2. **Tool latency is dwarfed by LLM time.** A `read_file` is <50 ms; an `IndexQuery::search_text` is <100 ms. The LLM round trip on a 26 B local model is several seconds. Even if three tools fire in parallel, best-case wall-clock saving is ~200 ms in a 5-10 s turn -- below the user-perceptible threshold.
3. **The "3-5x throughput" claim is cloud-agent math.** It assumes per-tool latency is the bottleneck. Locally it is not. Real saving is `t_total_serial - max(t_i)`, which on a workspace index with sub-100 ms tools is sub-second per turn at best.
4. **Concurrent index access is real engineering cost.** SQLite WAL handles concurrent readers, but `IndexQuery`'s prepared statements are not currently thread-safe. Making them thread-safe means either thread-local statement caches (more memory, more lifecycle) or serialising at the IndexQuery layer (which defeats the parallelism). Either way, the engineering bill is real for a sub-second win.
5. **The auto-approve-only carve-out kills the user-perceived benefit.** "approval-auto only -- approval-required tools stay serial" means the tools that are already fast (no human in the loop) get parallelised; the slow tools (waiting for user click) stay serial. Net user-experience improvement: zero.
6. **No frontend gain either.** The activity log streams events as they happen; the user doesn't notice tool calls completing 200 ms faster in aggregate when the next LLM round is still seconds away.

## Reactivation triggers

Bring this back if **any** of the following becomes true:

- Local models the project targets (Gemma 4 26B-class and below) start reliably emitting batched `tool_calls` arrays in real sessions, not as a synthetic test.
- A workload appears where tool-time dominates LLM time (e.g. a tool that calls a remote API with multi-second latency) such that parallelisation moves user-perceived wall-clock.
- `IndexQuery` is refactored for thread safety as a side effect of another stage, taking the engineering bill off this one's tab.

## Original spec (preserved for reference)

### Problem (original framing)

OpenAI tool-use spec supports `tool_calls: [...]` arrays in one assistant message. Serial execution leaves 3-5x throughput on the floor when the LLM requests read + outline + search together (the common case).

### Tasks (original)

- [ ] Detect multiple `tool_calls` in one LLM message
- [ ] Run read-only tools concurrently on a thread pool (approval-auto only -- approval-required tools stay serial)
- [ ] Aggregate results; inject back to LLM in original call order
- [ ] UI: approval panel handles batched approval for multiple pending tools
- [ ] Catch2 tests: result ordering preserved; `IndexQuery` thread-safety under concurrent access
