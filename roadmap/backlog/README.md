# Backlog

Unscheduled tasks. Items here are not part of any milestone -- they are recognized as worth doing eventually but have not been scoped, sequenced, or assigned to a milestone yet.

When an item is picked up:
- promote it into the right milestone's stage list (`roadmap/Mx/README.md`),
- give it a stage letter, and
- create the per-stage doc under `roadmap/Mx/Sx.<letter>-<slug>.md`.

Until then, items live here as one-line entries (or short paragraphs when the rationale is non-trivial). Items that already have a written-out spec live as their own files in this folder and are linked from the **Drafts** section below.

## Drafts (full specs, parked)

- [S4.C -- Auto-Verify Feedback Loop](S4.C-auto-verify.md). Originally an M4 stage; demoted because the "agent self-corrects from build/test error tail" premise breaks down on small local LLMs -- 7B-26B models thrash on C++ template / linker errors, the most common case in this very repo, burning tens of seconds of local-inference wall-clock on retries. `run_command` already exists for explicit invocation; the user-as-architect / transparency thesis means the user is in the loop every turn anyway and can call [S4.B](../M4/S4.B-checkpoint-undo.md) `/undo` or prompt a fix when they see broken code. Reactivate when local models reliably interpret build/test failure output, OR when a CPU-side weak model (parked [S4.Q](S4.Q-strong-weak-model.md) rewrite) can cheaply pre-process error tails into actionable summaries.
- [S4.E -- LSP Integration (via MCP)](S4.E-lsp.md). Originally an M4 stage; demoted because [S4.G -- MCP](../M4/S4.G-mcp.md) covers the most common LSP need that was real (external tools, compiler-side diagnostics via MCP-wrapped servers) and a custom in-process LSP client is unjustified maintenance. (The original demotion rationale also leaned on [S4.C -- Auto-Verify](S4.C-auto-verify.md) for diagnostics; that stage has since been parked too, so the residual case for LSP-via-MCP rests on MCP alone.) Revisit if a workflow demonstrates that MCP-wrapped LSP is meaningfully worse than native (latency-bound interactive use, large-batch reference lookups). The spec already captures the rationale and the reduced-scope "curate `mcp.json` snippets + a Settings preset picker" plan if a future maintainer wants a smaller landing path.
- [S4.U -- Subagents](S4.U-subagents.md). Originally an M4 stage; parked because subagents are a cloud-agent pattern that doesn't survive a single-GPU local-LLM runtime. Permanent ~250-token manifest tax, contradicts the user-as-architect / transparency thesis, no real parallelism on one GPU, and "summary not transcript" is just compaction with extra steps. Reactivate only when Locus gains real parallel inference (multi-GPU, batched vLLM, cloud fallback) or a measured workload proves an inline `search_semantic` + `read_file` loop is materially worse than a child agent.
- [S4.Q -- Strong/Weak Model Split](S4.Q-strong-weak-model.md). Originally an M4 stage; parked because the Aider cost-tier rationale doesn't translate to consumer GPUs (loading two model files in finite VRAM crashes; swapping eats wall-clock). The skeleton already shipped with [S3.B](../M3/S3.B-llm-client-split.md). The version that *would* land cleanly is "CPU-side weak model for non-latency-critical jobs" -- a different stage with a different rationale, captured in the parked spec. Reactivate by rewriting around CPU offload, not Aider's cost-tier story. (Renamed from "Multi-Model Split" to keep the "multi-model" label free for a future stage about running multiple model backends / families side-by-side -- a different scope.)
- [S4.H -- Parallel Tool Calls](S4.H-parallel-tools.md). Originally an M4 stage; parked because local 7B-26B models rarely emit batched `tool_calls` (the precondition rarely fires) and per-tool latency is dwarfed by LLM round-trip time anyway -- best-case saving is sub-second per turn. The auto-approve-only carve-out kills the user-perceived benefit. Reactivate when local models reliably emit parallel calls AND tool latency stops being a rounding error.
- [S5.H -- Per-Message Edit / Branch & Rewind + Move-to-new-tab-from-here](S5.H-edit-delete-branch.md). Originally an M5 stage; demoted 2026-05-16 after a value-vs-effort review. The load-bearing slice -- per-message delete -- moved into [S5.G](../M5/S5.G-chat-activity-restructure.md) as a hover-reveal X button (no archive infrastructure; "delete the last few messages and retype" covers the rewind workflow). Edit is a footgun: editing assistant messages or tool results injects hallucinated "ground truth" the next turn's KV-cache state already assumed otherwise; for secret redaction, delete is strictly safer (no half-tampered state); for trimming verbose tool results, [S5.F](../M5/S5.F-compaction-v2.md) Layer 2 does it automatically and better. Rewind / branch / restore-from-archive / "Move to new tab from here" (originally a [S5.I](../M5/S5.I-tabs-and-sessions.md) right-click) are all the same operation with different UX framings and all expensive infrastructure (per-archive metadata, GC, Settings recover-history menu) for a problem the simpler delete-and-retype path covers. **Reactivate when**: real usage shows users wanting to edit (not delete-and-retype) past messages -- e.g., a long planning conversation where retyping costs more than fixing one word -- OR a parallel-exploration workflow surfaces that delete + multi-tab ([S5.I](../M5/S5.I-tabs-and-sessions.md)) doesn't cover.
- [S5.E -- Per-Message Pin / Tag System](S5.E-pin-and-tag.md). Originally an M5 stage; demoted 2026-05-16 after a value-vs-effort review. Five of [S5.F](../M5/S5.F-compaction-v2.md)'s six compaction layers don't depend on pin state (only Layer 4 "drop low-priority tagged" does, and Layer 4 is the weakest layer -- skipped entirely if 1+2+3+6 free enough room). The "preserve critical messages" use case has cheaper covers: system prompt is already byte-stable ([S4.F](../M4/S4.F-kv-cache.md)), recent N turns can be auto-preserved by heuristic, attached-context messages can be auto-flagged where they're created, and S5.F's `CompactionView` already supports inline pin/unpin in the preview *before* the cascade runs (contextual, at the moment of decision, no persistent state needed). Persisting a 3-state field on every `ChatMessage` would have committed the saved-session JSON schema to a policy not yet validated against real workloads. **Reactivate when**: real usage of S5.F shows users repeatedly wanting to mark specific messages as "preserve always" or "drop first," AND the implicit heuristics plus CompactionView's pin-before-run aren't enough.

## Items

- **Raise workspace `index.max_file_size_kb` default from 1024 to ~4096.** [S5.N](../M5/S5.N-non-code-workspace-proof.md) found that rfc9114.pdf (1158 KB, the HTTP/3 spec) was silently dropped at the 1024 KB cap. The WS3 build script ([scripts/build_ws3.ps1](../../scripts/build_ws3.ps1)) sets 4096 explicitly to work around it, but the project-wide default should be saner for document workspaces. Single-line change in [src/core/workspace.h](../../src/core/workspace.h) `WorkspaceConfig::max_file_size_kb`; weigh against memory pressure during full re-index of a 4 MB file. (Companion fix: production `IndexQuery::search_text` FTS5 escaping was applied in S5.N -- this entry is the only remaining piece.)

- **Acronym <-> spelled-out form retrieval gap**. Same S5.N investigation: gold query "ARP address resolution protocol" missed rfc826.pdf on all three retrieval methods, because the 1982 RFC text uses only the spelled-out form "Address Resolution Protocol" and never the acronym "ARP" (the acronym only appears in our own notes). FTS5's implicit AND requires every token to match, semantic embedding leaned on the unfamiliar "ARP" token. Generic problem -- affects any acronym a user might type that the corpus doesn't author with the acronym form (Locus/LM Studio search "DNS"/"Domain Name System", "TLS"/"Transport Layer Security"). Possible v1 mitigations: expand obvious acronyms before FTS5 binding via a small static map; expand at index time and store both forms in the FTS row; or rely on query rewriting at the LLM agent level (the agent already gets to choose words for the tool call). **Reactivate when**: real-world workloads show users typing acronyms that miss, OR when an LLM-side query-rewrite stage is on the table for other reasons.

- **Agent-side history-archive recovery tool** (conditional on user demand). [S5.F](../M5/S5.F-compaction-v2.md) writes a chained history archive before each compaction and tells the *user* about it via a footnote in the new context. The LLM does not currently get a tool to read these archives -- a deliberate v1 cut to avoid the agent burning tokens re-reading content that was compacted on purpose. **Reactivate when**: a real workload demonstrates the agent saying "I summarized X earlier; I need the original," and the user friction of manual archive inspection outweighs the manifest cost of a `read_history_archive(path)` tool. Implementation would be small (one new tool, scoped to `.locus/sessions/<id>/history.before-*.json` paths) once the gate is met.
- **Stripped tool-result fetch by call_id**. Narrower companion to the history-archive recovery item above. [S5.F](../M5/S5.F-compaction-v2.md) Layer 2 strips large tool result bodies (replaces them with a header line). Today the body is gone; the agent cannot recover it. With this item: stripped bodies are written to `.locus/sessions/<id>/tool_results/<call_id>.txt` and a new tool `fetch_tool_result(call_id)` returns them on demand. Smaller scope than full history archive (no semantic search, no whole-conversation file), addresses the realistic 80% failure mode ("I need the body of that earlier `read_file`"). One new tool, ~150 prompt tokens of schema. **Reactivate when**: telemetry or anecdote shows the agent re-reading files via `read_file` to recover content that was just stripped, when the original was sitting in a stripped result the agent could have fetched cheaper.
- **Memory Bank auto-mining from sessions**. End-of-session sweep that runs an LLM call to extract durable factual claims from the conversation and proposes them as memory-bank entries (user approves which to commit). Inspired by MemPalace's auto-sweep. **Costs**: one extra LLM call per session (multi-second on local 26B); summary fidelity dependent on the model; privacy concerns ("the agent is summarising every conversation into a permanent DB"). **Reactivate when**: telemetry shows users repeatedly typing the same facts (i.e. the manual `/memorize` + right-click "Save as memory" paths from [S4.R](../M4/S4.R-memory-bank.md) aren't getting used and the agent is re-learning the same facts across sessions). Implementation would add an opt-in `memory.auto_mine` flag (default off) plus an end-of-session hook in `AgentLoop`; the proposed entries surface in a confirmation panel rather than auto-committing.
- **Memory Bank validity windows (v2)**. Temporal supersession for memory entries: "Fact F was true between dates X and Y," with invalidation + timeline queries. MemPalace's SQLite knowledge graph does this. **Reactivate when**: real workloads show the agent acting on stale memory entries because a fact changed (e.g. build command updated, naming convention switched). Until then, the simpler "edit the entry to update it" path that [S4.R](../M4/S4.R-memory-bank.md) v1 ships covers most cases.
- **RAG over compaction archive** (semantic search over prior conversation history). Each archived message gets chunked, embedded with the existing `bge-small`/`bge-m3` embedder, and stored in a session-scoped vector table parallel to the workspace's `chunk_vectors`. New tool `search_session_history(query, max_results)` lets the agent query semantically over compacted-out content when summary fidelity wasn't enough. Reuses the entire embedder + sqlite-vec infrastructure; storage cost is small. **Risks**: (1) vector search over short conversation snippets is noisier than over file content -- snippets often lack standalone semantic meaning; (2) ~150-200 tokens of permanent manifest schema; (3) agent over-fetch ("re-read content we explicitly compacted"). **Reactivate when**: (a) [S5.F](../M5/S5.F-compaction-v2.md) summaries demonstrably lose context the agent later needs, AND (b) the simpler `fetch_tool_result(call_id)` item above doesn't cover the failure (i.e., the missing content is conversational, not tool-result bodies). The two items are a graduated response -- start with the cheaper one, escalate to RAG only if the simpler tool isn't enough. Companion to [S4.R Memory Bank](../M4/S4.R-memory-bank.md) (which preserves *curated* facts always-in-context, not retrievably-on-demand).
- Tool-result caching (dedup identical tool calls in a session). Originally an S4.V bullet; dropped because it solves a problem that doesn't exist on local hardware -- a repeat `search_symbols("foo")` is sub-100 ms and agents rarely make exact-duplicate calls in practice. Engineering cost (cache key including watcher state, eviction) is real for invisible savings. Reactivate if telemetry surfaces a real workload where tool-time dominates.

- **Tool-call history compaction (chat panel).** Today every tool call in a turn renders as its own `msg-tool` bubble (success and failure alike -- the asymmetry users sometimes report is perceived, not real: failures often have larger result tails that visually dominate). On turns with many tool calls (8+ search/read/edit chains) this floods the chat history. Consolidate either always ("`read_file x 2, edit_file x 1, run_command x 1`" chip per turn, click to expand) or above a threshold (show first 3 individually, fold the rest into "+ N more"). Threshold-fold is the better default -- preserves transparency for typical turns, scales for tool-heavy ones. **Reactivate when**: a real workload demonstrates a turn-history readability problem the user hits often enough that the consolidation pays for the click-to-expand friction.

- **System-prompt capability-gate leak**. [S5.A](../M5/S5.A-capability-toggles.md) gating only filters the per-turn `tools: [...]` manifest in `ToolRegistry::build_schema_json(ws, mode)`. The system prompt still leaks every disabled tool through two sites: (1) the hardcoded "## Running shell commands" paragraph in [src/agent/system_prompt.cpp](../../src/agent/system_prompt.cpp) names `run_command_bg` / `read_process_output` / `stop_process` / `list_processes` in prose regardless of `capabilities.background_processes`; (2) the "## Available Tools" loop walks `tools.all()` and only checks `visible_in_mode(agent)`, not `available(ws)`, so every gated tool's name + description + params still goes through. Net effect: token savings from disabling a capability bucket are smaller than advertised, and the LLM keeps calling tools the dispatcher then rejects (the user sees the "X is off..." warning instead of the cleaner "model never knew about it"). Fix: thread `IWorkspaceServices&` into `SystemPromptBuilder::build`; filter the tools loop by `available(ws)`; conditionally drop the bg-shell prose section when none of its tools are available. Watch for the S4.F prompt-stability invariant -- the prompt becomes capability-dependent, so toggling buckets mid-session would invalidate the prefix cache (acceptable: the toggle is rare and already requires a settings round-trip). **Open question -- pair this with a per-tool "hide" approval policy?** `ToolApprovalPolicy` is currently `ask | auto_approve | deny`; a fourth `hide` value would let users not just block but completely strip a tool from the manifest + system prompt for cases like "I never want the agent to even consider `delete_file` in this workspace, don't waste tokens describing it." Would compose with capability buckets (capability=on but specific tool hidden) and give a finer-grained knob than the all-or-nothing bucket. Cost: one enum value + one branch in the same gating path the capability fix already touches, plus a Settings dropdown extension. Decide together because the same `build_schema_json` + `SystemPromptBuilder::build` filter is the natural home for both.

- **Remote / containerized terminal backends** for `ProcessRegistry`. Today every spawned process runs locally under a Win32 Job Object. Adding `ssh://host`, `docker://container`, or `wsl://distro` backends behind the same `BackgroundProcess` interface would unlock dev-container workflows and remote-host execution without touching `AgentCore` or the tool layer (the `run_command*` tools just hand the registry a target URI). Hermes Agent ships seven such backends (local / Docker / SSH / Singularity / Modal / Daytona / Vercel Sandbox); we don't need that many, but local + ssh + docker covers the common dev-container case. Reactivate when a real workspace lives on a remote host or inside a container the user doesn't want polluted.
- **Offline / reproducible build documentation.** Originally an S5.Z entry; demoted because there is no concrete user demand and the existing first-build-only network dependency (CMake caches `FetchContent` grammars + PDFium after the first successful configure) is not actually blocking anyone. Scope when reactivated: expand README / [CONTRIBUTING.md](../CONTRIBUTING.md) with (a) the exact list of network endpoints the first configure hits, (b) the `FETCHCONTENT_BASE_DIR` environment variable users can set to share the cache across machines / build trees, (c) how to vendor the grammars manually for true offline use (`FetchContent_Declare` with a local `GIT_REPOSITORY` pointing at a clone). Vendoring the grammars into the repo itself is overkill until someone actually needs an air-gapped build. **Reactivate when**: an air-gapped or CI-cache-cold-start workflow surfaces, or a contributor reports the first-configure download as a blocker.
