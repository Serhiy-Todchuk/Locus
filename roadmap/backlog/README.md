# Backlog

Unscheduled tasks. Items here are not part of any milestone -- they are recognized as worth doing eventually but have not been scoped, sequenced, or assigned to a milestone yet.

When an item is picked up:
- promote it into the right milestone's stage list (`roadmap/Mx/README.md`),
- give it a stage letter, and
- create the per-stage doc under `roadmap/Mx/Sx.<letter>-<slug>.md`.

Until then, items live here as one-line entries (or short paragraphs when the rationale is non-trivial). Items that already have a written-out spec live as their own files in this folder and are linked from the **Drafts** section below.

## Drafts (full specs, parked)

- [S4.C -- Auto-Verify Feedback Loop](S4.C-auto-verify.md). Originally an M4 stage; demoted because the "agent self-corrects from build/test error tail" premise breaks down on small local LLMs -- 7B-26B models thrash on C++ template / linker errors, the most common case in this very repo, burning tens of seconds of local-inference wall-clock on retries. `run_command` already exists for explicit invocation; the user-as-architect / transparency thesis means the user is in the loop every turn anyway and can call [S4.B](../M4/S4.B-checkpoint-undo.md) `/undo` or prompt a fix when they see broken code. Reactivate when local models reliably interpret build/test failure output, OR when a CPU-side weak model (parked [S4.Q](S4.Q-multi-model.md) rewrite) can cheaply pre-process error tails into actionable summaries.
- [S4.E -- LSP Integration (via MCP)](S4.E-lsp.md). Originally an M4 stage; demoted because [S4.G -- MCP](../M4/S4.G-mcp.md) covers the most common LSP need that was real (external tools, compiler-side diagnostics via MCP-wrapped servers) and a custom in-process LSP client is unjustified maintenance. (The original demotion rationale also leaned on [S4.C -- Auto-Verify](S4.C-auto-verify.md) for diagnostics; that stage has since been parked too, so the residual case for LSP-via-MCP rests on MCP alone.) Revisit if a workflow demonstrates that MCP-wrapped LSP is meaningfully worse than native (latency-bound interactive use, large-batch reference lookups). The spec already captures the rationale and the reduced-scope "curate `mcp.json` snippets + a Settings preset picker" plan if a future maintainer wants a smaller landing path.
- [S4.U -- Subagents](S4.U-subagents.md). Originally an M4 stage; parked because subagents are a cloud-agent pattern that doesn't survive a single-GPU local-LLM runtime. Permanent ~250-token manifest tax, contradicts the user-as-architect / transparency thesis, no real parallelism on one GPU, and "summary not transcript" is just compaction with extra steps. Reactivate only when Locus gains real parallel inference (multi-GPU, batched vLLM, cloud fallback) or a measured workload proves an inline `search_semantic` + `read_file` loop is materially worse than a child agent.
- [S4.Q -- Multi-Model Split (Weak + Strong)](S4.Q-multi-model.md). Originally an M4 stage; parked because the Aider cost-tier rationale doesn't translate to consumer GPUs (loading two model files in finite VRAM crashes; swapping eats wall-clock). The skeleton already shipped with [S3.B](../M3/S3.B-llm-client-split.md). The version that *would* land cleanly is "CPU-side weak model for non-latency-critical jobs" -- a different stage with a different rationale, captured in the parked spec. Reactivate by rewriting around CPU offload, not Aider's cost-tier story.
- [S4.H -- Parallel Tool Calls](S4.H-parallel-tools.md). Originally an M4 stage; parked because local 7B-26B models rarely emit batched `tool_calls` (the precondition rarely fires) and per-tool latency is dwarfed by LLM round-trip time anyway -- best-case saving is sub-second per turn. The auto-approve-only carve-out kills the user-perceived benefit. Reactivate when local models reliably emit parallel calls AND tool latency stops being a rounding error.

## Items

- **Agent-side history-archive recovery tool** (conditional on user demand). [S5.F](../M5/S5.F-compaction-v2.md) writes a chained history archive before each compaction and tells the *user* about it via a footnote in the new context. The LLM does not currently get a tool to read these archives -- a deliberate v1 cut to avoid the agent burning tokens re-reading content that was compacted on purpose. **Reactivate when**: a real workload demonstrates the agent saying "I summarized X earlier; I need the original," and the user friction of manual archive inspection outweighs the manifest cost of a `read_history_archive(path)` tool. Implementation would be small (one new tool, scoped to `.locus/sessions/<id>/history.before-*.json` paths) once the gate is met.
- **Stripped tool-result fetch by call_id**. Narrower companion to the history-archive recovery item above. [S5.F](../M5/S5.F-compaction-v2.md) Layer 2 strips large tool result bodies (replaces them with a header line). Today the body is gone; the agent cannot recover it. With this item: stripped bodies are written to `.locus/sessions/<id>/tool_results/<call_id>.txt` and a new tool `fetch_tool_result(call_id)` returns them on demand. Smaller scope than full history archive (no semantic search, no whole-conversation file), addresses the realistic 80% failure mode ("I need the body of that earlier `read_file`"). One new tool, ~150 prompt tokens of schema. **Reactivate when**: telemetry or anecdote shows the agent re-reading files via `read_file` to recover content that was just stripped, when the original was sitting in a stripped result the agent could have fetched cheaper.
- **Memory Bank auto-mining from sessions**. End-of-session sweep that runs an LLM call to extract durable factual claims from the conversation and proposes them as memory-bank entries (user approves which to commit). Inspired by MemPalace's auto-sweep. **Costs**: one extra LLM call per session (multi-second on local 26B); summary fidelity dependent on the model; privacy concerns ("the agent is summarising every conversation into a permanent DB"). **Reactivate when**: telemetry shows users repeatedly typing the same facts (i.e. the manual `/memorize` + right-click "Save as memory" paths from [S4.R](../M4/S4.R-memory-bank.md) aren't getting used and the agent is re-learning the same facts across sessions). Implementation would add an opt-in `memory.auto_mine` flag (default off) plus an end-of-session hook in `AgentLoop`; the proposed entries surface in a confirmation panel rather than auto-committing.
- **Memory Bank validity windows (v2)**. Temporal supersession for memory entries: "Fact F was true between dates X and Y," with invalidation + timeline queries. MemPalace's SQLite knowledge graph does this. **Reactivate when**: real workloads show the agent acting on stale memory entries because a fact changed (e.g. build command updated, naming convention switched). Until then, the simpler "edit the entry to update it" path that [S4.R](../M4/S4.R-memory-bank.md) v1 ships covers most cases.
- **RAG over compaction archive** (semantic search over prior conversation history). Each archived message gets chunked, embedded with the existing `bge-small`/`bge-m3` embedder, and stored in a session-scoped vector table parallel to the workspace's `chunk_vectors`. New tool `search_session_history(query, max_results)` lets the agent query semantically over compacted-out content when summary fidelity wasn't enough. Reuses the entire embedder + sqlite-vec infrastructure; storage cost is small. **Risks**: (1) vector search over short conversation snippets is noisier than over file content -- snippets often lack standalone semantic meaning; (2) ~150-200 tokens of permanent manifest schema; (3) agent over-fetch ("re-read content we explicitly compacted"). **Reactivate when**: (a) [S5.F](../M5/S5.F-compaction-v2.md) summaries demonstrably lose context the agent later needs, AND (b) the simpler `fetch_tool_result(call_id)` item above doesn't cover the failure (i.e., the missing content is conversational, not tool-result bodies). The two items are a graduated response -- start with the cheaper one, escalate to RAG only if the simpler tool isn't enough. Companion to [S4.R Memory Bank](../M4/S4.R-memory-bank.md) (which preserves *curated* facts always-in-context, not retrievably-on-demand).
- Tool-result caching (dedup identical tool calls in a session). Originally an S4.V bullet; dropped because it solves a problem that doesn't exist on local hardware -- a repeat `search_symbols("foo")` is sub-100 ms and agents rarely make exact-duplicate calls in practice. Engineering cost (cache key including watcher state, eviction) is real for invisible savings. Reactivate if telemetry surfaces a real workload where tool-time dominates.

Alternatives (vendor a multi-MB generated parser, use m-novikov's C++-scanner fork, switch to one of the dialect-specific grammars) all have issues; the right landing path is to introduce the Rust-based `tree-sitter` CLI as a build-time tool and let it regenerate the parser at configure time. - **System-prompt capability-gate leak**. [S5.A](../M5/S5.A-capability-toggles.md) gating only filters the per-turn `tools: [...]` manifest in `ToolRegistry::build_schema_json(ws, mode)`. The system prompt still leaks every disabled tool through two sites: (1) the hardcoded "## Running shell commands" paragraph in [src/agent/system_prompt.cpp](../../src/agent/system_prompt.cpp) names `run_command_bg` / `read_process_output` / `stop_process` / `list_processes` in prose regardless of `capabilities.background_processes`; (2) the "## Available Tools" loop walks `tools.all()` and only checks `visible_in_mode(agent)`, not `available(ws)`, so every gated tool's name + description + params still goes through. Net effect: token savings from disabling a capability bucket are smaller than advertised, and the LLM keeps calling tools the dispatcher then rejects (the user sees the "X is off..." warning instead of the cleaner "model never knew about it"). Fix: thread `IWorkspaceServices&` into `SystemPromptBuilder::build`; filter the tools loop by `available(ws)`; conditionally drop the bg-shell prose section when none of its tools are available. Watch for the S4.F prompt-stability invariant -- the prompt becomes capability-dependent, so toggling buckets mid-session would invalidate the prefix cache (acceptable: the toggle is rare and already requires a settings round-trip). **Open question -- pair this with a per-tool "hide" approval policy?** `ToolApprovalPolicy` is currently `ask | auto_approve | deny`; a fourth `hide` value would let users not just block but completely strip a tool from the manifest + system prompt for cases like "I never want the agent to even consider `delete_file` in this workspace, don't waste tokens describing it." Would compose with capability buckets (capability=on but specific tool hidden) and give a finer-grained knob than the all-or-nothing bucket. Cost: one enum value + one branch in the same gating path the capability fix already touches, plus a Settings dropdown extension. Decide together because the same `build_schema_json` + `SystemPromptBuilder::build` filter is the natural home for both.

- **Remote / containerized terminal backends** for `ProcessRegistry`. Today every spawned process runs locally under a Win32 Job Object. Adding `ssh://host`, `docker://container`, or `wsl://distro` backends behind the same `BackgroundProcess` interface would unlock dev-container workflows and remote-host execution without touching `AgentCore` or the tool layer (the `run_command*` tools just hand the registry a target URI). Hermes Agent ships seven such backends (local / Docker / SSH / Singularity / Modal / Daytona / Vercel Sandbox); we don't need that many, but local + ssh + docker covers the common dev-container case. Reactivate when a real workspace lives on a remote host or inside a container the user doesn't want polluted.
