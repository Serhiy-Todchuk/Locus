# S5.E -- Per-Message Pin / Tag System

**Status: PARKED (demoted from M5 on 2026-05-16).** Original spec preserved below for reactivation.

### Why parked

The pin/tag system was scoped before [S5.F](../M5/S5.F-compaction-v2.md) was designed in detail. Once S5.F's six-layer cascade was on paper, the value of pin states collapsed:

- Five of six S5.F compaction layers do not need pin state. Layers 1 (redundant tool results), 2 (strip large bodies), 3 (drop old reasoning), 5 (drop oldest turn pairs mechanically), and 6 (LLM summary) are all algorithmic. Only Layer 4 ("drop low-priority tagged") depends on S5.E -- and Layer 4 is the weakest layer (skipped entirely if 1+2+3+6 free enough room).
- The "preserve critical messages" use case has cheaper covers: the system prompt is already byte-stable (S4.F), recent N turns can be auto-pinned by heuristic, attached-context messages can be auto-flagged where they're created, and [S5.F](../M5/S5.F-compaction-v2.md)'s `CompactionView` already supports inline pin/unpin in the preview *before* running the cascade -- contextual, at the moment of decision, no persistent state needed.
- A persistent 3-state field on every `ChatMessage` commits the saved-session JSON schema to a policy that hasn't been validated against real workloads. Better to land S5.F with implicit heuristics and add S5.E if and only if real coding sessions on local LLMs surface the need.
- The spec itself defers the only thing that would make pins effective -- "Pin states are user-driven only in v1. No auto-tagging based on content type." Manual click-to-pin during agent turns is exactly the wrong time to ask the user for context curation.

### Reactivate when

Real usage of [S5.F](../M5/S5.F-compaction-v2.md) shows users repeatedly wanting to mark specific messages as "preserve always" or "drop first," AND the implicit heuristics (system always preserved, recent N turns preserved, attached context preserved) plus the CompactionView's inline pin-before-run aren't enough.

---

## Problem

[S5.D](../M5/S5.D-reserve-and-token-xray.md) makes the budget visible. This stage gives the user control over what survives compaction. Today, every message in history is treated equally -- mechanical compaction drops oldest, LLM summary picks what to summarize, the user has no way to say "this is critical, don't drop it" or "this is junk, drop it first."

A simple 3-state-per-message marker fixes that:

- **Pinned** -- never dropped, never summarized. Always present in the prompt. Cost: counts against the budget forever.
- **Default** -- normal compaction rules apply.
- **Low-priority** -- compacted first, before any default-tagged content.

Three states give the user useful expressivity without the dialog fatigue of a five-tier importance scale.

## Tasks

- [ ] `ChatMessage` gains a `pin_state` enum: `Default | Pinned | LowPriority`. Persisted in saved-session JSON. Default-state messages omit the field for compactness.
- [ ] `ConversationHistory` exposes `set_pin_state(message_id, state)` and `pin_state(message_id)`. Thread-safe via the existing `ConversationOwnerScope` discipline.
- [ ] Chat panel renders a small pin-state icon next to each message:
  - Pinned: solid pin (`PIN_FILLED`)
  - Default: empty / hollow pin (`PIN_OUTLINE`) -- always visible so the affordance is discoverable
  - Low-priority: down-arrow / decrement icon (`ARROW_DOWN`)
- [ ] Click cycles `Default -> Pinned -> LowPriority -> Default`. Tooltip shows current state + "click to mark as <next>". Single click, no popover, no dialog.
- [ ] Right-click on a message opens a context menu including the three states explicitly (in case the cycle order is non-obvious for new users) plus "Show token cost" (jumps to the X-ray chip).
- [ ] System prompt bubble: pin-state UI is suppressed -- system prompt is implicitly pinned and cannot be downgraded from the chat. The pin button just shows "always pinned" on hover.
- [ ] Persistence: pin states round-trip in saved sessions (JSON field per message). Loading an older session without the field treats every message as Default.
- [ ] Visibility integration with [S5.D](../M5/S5.D-reserve-and-token-xray.md)'s stacked-bar context meter: pinned content gets its own stack segment (typically darker / bolder color) so users see at a glance how much of their budget pinned content is consuming.
- [ ] Compaction strategies in [S5.F](../M5/S5.F-compaction-v2.md) read pin states:
  - Pinned messages skipped by every mechanical strategy and excluded from LLM-summary input
  - Low-priority messages compacted in their own pass before defaults
  - Default messages compacted with normal recency / token-budget heuristics
- [ ] Catch2 tests:
  - cycle order correctness on click
  - persistence round-trip in JSON
  - pin states honored by mechanical drop strategies (mock compaction pipeline)
  - system-prompt bubble cannot be downgraded

## Implementation notes

- The chat-merge work in [S5.G](../M5/S5.G-chat-activity-restructure.md) puts every LLM-visible message in the chat as its own bubble. Pin UI thus naturally lives in chat, not in a separate inspector panel. This stage assumes the per-message bubble is the affordance surface; if S5.G hasn't landed yet, the pin UI lives on whatever surface the message currently has (tool bubbles, assistant bubbles).
- "Low-priority" is a softer signal than "drop me." It means *prefer to compact this first*, not "delete me now." Users who actually want to delete a message use the hover-reveal delete control that lives in [S5.G](../M5/S5.G-chat-activity-restructure.md) (the load-bearing slice extracted from the parked [S5.H](S5.H-edit-delete-branch.md)).
- The 3-state cycle was chosen over a popover after weighing discoverability vs. accidental misclicks. Mitigations: (1) the state icon is unambiguous, (2) the right-click context menu offers explicit selection, (3) the state change is reversible by clicking again.
- Pin states are user-driven only in v1. No auto-tagging based on content type. Auto-policies (e.g., "every tool result over N tokens auto-tagged low-priority") are deliberately deferred; they would couple the pin system to compaction policy and we want to see the manual UX in the wild first.
- Pin status does **not** affect the system prompt or the LLM's view -- the LLM sees normal `ChatMessage` content. Pin is a Locus-internal compaction policy, never a hint to the model.

## Manifest cost

**Zero new tools.** UI + persistence-only feature.
