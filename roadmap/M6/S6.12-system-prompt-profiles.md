# S6.12 -- System-Prompt Profiles (Full / Compact / Minimal)

## Problem

The system prompt's prose sections (Rules / Editing and creating files / Running shell commands / Windows C++ pitfalls) consume ~700-1000 t per turn even with [S6.11](S6.11-lazy-tool-manifest.md)'s lazy tool manifest in place. The content was written for capable models that benefit from worked guidance; on a fine-tuned coding agent like Qwen3-Coder it's largely redundant -- the model already knows to use tools, paginate reads, prefer edit over write, etc.

See [ADR-0007](../../architecture/decisions/0007-context-budget-reshape-lazy-manifest-and-profiles.md) for the cross-cutting rationale.

## Lands in M6 because

- Companion to [S6.11](S6.11-lazy-tool-manifest.md); same ADR.
- Surface is one file ([`system_prompt_assembly.cpp`](../../src/agent/system_prompt_assembly.cpp)) plus the Settings panel and config plumbing -- modest blast radius.
- Should land after S6.11 so the `## Available Tools` section's render path is already abstracted behind the lazy_manifest flag and the profile cuts don't accidentally collide with the tools-section work.

## Design

New enum `SystemPromptProfile { Full, Compact, Minimal }` with `Full` as the default. Each profile picks a different rendered body for the prose sections; everything else in `SystemPromptAssembly` (workspace metadata, LOCUS.md, memory bank, tools section, format addendum) is identical across profiles.

### Content cuts per profile

| Section | Full | Compact | Minimal |
|---|---|---|---|
| Lead-in ("You are Locus...") | Today | Today | Today |
| `## Rules` (6 bullets) | All 6 bullets | Bullets 1, 2, 3, 6 (drop "show plan", "no destructive without confirmation" -- approval gate enforces both anyway) | Single line: "Use tools. Paginate. Be concise." |
| `## Editing and creating files` (5 bullets) | All 5 | Bullets 1, 3 (edit-vs-write preference; edit_file precondition). Drop the long write_file explanation + old_string uniqueness bullet (the lazy_manifest's describe_tool covers the actual schema). | Single line: "edit_file requires prior read_file in this turn." |
| `## Running shell commands` (2 bullets) | Both | Both, but compressed to one line each | Omitted entirely -- `describe_tool('run_command')` carries everything the model needs. |
| `## Windows C++ / MSVC pitfalls` (2 bullets) | Both | Both (retained -- they pay for themselves) | Omitted. |

Estimated rendered token counts:

| Profile | Rough size |
|---|---:|
| Full (today) | ~700-1000 t |
| Compact | ~300 t |
| Minimal | ~80 t |

Numbers are estimates; the actual test of record is the `system prompt ~N tokens` log line and the new Catch2 test that pins each profile's size to a band.

### S4.F invariant

Switching profile mid-session invalidates the LM Studio prefix cache for the duration of the session change. Within a single profile choice the system prompt remains byte-stable across turns. The Settings UX should group Profile + Lazy manifest under a single "Prompt Cost" section so users understand they're a tuning pair, not independent knobs in different tabs.

### Task C -- Compact body

The compact body is the "small but everything load-bearing still present" variant. Drafted here so the implementation has a target:

```text
You are Locus, a workspace-bound assistant. Use tools to read and modify files in this workspace.

## Rules
- Use tools to read files and search the index. Never guess file contents.
- Paginate file reads. Never request a whole large file.
- Summarize tool results before responding.
- Be concise.

## Editing files
- Prefer edit_file over write_file for existing files. edit_file requires you to have called read_file (or write_file) on the path earlier in this turn.
- Use write_file only for new files. Set overwrite=true to replace; default refuses.

## Running shell commands
- run_command is synchronous; run_command_bg returns a process id for long-running processes you must drain with read_process_output.

## Windows C++ / MSVC pitfalls
- #define NOMINMAX before #include <windows.h> (the header otherwise defines min/max as macros).
- Do not assign braced initializer lists to std::array members after construction.
```

Reviewers should add/remove lines based on real-model behaviour during S6.12 verification.

### Task D -- Minimal body

```text
You are Locus, a workspace-bound assistant.
- Use tools, do not guess.
- Paginate reads.
- edit_file requires a prior read_file on the same path this turn.
- Be concise.
```

Five lines, fits in ~80 tokens. Intentionally Spartan: users who pick this opt into a power-user setup.

## Notes

- The Compact body's "Running shell commands" line says `run_command_bg ... drain with read_process_output` -- but in a workspace where `capabilities.background_processes` is off, that line is misleading. Either gate per-line by capability (preferred, follows the same path as the backlog item "System-prompt capability-gate leak"), OR leave the line in and trust the model to consult the live manifest. The capability-leak fix is broader than this stage; suggest landing it under [S6.10](S6.10-small-model-robustness.md) or its own item rather than coupling here.
- The Minimal profile drops the "no destructive without confirmation" rule from the prose. Approval gate still enforces it; the prose was redundant.
