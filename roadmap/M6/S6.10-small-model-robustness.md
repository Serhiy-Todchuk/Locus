# S6.10 -- Small-Model Robustness Pass

## Problem

Locus already covers the structural gaps in coding-agent design: multi-decoder stream parsing (S4.N), checkpoint/undo (S4.B), plan mode (S4.D), MCP (S4.G), memory bank (S4.R), per-turn file-change awareness (S4.T), per-model presets (S4.V Task 6). What it does NOT yet handle is eight concrete failure modes that surface when the model is a local 7-26B (Gemma E4B, Qwen3-Coder-7B, Qwen3.6-35B-A3B, DeepSeek-R1 distills) instead of a frontier API:

1. **Malformed JSON in tool-call bodies.** Small models emit `{"path":"foo.cpp",}` (trailing comma), `{path: "foo.cpp"}` (unquoted key), or close-brace-counts-off output. Current decoder path: try `nlohmann::json::parse`, on failure log a warn, drop the call. The model then says "let me try again" and burns a round on the same content. SmallCode and little-coder both pay this cost up-front with a repair pre-pass and get the call through.

2. **Stalled / looping / hallucinated tool calls.** The same tool with the same args fires three turns in a row. The model emits a tool name that doesn't exist. The assistant response is empty (no text, no tool call). Locus today loops or terminates the turn and surfaces nothing to the model that would help it recover. little-coder's `quality-monitor` catches these four cases at `tool_result` and injects a specific corrective message into the next turn -- no re-prompt from scratch.

3. **Thinking blocks pile up across history.** Reasoning models (Qwen3-Thinking, DeepSeek-R1-Distill, GLM-Z1) emit 5-20k thinking tokens per turn. Locus already routes them to `on_reasoning` (separate channel from `on_text`) but `ConversationHistory` stores them and ships them back to the LLM next turn as part of the assistant message. After 3-4 thinking-heavy turns the prefix cache is shot and the next request blows the context window. Past thinking is a scratchpad for the decision that was already made -- it adds no information once the tool result is back in the conversation.

4. **Server-side grammar-constrained decoding is unused.** llama.cpp accepts a GBNF grammar; LM Studio and vLLM accept `response_format: { type: "json_schema", ... }`. With the grammar set, the token sampler can only emit tokens that keep the output parseable -- malformed JSON tool calls become physically impossible to generate. Locus currently relies entirely on post-hoc parsing + (with Task A) post-hoc repair. The grammar path is strictly better where the server supports it.

5. **Tool descriptions lack examples.** Small models follow tool schemas literally. Locus's `description()` strings today are prose explanations of the parameters with no worked examples. Frontier models gloss over the difference; local 7-26B does not. Concrete loss: `search` with five modes (text / regex / symbols / semantic / hybrid) gets the wrong mode picked routinely; `edit_file` with the batch-of-edits shape (`edits: [{oldText, newText}, ...]`) confuses models that haven't seen that exact shape in training; `run_command_bg` -> `read_process_output` handoff is opaque.

6. **Wrong preset silently applied across model swaps.** Users running multiple models against the same LM Studio instance (swap Qwen-Coder for Gemma between sessions) must remember to re-pick the `tool_format` and chat-template preset in Settings. Today the preset doesn't auto-sync to the actually-loaded model. Wrong combination drops tool-call success by 30-50%.

7. **Code writes can be silently truncated.** Small models stop mid-file at line 140 of 200, emit a `// ... rest of the function` comment or a literal `...` and end the message. `WriteFileTool` writes the truncated content; the user has to notice the file is broken. No detector exists for trailing-ellipsis / unbalanced-brace / "TODO: implement" patterns.

8. **Sampler knobs default to server-side defaults that are often wrong.** `ModelPreset` (S4.V Task 6) carries `temperature` / `max_tokens` / `tool_format` but not `top_k` / `top_p` / `repetition_penalty` / `min_p`. The Settings UI exposes these (S4.V Task 5) at the workspace level but per-model defaults don't capture them. Real cases: Qwen3-Coder wants `temp=0.1, top_p=0.9`; Gemma 4 wants `temp=0.7, top_k=64`; DeepSeek-R1-Distill wants `temp=0.6, top_p=0.95`.

All eight target the same target user: someone running a 7-26B model on consumer hardware. None of them touch the byte-stable system prompt invariant (S4.F).

## Lands in M6 because

- M5 finished the polish pass; the engine is now stable enough that turn-level recovery behaviour is the next observable quality lever.
- All three changes hook into surfaces that exist today: stream decoder JSON parse points, `ToolDispatcher` between-round seam, `LLMContext` payload assembly. No new subsystem.
- Independent of S6.0-S6.9 connectivity work; can land before or after.

## Tasks

### Task A -- JSON repair pre-pass for tool-call bodies

- [ ] New helper `src/llm/json_repair.h/cpp`: pure function `std::optional<std::string> repair_for_parse(std::string_view input)`. Stages:
  - Strip trailing commas before `}` and `]`.
  - Wrap unquoted identifier keys: `{path: "foo"}` -> `{"path": "foo"}`. Regex-bounded, only at object-key positions.
  - Convert single-quoted strings to double-quoted, but only if no double quotes present in the input (avoids breaking embedded quoted text).
  - Patch missing closing braces / brackets by counting opens vs closes and appending the difference.
  - Fix literal-vs-escaped newlines in string values (heuristic: `"foo\nbar"` raw newline -> `"foo\\nbar"`).
  - Fallback: regex-extract the first balanced `{...}` block from a longer string.
- [ ] `XmlToolCallExtractor` (`src/llm/xml_tool_call_extractor.cpp`): when the body fails to parse, run `repair_for_parse`, retry parse. Log at trace which stage fired. Both Qwen and Claude XML decoders inherit the fix for free.
- [ ] `ToolDispatcher` (`src/agent/tool_dispatcher.cpp`): same pre-pass before parsing the canonical OpenAI-native `arguments` JSON string. This covers the case where the model emitted a malformed `arguments` field that the stream decoder accumulated faithfully but it doesn't parse.
- [ ] Field-name aliases: accept `parameters`, `args`, `arguments`, `input` interchangeably at the tool-arg-binding layer (separate from JSON repair -- this is shape repair, not syntax repair). Some models emit `"parameters"`; Locus's tool schemas expect named fields directly inside the args object, not under a `parameters` wrapper.
- [ ] Catch2 tests in `tests/test_json_repair.cpp` `[s6.10]`: one case per repair stage with a representative broken input + expected fixed output. One case verifying clean input round-trips unchanged. One end-to-end through `XmlToolCallExtractor` confirming a Qwen-XML body with two errors (trailing comma + unquoted key) gets dispatched.
- [ ] Integration test in `tests/integration/test_int_tool_calls.cpp` `[integration][s6.10]`: feed a synthetic broken-JSON tool call through the dispatcher, assert success after repair.

### Task B -- Quality monitor (turn-level auto-correction)

- [ ] New class `src/agent/quality_monitor.h/cpp`. Stateless helpers that inspect the latest assistant message + tool-call history; returns an optional `QualityCorrection { Kind, std::string corrective_message }` for the next turn's prepend.
- [ ] Four detectors:
  - **`empty_response`**: assistant message has no `content`, no `tool_calls`, no surfaced reasoning that became text. Corrective: "Your previous response was empty. Respond with either text or a tool call."
  - **`unknown_tool:<name>`**: tool name not in the manifest visible to the model this round (after MCP namespacing + capability gating). Corrective: lists the valid tools that WERE in scope, prefixed with "Tool `<name>` does not exist. Available tools this round: ...".
  - **`repeated_tool_call`**: same tool name + canonical-JSON-stringified args as the previous tool call (or two consecutive). Corrective: "You just called `<name>` with the same arguments and it didn't produce new information. Try a different approach -- different arguments, a different tool, or a question."
  - **`malformed_args:<name>`**: tool dispatcher reports JSON parse failure even after Task A repair. Corrective: "Tool `<name>` arguments were not parseable as JSON. Provide them as a proper JSON object."
- [ ] Hook in `AgentCore` / `AgentLoop`: after `ToolDispatcher` returns, before the next round's user message is composed, run the monitor. If a correction fires, prepend it to the next round's user message (NOT the system prompt -- preserves S4.F).
- [ ] Cap: at most 2 quality corrections per user-message turn. If 2 fired and a third would fire, surface "agent appears stuck" to the user and pause for input instead of correcting again.
- [ ] Settings flag: `WorkspaceConfig::quality_monitor_enabled` (default `true`). Settings tab gets a checkbox under the LLM tab next to the existing tool-format dropdown.
- [ ] Activity log: each fired correction emits a `quality_correction { kind, summary }` event so the user sees what happened (and why the model "magically improved" mid-turn).
- [ ] Chat-footer chip: `auto-corrections: N` per session, click to filter activity log to the `quality_correction` events. Hidden when N == 0.
- [ ] Catch2 tests `tests/test_quality_monitor.cpp` `[s6.10]`: one per detector kind, plus a cap test (3rd correction blocked), plus a feature-flag test (disabled -> no detection).

### Task C -- Strip past-turn thinking from LLM payload

- [ ] `ChatMessage` already exposes `reasoning_content` separately from `content` (S4.N stream-decoder split). Verify the storage layer (`ConversationHistory::to_json` / `from_json` in `src/agent/conversation.cpp`) round-trips it as its own field. Add a Catch2 round-trip test if not already covered.
- [ ] `LLMContext::build_request_messages()` (or equivalent in `src/agent/llm_context.cpp` / `agent_loop.cpp`): when assembling the `messages[]` array to send to the LLM, strip `reasoning_content` from every assistant message EXCEPT the most recent if and only if the next call is a same-turn tool-result follow-up (the model may still be mid-decision-chain).
- [ ] **Edge case: assistant message has ONLY reasoning** (no `content`, no `tool_calls`). Rare but happens on stalled streams or thinking-budget aborts. Preserve a short version instead of stripping completely: first 3 sentences + ` [...] ` + last 3 sentences of the reasoning. Heuristic to avoid sending the model into the next turn with literally zero context about what it was just doing.
- [ ] Display layer unchanged: chat panel keeps the collapsible reasoning bubble, activity log keeps the full reasoning event, `.locus/locus.log` keeps it under `trace`. The strip is payload-only.
- [ ] Saved session JSON keeps full reasoning. Re-opening a saved session and continuing -- the strip happens fresh at LLM-payload-build time, not at session-load time. The on-disk archive is the source of truth.
- [ ] Token-cost log: per round, log `prompt_tokens_before_strip` vs `prompt_tokens_after_strip` at trace level so the savings are measurable. Expect 5-30% reduction on thinking-heavy models, near-zero on non-thinking models (which is fine -- no-op cost is negligible).
- [ ] Catch2 tests `tests/test_thinking_strip.cpp` `[s6.10]`:
  - Strip behaviour: 3-turn history with reasoning on every assistant message -> built payload has reasoning only on the most recent.
  - Edge case: assistant message with ONLY reasoning -> payload contains the head+tail-3-sentence form, not empty.
  - Save/load round-trip: full reasoning preserved on disk; strip only at payload build.
  - Feature flag: `WorkspaceConfig::strip_past_thinking` (default `true`); disabling restores current behaviour for users who want the old shape.

### Task D -- Server-side grammar-constrained decoding for tool calls

- [ ] Extend `ModelPreset` (S4.V Task 6) with `grammar_mode: { off | strict | best_effort }`. Default `best_effort` for known-good families (Qwen / Llama 3.x), `off` for legacy presets.
- [ ] At session start (`LMStudioClient::query_model_info` or first request), probe server capability:
  - llama.cpp: `GET /props` -- check for `chat_template` + `grammar` support.
  - LM Studio: try a tiny request with `response_format: { type: "json_schema", json_schema: { ... } }`; success means structured output is wired.
  - vLLM: same `response_format` probe.
  Cache the result on the client for the session.
- [ ] Grammar source: derive from `IToolRegistry::build_schema_json` at request-build time. For OpenAI-native tool format the existing `tools: [...]` array stays; ADD `tool_choice` constraint and (for LM Studio / vLLM) a `response_format` enforcing the union-of-tool-calls schema. For Qwen-XML / Claude-XML formats, derive a GBNF grammar that constrains the body to the union schema -- llama.cpp's grammar field accepts arbitrary GBNF, not just JSON-schema-derived. Helper lives at `src/llm/tool_call_grammar.h/cpp`.
- [ ] Behaviour matrix:
  - `grammar_mode: strict`: refuse to send the request if server probe failed; surface clear error.
  - `grammar_mode: best_effort`: attach grammar if server supports it, otherwise send unconstrained + log at trace. Locus's existing decoder + Task A repair still run.
  - `grammar_mode: off`: never attach grammar. Useful for debugging or for models the grammar hurts (older non-OpenAI-tools-trained models can degrade under grammar constraint).
- [ ] Settings UI: dropdown on the LLM tab next to `tool_format`, three values, with help text "Use server-side grammar to force valid tool-call JSON. Requires llama.cpp / LM Studio / vLLM."
- [ ] Activity log: each request logs `grammar_attached: yes|no` at trace.
- [ ] Catch2 tests `tests/test_tool_call_grammar.cpp` `[s6.10]`:
  - GBNF generation from a 3-tool schema is well-formed.
  - JSON-schema generation matches OpenAI spec shape.
  - Probe success / failure paths.
- [ ] Integration test `[integration][llm][s6.10]`: run a known-bad model (one that emits malformed JSON in unconstrained mode) and verify it produces a valid tool call with `grammar_mode: strict`.

### Task E -- Few-shot examples in built-in tool descriptions

- [ ] Audit every built-in tool's `description()` string in `src/tools/`. Today most are prose-only.
- [ ] Required pattern: each description ends with one `Example:` block showing a canonical call. Format: deliberate, minimal, ASCII-safe (we already enforce no-em-dash via `scripts/check_no_em_dashes.py`).
- [ ] Tools with the highest leverage:
  - `search` (`src/tools/search_tools.cpp`): one example per mode (text / regex / symbols / semantic / hybrid). 5 examples.
  - `edit_file` (`src/tools/file_tools.cpp`): two examples -- single edit, batch of two edits. Highlights `edits` array shape and the "all match against original simultaneously" semantics.
  - `write_file`: one example with `overwrite: false` (default) and a note that it refuses on existing files.
  - `run_command_bg` + `read_process_output` + `stop_process`: one chained example showing the id handoff between the three tools.
  - `propose_plan` (plan mode): one example with 3 steps + `tools_needed`.
  - `add_memory` + `search_memory`: one canonical pair.
  - `read_file`, `list_directory`, `get_file_outline`: one example each.
  - `ask_user`: one example with `options` array.
- [ ] MCP tools out of scope -- their `inputSchema` comes from the server.
- [ ] Catch2 lint test `tests/test_tool_descriptions.cpp` `[s6.10]`: walk every registered built-in tool, assert `description()` contains the literal substring `Example:` (cheap forced-discipline gate). If a future tool author forgets the example block the build fails.
- [ ] Token impact: each example adds ~30-80 tokens to the per-turn manifest. Net cost across all built-in tools roughly +500-700 tokens. Acceptable given the failure-rate reduction.

### Task F -- Auto-detect loaded model and auto-apply preset

- [ ] At `LocusSession` construction (after `ILLMClient` is created), call `ILLMClient::query_model_info()`. Both LM Studio (`GET /v1/models`) and llama.cpp (`GET /v1/models`) return enough.
- [ ] New matcher in `src/llm/model_presets.cpp`: `find_preset_for_model(string model_id) -> optional<ModelPreset>`. Per-preset regex (`qwen.?3.*coder`, `gemma.?4`, `deepseek.*r1`, etc) matched case-insensitive against the returned model id.
- [ ] Behaviour: if the user has an explicit preset selected in workspace config (non-default), respect it. If `preset = auto` (new default), apply the matched preset and persist `last_applied_preset` in `.locus/ui_state.json`. Log at info: `auto-applied preset '<name>' for model '<id>'`. No silent surprise: chat-footer chip or one-time toast notifies the user.
- [ ] Workspace config addition: `WorkspaceConfig::auto_detect_model_preset` (default `true`). Settings tab adds a checkbox.
- [ ] Re-detect on model swap: if the user changes the model in LM Studio mid-session, the next request gets the new `model_id`; if it differs from `last_applied_preset` source, re-match. Avoid re-applying every turn (cache).
- [ ] No match -> log at warn, keep current preset, no error. Conservative default.
- [ ] Catch2 tests `tests/test_model_preset_matching.cpp` `[s6.10]`: regex matcher against a list of real model ids (`qwen3-coder-7b-instruct-q4_k_m`, `gemma-4-9b-it`, etc) asserts the right preset is returned.

### Task G -- Anti-truncation detection for code writes

- [ ] New detector in `src/tools/truncation_detector.h/cpp`. Pure function `optional<TruncationReason> detect_truncation(language, content)`. Reasons enum: `unbalanced_braces`, `trailing_ellipsis`, `placeholder_comment`, `bare_pass_after_def`.
- [ ] Brace / paren / bracket balance check: reuse the workspace's `TreeSitterRegistry` (`src/index/tree_sitter_registry.h`) -- parse the content, look for `ERROR` nodes near the end of the file or unclosed scopes. Language-aware.
- [ ] Literal patterns (language-agnostic regex):
  - Trailing `...` or `// ...` or `# ...` or `/* ... */` at end of file (last 200 chars).
  - Phrases: `// rest of the code`, `// rest of the implementation`, `// TODO: implement`, `# (continued)`, `// (continued)`, `// ... existing code ...`.
- [ ] Python-specific: detector for `def foo(...): \n    pass` at end of file when the function is non-trivial (heuristic: name suggests implementation, like `parse_*` or `compute_*`).
- [ ] Hook in `WriteFileTool` and `EditFileTool`: after content is assembled but BEFORE `write_atomic` fires, run the detector. On detection, refuse the operation, return `ToolResult{success=false, content="..."}` with a specific message: "The file appears truncated: <reason>. Emit the complete content."
- [ ] Settings flag: `WorkspaceConfig::detect_write_truncation` (default `true`).
- [ ] Activity log: emits a `truncation_blocked` event with the reason; pairs with the S6.10 Task B quality-correction display.
- [ ] Catch2 tests `tests/test_truncation_detector.cpp` `[s6.10]`: positive cases (one per reason kind, one per supported language), negative cases (legitimately short files, comments containing `...` in the middle of valid code, real `pass` statements that aren't placeholders).

### Task H -- Per-model sampler defaults (top_k / top_p / repetition_penalty / min_p)

- [ ] Extend `ModelPreset` struct (`src/llm/model_presets.h`) with optional fields: `top_k`, `top_p`, `repetition_penalty`, `min_p`. `std::optional<float>` (or sentinel) so "not specified" means "use workspace default / server default".
- [ ] Propagate through `LLMConfig` -> `OpenAiTransport` -> request body. LM Studio + llama.cpp + vLLM all accept these as top-level request fields on `/v1/chat/completions`.
- [ ] Populate `builtin_presets()` in `src/llm/model_presets.cpp` with the values that are documented per model family:
  - Qwen3-Coder: `temp=0.1, top_p=0.9`
  - Qwen3 (general): `temp=0.6, top_p=0.95, top_k=40`
  - Gemma 4: `temp=0.7, top_k=64`
  - DeepSeek-R1-Distill: `temp=0.6, top_p=0.95`
  - Llama 3.x: `temp=0.6, top_p=0.9`
  - Generic / non-tool-trained: keep current defaults.
- [ ] Settings UI: the existing S4.V Task 5 sampler-controls panel already renders these; new behaviour is just that the preset can pre-fill them. Add a "Reset to preset defaults" button to the panel.
- [ ] No new tests required beyond verifying the preset values round-trip into the outbound request body. Add one Catch2 case in `tests/test_model_presets.cpp` that asserts each named preset's expected sampler tuple.

## Dependencies

- None blocking. Lands when scheduled. Independent of the connectivity stages (S6.0-S6.6) and the storage migration (S6.8).
- Internal task order: A and D are complementary (D obsoletes A's repair pipeline where servers support it; A is the universal fallback). Ship A first as the safety net, then D as the optimization. B and G compose (G's truncation-blocked events surface through B's quality-correction display path). C is independent of all others. E and F and H are independent of each other and of A-D.

## Implementation notes

- Task A is reused by all three XML decoder variants (Qwen / Claude / Auto) plus the OpenAI-native path. Keep the helper in `src/llm/` with no decoder-specific includes so all four call it.
- Task B's correction messages are templated strings, not LLM-generated. Keep them deterministic -- this is feedback, not summarization.
- Task C is the most measurable win. On a Qwen3-Thinking 14B running locally, a single thinking-heavy turn can push 8k tokens into history. Three turns and the prefix cache no longer fits. The strip is the cheapest fix in the stage by far.
- Task D's grammar source is the same JSON schema Locus already builds for the OpenAI-native `tools: [...]` array. The work is wiring it into a second header (`response_format` / `grammar`), not authoring schemas from scratch.
- Task E enforcement is a build-time lint, not runtime. If the assertion fires, the build fails -- no risk of shipping a tool without an example by accident.
- Task F's matcher is conservative: unknown model -> no preset change, never an error. The detection is best-effort signalling, not authoritative configuration.
- Task G reuses the `TreeSitterRegistry` already linked for the indexer; no new parser dependency. The placeholder-comment regex pass is language-agnostic and runs even on languages without a grammar.
- Task H values come from each model's published model card. Treat them as starting points, not gospel -- the Settings UI exposes overrides anyway.
- All eight tasks ship behind feature flags so a regression in any one can be disabled without disabling the others. Default-on for A / C / E / F / G / H (low-risk); B and D default-on but settings-toggleable.
- No new dependencies. Pure C++ + Catch2. The grammar generation in Task D produces strings handed to the existing transport layer; no new HTTP client work.

## Open questions

- Should Task A's repair stages run in a fixed order or be tried independently and the cheapest fix that parses cleanly wins? Fixed-order is simpler and matches what SmallCode's parser does; consider only changing if test cases surface a conflict between stages.
- Should Task B's `repeated_tool_call` detector look back 1 turn (strict) or 2-3 turns (catches alternating-pair loops like `read_file A -> read_file B -> read_file A -> ...`)? Start with 1, widen if telemetry shows alternating loops.
- Should Task C's "only-reasoning" edge case retain head+tail sentences, or a fixed token cap, or both? Sentence-based is more readable but token-based is more predictable. Default to sentences with a fallback token cap of e.g. 200.
- Should Task D's `best_effort` mode log a one-time warn when the server doesn't support grammar, or stay silent? Lean silent at info, trace-log per request -- otherwise users on older llama.cpp see warns every session.
- Should Task E examples be shown in the model's manifest verbatim, or templated through `LOCUS.md`-style substitution for workspace-specific paths? Verbatim. The whole point is that small models follow exact patterns; substitution that the model can't see breaks the pattern.
- Task F: how long to wait for the model-info probe before falling back to "no preset change"? 500ms hard cap; if the server is slow on `/v1/models` the auto-detect just doesn't fire that session.
- Task G: should `WriteFileTool` refusal be a hard error or a soft warning the user can override? Hard error initially; demote to soft if false-positive rate is high in practice.
- Task H: should sampler values that come from a preset be visible in the Settings UI as "(from preset)" annotations, or just silently pre-fill the fields? Pre-fill with a visible "from preset: <name>" badge under each field.

## Inspiration / reference

- [smallcode](https://github.com/Doorman11991/smallcode) `bin/governor.js` + the "forgiving tool call parser" mechanism (JSON / YAML / XML / Hermes / plain text repair) -- Task A is the JSON slice of this.
- [little-coder](https://github.com/itayinbarr/little-coder) `.pi/extensions/quality-monitor/quality.ts` -- Task B follows the same four detector kinds (empty / unknown_tool / repeated_tool_call / malformed_args) with the same "synthetic corrective message into the next turn" pattern.
- [little-coder](https://github.com/itayinbarr/little-coder) `.pi/extensions/thinking-budget/index.ts` -- Task C is the strictly-better proactive variant of this extension's reactive abort-and-retry model.
- llama.cpp GBNF docs + LM Studio "Structured Output" API + OpenAI `response_format: json_schema` spec -- Task D wires Locus into the constrained-decoding ecosystem that already exists.
- Tool-description conventions from Anthropic Claude tool-use docs and OpenAI function-calling guide (both recommend examples) -- Task E formalizes a discipline most schemas already gesture toward.
- Model-family cards on HuggingFace for Qwen3-Coder, Gemma 4, DeepSeek-R1-Distill, Llama 3.x -- Task H values come from these.
