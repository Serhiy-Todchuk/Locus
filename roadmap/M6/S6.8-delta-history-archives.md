# S6.8 -- Delta History Archives

## Problem

[S5.F](../M5/S5.F-compaction-v2.md) shipped a snapshot-based history archive: before each compaction, the full pre-compaction history is written to `<sessions_dir>/<session_id>/history.before-compact-<N>.json`. The model is correct but has two costs that compound over the lifetime of a long-running session:

1. **Storage redundancy.** After K compactions, the session directory holds K complete copies of mostly-overlapping history. Each archive contains everything that survived previous compactions PLUS whatever this compaction is about to drop. Across a session that's been compacted six times, ~80% of the on-disk bytes is the same content repeated.
2. **User-readability mismatch.** Opening `before-compact-3.json` to ask "what did compaction 3 actually drop?" shows the entire conversation up to that point, not a focused diff. The user has to read two adjacent archives and reconcile them in their head.

Delta archives close both: each archive file contains only the slice that compaction N removed, with a parent pointer to the previous archive. Storage scales with what was actually dropped; the file itself is already the "what was removed at step N" view.

## Why M6 (and not sooner)

- **No measured pain in M5 timeframe.** Sessions are kB-MB JSON; `archive_keep_count=5` caps worst case at ~5x history size; disk is cheap on a developer workstation. Snapshots work.
- **Remote-access workloads make it visible.** [S6.4](S6.4-remote-access.md) (Remote Access) and [S6.5](S6.5-web-frontend.md) (Web Frontend) raise the realistic upper bound on session lifetime -- a session kept open across devices for a week through many compactions exposes the redundancy more sharply than a short local-dev loop.
- **The Manage Sessions dialog viewer benefits the web frontend.** Focused delta files are much friendlier to render in a browser than 50-message snapshots; the dialog from [S5.I](../M5/S5.I-tabs-and-sessions.md) translates cleanly to the web shell when delta archives are the source data.

## On-disk layout

```
<sessions_dir>/
├── <session_id>.json                          ← live session (references archive head)
└── <session_id>/
    ├── history.compacted-1.json               ← what compaction 1 removed (parent: null)
    ├── history.compacted-2.json               ← what compaction 2 removed (parent: 1)
    └── history.compacted-3.json               ← what compaction 3 removed (parent: 2)
```

Filename changes from `history.before-compact-<N>.json` (snapshot semantics) to `history.compacted-<N>.json` (delta semantics) so a glance at the directory tells you which model the session is using during the migration window.

### Archive file shape

```json
{
  "counter": 3,
  "parent_counter": 2,
  "compaction_timestamp": "2026-...",
  "layers_run": ["1: redundant tool results", "2: strip large bodies", "6: llm summary"],
  "dropped_messages": [ {ChatMessage}, ... ],
  "stripped_tool_results": [ {"call_id": "...", "original_body": "..."}, ... ],
  "dropped_reasoning_blocks": [ {"message_id": N, "original_text": "..."}, ... ],
  "inserted_summary_message_id": N
}
```

Live session JSON gains an `archive_head: <counter>` field pointing to the latest archive (null if no compaction has run). Backward compatibility: missing field treated as `null`.

## Recovery

To reconstruct full pre-compaction state at any point in the chain:

1. Start with the live session.
2. Walk back via `archive_head` -> `parent_counter`.
3. At each step, splice the archive's `dropped_messages` back into their original positions, restore `stripped_tool_results` bodies into the matching tool-result messages, restore `dropped_reasoning_blocks` into the matching assistant messages, and remove the `inserted_summary_message_id` (if Layer 6 fired at that step).

Helper: `HistoryArchive::reconstruct_at(session_id, counter)` returns a `ConversationHistory` representing the state immediately before compaction `<counter>` ran.

## Migration from S5.F snapshots

On first workspace open after S6.8 lands, the workspace boot path scans `<sessions_dir>/<session_id>/` directories for legacy `history.before-compact-<N>.json` files. For each such directory:

1. Read snapshots in counter order (1, 2, 3, ...).
2. Compute the pairwise delta between adjacent snapshots (and between the last snapshot and the live session).
3. Write `history.compacted-<N>.json` files with the deltas.
4. Update the live session's `archive_head`.
5. Delete the legacy snapshots only after all deltas are successfully written (atomic per session: if any delta fails, leave the snapshots intact for that session, log a warning).

Backward-compat: a workspace opened with an older Locus version sees both legacy `history.before-compact-*.json` and new `history.compacted-*.json` cleanly because the filenames differ. The HistoryArchive class supports reading either layout during the migration window; new writes are always deltas.

## GC

S5.F's GC pruned oldest snapshots first. With deltas, dropping the middle of the chain breaks parent pointers in later archives. New GC policy:

- Always drop **from the oldest end of the chain**.
- After drop, set the next-oldest archive's `parent_counter` to `null` (it becomes the new chain root; the content it once referenced is gone).
- `archive_keep_count` semantics unchanged: cap on number of archives, configured per workspace.

Pruning the oldest archive means the content it captured is permanently unrecoverable (same as today). What changes is that subsequent archives stay walkable.

## Out of scope

- **Agent-side recovery tool.** Still parked, per the S5.F deliberate choice. The agent gets no tool to read deltas, same as it got no tool to read snapshots. The same backlog item tracking "agent reads archives if real workload demands it" applies unchanged.
- **Cross-session archive sharing.** Each session's chain is independent.
- **Reversibility beyond the chain.** Once the oldest archive is GC'd, the content it held is unrecoverable -- same semantics as today.
- **Tree-shaped archive history.** If the parked [S5.H](../backlog/S5.H-edit-delete-branch.md) edit / rewind / branch operations reactivate, they would add archive entries representing user-initiated edits. Out of scope here; the chain stays linear (compaction-only) in this stage.

## Manifest cost

**Zero new tools.** Agent-facing surface unchanged; this is on-disk format and viewer work only.
