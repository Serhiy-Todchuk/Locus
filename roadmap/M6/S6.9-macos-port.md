### Next session (start here)

1. **GUI WKWebView JS-readiness bug.** Symptom on launch: a dialog "Error running
   JavaScript: ReferenceError: Can't find variable: addSystemPromptMsg". Cause:
   `ChatPanel::run_script` queues scripts until `page_ready_`, which flips true on
   `wxEVT_WEBVIEW_LOADED` ([chat_panel.cpp:235](../../src/frontends/gui/chat_panel.cpp#L235))
   and then flushes the queue. On WebView2 that event fires AFTER the page's inline
   `<script>` runs; on WKWebView it fires BEFORE the functions (`addSystemPromptMsg`
   et al. in [chat_html_template.cpp](../../src/frontends/gui/chat_html_template.cpp))
   are defined, so the flush calls an undefined function. Fix: stop trusting
   `wxEVT_WEBVIEW_LOADED` for readiness -- have the page's inline script, after
   defining everything, signal C++ over the already-intercepted `locus://` scheme
   (navigate to `locus://ready`; `on_webview_navigating` flips `page_ready_` + flushes).
   Reliable on both backends. Optionally also probe `typeof addSystemPromptMsg ===
   'function'` before flushing. Then re-run WKWebView visual QA (system-prompt bubble,
   file tree, send a message, markdown + Prism render).
2. **Stage B -- Shell + MCP parity.** POSIX `run_command` / `run_command_bg` / MCP

## Problem

Locus is Windows-only at v1. The README at [README.md:60-67](../../README.md#L60-L67)
already documents this and lists the runtime surfaces that currently return
`[platform: <os>]` rather than working: shell execution (`run_command`,
`run_command_bg`), MCP stdio transport, autostart on login, recent-workspaces
integration, notification sounds, and the UI automation harness.

A walk of the actual code shows the situation is *less* bleak than the README
suggests. The cross-platform plumbing for the file lock, time functions, and
filesystem paths is already in place ([workspace_lock.cpp:205](../../src/core/workspace_lock.cpp#L205)
has a working POSIX `flock` branch); most files with `_WIN32` guards already
have an `#else` arm; the static-CRT workaround driving `x64-windows-static` is
MSVC-specific and harmless under clang. What blocks the build today is mostly
in `CMakeLists.txt` (PDFium URL hard-pinned to `pdfium-win-x64`, unconditional
`unofficial-webview2` requirement) plus three load-bearing runtime subsystems
(`ProcessRegistry`, `StdioTransport`, the GUI's WebView2 integration) that
genuinely have no POSIX branch yet.

Lands in M6 because (a) it's not "polish" -- M5's frame -- and (b) it slots
naturally into the catch-all "Misc" half of M6. No dependencies on other M6
stages; can land in parallel with S6.1/S6.2/S6.3 if useful.

## Tasks

Three sub-stages, each independently shippable. A Mac user can stop after A
for chat-only, after B for full agent parity minus polish, or carry through C
for a distributable .app bundle.

### Stage B -- Shell + MCP Parity (1-2 days)

Goal: `run_command`, `run_command_bg`, and MCP stdio servers work the same on
macOS as they do on Windows.

- [ ] POSIX process spawn in `ProcessRegistry` ([process_registry.cpp](../../src/tools/process_registry.cpp)).
  - Use `posix_spawn` + `pipe2(O_CLOEXEC)` for stdin / stdout / stderr.
  - Job-object equivalent for killing the whole process tree: `setpgid(0, 0)`
    in the child, then `killpg(-pgid, SIGTERM)` on cleanup. The Win32
    `KILL_ON_JOB_CLOSE` semantic doesn't have an exact macOS equivalent, but
    process groups cover the common case (`cmd /c foo && bar` -> `sh -c
    'foo && bar'`).
  - Reader thread per process draining stdout+stderr into the existing ring
    buffer. The current Win32 code uses overlapped I/O + `GetOverlappedResult`;
    on POSIX a simple `read()` loop on the pipe FD is fine.
  - stdin pipe with non-inheritable parent-side write handle: use
    `fcntl(F_SETFD, FD_CLOEXEC)` after the spawn, mutex-guarded the same way.
- [ ] POSIX synchronous `run_command` in `process_tools.cpp`. Mirror the bg
  path but block on `waitpid` for the exit status.
- [ ] POSIX `StdioTransport` in [stdio_transport.cpp](../../src/mcp/stdio_transport.cpp).
  Same `posix_spawn` + pipes + reader-thread shape, with `CancelIoEx`-style
  shutdown replaced by closing the read pipe FD (the reader thread's blocking
  `read()` returns 0 = EOF, thread exits cleanly).
- [ ] Tests:
  - [tests/test_process_registry.cpp](../../tests/test_process_registry.cpp) already
    exists; tag the existing cases with both Windows and macOS where they apply,
    add macOS-specific cases for `killpg`-on-child-tree.
  - Integration tests under `[shell]` and `[bg]` and `[mcp]` must pass on
    macOS without changes.
- [ ] Stage B acceptance: a `run_command_bg` running `tail -f /tmp/foo.log`
  streams to the terminal panel; kill from the right-click menu actually
  ends the process and its `tee` child. MCP server (e.g. `npx
  @modelcontextprotocol/server-filesystem`) starts, advertises tools,
  responds to `tools/call`, and shuts down cleanly when the workspace closes.

## Implementation notes

- **vcpkg triplet**. `arm64-osx` ships as dynamic by default; we don't need a
  static-CRT custom triplet to mirror `x64-windows-static`'s overlay. The
  reason we have an overlay on Windows is `-D_USE_STD_VECTOR_ALGORITHMS=0` --
  MSVC STL only. Under clang the define is a no-op, so it can stay in
  `CMakeLists.txt` as `add_compile_definitions(_USE_STD_VECTOR_ALGORITHMS=0)`
  for everyone with no harm. Whether to add a stub `arm64-osx.cmake` overlay
  for symmetry with the Windows commands is a doc-ergonomics call; not load-bearing.

- **PDFium hash discipline**. The Windows pin is to chromium/6996. macOS
  binaries for that same chromium tag exist in the same bblanchon release.
  Use them, not a newer release -- this guarantees PDF extraction parity
  across platforms (same parser, same glyph quirks).

- **WebView2 / WebKit divergence**. The chat panel renders markdown +
  bundled Prism via `wxWebView`. On Windows the backend is WebView2 (Edge);
  on macOS it's `wxWEBVIEW_BACKEND_WEBKIT` (WKWebView). The bundled
  `resources/prism/` assets and the `locus://` scheme interception both work
  through `wxWebView`'s abstract API -- no per-backend code in our codebase.
  But there will be subtle CSS / JS engine differences (Safari vs Edge);
  budget visual QA time in Stage A acceptance.

- **`run_command` semantics divergence**. Windows shells out via `cmd /c`;
  macOS will shell via `/bin/sh -c`. Quoting rules differ. Tool documentation
  in `process_tools.cpp` should not promise cross-platform command-line
  syntax -- agents will need to learn (or be told via system prompt) what
  shell they're running on. The platform string is already in
  `WorkspaceMetadata`; expose it in the system prompt so the model adapts.

- **`%APPDATA%` -> `~/Library/Application Support/Locus/`**. Already abstracted
  in [global_paths.cpp](../../src/core/global_paths.cpp); audit the file
  carefully -- it's the right place for the `if(APPLE)` macOS branch (and a
  Linux/XDG branch eventually). Don't sprinkle path resolution across call
  sites.

- **CI**. macOS runner on GitHub Actions can build + unit-test on every PR
  after Stage A. Integration tests stay manual (LM Studio + a real model).
  Cost is non-trivial; consider running macOS CI only on main + release
  branches, not every push.

- **What we deliberately don't port**: Windows-only registry / jump-list /
  COM-based features that have no macOS analogue stay Windows-only; they're
  already platform-guarded and shouldn't be re-architected for parity.
  Likewise the Windows installer story (signed MSI) is separate from the
  macOS .app bundle and lives in its own future stage if we ever need a
  cross-platform installer pipeline.

## Open questions

- **Linux at the same time?** The POSIX branches written for macOS in
  Stage B (process spawn, MCP stdio) are 90% Linux-ready already (`epoll`
  vs `kqueue` is the main divergence in the reader thread; both have a
  blocking `read()` fallback that works). Worth a small extra effort to
  test the Stage B code on Linux too, even if Stage C polish (LaunchAgent,
  .app bundle, NSSound) doesn't carry over. Decide before Stage B starts.

- **Single .app bundle or two binaries?** The Windows layout ships
  `locus.exe` (CLI) and `locus_gui.exe` side by side. On macOS the
  convention is one .app per visible-to-Finder application. Three options:
  (1) `Locus.app` contains both `MacOS/locus_gui` and `MacOS/locus`, with
  `locus` exposed via a `/usr/local/bin` symlink during install; (2) ship
  `Locus.app` (GUI) and a separate `locus` CLI installed via Homebrew tap;
  (3) ship `Locus.app` and have the CLI live inside it (`Locus.app/Contents/MacOS/locus`
  reachable only via full path). (1) matches Windows ergonomics best; (2)
  is most idiomatic for macOS power users. Open.

- **Universal binary or arm64-only?** Apple Silicon is the majority of
  active Mac developers now. Shipping arm64-only is faster to build, half
  the bundle size, and excludes ~15% of Mac users (rough; trending down).
  Recommendation: arm64-only for v1; revisit if user feedback demands x64.

- **Notarisation cadence**. First notarisation can take 15-60 min from
  Apple. Subsequent ones are usually fast. Whether Stage C blocks on a
  fully-set-up Developer ID account is an open question -- it might be
  reasonable to ship Stage A + B as a "build it yourself" macOS path and
  defer Stage C's signed-bundle story to a follow-up once we know users
  actually want it.
