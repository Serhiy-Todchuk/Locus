# S6.15 -- Session Restore Parity

## Problem

`SessionManager::save` / `load` already round-trips every `ChatMessage`
field on the wire -- `content`, `reasoning_content` (saved since S6.10
Task C), `tool_calls`, `tool_call_id`. But the GUI render path
(`ChatPanel::render_loaded_history`) dropped two of them on the floor:

1. `reasoning_content` was persisted to disk and ignored at render time.
   Opening a saved session showed only the assistant's final answer; the
   "Thinking..." bubble that streamed live during the original turn was
   invisible on restore.
2. When an assistant turn emitted only tool calls (no text body), the
   restore path skipped the assistant message entirely. The matching
   tool-result bubble then appeared with only the tool name -- no
   `tool-preview` line ("read_file(foo.cpp)") above it that live mode
   gets from `on_tool_pending`.

Two other gaps were one step further out:

3. `agent_mode` (chat / plan / execute) and any in-flight `Plan` state
   weren't saved at all. Re-opening a plan-mode session dropped you back
   into chat mode with no plan bubble.
4. The `ActivityLog` was in-process only. After workspace close the
   per-turn timeline (system_prompt / tool_call / tool_result / index_event
   / ...) was gone -- a real loss for multi-day sessions.

Manage Sessions also rendered ages with `5mo ago` / `38d ago` strings
that get useless once a workspace accumulates more than a couple of
weeks of history.

## Lands in M6 because

- Builds on S6.10 Task C (assistant `reasoning_content` is already persisted) and S5.I (multi-tab session lifecycle).
- Independent of S6.13 (reasoning watchdog) and S6.14 (thinking knob) -- those are stream-time concerns.

## Tasks

### Task A -- Render parity on restore

- [x] `ChatPanel::render_loaded_history` emits a `msg-reasoning <details>`
      bubble (via the same `addReasoning` / `setReasoningBody` /
      `finalizeReasoning` JS helpers the live path uses) whenever a
      restored assistant message carries `reasoning_content`. The summary
      collapses to `Thinking (restored)` so the user can tell the bubble
      came from disk and not from a live stream.
- [x] Same call site now accepts an optional `IToolRegistry*`. When
      supplied, every tool-result bubble's head re-runs
      `ITool::preview(call)` against the saved
      `assistant.tool_calls[].arguments` JSON so the `tool-preview` line
      matches what live mode shows. Missing registry, unknown tool, or
      parse failure all degrade gracefully to "tool-name only" -- the
      existing rendering.
- [x] `LocusFrame::create_tab_ui` passes `tab.agent().tools()` into the
      call so the GUI restore path picks up real previews; the no-arg
      overload remains for any future caller that doesn't have a registry
      handy.

### Task B -- Persist + restore agent mode + plan state

- [x] `AgentCore::save_session` writes `agent_mode` always and `plan` /
      `plan_awaiting_decision` only when a plan is currently live (the
      `current_plan_` lock-guarded snapshot).
- [x] `AgentMode` / `Plan` already had `to_string` / `plan_to_json`; new
      `agent_mode_from_string` was already there. Added
      `plan_from_json` + `plan_step_status_from_string` in
      [agent_mode.h](../../src/agent/agent_mode.h).
- [x] New `SessionManager::load_full(id) -> nlohmann::json` returns the
      whole session file including the top-level extras the existing
      `load(id)` discards.
- [x] `AgentCore::load_session` reads the extras up front, re-applies
      `mode_` directly (the agent thread is idle at that point so the
      queued `set_mode` path isn't needed), restores `current_plan_` +
      `plan_awaiting_decision_` under the existing mutex, and broadcasts
      `on_mode_changed` + `on_plan_proposed` BEFORE `on_session_reset`
      so the chat panel paints the right mode tab + plan bubble on the
      first frame.

### Task C -- Opt-in activity sidecar JSONL

- [x] `WorkspaceConfig::Sessions::persist_activity` (default off). JSON
      round-trip wired in `workspace_config_json.cpp`.
- [x] `ActivityLog::set_persist_path(path)` and `replay_from(path)`:
      append-only JSONL sidecar at `.locus/sessions/<id>.activity.jsonl`.
      Coalesced index_event rows get one line per emit -- the replay
      reader keeps last-wins by id so the in-memory state matches the
      live behaviour. Frontend broadcasts re-fire from `replay_from` so a
      newly-attached Activity panel paints the restored rows.
- [x] `ActivityLog::clear()` resets the in-memory ring + id counter
      between sessions so old activity doesn't bleed into a freshly-loaded
      one.
- [x] `AgentCore::save_session` sets the persist path after the save
      returns the id; `load_session` clears + replays + re-points the
      persist path so subsequent events continue the file.
- [x] `SessionManager::remove(id)` deletes the sidecar alongside the
      session JSON + archive dir.
- [x] New "Sessions" tab in Settings with the toggle (plus
      `restore_last`, which never had a GUI before). Tab id
      `ui_names::kSettingsTabSessions`; controls
      `kSettingsSessionsRestoreLast` and
      `kSettingsSessionsPersistActivity`.

### Task D -- Absolute timestamp for old sessions

- [x] `ManageSessionsDialog::format_age` keeps the "%llds / m / h / d
      ago" string for sessions touched in the last week; older sessions
      get a local-time `%Y-%m-%d %H:%M` stamp instead. Same shape the
      session-id filename uses on disk, so the dialog reads consistently.

### Task E -- Tests

- [x] Unit suite (`tests/test_session_restore.cpp`): per-field
      ChatMessage round-trip, `plan_to_json` <-> `plan_from_json` parity,
      top-level extras (`agent_mode` / `plan` /
      `plan_awaiting_decision`) survive `SessionManager::save` ->
      `load_full`, activity sidecar replay, coalesced-row dedupe on
      replay, `SessionManager::remove` cleans up the sidecar, `load_full`
      throws on missing.
- [x] Live-LLM integration (`tests/integration/test_int_session_restore.cpp`):
      drive a tool-using turn, close the session, reopen via
      `LocusSession::add_tab_for_session`, assert every non-system
      `ChatMessage` survives field-for-field including reasoning + tool
      args.
- [x] UIA visual parity
      (`tests/ui_automation/scripts/session_restore_round_trip.json`):
      seeds a workspace with a complete saved session (reasoning + tool
      call + tool result + final answer) plus a `ui_state.json` that
      auto-opens it; asserts the restored chat WebView contains every
      marker. Uses two new script-runner setup primitives:
      `sessions_seed[].messages` and `setup.ui_state`.

## Implementation notes

- Reasoning bubble on restore deliberately uses a different summary
  label ("Thinking (restored)") so a screenshot diff can distinguish a
  replayed bubble from a live stream's collapsed end-state. Cheap
  affordance, no behaviour change.
- Activity sidecar uses `app | binary` open mode so partial writes after
  a crash leave the file readable -- the parse loop simply drops any
  truncated last line.
- The Sessions settings tab is the natural home for future cleanup
  knobs (`keep_last_count`, `delete_after_days`); they stay
  config-file-only until someone needs them in the UI.
