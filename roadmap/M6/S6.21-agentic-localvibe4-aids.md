## Problem (findings.md)

Across the session, with the existing S6.10/S6.13/S6.19/S6.20 machinery in
place, the agent still:

- **Looped silently when ALL tool calls in a turn were dropped as malformed**
  (gemma-4-31b finding #2). When every tool call is dropped (here: corrupt
  argument JSON), the turn ends with only a bare `Dropped N malformed tool
  call(s)` error and no tool output. The S6.20 build-error streak detector keys
  on tool RESULTS, and the S6.10 repeated-call QualityMonitor keys on identical
  (name, args) -- a turn that produces ZERO tool results trips neither, so the
  model re-emits the same corrupt call next turn forever. No nudge, no abort.
- **Got stuck on `edit_file` "old_string matches N locations" / "not found"**
  (gpt-oss finding #8; recurred on the local run). Small models loop re-reading
  the file and re-guessing the anchor; they never discover `replace_all=true` or
  fall back to `write_file`. The error string states the remedy but the model
  doesn't act on it.
- **Falsely reported success** (gpt-oss finding #7): "the build now succeeds and
  produced frame0.ppm" while the on-disk build was still exit 1 and no artifact
  existed. The turn had run NO `run_command` that round -- the claim was
  ungrounded. Nothing flagged the assistant asserting a verified outcome it
  never verified.
- **Could not converge at 16k context** (local gemma-12b finding #9): the
  auto-compact band fired every few rounds and hard-trimmed the working memory of
  WHICH build error it was mid-fixing, so it restarted thin and re-derived --
  reaching round 19-24 without landing a single edit to the broken readback
  block. Compaction kept the session alive (correct) but erased the build-loop
  thread faster than the model could close it.

## Tasks

### 1. "All tool calls dropped" nudge/abort (highest impact)

Today a round where the stream delivered tool calls but ALL were dropped as
malformed (`LMStudioClient: dropping malformed tool call ... 0 delivered, N
dropped`) ends the turn on a bare error. Add a per-turn counter on
`AgentTurnRunner` (sibling to the S6.20 build-error streak): when a round nets
**0 delivered / >=1 dropped**, inject ONE steering nudge ("your last tool
call's arguments were not valid JSON -- re-send the call with a correctly-quoted
JSON object; do not repeat the previous malformed call verbatim"), counted
against the shared 2-cap. A second all-dropped round after the nudge aborts the
turn with "Agent appears stuck (tool calls repeatedly malformed)". A round with
>=1 delivered call resets the counter. Gated by `agent.quality_monitor_enabled`.

Plumbing: `AgentStepResult` already carries delivered/dropped counts at the
`AgentLoop` layer (the log line is sourced there) -- surface `dropped_count`
(and `delivered_count`) on the result so `AgentTurnRunner` can read it without
re-parsing the log. The nudge text is static; no new schema.

### 1b. Malformed-tool-call error must carry the diagnostic (activity panel)

The user-facing error is generic. [LMStudioClient](../../src/llm/llm_client.cpp)
emits `Dropped N malformed tool call(s) from the model response` (plus the
max_tokens hint when truncation is suspected) -- but the per-call detail (which
tool, why it failed) only lives in the trace-level `dropping malformed tool call
(name='edit_file', id='...', args_len=334): args do not parse as JSON` log line.
So the Activity panel / chat error shows "Dropped 1 malformed tool call(s)" with
NO hint about what was wrong, and a user (or QA-LLM driver) watching the panel
can't tell a JSON-quoting slip from a max_tokens truncation from provider-side
corruption without opening `.locus/locus.log` at trace level.

Fix: thread the per-call diagnostics up into the user-visible error. Each dropped
call already computes `(name, id, args_len, reason)` at the warn site; collect
them and fold a compact summary into `err_msg` -- e.g. `Dropped 1 malformed tool
call(s): edit_file (args do not parse as JSON, 334 bytes). Re-send with a valid
JSON object.` Cap the included arg preview (reuse `truncate_for_log`-style
bounding) so a multi-KB body doesn't dominate the panel. This is the
visibility-tripwire complement to Task 1's nudge: Task 1 steers the *model*, 1b
tells the *user* what actually went wrong. No new activity kind -- it rides the
existing `on_error` -> Activity panel path; just a richer string.

### 2. edit_file ambiguity escape hatch

Two parts, both narrow:

- **Self-healing error text.** When `EditFileTool` returns the "`old_string`
  matches N locations" or "`old_string` not found" error, append a one-line
  remedy to the `ToolResult` content the model sees: for the ambiguous case,
  "set `replace_all=true` to replace all N, or add more surrounding context to
  the anchor"; for not-found, "re-read the file -- the exact text including
  whitespace must match; for a large change prefer `write_file overwrite=true`".
  (The strings exist for humans today; this makes them model-actionable in the
  result body, mirroring S6.20 Task 3's schema-on-throw approach.)
- **Optional `AgentTurnRunner` nudge.** If the SAME edit_file ambiguity/not-found
  error recurs >=2x in a turn, inject the escape-hatch hint once (shared 2-cap).
  This catches the loop the local-gemma run got stuck in even when the per-result
  text isn't enough on its own.

### 3. No-progress / unverified-success guard

When the assistant emits a turn-ending message asserting a verifiable outcome
("builds", "compiles", "succeeds", "frame0 / screenshot created", "the exe
runs") in a turn that ran NO confirming tool call that round (no `run_command`,
no `read_file` of the claimed artifact, no `list_directory` showing it), surface
a non-blocking footer/activity note: "Assistant claimed success without a
verifying tool call this turn." This is a *visibility tripwire for the watching
user*, not a model correction -- the agentic `get_chat_status` / activity scrape
picks it up so a driver (human or QA-LLM) doesn't trust the claim. Keep the
trigger conservative (phrase match + "zero verifying tool calls this round") to
avoid false positives on legitimate summaries. No turn abort, no nudge.

### 4. Compaction breadcrumb (highest leverage for small-context local models)

When a hard-trim fires mid-build-loop, the model loses the thread of which error
it was fixing. After `CompactionPipeline` hard-trims, if the turn has a live
build-error signature (S6.20's `extract_error_signature` over the most recent
tool output is non-empty), prepend a one-line breadcrumb to the next user/turn
payload: "[Resuming: last build error was `<signature head>`; you were editing
`<file>`. Make a minimal targeted edit.]". Cheap (one synthesized line, reuses
the S6.20 detector + the S6.18 hard-trim seam), and it is the single
highest-leverage change for letting 16k-class local models converge on iterative
build tasks. Gated by `agent.quality_monitor_enabled`; the breadcrumb is dropped
if no build-error signature is live (pure chat compaction is unaffected).

## Out of scope -- provider / model behaviour (documented, no Locus change)

These findings are real but are not Locus bugs; recorded in findings.md and here
so a future session doesn't re-investigate:

- **gemma-4-family-on-OpenRouter `<`-in-tool-args corruption** (finding #1).
  Direct probes (Locus bypassed) show the provider corrupts every `<` inside
  tool-call argument strings (`a<b<c` -> `a<<bb<<cc`), streaming AND non-stream;
  plain content streaming is clean. A `<<`-collapse repair was considered and
  rejected as lossy on legitimate `<<` (bit-shift / `std::cout <<`). The
  free-model sweep confirmed this is Gemma-4-specific on OpenRouter -- every other
  tool-capable free model emits `<` cleanly. Recommendation lives in findings.md
  (use a clean model; gemma-4 via OpenRouter is chat-only).
- **Free-tier 429 walls** (finding #4) -- S6.19's `classify_retry` already handles
  these correctly (verified: 5 retries with 8s-floor backoff, clean terminal
  error). No change.
- **"Describe then stop" refusal-to-build** (finding #3) -- partly addressed by
  S6.13's auto-nudge for reasoning spirals, but the stop-with-finish_reason-stop
  case is model-side; the success-guard (Task 3) at least makes it visible.
