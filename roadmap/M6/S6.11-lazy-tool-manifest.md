# S6.11 -- Lazy Tool Manifest

## Problem

Every turn ships the entire tool catalog to the model in two duplicate forms (see [ADR-0007](../../architecture/decisions/0007-context-budget-reshape-lazy-manifest-and-profiles.md)):

- System prompt's `## Available Tools` section: every tool's name + description + every `ToolParam` with type + description. Walked from `tools.all()` in [`system_prompt_assembly.cpp`](../../src/agent/system_prompt_assembly.cpp).
- API request `tools: [...]` array: same content as full OpenAI-format JSON schema, built by `tool_registry::build_schema_json()`.

Measured per-turn cost on the 12 built-in tools: ~1000 t in the system prompt + 2854 t in the API array. On a 16K-context model that's 23% of the budget gone before the user message lands.

## Lands in M6 because

- Companion to [S6.12](S6.12-system-prompt-profiles.md); both consume the same ADR.
- Independent of S6.13/S6.14 (those are stream-time, not prompt-time).
- Touches `ToolRegistry`, `SystemPromptAssembly`, `WorkspaceConfig`, and the Settings UI -- surfaces that already exist.

## Design (summary, full rationale in [ADR-0007](../../architecture/decisions/0007-context-budget-reshape-lazy-manifest-and-profiles.md))

When `WorkspaceConfig::tools.lazy_manifest = true`:

1. The system-prompt `## Available Tools` section degrades to **summaries**: `**<name>**: <one-line description>` -- no `ToolParam` lines.
2. The API `tools: [...]` array carries the same tool list, but each entry's `parameters` schema is replaced with `{"type":"object","properties":{},"additionalProperties":true}`. The model can still emit a syntactically valid tool call but does not see the real arg shape upfront.
3. A new built-in meta-tool **`describe_tool(name: string)`** returns the full OpenAI-format JSON schema as its result content. Auto-approve. Always visible in the manifest (counts toward the budget at ~80 t).
4. `ToolDispatcher`'s existing tool-name validation gets a friendlier error when the model picks an unknown name: "Tool `<name>` does not exist. Call `describe_tool('<closest_match>')` for its schema, or pick another tool."
5. Missing-required-arg errors (already produced today) gain a hint: "Tool `<name>` requires arg `<key>`. Call `describe_tool('<name>')` to see the full schema."

When `lazy_manifest = false` (default): today's behaviour unchanged.

### Task B -- `describe_tool` meta-tool

- [ ] New header [`src/tools/describe_tool.h/cpp`](../../src/tools/describe_tool.h):
  ```cpp
  class DescribeTool : public ITool {
   public:
      explicit DescribeTool(const IToolRegistry& reg);
      std::string name() const override { return "describe_tool"; }
      std::string description() const override;
      std::vector<ToolParam> params() const override;  // single string `name`
      ToolApprovalPolicy approval_policy() const override { return ToolApprovalPolicy::auto_approve; }
      ToolMode visible_in_mode(ToolMode m) const override;  // visible in all modes
      ToolResult execute(IWorkspaceServices& ws, const ToolCall& call) override;
   private:
      const IToolRegistry& registry_;
  };
  ```
- [ ] Returns full OpenAI-format schema for the requested tool as JSON-string content. Unknown name -> `success=false` with `"Tool '<name>' is not registered. Available: <list>"`.
- [ ] Registered in `register_builtin_tools()` ALWAYS (independent of `lazy_manifest` setting -- it's cheap to keep available; the model just won't be told to use it when `lazy_manifest=false`).
- [ ] **Always visible in the per-turn manifest** when `lazy_manifest=true`; **hidden from the manifest** when `lazy_manifest=false` (no point telling the model to use a tool whose payload is already redundant with the existing schema). Filtering logic: `if (!ws.config().tools.lazy_manifest && tool->name() == "describe_tool") skip;`

  - Known tool -> success + JSON schema matches `ToolRegistry::build_schema_json` slice for that tool.
  - Unknown tool -> success=false + helpful message.
  - Approval policy is auto-approve.

## Notes

- `describe_tool` description should itself be short -- it's in every lazy manifest. Target: under 30 tokens.
- The `additionalProperties:true` permissive schema on lazy-manifest tools means servers that strictly validate against the schema (LM Studio's strict mode, vLLM with `guided_json`) would happily accept any args, including malformed ones. That's fine -- the existing post-dispatch validation catches missing required args. If a future stage needs strict server-side validation, it'd be incompatible with lazy_manifest in its current form; document that.
- Composes with [S6.12](S6.12-system-prompt-profiles.md): the profile setting controls the prose sections; this setting controls the tools section. They never collide on text.
