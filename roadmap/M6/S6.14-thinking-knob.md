# S6.14 -- Thinking ON/OFF/auto knob in LLM settings -- done

## Problem

Locus already handles reasoning content on the read side: `LMStudioClient` routes `reasoning_content` through `on_reasoning_token` to a separate DOM channel, the chat panel renders a collapsible "Thoughts" bubble, and Layer 3 compaction strips leaked `<think>...</think>` blocks. What is still missing is a way to turn thinking OFF *at the source* for the next request.

Concrete evidence:
- The 2026-05-24 TestLocalVibe agentic run produced a single LLM round on `qwen/qwen3.6-27b` that streamed 666 KB / 2500+ chunks of reasoning content before hitting the harness's 30-min timeout, with no tool call ever emerging. A second run on the same workspace with `/no_think` appended to the user prompt finished the same task shape in 183 seconds across 5 rounds.
- The earlier agentic Tetris run 1 round 10 on `qwen/qwen3.6-35b-a3b` produced 12602 chars of reasoning in ~10 min while the model thrashed on a SHAPE-array bounds error -- more thinking actively hurt.

Companion stage [S6.13 (reasoning watchdog)](S6.13-reasoning-watchdog.md) addresses the *symptom* (cap the runaway), but the root-cause fix is to disable thinking up front when the user knows the task doesn't need it.

## Lands in M6 because

- Hooks the LLM-request assembly path (`LMStudioClient::stream_completion`) which is the same surface [S6.10](S6.10-small-model-robustness.md) Task C touches; small change, no new subsystem.
- Independent of S6.0-S6.13. Can land any time within M6.
- Companion to S6.13 (watchdog) -- one disables thinking, the other caps it. Both useful, neither subsumes the other.

## Tasks

### Task A -- `LLMConfig::enable_thinking` tri-state

- [x] Add `enum class ThinkingMode { Auto, On, Off }` to `src/llm/llm_client.h`. Default `Auto`.
- [x] `LLMConfig` gains `ThinkingMode enable_thinking = ThinkingMode::Auto;`.
- [x] `WorkspaceConfig::Llm` gains a string field `thinking_mode` (`"auto"` / `"on"` / `"off"`). `workspace_config_json.cpp` round-trips it.
- [x] `LocusSession` ctor maps the config string -> `LLMConfig::enable_thinking` (in `merge_workspace_defaults`).

### Task B -- Per-model-family request injection in `LMStudioClient`

When `enable_thinking != Auto`, `LMStudioClient::stream_completion` augments the request per the loaded model's family. Detection piggybacks on the existing `tool_format` heuristic in [`model_presets.cpp`](../../src/llm/model_presets.cpp):

| Model family | Mechanism for `Off` | Mechanism for `On` |
|---|---|---|
| Qwen 3.x hybrid (`qwen3.6-*`, `qwen3-thinking-*`, etc.) | `extra_body.chat_template_kwargs.enable_thinking = false` | `enable_thinking = true` |
| Qwen 2.x (`qwen2-*`, older `qwen-*`) | Append `\n/no_think` to the last user message content | Append `\n/think` |
| o1-style (`o1-*`, `deepseek-r1-*`) | `reasoning_effort: "low"` | `reasoning_effort: "high"` |
| Unknown | No-op (and warn-log once: "thinking_mode forced to On/Off but no known mechanism for model 'X'; ignoring") | Same |

`Auto` always no-ops -- the server-default behaviour runs.

- [x] New helper `src/llm/thinking_injection.h/cpp`: `void apply_thinking_mode(nlohmann::json& request, std::vector<ChatMessage>& messages, ThinkingMode mode, std::string_view model_id)`. Pure function (one warn log on unknown family); tested separately. Also exposes `classify_thinking_family(model_id)` + `ThinkingFamily` for the unit tests.
- [x] Hook the helper in `LMStudioClient::build_request_body` (the single body-assembly site that `stream_completion` -> `do_stream_completion` routes through) after `body["messages"]` is set, on a local copy of the message vector (build_request_body's `messages` param is const).
- [x] **S4.F invariant**: the Qwen 2.x mechanism mutates `messages[last_user]` content per request. That breaks byte-stable system-prompt prefix-cache reuse on the LAST user message but NOT on `messages[0]` (system). Documented in the helper header and in `architecture/agent-loop.md`. Out-of-scope to preserve KV-cache here -- thinking-mode toggles are explicit user actions.

### Task C -- Settings UI

- [x] Settings -> LLM tab gains a 3-state "Thinking" control below the grammar-mode dropdown:
  - **Auto** (use the model's default behaviour) -- default
  - **On** (force on)
  - **Off** (force off)
  - Implemented as a `wxChoice` (Auto/On/Off) to match the sibling tool-format / grammar-mode dropdowns and lay out cleanly in the same FlexGridSizer; semantically the same 3 mutually-exclusive options the spec's "radio" asks for.
- [x] Tooltip lists the per-family mechanism so users understand what gets injected.
- [x] On OK, `commit_to_config` writes `WorkspaceConfig.llm.thinking_mode`; `settings_dialog.cpp` folds it into `llm_changed_` so the user gets the existing "Restart Locus to apply" prompt (LLM settings are resolved at workspace open, not hot-swapped -- same path as endpoint/model/temperature).
- [x] Named widget `locus.settings.llm.thinking_mode` addressable from UIA.

### Task D -- Tests

- [x] `tests/test_thinking_injection.cpp` `[s6.14]` (10 cases / 49 assertions, exceeds the spec's 5):
