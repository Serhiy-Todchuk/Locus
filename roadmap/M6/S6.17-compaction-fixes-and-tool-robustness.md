# S6.17 -- Compaction Fixes + Tool-Arg Robustness (TestLocalVibe2 follow-up)

## Problem

The 2026-05-25/26 TestLocalVibe2 agentic-mode run on `qwen/qwen3.6-27b` (five passes, 16384-token LM Studio context, full report at [findings.md](../../tests/ui_automation/output/agentic_TestLocalVibe2/findings.md)) exposed two structural problems plus a bag of surface-level bugs that compound under realistic load.

**Compaction is broken on open-ended tasks.** Pass 5 (open-ended refactor: "extract 3 of 10 subsystems into `src/*.h/cpp`") ran for 35.6 min, fired four compactions, and produced **zero output files**. Three of the four compactions reported `reached=no` and stripped nothing -- each "compaction" added only the 37-token sentinel breadcrumb, so the next round was *larger* than the previous one:

```
auto-compact firing (12088 / 13108 = 92% of effective_limit, threshold 85%)
  pipeline compaction 10234 -> 10271 (target 9831, reached=no)    Delta +37
auto-compact firing (13115 / 13108 = 100% of effective_limit, threshold 85%)
  pipeline compaction 11343 -> 11380 (target 9831, reached=no)    Delta +37
auto-compact firing (14316 / 13108 = 109% of effective_limit, threshold 85%)
  pipeline compaction 12427 -> 12464 (target 9831, reached=no)    Delta +37
```

The strip-large-tool-bodies layer (which carried Pass 4's clean 89% reduction) had nothing to strip in Pass 5 because every tool result was a small `run_command` output already trimmed to 60+60 lines by `process_output_filter`. The other layers (drop_redundant_tool_results / drop_old_reasoning / drop_oldest_turns) also had nothing to grab on the conversation shape Pass 5 produced. The pipeline gave up three times in a row while context kept growing, the agent eventually saturated the LLM's 16k ceiling, slow streaming + watchdog 3-strike abort, no files written.

**Lazy tool manifest amplifies silent-fallback bugs.** With the per-turn manifest collapsed to one-line summaries, the model can't see arg names until it calls `describe_tool`. Multiple tools silently swallow unknown keys via `nlohmann::json::value(key, default)`:

- `read_file` with `lines:"100-152"` -> silently returned lines 1-100 (defaulted to `offset=1, length=100`); the agent retried 6 times across Passes 2 and 5 before working around it via PowerShell `Get-Content`.
- `SearchTool` with `type:"regex"` -> silently ran as `mode:"text"`; the resulting FTS5-tokenised search returned 1 hit on a query that genuinely had 4 matching lines.
- `run_command` with `cmd:` -> rejected with `'command' parameter is required` (good error, but the model needed a describe_tool round to discover the right key).

Each individual bug is recoverable. Under realistic load they stack: Pass 5 burned roughly 12 rounds on tool-schema discovery alone before hitting the compaction cliff.

**Removal candidate exposed.** Pass 5 message [15] caught the model reaching for `filter_output` expecting a grep-over-file shape. The tool overlaps with two existing capabilities (inline `output_filter_mode` on `run_command` / `read_process_output`; `search` with `mode=regex`) and its own header concedes "the agent rarely needs to chain a separate filter_output call."

## Lands in M6 because

- Continuation of the [S6.10](S6.10-small-model-robustness.md) small-model robustness theme: same hardware (Qwen 3.6 27B), same goal (keep slow local models productive), same surfaces (AgentLoop, ToolDispatcher, CompactionPipeline, prompt assembly).
- Independent of the connectivity stages (S6.1-S6.9).
- Compaction subsystem refactor is the biggest piece and gates whether agentic-mode on a 16k-context local model is viable for open-ended tasks at all.

## Design

The work splits into a **compaction overhaul** (one large subsection, must land together), a **tool-arg robustness pass** (cross-tool helper + per-tool fixes), a **manifest-shape change** (search modes split per ADR-0008), and a **polish bucket**. Order is set so the highest-impact change lands first.

### Compaction overhaul (Task B -- one cohesive subsection)

Three things must land together because they jointly determine whether the next Pass-5-shaped run actually makes progress. Independent half-fixes are worse than nothing.

1. **Layer escalation on `reached=no`**. The current pipeline runs the configured layers once and reports the outcome. Change: when a compaction reports `reached=no`, the *next* compaction attempt in the same turn force-enables the next-most-aggressive layer regardless of user config, in this fixed order:
   1. `layer_drop_redundant_tool_results`
   2. `layer_strip_large_tool_bodies`
   3. `layer_drop_old_reasoning`
   4. `layer_llm_summary`
   5. `layer_drop_oldest_turns`

   If layer 5 still reports `reached=no`, trim turn-by-turn from oldest until target is hit, and emit a warn-level log line naming the saturated state.

   Also add a **count-based heuristic** to `layer_strip_large_tool_bodies`: when the most recent N rounds carry M+ tool results (today's default config: N=10, M=12), strip the older half regardless of individual byte size. This catches the Pass-5 shape where each result was small but the count was large.

2. **Progress-preserving sentinel**. Today's sentinel is generic: `[Earlier context compacted; full pre-compaction history archived at ...]`. After 4 of these stacked (Pass 5 messages 0-3) the model had no idea what task phase it was in. Two complementary changes:

   - Fire `layer_llm_summary` whenever any *other* layer reports `reached=no`, not just when it's the configured next layer. Prepend the summary to the sentinel: `[Earlier context compacted. Progress so far: <one-sentence summary>. Full pre-compaction history at ...]`.
   - Always preserve the most recent assistant message that has `content` (text) and no `tool_calls` -- the "what I just figured out" notes -- verbatim across compaction. Today's pipeline can strip it under `drop_old_reasoning` / `drop_oldest_turns`; pin it.

3. **Five-layer boolean config -> one preset**. The 5 `layer_*` booleans in `WorkspaceConfig::Compaction` are too granular -- a user can have all defaults "on" and still hit Pass 5's outcome. Replace the surfaced UI with:

   ```
   compaction.aggressiveness = "gentle" | "balanced" | "aggressive"
   ```

   Mapping (the booleans become the under-the-hood matrix the way [S5.S](../M5/S5.S-permission-presets.md) permission_presets works):

   | preset | layers enabled |
   |---|---|
   | gentle | drop_redundant_tool_results + strip_large_tool_bodies |
   | balanced (default) | + drop_old_reasoning + layer_llm_summary |
   | aggressive | + drop_oldest_turns |

   Individual booleans stay parseable from `config.json` for power users / tests. Settings UI surfaces only the preset.

4. **Telemetry / fault visibility**. `reached=no` is currently info-level. Promote to warn the first time it happens in a turn, error if two in a row. The chat-footer `compacted_chip` today shows a count of compaction events; split it so "no-op compactions" are visible separately: `compacted: N (M no-op)`.

### Tool-arg robustness pass (Task D)

A cross-tool helper in `src/tools/shared.h`:

```cpp
std::optional<ToolResult> reject_unknown_keys(
    const ToolCall&                 call,
    std::initializer_list<const char*> allowed);
```

Each affected tool's `execute()` calls it as the first line:

```cpp
if (auto err = reject_unknown_keys(call, {"path", "offset", "length"})) return *err;
```

Error format includes a suggestion when the unknown key is a known alias of an expected one:

> `Error: unrecognized argument 'lines'. Did you mean 'offset' + 'length'? Example: {"path":"...","offset":<int>,"length":<int>}.`

Alias map (lives next to the helper):

| tool | alias seen in this run | suggest |
|---|---|---|
| read_file | `lines`, `line_range`, `range` | `offset` + `length` |
| search* (post-Task-G; pre-Task-G applies to unified search) | `type`, `kind`, `search_mode` | `mode` |
| run_command | `cmd`, `shell`, `exec` | `command` |
| edit_file | (covered by Task E's shorthand acceptance, no alias needed) | -- |
| write_file | `body`, `text` | `content` |

For content-returning tools (read_file, search_text, search_regex), the result header carries the **args actually applied** so the model self-corrects when its arg was honoured-as-default:

> Today: `[snake.cpp lines 1-100 of 152]`
> After: `[snake.cpp lines 1-100 of 152 (offset=1 length=100)]`

### Search modes -> separate tools (Task G + ADR-0008)

See [ADR-0008](../../architecture/decisions/0008-split-unified-searchtool.md). Splits `SearchTool` back into `search_text` / `search_regex` / `search_symbols` / `search_semantic` / `search_hybrid` / `search_ast` as top-level tools. Removes the mode-discriminator dispatch step that is the root of F11 (model didn't reach for symbols mode), F15 (alias arg silently fell through), and F20 (model reached for `filter_output` because `search` didn't read as "grep").

### Prompt-cost preset (Task H)

Today three orthogonal toggles interact: `agent.lazy_tool_manifest` (S6.11), `agent.system_prompt_profile` (S6.12), and `llm.tool_format` (the `openai_json` path triggers S6.10 Task J). ADR-0007's matrix is the right mental model; the user shouldn't have to read an ADR to pick a row.

Add `agent.prompt_cost: minimal | balanced | verbose`:

| preset | lazy_tool_manifest | system_prompt_profile | tool_format hint |
|---|---|---|---|
| minimal | on | minimal | prefer openai_json |
| balanced (default for ctx <= 16k) | on | compact | prefer openai_json |
| verbose (default for ctx > 64k) | off | full | auto |

Individual flags stay parseable from `config.json`; Settings UI surfaces the preset only.

### Watchdog improvements (Task C)

Two narrow fixes, both motivated by clear evidence:

- **Timer-driven tick (F3)**: `AgentLoop::check_watchdog()` is currently called from inside per-chunk stream callbacks only. Pass 1's 499-second silent inter-chunk gap never tripped the 180s watchdog because no chunks arrived to drive the check. Add a 5s `std::thread` timer that calls `check_watchdog()` independent of chunk arrival; stop on stream-end / cancel / round-complete.
- **Don't 3-strike during commit phase (F8)**: Pass 3 and Pass 5 both died because the watchdog tripped during slow text-only summarisation. Heuristic: if the in-flight round has zero accumulated tool_call deltas AND accumulated_text is growing, suppress the auto-nudge and don't count the trip toward the 3-strike cap. Reserve the strikes for "model is reasoning but not committing".

### Removal: `filter_output` (Task A)

Three pieces of evidence:

1. Pass 5 message [15] caught the LLM expecting a file-grep shape; the actual signature (`text` arg, no path) doesn't match the use case.
2. `run_command` and `read_process_output` already trim output inline via `output_filter_mode` / `_pattern` / `_lines` / `_context` -- this covers the >90% case `filter_output` was added for.
3. `search` with `mode=regex` (and post-Task-G, `search_regex`) covers the remaining "extract lines matching X" case directly against indexed files.

Cuts:
- `src/tools/filter_output_tool.h` + `.cpp`
- `register_builtin_tools()` entry in `src/tools/tools.cpp`
- `builtin_tool_category()` entry in `src/tools/permission_presets.cpp`
- Code Map row in `CLAUDE.md`
- Any references in `process_output_filter.h` doc comments

No replacement needed -- the inline filter mode + search_regex cover the call sites.

### Task L -- Index storage shrink (vec0 lazy alloc + chunker leaf-symbol fix)

The 2026-05-26 follow-up investigation against the TestLocalVibe2 vectors.db found two distinct bloat sources stacked behind the original F2 finding (4.5 MB vectors.db-wal):

- `chunk_vectors_vector_chunks00` is allocated **exactly 4 MiB** the moment the vec0 table is created (sqlite-vec stores vectors in fixed-size 1024-vector buckets). For any workspace with `semantic_search.enabled=false`, or with zero indexable files, or with <1024 chunks total, this 4 MiB sits empty.
- `chunks` table had **238 rows for a 3-file workspace** with **220 of them overlapping at least one other in the same file** and **43 rows carrying <=50 bytes of content (13 of which are zero bytes)**. Root cause is in `chunk_code` ([src/index/chunker.cpp](../../src/index/chunker.cpp)): the symbol extractor emits 112 symbols for big_module.cpp with 111 nested inside another (namespace > class > methods), and the chunker's walk has two bugs that interact poorly with nested input -- degenerate chunks when the cursor-clamp inverts the range, and cursor moving backward when a nested symbol ends earlier than the current cursor. Detailed trace in the 2026-05-26 conversation log.

#### L.1 -- Lazy vec0 table creation

- [ ] [`src/core/database.cpp`](../../src/core/database.cpp) `Database::ensure_vectors_schema(dim)`: defer the `CREATE VIRTUAL TABLE chunk_vectors USING vec0(...)` statement (and the memory_vectors equivalent) until the first insert. Today's "create on workspace open" path pays 4 MiB + 4 MiB = 8 MiB up front; lazy creation drops this to 0 for workspaces with no embeddings.
- [ ] Implementation shape: track schema state with a boolean `vectors_table_present_`; first insert path checks the flag, runs the CREATE if false, then proceeds with the INSERT. Reads check the same flag and short-circuit to "no results" if the table isn't there yet (the existing query path's "table doesn't exist" branch should already handle this; verify).
- [ ] Acceptance: open an empty workspace with `semantic_search.enabled=true`; `vectors.db` size on disk after first launch is < 100 KB (today: 4.0+ MiB). Index a single small file; `vectors.db` grows to the expected size after the embedding worker drains.

#### L.2 -- Chunker `chunk_code` defensive walk

- [ ] [`src/index/chunker.cpp`](../../src/index/chunker.cpp) `chunk_code` loop: after the `sym_begin < cursor -> sym_begin = cursor` clamp, add the inverse check `if (sym_begin >= sym_end) continue;` -- this drops nested symbols entirely instead of emitting a degenerate `start_line > end_line, content=""` chunk and moving the cursor backward.
- [ ] Unit test `tests/test_chunker.cpp` `[s6.17]`: input with a namespace spanning the whole file + several nested classes + nested methods -- verify no chunk has `start_line > end_line`, no chunk has empty `content`, and the cursor never moves backward (assert on per-chunk monotonicity).

#### L.3 -- Leaf-symbol pre-filter in indexer

- [ ] [`src/index/indexer.cpp`](../../src/index/indexer.cpp) around line 479 (the `if (!symbol_spans.empty())` branch before `chunk_code`): pre-filter `symbol_spans` to drop spans that fully contain another span. O(n^2) check; n is the per-file symbol count which is typically <500. Pseudocode:
  ```cpp
  std::vector<SymbolSpan> leaf_spans;
  for (size_t i = 0; i < symbol_spans.size(); ++i) {
      const auto& s = symbol_spans[i];
      bool contains_another = false;
      for (size_t j = 0; j < symbol_spans.size(); ++j) {
          if (i == j) continue;
          const auto& other = symbol_spans[j];
          if (s.line_start <= other.line_start && s.line_end >= other.line_end
              && (s.line_start < other.line_start || s.line_end > other.line_end)) {
              contains_another = true;
              break;
          }
      }
      if (!contains_another) leaf_spans.push_back(s);
  }
  chunks = chunk_code(content, leaf_spans, ...);
  ```

#### L.5 -- Fix `chunk_vectors` UNIQUE-constraint warnings (rowid-reuse race)

Surfaced 2026-05-27 during the full integration run (74/74 passed; the warning fired 3 times for chunk_ids 21221, 21397, 21401). Root cause: `chunks.id` is `INTEGER PRIMARY KEY` *without* `AUTOINCREMENT`, so SQLite reuses freed rowids after `DELETE FROM chunks WHERE file_id=N`. When a file is modified faster than the embedding worker drains the queue, the new chunks get the same ids as the old ones, the worker sees duplicates, and the second `INSERT INTO chunk_vectors` collides with the first. In production this drops the second embedding silently -- the rare-but-real correctness bug under it is that the embedding for the *current* chunk content is the one that gets lost.

- [ ] [`src/index/embedding_worker.cpp`](../../src/index/embedding_worker.cpp) line 95: change `INSERT INTO chunk_vectors` to `INSERT OR REPLACE INTO chunk_vectors`. Keeps the embedding for the current chunk content rather than dropping it on the conflict.
- [ ] Demote the warning at line 150 (`Failed to insert vector for chunk N: UNIQUE constraint failed`) to trace level, OR remove it (the OR REPLACE path no longer triggers it). Other SQLite errors should still surface.
- [ ] Optional follow-up (different line count -- could be deferred): `EmbeddingWorker::enqueue` deduplicates against a `std::unordered_set<int64_t> in_queue_` so the worker doesn't compute the same embedding twice in a rapid-re-index burst. ~20 lines including the lock dance. Not strictly necessary -- OR REPLACE makes the redundant work harmless -- but saves an embedder forward pass per duplicate.
- [ ] Unit test `tests/test_embedding_worker.cpp` `[s6.17]`: enqueue the same chunk_id twice with different content blobs; assert no warning fires and the row in `chunk_vectors` carries the *second* content's embedding.
- [ ] Integration test: replay the rapid-re-index pattern (write file -> wait for embed start -> write file again before drain) and assert zero UNIQUE-constraint warnings in `.locus/locus.log`.

Out of scope for L (per user 2026-05-26): filtering chunks below a minimum byte/line size -- the small chunks remain useful for document-workspace headings (e.g. a one-line H2 heading is a legitimate retrieval anchor even at 30 bytes). L.1-L.4 fix the structural defects without touching the size policy.
