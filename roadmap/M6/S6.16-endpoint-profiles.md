# S6.16 -- Endpoint Profiles + Chat-Footer Endpoint Switcher

**Done 2026-05-29.** All tasks A-I shipped. Implementation notes vs. this draft:
- Store path is `~/.locus/endpoints.json` (via `global_paths::global_dir()`), not the draft's `%APPDATA%/Locus` -- the codebase settled on `~/.locus` in S5.M.
- Profile field landed on `WorkspaceConfig::Llm::active_endpoint` (nested), not a top-level `active_endpoint_name`.
- The startup-vs-switch layering is one shared `apply_endpoint_profile(cfg, profile, force)` helper in `endpoint_profile.cpp` (force=false = sentinel-guarded "legacy wins"; force=true = unconditional overlay), used by both `LocusSession::load_shared_resources` and `AgentCore::apply_pending_endpoint_switch`.
- Added `join_api_url()` in `llm_client.h`: hosted base_urls carry `/v1` (NVIDIA / Ollama / OpenRouter) and would otherwise double to `.../v1/v1/chat/completions` and 404. The transport + `query_model_info` both route through it.
- Hot-swap reseats a pointer inside `AgentLoop` (`set_llm`); `AgentCore` owns the swapped client in `owned_llm_`. The system prompt stays byte-stable for the session (only the tool-format addendum text could go stale, which is benign); the decoder DOES honour the new `tool_format` because it's baked into the new client.
- Migration is centralized in `LocusSession` (covers both CLI + GUI) rather than duplicated in `main.cpp` / `LocusApp::OnInit`.

## Problem

Locus today is hard-wired to one LLM source: a single LM Studio-shaped OpenAI-compatible endpoint, configured via a flat set of `WorkspaceConfig::llm_*` fields. There is no way to keep more than one endpoint configured at the same time, no way to use a hosted provider that requires an API key (NVIDIA, OpenRouter, OpenAI direct, Anthropic via an OpenAI-compat proxy), and no way to flip between sources without editing config and reopening the workspace.

Real workflows that fail today:

1. Local Gemma 4 E4B for cheap retrieval / outline rounds, hosted Qwen3-Coder-480B (NVIDIA free tier) for the actual code-writing turn -- impossible to switch mid-session.
2. "My local box is down, point Locus at Anthropic for this one task" -- needs a config edit and full workspace reopen.
3. A/B-ing the same prompt against two providers -- requires two workspaces.

The footer of the chat panel already hosts a `preset_choice_` `wxChoice` for permission presets ([src/frontends/gui/chat_panel.h:320](../../src/frontends/gui/chat_panel.h#L320)). The same widget shape is the natural home for an endpoint picker.

## Lands in M6 because

- Stand-alone: independent of the rest of M6 (web RAG / Crow / VS Code shim / Skills / archives / macOS port / small-model robustness / lazy manifest / reasoning watchdog / thinking knob / session restore). No other stage blocks it and it blocks none.
- Surfaces involved already exist: [`LLMConfig`](../../src/llm/llm_client.h), [`OpenAiTransport`](../../src/llm/openai_transport.h), [`LocusSession::load_shared_resources`](../../src/core/locus_session.cpp), [`ChatPanel` footer action row](../../src/frontends/gui/chat_panel.h), the per-tab `ISettingsPanel` split from S5.Q.
- Composes well with [S6.10](S6.10-small-model-robustness.md) Task F (`find_preset_for_model`) -- preset auto-detect is re-fired on endpoint switch with no extra plumbing.

## Decisions (answered before drafting)

| Question | Answer | Implication |
|---|---|---|
| What's "Claude Code" as a source? | OpenAI-compat proxy in front of Claude (LiteLLM, `claude-code-proxy`, etc.) | **Zero new transport / decoder code.** Claude lands as just another endpoint profile pointing at the user's local proxy. Native Anthropic `/v1/messages` is out of scope; revisit in a separate stage if proxies prove insufficient. |
| Where do API keys live? | Plaintext at `%APPDATA%/Locus/endpoints.json` (Windows) / `~/.config/locus/endpoints.json` (XDG) | Global, never inside a workspace, never git-committed. OS file ACLs are the security boundary (Locus is single-user-desktop). No keychain dependency. |
| Mid-session switch semantics? | Preserve history; switch applies on next user message; refused mid-turn | Switch is queued onto the agent thread; takes effect at the top of `process_message` before the next round builds. Different tokenizer => the context-meter number shifts after switch -- expected; logged but not treated as anomaly. |

## Design

### Data model

New file `src/llm/endpoint_profile.h/cpp`:

```cpp
struct EndpointProfile {
    std::string name;                 // user-visible label, unique within store
    std::string base_url;             // e.g. https://integrate.api.nvidia.com/v1
    std::string api_key;              // empty for local-no-auth endpoints
    std::string default_model;        // "" = use server default / query_model_info
    int         default_context_limit = 0;          // 0 = ask the server
    std::string tool_format = "auto";                // overrides workspace tool_format on activation
    std::map<std::string,std::string> extra_headers; // future-proofing, empty default
    bool        builtin = false;                     // seed profiles; survive store regeneration
};
```

`EndpointProfileStore` (same file) owns load/save against the global path, exposes `list() / find(name) / add / update / remove / save()`, and seeds six rows on first open if the file is absent. All six are OpenAI-compatible -- the `api_key` + `extra_headers` plumbing on `LLMConfig` / `OpenAiTransport` is the entire transport layer; no new client classes.

| Name | base_url | Key required | Default model | tool_format | Notes |
|---|---|---|---|---|---|
| **LM Studio (local)** | `http://127.0.0.1:1234` | no | empty (server default) | `auto` | Default active. |
| **Ollama (local)** | `http://127.0.0.1:11434/v1` | no | empty (server default -- `ollama run <model>` sets it) | `auto` | Ollama's OpenAI-compat shim at `/v1/*`. Tool-calling support depends on the loaded model. |
| **NVIDIA Build (hosted)** | `https://integrate.api.nvidia.com/v1` | yes | `qwen/qwen3-coder-480b-a35b-instruct` | `openai` | Free tier with rate limits. |
| **OpenAI (hosted)** | `https://api.openai.com/v1` | yes | empty (user picks: `gpt-4o-mini` / `gpt-4o` / `o1` / etc.) | `openai` | Paid. |
| **OpenRouter (hosted)** | `https://openrouter.ai/api/v1` | yes | empty (single account fronts hundreds of models -- user picks) | `auto` | Paid. Optional `extra_headers={"HTTP-Referer":"...", "X-Title":"Locus"}` for OpenRouter's leaderboard / per-app analytics; the API works without them. |
| **Claude via OpenAI-compat proxy** | `http://127.0.0.1:8082/v1` (user adjusts) | proxy-dependent | empty | `claude` | Requires the user to run LiteLLM / `claude-code-proxy` / equivalent locally. Native Anthropic `/v1/messages` is parked as a backlog draft ([claude-native-api.md](../backlog/claude-native-api.md)). |

All six carry `builtin=true`. Builtins can be edited (key, model, base_url, extra_headers) but not deleted; the row stays in the store even if all fields are cleared (the user can hide it by clearing the name -- the UI filters empty names).

### Persistence

- File path resolved via a new `endpoints_path()` free function: `%APPDATA%/Locus/endpoints.json` on Windows; `${XDG_CONFIG_HOME:-~/.config}/locus/endpoints.json` on Unix. Same dir family as the existing global MCP / prompt-templates dirs.
- JSON shape:
  ```json
  {
    "version": 1,
    "active": "LM Studio (local)",
    "profiles": [
      {"name":"LM Studio (local)", "base_url":"http://127.0.0.1:1234", "api_key":"", "default_model":"", "tool_format":"auto", "builtin":true},
      {"name":"NVIDIA Build (hosted)", "base_url":"https://integrate.api.nvidia.com/v1", "api_key":"nvapi-...", "default_model":"qwen/qwen3-coder-480b-a35b-instruct", "tool_format":"openai", "builtin":true}
    ]
  }
  ```
- `active` lives in the global file (not `WorkspaceConfig`) by default so a new workspace inherits "what I was using". `WorkspaceConfig::active_endpoint_name` overrides per-workspace when set (empty string = follow global).
- Write is atomic temp+rename (reuse `tools::write_atomic`).
- Loud-warn (once per session) if file permissions are world-readable on Unix; no-op on Windows (NTFS default ACL is already user-only in `%APPDATA%`).

### Legacy migration

First run with no `endpoints.json`: synthesise the three seed profiles. If `WorkspaceConfig::llm_endpoint` is non-default (i.e. not `http://127.0.0.1:1234`), also synthesise a fourth profile `Migrated (<workspace-name>)` carrying the legacy `llm_endpoint` / `llm_model` / `llm_tool_format` so the user's pre-S6.16 setup keeps working. Set `WorkspaceConfig::active_endpoint_name` to that migrated profile. Subsequent runs: the WorkspaceConfig legacy fields are still honoured as **per-workspace overrides on top of the active profile** -- if both `active_endpoint_name` and a non-default `llm_endpoint` are set, the legacy fields win (one-way ratchet so we can drop the legacy fields cleanly in a future stage).

### LLMConfig + transport

[`src/llm/llm_client.h`](../../src/llm/llm_client.h) `LLMConfig` gains:

```cpp
std::string api_key;                                    // empty -> no Authorization header
std::map<std::string,std::string> extra_headers;        // appended verbatim to the cpr::Header
```

[`src/llm/openai_transport.cpp`](../../src/llm/openai_transport.cpp) builds its `cpr::Header` block from config:

```cpp
cpr::Header headers{
    {"Content-Type", "application/json"},
    {"Accept",       "text/event-stream"}
};
if (!config_.api_key.empty())
    headers.emplace("Authorization", "Bearer " + config_.api_key);
for (auto& [k,v] : config_.extra_headers)
    headers.emplace(k, v);
```

Key is **not** logged at any level -- the existing info-level "LLM POST" trace already redacts headers; assert in code review that a future stack-trace dump doesn't leak `config_.api_key`.

### Resolution path (`LocusSession::load_shared_resources`)

Current code layers: defaults -> `.locus/config.json` -> caller seed -> live `query_model_info`. After S6.16:

1. Defaults (unchanged).
2. **Active profile** resolved via `EndpointProfileStore::find(workspace_config.active_endpoint_name or store.active())`. If the named profile is missing (deleted out-of-band), fall back to the first profile and log a warning.
3. Profile contributes `base_url` / `api_key` / `default_model` / `default_context_limit` / `tool_format` / `extra_headers` into `LLMConfig`.
4. `.locus/config.json` legacy `llm_*` fields override (back-compat).
5. Caller seed.
6. Live `query_model_info` (unchanged).

S6.10 Task F's `find_preset_for_model` keeps running at the same point in the chain.

### Hot-swap on `AgentCore`

New method on [`ILocusCore`](../../src/core/frontend.h):

```cpp
virtual void request_endpoint_switch(std::string profile_name) = 0;
```

Implementation in `AgentCore`:

- Queue a `SwitchEndpoint` message onto the agent thread (same mechanism as `process_message` / `request_commit_now`).
- At drain time, if `current_turn_in_flight_`, defer the switch to end-of-turn (don't refuse -- the user clicked the chip while waiting on a stuck round; they meant it for the next turn). Log `endpoint_switch deferred until turn complete`.
- Construct the new `ILLMClient` via the factory, replace `llm_` atomically (the agent thread is the sole user of `llm_`), invoke `query_model_info()` once to refresh context-limit / preset-detect, broadcast `IFrontend::on_endpoint_changed(name, model, context_limit)`.
- The S4.F byte-stable system-prompt invariant is **per-endpoint**. Switching endpoints rebuilds `SystemPromptAssembly` on the next turn; this is correct behaviour (different server, different prefix cache anyway). The invariant within an endpoint-bound run is unchanged.

### ChatPanel footer chip

[`src/frontends/gui/chat_panel.h`](../../src/frontends/gui/chat_panel.h): new `wxChoice* endpoint_choice_` adjacent to `preset_choice_`. Named widget `locus.chat.endpoint_choice`.

- Populated from `EndpointProfileStore::list()` at construction; appended sentinel row `"Edit endpoints..."` at the bottom.
- Selection change with a normal row -> `on_endpoint_pick_(profile_name)` callback (wired by `LocusFrame` to the active tab's `ILocusCore::request_endpoint_switch`).
- Selection change with the sentinel row -> opens Settings dialog at the Endpoints tab; selection reverts to the previously active row.
- Tooltip shows `base_url` + masked key suffix (`****abcd`) so the user can verify which profile is wired.
- The choice does NOT show key contents; a missing-key profile renders with a yellow chip colour (`*` suffix on the label) the same way `preset_choice_` colour-cues runtime overrides.

### Settings dialog: Endpoints tab

New `src/frontends/gui/settings/endpoints_settings_panel.h/cpp` implementing `ISettingsPanel`:

- `wxListCtrl` with columns `Name | URL | Model | Key`; key column shows `(none)` / `****abcd`.
- Buttons: `Add` / `Edit` / `Remove` / `Set Active`. Builtin rows have `Remove` greyed out.

## Notes

- **Anthropic native** (`/v1/messages`, content blocks, `tool_use`/`tool_result`) is **not** in scope. The decision to route Claude through an OpenAI-compat proxy keeps this stage at "config + UI" size; a separate stage can introduce `AnthropicClient : ILLMClient` + a native SSE decoder if proxies prove insufficient. Add to backlog at stage-close if interest persists.
- **Per-tab endpoint binding** was considered and rejected: the chip lives on the chat panel but the binding is per-tab (each tab has its own `AgentCore`), so different tabs can already be on different endpoints if the user switches the chip after activating a tab. We don't need a separate "tab inherits endpoint at open" mechanism.
- **`LlmRouter`** (S4.Q hook, parked) overlaps conceptually -- it imagined a strong/weak split where Locus auto-selects. That's a separate concern: this stage gives the user **manual control over which single endpoint is active**, leaving the router as a future consumer of the same `EndpointProfileStore`. Don't try to wire both at once.
- **Per-endpoint model presets** (S6.10 Task F's `find_preset_for_model`) -- already keyed on model id, not endpoint, so the existing regex matcher works unchanged. The auto-detect re-runs on endpoint switch because `query_model_info` runs again. No new code path.
- **Streaming decoder format auto-detect** -- `tool_format=auto` keeps working across endpoints. NVIDIA hosting Qwen3-Coder emits standard OpenAI JSON tool_calls, so `auto` resolves the same way as for LM Studio. The per-profile `tool_format` override is there for the cases where the user knows better (Claude-via-proxy benefits from `claude`).
- **Key rotation** is manual: edit the profile in Settings, save. No auto-refresh, no OAuth flows -- those belong to a hypothetical future stage if hosted providers move to short-lived tokens.
- **Cost / quota awareness** is not in scope. Hosted providers have rate limits and budgets the user has to track themselves; surfacing them belongs in a future Metrics-tab extension.
- **Tool-call format mismatch on switch**: if the user switches from a Qwen-XML-emitting endpoint to one that expects OpenAI JSON, the previous turn's `assistant` message in history still contains the old tool-call shape. This is benign -- past `tool_calls` arrays serialise the same way regardless of endpoint; only future turns honour the new format. Document but don't engineer around.
- **UIA tab-index drift (2026-06-07 post-stage fix).** Inserting the "Endpoints" tab at index 1 ([settings_dialog.cpp](../../src/frontends/gui/settings_dialog.cpp) `AddPage` order: LLM=0, Endpoints=1, Index=2, Agent=3, Capabilities=4, Tool Approvals=5, MCP Servers=6, Notifications=7, Sessions=8) shifted every later tab by +1, silently breaking UIA scripts that `select_tab` the settings notebook by index. `mcp_open_json` / `mutating_tool_friction` / `permission_presets_settings` / `prompt_cost_and_watchdog_settings` / `settings_tour` / `capabilities_first_open` were re-pinned to the new indices. `select_tab` by `name` was tried first but wx's UIA tree does NOT expose the page title as the TabItem Name, so index is the only working selector. **Any future stage that inserts or reorders a Settings tab MUST re-pin these scripts' `select_tab` indices.** Separately, `plan_mode.json` now sets `LOCUS_GLOBAL_DIR={workspace}/global` so the GUI gets the seeded-default active endpoint (LM Studio local) instead of inheriting whatever hosted endpoint the developer's global `endpoints.json` last had active (an NVIDIA Build stall caused a false plan_mode failure) -- the isolation pattern any LLM-dependent UIA script should follow.
