## Tasks

### Task A -- Index storage hygiene

Was Task L in S6.17 (originally L.1-L.5) plus the F2 WAL-checkpoint polish item from S6.17's K bucket. Six small fixes targeting the same outcome: keep `vectors.db` proportional to actual content, stop dropping embeddings on rowid-reuse races, stop leaking WAL bytes.

The 2026-05-26 follow-up investigation against the TestLocalVibe2 vectors.db found two distinct bloat sources stacked: a 4 MiB vec0 bucket allocated up front, and 238 chunks for a 3-file workspace with 220 overlapping at least one other in the same file. Root cause for the chunk bloat is in `chunk_code` ([src/index/chunker.cpp](../../src/index/chunker.cpp)): the symbol extractor emits 112 symbols for big_module.cpp with 111 nested (namespace > class > methods), and the chunker has two bugs that interact with nested input -- degenerate chunks when the cursor-clamp inverts the range, and cursor moving backward when a nested symbol ends earlier than the cursor.

#### A.1 -- Lazy vec0 table creation

- [ ] [`src/core/database.cpp`](../../src/core/database.cpp) `Database::ensure_vectors_schema(dim)`: defer the `CREATE VIRTUAL TABLE chunk_vectors USING vec0(...)` statement (and the `memory_vectors` equivalent) until the first insert. Today's "create on workspace open" path pays 4 MiB + 4 MiB = 8 MiB up front; lazy creation drops both to 0 for workspaces with no embeddings yet.
- [ ] Implementation: track schema state with a `bool vectors_table_present_` member; the first `INSERT INTO chunk_vectors` (in `embedding_worker.cpp`) calls a new `Database::ensure_chunk_vectors_table()` that issues the CREATE if the flag is false. Same shape for `memory_vectors` via `Database::ensure_memory_vectors_table()`. Read paths short-circuit to "no results" if the flag is false (the existing query path's "table doesn't exist" branch should already handle this; verify and add a flag check if cleaner).
- [ ] The `meta` table's `vector_dim` row continues to be written on `ensure_vectors_schema(dim)` -- that's the dim-mismatch wipe trigger and must stay eager so a model swap is detected at workspace open, not at first embed.
- [ ] Acceptance: open an empty workspace with `semantic_search.enabled=true`; `vectors.db` on disk after first launch is < 100 KB (today: 4.0+ MiB). Index a single small file; `vectors.db` grows to the expected size after the embedding worker drains.

#### A.2 -- `chunk_code` defensive walk

- [ ] [`src/index/chunker.cpp`](../../src/index/chunker.cpp) `chunk_code` loop: after the `sym_begin < cursor -> sym_begin = cursor` clamp, add the inverse check `if (sym_begin >= sym_end) continue;` -- drops nested symbols entirely instead of emitting a degenerate `start_line > end_line, content=""` chunk and moving the cursor backward.
- [ ] Unit test `tests/test_chunker.cpp` `[s6.18]`: input with a namespace spanning the whole file + several nested classes + nested methods. Verify no chunk has `start_line > end_line`, no chunk has empty `content`, and cursor monotonicity holds.

#### A.3 -- Leaf-symbol pre-filter in indexer

- [ ] [`src/index/indexer.cpp`](../../src/index/indexer.cpp) around the `if (!symbol_spans.empty())` branch before `chunk_code`: pre-filter `symbol_spans` to drop spans that fully contain another span. O(n^2) sweep is fine -- n is typically <500 per file. Shape:
  ```cpp
  std::vector<SymbolSpan> leaf_spans;
  for (size_t i = 0; i < symbol_spans.size(); ++i) {
      const auto& s = symbol_spans[i];
      bool contains_another = false;
      for (size_t j = 0; j < symbol_spans.size(); ++j) {
          if (i == j) continue;
          const auto& other = symbol_spans[j];
          if (s.line_start <= other.line_start && s.line_end >= other.line_end
              && (s.line_start < other.line_start || s.line_end > other.line_end)) {
              contains_another = true;
              break;
          }
      }
      if (!contains_another) leaf_spans.push_back(s);
  }
  chunks = chunk_code(content, leaf_spans, ...);
  ```
- [ ] Acceptance: re-index TestLocalVibe2 against the fixed code. `SELECT COUNT(*) FROM chunks WHERE file_id=3` drops from ~230 to ~100. `SELECT COUNT(*) FROM chunks WHERE LENGTH(content) = 0` is 0. Overlap rate falls to <10% (a small natural overlap from the existing `overlap` parameter is expected).
- [ ] Side benefit: embedder workload halves on code-heavy files.

#### A.4 -- Skip empty chunks defensively

- [ ] [`src/index/chunker.cpp`](../../src/index/chunker.cpp) `make_chunk`: refuse to emit a chunk where `begin_idx >= end_idx`. Returns `std::nullopt`. Belt-and-suspenders behind A.2.
- [ ] [`src/index/indexer.cpp`](../../src/index/indexer.cpp) chunk-insert loop: skip any chunk with `content.empty()` before persisting. Belt-and-suspenders behind A.3.
- [ ] Unit test in `tests/test_chunker.cpp` `[s6.18]`: feed `make_chunk` with `begin >= end` -- expects an empty optional, never a stored row.

#### A.5 -- `INSERT OR REPLACE` for chunk_vectors

Surfaced 2026-05-27 during the full integration run (74/74 passed; the warning fired 3 times for chunk_ids 21221 / 21397 / 21401). `chunks.id` is `INTEGER PRIMARY KEY` *without* `AUTOINCREMENT`, so SQLite reuses freed rowids after `DELETE FROM chunks WHERE file_id=N`. When a file is rewritten faster than the embedding worker drains, new chunks land on the same rowids as the old ones, and the second `INSERT INTO chunk_vectors` collides with the first. Silent-drops the second embedding -- the rare-but-real correctness bug is that the embedding for the *current* chunk content is the one lost.

- [ ] [`src/index/embedding_worker.cpp`](../../src/index/embedding_worker.cpp) line ~95: change `INSERT INTO chunk_vectors` to `INSERT OR REPLACE INTO chunk_vectors`. Keeps the embedding for the current chunk content.
- [ ] Demote the warning at line ~150 (`Failed to insert vector for chunk N: UNIQUE constraint failed`) to trace level, OR drop it (the OR REPLACE path should never trigger it; other SQLite errors should still surface).
- [ ] Optional follow-up: `EmbeddingWorker::enqueue` deduplicates against a `std::unordered_set<int64_t> in_queue_`. Not required -- OR REPLACE makes the redundant work harmless -- but saves an embedder forward pass per duplicate.
- [ ] Unit test `tests/test_embedding_worker.cpp` `[s6.18]`: enqueue the same chunk_id twice with different content blobs; assert no warning fires and the row in `chunk_vectors` carries the *second* content's embedding.

#### A.6 -- WAL checkpoint on drain + shutdown

- [ ] [`src/index/embedding_worker.cpp`](../../src/index/embedding_worker.cpp): when the queue drains, run `PRAGMA wal_checkpoint(PASSIVE)` on the vectors.db connection. Trace `EmbeddingWorker: wal_checkpoint after drain (frames=X)`.
- [ ] [`src/index/indexer.cpp`](../../src/index/indexer.cpp) `Indexer::shutdown()` (or the dtor): same checkpoint on the main.db connection.
- [ ] Acceptance: open + index + close TestLocalVibe2; `vectors.db-wal` and `index.db-wal` on disk after clean shutdown are 0 bytes (today: 4.5 MB on vectors.db-wal).

## Implementation notes

- Task A's lazy-create gate must be per-connection, not process-global.
- Task A.5's `INSERT OR REPLACE` is the simplest fix; the alternative (AUTOINCREMENT on `chunks.id`) would require schema migration + chunk_vectors rowid remap. Strictly worse trade-off for the same outcome.
- Task A.6's `PASSIVE` checkpoint is the right level -- never blocks readers / writers, opportunistically commits whatever pages it can.
- Task B.1's "preserved set" is the cleanest API for the pin -- pass an `unordered_set<int> preserved_history_ids` through the pipeline layer protocol so each layer's strip path can skip preserved messages without each layer reimplementing the rule.
- Task C.3's `IFrontend::on_compaction_archived` signature widening is the same pattern as the S6.13 `on_round_progress` addition -- one new argument, default-constructed for tests that don't care, forwarded through `WxFrontend`.
- Task D.1 -- the per-tick delta tracking is best implemented as a single atomic shadow counter that records the previous total; the tick reads `current - shadow` and updates `shadow = current`. Single atomic load + store per tick.
- Task F's canonicalisation should be one-direction (recommend lowercase forward-slash everywhere keyed) and applied at every insert site + every lookup site. A new helper `index_paths::canonicalize(rel)` in `src/index/` keeps the rule in one file.
