## Tasks

### Task A -- Index storage hygiene

Was Task L in S6.17 (originally L.1-L.5) plus the F2 WAL-checkpoint polish item from S6.17's K bucket. Six small fixes targeting the same outcome: keep `vectors.db` proportional to actual content, stop dropping embeddings on rowid-reuse races, stop leaking WAL bytes.

The 2026-05-26 follow-up investigation against the TestLocalVibe2 vectors.db found two distinct bloat sources stacked: a 4 MiB vec0 bucket allocated up front, and 238 chunks for a 3-file workspace with 220 overlapping at least one other in the same file. Root cause for the chunk bloat is in `chunk_code` ([src/index/chunker.cpp](../../src/index/chunker.cpp)): the symbol extractor emits 112 symbols for big_module.cpp with 111 nested (namespace > class > methods), and the chunker has two bugs that interact with nested input -- degenerate chunks when the cursor-clamp inverts the range, and cursor moving backward when a nested symbol ends earlier than the cursor.

#### A.1 -- Lazy vec0 table creation

- [x] [`src/core/database.cpp`](../../src/core/database.cpp) `Database::ensure_vectors_schema(dim)`: defer the `CREATE VIRTUAL TABLE chunk_vectors USING vec0(...)` statement (and the `memory_vectors` equivalent) until the first insert. Today's "create on workspace open" path pays 4 MiB + 4 MiB = 8 MiB up front; lazy creation drops both to 0 for workspaces with no embeddings yet.
- [x] Implementation: track schema state with a `bool vectors_table_present_` member; the first `INSERT INTO chunk_vectors` (in `embedding_worker.cpp`) calls a new `Database::ensure_chunk_vectors_table()` that issues the CREATE if the flag is false. Same shape for `memory_vectors` via `Database::ensure_memory_vectors_table()`. Read paths short-circuit to "no results" if the flag is false (the existing query path's "table doesn't exist" branch should already handle this; verify and add a flag check if cleaner).
- [x] The `meta` table's `vector_dim` row continues to be written on `ensure_vectors_schema(dim)` -- that's the dim-mismatch wipe trigger and must stay eager so a model swap is detected at workspace open, not at first embed.
- [x] Acceptance verified 2026-05-28 (bge-m3, dim=1024): empty WS `vectors.db` = **32 KB** (target <100 KB; pre-fix 4 MiB+). Log line `(chunk_vectors lazy, memory_vectors lazy)` confirms tables not created. Adding one small `.cpp` file then re-opening: `vectors.db` grows to 4.2 MiB after the worker drains; log shows `Created chunk_vectors table (dim=1024)` fired exactly once at first INSERT.

#### A.2 -- `chunk_code` defensive walk

- [x] [`src/index/chunker.cpp`](../../src/index/chunker.cpp) `chunk_code` loop: after the `sym_begin < cursor -> sym_begin = cursor` clamp, add the inverse check `if (sym_begin >= sym_end) continue;` -- drops nested symbols entirely instead of emitting a degenerate `start_line > end_line, content=""` chunk and moving the cursor backward. Also enforced cursor monotonicity (`if (sym_end > cursor) cursor = sym_end;`) so a same-line trailing leaf can't pull the cursor back.
- [x] Unit test [`tests/test_chunker_s618.cpp`](../../tests/test_chunker_s618.cpp) `[s6.18]`: degenerate-span, nested namespace + classes, and overlapping-spans cases. All assert no `start_line > end_line`, no empty `content`, and cursor monotonicity holds.

#### A.3 -- Leaf-symbol pre-filter in indexer

- [x] [`src/index/indexer.cpp`](../../src/index/indexer.cpp) around the `if (!symbol_spans.empty())` branch before `chunk_code`: pre-filter `symbol_spans` to drop spans that fully contain another span. O(n^2) sweep is fine -- n is typically <500 per file. Shape:
  ```cpp
  std::vector<SymbolSpan> leaf_spans;
  for (size_t i = 0; i < symbol_spans.size(); ++i) {
      const auto& s = symbol_spans[i];
      bool contains_another = false;
      for (size_t j = 0; j < symbol_spans.size(); ++j) {
          if (i == j) continue;
          const auto& other = symbol_spans[j];
          if (s.line_start <= other.line_start && s.line_end >= other.line_end
              && (s.line_start < other.line_start || s.line_end > other.line_end)) {
              contains_another = true;
              break;
          }
      }
      if (!contains_another) leaf_spans.push_back(s);
  }
  chunks = chunk_code(content, leaf_spans, ...);
  ```
- [x] Acceptance verified 2026-05-28 against WS1 (the project's own code -- symbol-heavy, equivalent shape to TestLocalVibe2): `vectors.db` post-reindex is **10 MiB** (down from 23 MiB pre-S6.18). Total chunks 7440 across 672 files. **0** empty-content chunks. **0** inverted-range chunks. Aggregate overlap rate **9.3%** (target <10%). Per-file worst cases: agent_core.cpp 18.3% (60 leaf symbols, expected sliding-window boundary overlap), sqlite-vec.c 14.4% (single flat C file -- vendored third-party code). TestLocalVibe2 corpus-level numbers to be confirmed in the next agentic run.
- [x] Side benefit: embedder workload halves on code-heavy files.

#### A.4 -- Skip empty chunks defensively

- [x] [`src/index/indexer.cpp`](../../src/index/indexer.cpp) chunk-insert loop: skip any chunk with `content.empty()` before persisting. Belt-and-suspenders behind A.3.
- [x] Unit test in [`tests/test_chunker_s618.cpp`](../../tests/test_chunker_s618.cpp) `[s6.18]`: empty-content sliding window + high-overlap stress -- both assert no empty chunks slip through.

#### A.5 -- `INSERT OR REPLACE` for chunk_vectors

Surfaced 2026-05-27 during the full integration run (74/74 passed; the warning fired 3 times for chunk_ids 21221 / 21397 / 21401). `chunks.id` is `INTEGER PRIMARY KEY` *without* `AUTOINCREMENT`, so SQLite reuses freed rowids after `DELETE FROM chunks WHERE file_id=N`. When a file is rewritten faster than the embedding worker drains, new chunks land on the same rowids as the old ones, and the second `INSERT INTO chunk_vectors` collides with the first. Silent-drops the second embedding -- the rare-but-real correctness bug is that the embedding for the *current* chunk content is the one lost.

- [x] [`src/index/embedding_worker.cpp`](../../src/index/embedding_worker.cpp): the INSERT is now `INSERT OR REPLACE INTO chunk_vectors`. Keeps the embedding for the current chunk content. Lazy-prepared so an empty workspace pays nothing.
- [x] Warning kept (now reads "Failed to insert vector for chunk N: <err>" -- with OR REPLACE the UNIQUE-constraint silent-drop path is closed; any remaining error is real and worth surfacing). Verified 2026-05-28: full re-index of WS1 produced **0** `UNIQUE constraint failed` warnings (vs 183 historical occurrences on May 13-18 pre-fix).
- [ ] Optional follow-up: `EmbeddingWorker::enqueue` deduplicates against a `std::unordered_set<int64_t> in_queue_`. Not required -- OR REPLACE makes the redundant work harmless -- but saves an embedder forward pass per duplicate. (Skipped intentionally; OR REPLACE makes the redundant work harmless.)
- [ ] Unit test `tests/test_embedding_worker.cpp` `[s6.18]`: enqueue the same chunk_id twice with different content blobs; assert no warning fires and the row in `chunk_vectors` carries the *second* content's embedding. (Deferred -- needs a real embedder fixture; the SQL behaviour is straightforward and covered by manual smoke.)

- [x] [`src/index/indexer.cpp`](../../src/index/indexer.cpp) `Indexer::~Indexer`: checkpoint both main.db and vectors.db on workspace close.

## Implementation notes

- Task A's lazy-create gate must be per-connection, not process-global.
- Task A.5's `INSERT OR REPLACE` is the simplest fix; the alternative (AUTOINCREMENT on `chunks.id`) would require schema migration + chunk_vectors rowid remap. Strictly worse trade-off for the same outcome.
- Task A.6's `PASSIVE` checkpoint is the right level -- never blocks readers / writers, opportunistically commits whatever pages it can.
- Task B.1's "preserved set" is the cleanest API for the pin -- pass an `unordered_set<int> preserved_history_ids` through the pipeline layer protocol so each layer's strip path can skip preserved messages without each layer reimplementing the rule.
- Task C.3's `IFrontend::on_compaction_archived` signature widening is the same pattern as the S6.13 `on_round_progress` addition -- one new argument, default-constructed for tests that don't care, forwarded through `WxFrontend`.
- Task D.1 -- the per-tick delta tracking is best implemented as a single atomic shadow counter that records the previous total; the tick reads `current - shadow` and updates `shadow = current`. Single atomic load + store per tick.
- Task F's canonicalisation should be one-direction (recommend lowercase forward-slash everywhere keyed) and applied at every insert site + every lookup site. A new helper `index_paths::canonicalize(rel)` in `src/index/` keeps the rule in one file.
