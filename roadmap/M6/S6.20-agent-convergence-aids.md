# S6.20 -- Agent Convergence Aids (stuck-error detector, edit-over-rewrite nudge, self-correcting tool errors, retry UI)

**Done 2026-06-07.** Four agent-productivity fixes, all surfaced by the same
agentic-mode session as [S6.19](S6.19-tool-arg-coercion-and-endpoint-retry.md)
(driving `qwen/qwen3-coder-480b` on the NVIDIA Build hosted endpoint through the
DX12 + DXR app build, findings in
[tests/ui_automation/output/agentic_TestQwenVibeGI/findings.md](../../tests/ui_automation/output/agentic_TestQwenVibeGI/findings.md)).
S6.19 closed the *crash / abort* holes; S6.20 closes the *convergence* gap --
the model authored good code but looped 38-72 rounds on a large file without
ever converging, with nothing telling it (or the user) it was stuck.

## Problem (findings.md "Key product finding")

On the DX12 build the agent repeatedly:
- rewrote a 27-61 KB `main.cpp` wholesale every round via `write_file overwrite`,
  reintroducing the same compile errors each time (token-heavy -> more
  empty-response / 429 exposure; error-reintroducing);
- hit the SAME compiler error (e.g. `C2102: '&' requires l-value`, `CD3DX12_*
  not found`) round after round with no signal it was looping -- the existing
  repeated-tool-call QualityMonitor (S6.10) never fired because the *calls*
  differed each round (different rewrites), only the resulting *error* was the
  same;
- surfaced opaque "tool failed with an internal error" results on a mistyped
  arg, so the model re-sent the same call and tripped the nudge cap;
- went silent for minutes during endpoint backoff (S6.19's retry) with no UI
  indication, so a watching human couldn't tell thrash from progress.

## Tasks

### 1. "Stuck on the same build error" detector (highest impact)

New pure helper [src/agent/build_error_detector.{h,cpp}](../../src/agent/build_error_detector.h):
`extract_error_signature(tool_output)` distills compiler/linker diagnostics into
a signature that is invariant to the noise that changes round-to-round but is
irrelevant to "is this the same failure" -- line/column numbers, absolute paths,
hex addresses, counts ("3 unresolved externals"), and diagnostic ORDER (sorted +
de-duplicated). Error CODES are preserved (`C2102` stays `C2102`, not `C#`) so
different errors don't collide. Recognizes MSVC (`error C####` / `fatal error
LNK####`) and GCC/Clang (`path:line:col: error:`) shapes; ignores warnings (a
repeating warning is not a stuck signal).

`AgentTurnRunner` (per-turn state) aggregates each round's tool output, computes
the signature, and tracks a consecutive-identical streak. On the **3rd** repeat
it injects ONE steering nudge ("you've hit the same error N times -- stop
rewriting the whole file, read the failing lines and make a minimal edit"). If
the same signature recurs **after** the nudge, the turn aborts with "Agent
appears stuck on a repeating build error". A round with no build error resets the
streak. Reuses the existing `inject_nudge` 2-cap machinery (so it shares the cap
with the reasoning watchdog + other quality nudges); gated by
`agent.quality_monitor_enabled`.

### 2. Soft nudge: edit_file over whole-file rewrites

`AgentTurnRunner::note_write_call` flags when a dispatched `write_file
overwrite=true` carries a large body (>= ~16 KB / ~400 lines). When a build
error then lands, `maybe_hint_large_overwrite` injects a one-time-per-turn hint
toward `edit_file` for targeted fixes. Deliberately narrow: only fires on a
LARGE overwrite that is FOLLOWED by a build failure -- never interferes with a
legitimate one-shot full rewrite that compiles. Fires before the stuck-error
escalation, so it often heads the streak off early.

### 3. Self-correcting tool-error messages

[ToolDispatcher](../../src/agent/tool_dispatcher.cpp)'s execute() catch-all now
appends the tool's canonical schema (`tools::render_tool_schema`) to the error
when a tool throws. nlohmann's `type_error` what() ("type must be number, but is
string") is informative but doesn't name the arg or its expected shape; with the
schema attached the model fixes the arg type in one round instead of re-sending
the same mistyped call and tripping the repeated-call cap. (Companion to S6.19's
`coerce_*` helpers, which make the *common* mistypes succeed; this covers the
residual throwing cases.)

### 5. Surface retry/latency state in the footer

S6.19 added transient-failure retry but it was silent. New optional
`StreamCallbacks::on_status` (-> `OpenAiTransport::Callbacks::on_status`) fires
before each backoff with a short string ("HTTP 429 (rate limited) -- retrying
2/5 in 8s"). `AgentLoop` broadcasts it to frontends via the new
`IFrontend::on_llm_retry`; the GUI routes it (`EVT_AGENT_LLM_RETRY` ->
`AgentEventRouter` -> `ChatPanel::set_transient_status` ->
`ChatFooterChips::set_transient_status`) into the footer round-chip area, where
it is naturally superseded by the next round-progress / turn-complete event. No
new widget (footer-declutter-compatible); the agentic `get_chat_status` scrape
picks it up via the round chip. CLI inherits the IFrontend default no-op.

## Tests

- [tests/test_build_error_detector.cpp](../../tests/test_build_error_detector.cpp)
  tag `[build_error_detector]` (18 assertions): empty/no-error -> empty;
  warnings ignored; MSVC + GCC/Clang + linker shapes captured; file/line/path
  noise stripped; SAME error at different lines / reordered diagnostics / count
  drift -> identical signature; DIFFERENT errors -> different signatures; error
  codes preserved.

  the footer plumbing) are integration-shaped (need a live multi-round turn);
  the pure detector is the unit-tested core. Full suite green (3590 assertions).
