# S5.P -- Source Encoding Audit & Lint

**Theme**: Hygiene / Tooling

**Status**: done

## Goal

Sweep the existing repo for em-dashes and other non-ASCII drift, then add a build-time check that keeps them out. The "no em-dashes anywhere" rule in CLAUDE.md is the project's own rule and is currently enforced by trust alone.

## Motivation

CLAUDE.md states:

> **No em-dashes anywhere.** Use ASCII `-` (or `--`) in code, comments, commit messages, docs, test names, and log strings. Em-dashes (`--`, U+2014) break ctest test-name dispatch on Windows (no CP437 mapping) and complicate grep/diff. This rule applies to anything written into the repo, including this file.

A scan of `src/` finds em-dashes still present in at least:

- [src/tools/shared.cpp:14](../../src/tools/shared.cpp#L14) -- comment
- [src/agent/tool_dispatcher.cpp:277](../../src/agent/tool_dispatcher.cpp#L277), [310](../../src/agent/tool_dispatcher.cpp#L310) -- comment + log string (the log string is the riskier one; emitted to `locus.log`, may surface in `/metrics` activity events)
- [src/tools/file_tools.cpp:93](../../src/tools/file_tools.cpp#L93), [124](../../src/tools/file_tools.cpp#L124), [333](../../src/tools/file_tools.cpp#L333), [348](../../src/tools/file_tools.cpp#L348) -- comments + one user-visible error string ("no changes written (edits are atomic)" carries an em-dash that the LLM will see in tool result content)

62 source files in `src/` contain at least one non-ASCII byte. Many are deliberate UI glyphs in `chat_panel.cpp`'s HTML/CSS, but the bulk is comment + log drift.

The risk surfaces in three places:

1. **Tool result strings**: anything that flows into `ConversationHistory` is sent to the LLM. Mojibake in the tool error message poisons the next round if LM Studio's chat template can't render it (`charmap` codec failures in the Python-based templates do happen).
2. **CTest output**: test names with em-dashes get mangled on Windows (the CLAUDE.md rule's stated original justification).
3. **Patch/diff noise**: `git diff` shows em-dashes as multi-byte sequences in any console with a non-UTF-8 active codepage (CP437/CP1252 default on many Windows terminals); reviewers see garbage where there should be text.

## Design

Two phases: cleanup and enforcement.

### Phase 1 -- one-shot cleanup

A small Python or PowerShell script ([scripts/strip_em_dashes.ps1](../../scripts/strip_em_dashes.ps1), new) walks `src/` + `tests/` + `architecture/` + `roadmap/` + `*.md` and rewrites:

- U+2014 (em dash) -> `--`
- U+2013 (en dash) -> `-`
- U+2192 (right arrow) -> `->`
- U+2190 (left arrow) -> `<-`
- U+2026 (ellipsis) -> `...`
- U+2018 / U+2019 (curly quotes single) -> `'`
- U+201C / U+201D (curly quotes double) -> `"`
- U+00A0 (non-breaking space) -> ` `

Files outside that allow-list (binary models, vendored grammar `parser.c` files under `build/`) are skipped. UI glyphs that genuinely need to render in chat (the streaming-cursor character in `chat_panel.cpp`'s CSS, the activity-log status icons) are kept on a per-file allow-list inside the script.

Run, review the diff, commit as a single sweep commit. Touched-file count is expected to be small (under 70) and largely comment-only.

### Phase 2 -- build-time lint

New CMake `scripts/check_no_em_dashes.cmake` runs as a `POST_BUILD` step on `locus_core` (cheap; only re-runs when sources change). For each `.h` / `.cpp` / `.md` file in the project source tree, scan for the same character set. If any are found outside the allow-list, fail the build with a clear message:

```
ERROR: 3 source file(s) contain non-ASCII characters that are not on the allow-list:
  src/tools/shared.cpp:14 -- U+2014 (em dash)
  src/agent/tool_dispatcher.cpp:310 -- U+2014 (em dash)
  src/agent/tool_dispatcher.cpp:277 -- U+2014 (em dash)
See CLAUDE.md "No em-dashes anywhere." for the rationale.
```

The check is deliberately a CMake step rather than a git pre-commit hook -- `git config core.hooksPath` is not portable, and the user has explicitly asked for hooks not to be skipped. CMake POST_BUILD runs locally on every build and on CI.

Allow-list is a small text file ([scripts/non_ascii_allowlist.txt](../../scripts/non_ascii_allowlist.txt)): one entry per line, `<file_path>:<U+codepoint>` form. Bare path matches every codepoint in that file (used for `chat_panel.cpp`'s HTML/CSS UI text).

### Phase 3 (optional, deferred to S5.Z) -- editor hint

A `.editorconfig` with `charset = utf-8` and a `.vscode/settings.json` snippet warning against curly quote auto-pair rules. Out of scope for the stage core because the build-time lint already catches what the editor lets through.

## Tests

- New `tests/test_source_hygiene.cpp` tagged `[s5.p]`: parses the allow-list and asserts each entry matches a real file. Catches stale allow-list entries when files are renamed or deleted.
- The CMake check itself is the primary test surface -- the build refuses to complete if a violation slips in.
- Run `locus_tests --list-tests` after the cleanup sweep -- no test name should contain a non-ASCII byte (covers the CTest dispatch concern that motivated the original rule).

## Out of scope

- Per-file source encoding declaration (BOM vs no BOM). MSVC handles either; the project doesn't standardize today and the lint doesn't need it to.
- Non-ASCII bytes in workspace test fixtures that the indexer reads (those exercise the extractors and should stay intentionally non-ASCII).
- Localization / i18n of UI strings -- separate concern, not in M5.

## Manual test plan

None required -- the build either passes or fails the lint. If the cleanup sweep accidentally rewrites a UI glyph that was meant to render literally, the chat-panel manual test ([tests/manual/inline-diffs-in-chat.md](../../tests/manual/inline-diffs-in-chat.md)) and the activity-log smoke surface the regression visually within minutes.
