# S5.L -- UI Automation Test Driver done

## Problem

Today the only way to verify GUI behaviour is a human running through one of the [tests/manual/](../../tests/manual/) plans. Unit tests cover `locus_core`; integration tests in [tests/integration/](../../tests/integration/) drive `AgentCore` end-to-end against a live LM Studio but never touch the wx widgets. There's a real gap: every chat-panel regression, every settings-dialog wiring slip, every AUI-layout breakage today only gets caught when a human notices it.

Claude (and the user) want a scripted way to drive [locus_gui.exe](../../src/frontends/gui/locus_app.cpp) -- launch it, find a control, click it, type into it, screenshot the result, assert state -- so smoke-level GUI regressions become catchable from a subprocess. The agent should be able to invoke the driver from `Bash`, read pass/fail from the exit code, and read screenshots + logs as artifacts.

## Why UIA, not the alternatives

Four routes were considered:

- **Windows UI Automation (UIA) via a small C++ driver** -- chosen. Exercises the real wx event path, ships zero new product code, and uses a well-trodden Windows API that's already the substrate underneath FlaUI and WinAppDriver. Cost: scripts are coupled to the desktop session, so this is a developer-machine / CI-runner thing, not a unit-test-time thing.
- **In-process `--ui-test <script>` flag inside `locus_gui`.** Cleanest determinism (no UIA timing flake) but introduces a new test-only code path in the GUI binary that has to stay working forever. Reserve for the day UIA flake forces our hand; not v1.
- **WinAppDriver / FlaUI.** Both wrap UIA under heavier dependency surfaces (Java + Appium server; .NET runtime). Going to the substrate directly is leaner and matches the project's no-runtime-dep stance.
- **Screenshot + vision model.** Slow, flaky, no determinism gate. Never the right regression substrate.

## Scope -- v1 prototype, not a full QA replacement

This stage delivers the driver + a minimum credible script suite. It explicitly does **not** replace the [tests/manual/](../../tests/manual/) plans -- those still cover what UIA structurally can't reach (drag-and-drop visuals, focus order, file-dialog interactions, third-party-software workflows).

- **Windows only.** Cross-platform UI testing is a separate problem (AT-SPI on Linux, `AXUIElement` on macOS). When Locus crosses platforms, revisit; until then UIA is the right tool.
- **Manual run, not in `ctest`.** Like the integration tests and the retrieval eval, the driver runs against a real desktop session and (for chat scripts) a live LM Studio. Same protocol as `tests/integration/`: explicit on user request, never per-build.
- **Three demo scripts ship in v1.** Smoke (launch + main window appears + screenshot + clean quit), settings-dialog tour (open dialog, click each of the four tabs, screenshot each, close), and chat round-trip (type a message, hit send, assert it appears in the chat WebView). Anything beyond these is post-v1; the goal is to prove the harness, not to back-fill coverage in one stage.

## Implementation notes

- **UIA timing**: never `sleep_for(N)` blindly. UIA exposes property-change events and a structural change handler; the driver's `wait_for(predicate, timeout_ms)` polls at 50 ms with an exponential back-off. Explicit waits keep scripts fast on a quick machine and stable on a slow one. Default per-step timeout: 5 s.
- **Focus stealing**: a UIA script driving the real desktop pops windows on the developer's screen, steals focus, and is generally obnoxious. The README documents the workaround: run inside a separate Windows session (`runas /user:locustest`) or an RDP loopback session. Github Actions Windows runners have a desktop session out of the box, so CI is plausible later.
- **Test isolation**: each script launches `locus_gui.exe` against a temp workspace dir under `%TEMP%\locus_ui_tests\<script>\` (created in `setup`, removed in `cleanup`). No shared state across scripts. The driver passes the workspace path on the command line so the GUI doesn't show the workspace picker.
- **wxAccessibility coverage caveats** worth flagging up front:
  - `wxAuiNotebook` tabs surface as UIA `TabItem` controls -- works.
  - `wxStyledTextCtrl` (Scintilla) has limited UIA support. For the tool approval panel's JSON / diff view, fall back to reading via `SendMessage`/`SCI_GETTEXT` through the HWND if the v1 scripts need it. None of the three demo scripts hit this path, so it stays a documented limit, not a blocker.
  - `wxWebView` (Edge backend) exposes the full DOM as UIA -- the chat-round-trip script depends on this; verify early.
- **JSON script format vs a real DSL**: JSON is verbose but free (no parser to write). The driver embeds [nlohmann/json](../../src/) -- already a vcpkg dep elsewhere in the project. If scripts grow past ~30 each, revisit; for v1 the verbosity is fine.
- **Why no `--mock-llm` mode for chat tests in v1**: building a stub LLM is a separate subsystem with its own scoping conversation. Until that exists, the chat-round-trip script asserts the **user-side** of the round-trip only (user message renders into the chat view), which doesn't need an LLM response. Assertions about agent replies stay in [tests/integration/](../../tests/integration/) where real model behaviour is what's being checked anyway.
- **Subprocess invocation contract**: Claude should be able to run `build/release/tests/ui_automation/Release/locus_ui_tests.exe --script tests/ui_automation/scripts/smoke.json` and rely on (a) exit code 0/non-zero for pass/fail, (b) `tests/ui_automation/output/smoke/` for screenshots + logs after the run. This is the same contract `tests/integration/` already has, deliberately.

## Out of scope (deferred to later M5 stages or post-M5)

- Cross-platform UI testing.
- Image-diff regression gates.
- An in-process `--ui-test` hook (revisit only if UIA flake makes it necessary).
- A test corpus large enough to mean anything statistically -- v1 is three scripts proving the harness.
- CI wiring. Document the plausibility; don't wire it.

## Manifest cost

**Zero.** No agent-facing changes, no new tools, no new prompts. Pure test infrastructure.

## Post-S5.L expansion (2026-05-16)

The original 3 demo scripts proved the harness; this expansion brought the script count to **17** (3 original + 1 LLM-dependent `plan_mode.json` from a later patch + 13 added now) and named several previously-anonymous controls so newer surfaces can be addressed.

**New scripts.** Ten deterministic (no LM Studio): `menu_and_view_toggles.json` (Ctrl+` terminal accelerator), `capabilities_first_open.json` (S5.A modal flow), `settings_persistence.json` (model field round-trips through `.locus/config.json`), `settings_preset_dropdown.json` (S4.V Task 6 preset -> tool_format wiring), `save_as_global_defaults.json` (S5.M global config write), `mcp_open_json.json` (stub-on-missing path), `slash_popup.json` (popup + accept + Enter side-effect), `help_slash.json` (deterministic `/help` slash), `compaction_dialog.json` (CompactionDialog opens + radios reachable), `mutating_tool_friction.json` (Tool Approvals tab dropdowns reachable). Three LLM-dependent: `tool_approval_dialog.json` (approval gate fires when `write_file` is forced to `ask` via the new `setup.tool_approvals_override`), `inline_diff_render.json` (S5.C diff visible in WebView), `terminal_panel_run.json` (S5.B sink-broker -> panel visible after `run_command`).

**Harness extensions.** `setup.allow_first_time_prompts` (bool, default false) skips the tmp-workspace `.locus/config.json` pre-seed AND the `--no-first-time-prompts` launch flag so the Capabilities first-open modal can fire. `setup.env` (map) sets env-vars via `SetEnvironmentVariableA` before launch with `{workspace}` substitution -- used for `LOCUS_GLOBAL_DIR`. `setup.tool_approvals_override` (map) merges into the pre-seed so a script can force a tool back to `ask`. `wait_for_window` now claims foreground via `AttachThreadInput + SetForegroundWindow + BringWindowToTop` so back-to-back script runs don't lose focus to the parent terminal (was the root cause of most batch-run flakes). `press_key` key table grew from 24 to 32 entries (added `BACKTICK` / `TILDE` / `HOME` / `END` / `PAGEUP` / `PAGEDOWN` / `DELETE` / `F12`).

**Widget naming additions.** [src/frontends/gui/ui_names.h](../../src/frontends/gui/ui_names.h) grew by ~30 entries: chat-footer chips (plan / commit), the two popup window names (slash + mention), Tool Approval Dialog (panel + 3 buttons + Scintilla args view + safety banner + ask-input + name label), Capabilities Dialog (dialog + 5 checkboxes), Compaction Dialog (dialog + 2 radios + slider + preview list), Settings sub-controls (LLM tab fields: endpoint / model / temperature / context_limit / max_tokens / preset_choice / preset_apply_btn / tool_format; Save-as-global button; Approvals list scrollwindow; per-tool dropdowns `locus.settings.approvals.choice.<tool_name>` composed at construction; MCP tab list + restart_btn + open_json_btn + trust_check). Matching `SetName(...)` + `gui::apply_locus_accessible_name(...)` calls land in the corresponding `*_panel.cpp` / `*_dialog.cpp` files. Strictly additive -- no behaviour change.

**Incidental fix.** `CapabilitiesDialog` now calls `outer->SetSizeHints(this)` after `Layout()` so the dialog grows beyond its ctor fixed-460-px height when all 5 capability rows + hint text + buttons don't fit. Without this, `cb_web` and its hint were rendered below the visible client area and disappeared from the UIA tree -- a real UI bug the new `capabilities_first_open.json` script caught.

**Known limits documented in the README.** Reading Scintilla content via UIA is unreliable (tool-approval JSON pane, terminal-panel tabs); the scripts assert surfaces (visibility, parent presence, tab existence) and use the chat WebView as a parallel signal where the same content is mirrored there. wxChoice `CBS_DROPDOWNLIST` selection-by-key doesn't reliably fire `wxEVT_CHOICE` through PostMessage / SendInput on all Windows builds, so `mutating_tool_friction.json` asserts the dropdown is reachable but defers the friction-modal trigger to the manual test plan. wxPopupTransientWindow's UIA visibility varies; scripts observe popup side-effects on the parent control instead. The `@`-mention popup test is deferred until the harness gains a `setup.seed_files` option (the candidate list is empty in a fresh tmp workspace). No right-click op yet; the file-tree "Attach to context" path is manual-only.

All 13 deterministic scripts pass green back-to-back in ~60s total. LLM-dependent scripts (the four under the `yes` column in the README's table) inherit `plan_mode.json`'s "fails loudly if LM Studio isn't reachable" behaviour; the README documents the model requirement.
