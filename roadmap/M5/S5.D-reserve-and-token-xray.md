# S5.D -- Reserve Enforcement + Token-Cost X-Ray

## Problem

Two foundational gaps that everything else in the M5 context-management work depends on. They are different in nature but share the same target -- making the model's context budget visible and protected.

1. **No reserve enforcement.** The agent loop today refuses to send only when the prompt strictly exceeds the limit. There is no "and leave room for the response" guard. On a 32k local model with a 30k prompt, the model has 2k of generation room -- often not enough for a meaningful tool-call argument or a compaction summary, and zero room if the turn produces a long answer. Pi reserves 16k by default; Claude Code reserves 33k. Locus reserves nothing. This is the reason the existing soft / hard threshold design is structurally wrong: by the time we ask the user to compact, the LLM already cannot generate a useful summary.

2. **No per-message token visibility.** The context meter shows total usage, but the user has no signal about *where* the tokens are -- system prompt, LOCUS.md, the tool manifest, a single fat tool result, the last 12 turns of conversation. On a tight budget the answer matters; today the user is flying blind.

## Scope

This stage delivers the two primitives. Everything in [S5.E](S5.E-pin-and-tag.md) -- [S5.H](S5.H-edit-delete-branch.md) builds on them.

### Reserve enforcement

- New config: `compaction.reserve_tokens` (default `~20%` of `context_limit`, capped at `4096`). Documented as "minimum response headroom the agent loop guarantees the LLM."
- `ContextBudget` learns about the reserve. New method `effective_limit() = context_limit - reserve_tokens`. The existing `check_overflow()` thresholds are recomputed against `effective_limit()` rather than `context_limit`.
- The agent loop in [src/agent/agent_loop.cpp](../../src/agent/agent_loop.cpp) refuses to dispatch a new LLM round when `current + reserve > limit` -- the user's input is queued until compaction frees room. The refusal surfaces in the chat as an actionable message, not a silent stall.

### Per-message token X-ray

- `ChatMessage` gains a `token_estimate` field, populated by `TokenCounter::estimate(text)` at the point the message is added to history. Cached so we don't re-estimate on every render.
- `IFrontend::on_activity` events for `user_message` / `llm_response` / `tool_call` / `tool_result` / `system_prompt` carry the per-message token count.
- Chat panel renders the count next to (or inside) each bubble -- small grey label like `(312 t)` in the corner. Exact placement TBD during implementation; reuse existing typography.
- Tool-call bubbles split the count into `args` and `result` portions when both are present.
- System-prompt bubble (introduced in [S5.G](S5.G-chat-activity-restructure.md)) shows its breakdown: base / LOCUS.md / metadata / tool manifest / format addendum.

## Implementation notes

- The "20% capped at 4096" default is the smallest reserve that reliably fits a Pi-style structured-summary LLM call (~1.5-2k for the summary itself + 1-2k for the post-compaction next-turn assistant response). On a 32k model this is 4096 / 32768 = 12.5%, close to Pi's 16384 / context defaults. On an 8k model the 20% rule yields 1638, cramped but workable.
- Reserve is **a budget cap on the prompt**, not a separate buffer. Generation still uses whatever's left of the context window after the prompt is sent. The LLM API itself doesn't know about our reserve; it's a Locus-side discipline that prevents us from squeezing the model.
- `ContextBudget` stores the reserve but the auto computation (`min(limit/5, 4096)`) lives in `LLMContext::update_reserve()`, called once at construction and re-callable when the workspace config changes. `FakeWorkspaceServices::workspace()` returns `nullptr` in tests, so `update_reserve()` is a no-op there -- tests call `ctx->budget().set_reserve(n)` directly.
- The reserve-breach guard in `AgentLoop::run_step()` fires on the estimated prompt size of the about-to-be-sent messages, not the server-reported total from the last round. This is intentional: we don't yet have the server total for the new turn, and the estimate is a conservative over-approximation.
- `TokenCounter::estimate` remains a heuristic (~4 chars/token); a future stage may swap in a real BPE tokenizer per [S4.F](../M4/S4.F-kv-cache.md). The X-ray view doesn't care which estimator is in use as long as it's stable across the session.
- Context meter color thresholds (yellow at 75%, red at 90%) are now evaluated against `eff_limit = limit - reserve` rather than `limit`. Full stacked-bar view (system / LOCUS.md / tool manifest / pinned / recent / oldest) is deferred to S5.G, which restructures the chat panel to expose the system prompt directly.
- Per-message token chip in chat reuses the existing chat CSS typography; `.token-chip` adds only a grey color and opacity reduction to stay unobtrusive. User-message chips are injected at submit time using `TokenCounter::estimate` + 4 framing overhead; assistant chips fire in `on_turn_complete` using the server-reported `completion_tokens` from the last round (more accurate than estimating the streamed text).

## Manifest cost

**Zero new tools.** Pure agent-loop and frontend changes. The model is unaware of the reserve; it sees a normal prompt that happens never to exceed `context_limit - reserve_tokens`.
