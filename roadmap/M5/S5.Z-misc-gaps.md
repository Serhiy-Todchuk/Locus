1. [x] **Bug**: assistant text bubble extends past tool calls in chat history. In [src/frontends/gui/chat_panel.cpp](../../src/frontends/gui/chat_panel.cpp), `assistant_id_` is created once per turn (`start_message`, line ~483) and only reset on `clear_chat()`. After a tool call fires, `on_tool_call()` adds a separate `msg-tool` bubble with a fresh `message_id_` but **never touches `assistant_id_` or `current_response_`** -- so when the LLM resumes streaming after the tool result, the new tokens get appended to `current_response_` and re-rendered into the original (pre-tool) assistant bubble, visually breaking the chronological order (old assistant text + tool bubble + *more text appearing inside the old bubble*). Fix is the seal-and-restart pattern: in `on_tool_call`, flush any remaining tokens, strip the `streaming-cursor` class from the current assistant bubble, set `assistant_id_ = 0`, clear `current_response_` + `token_buffer_`. In `on_token`, if `assistant_id_ == 0`, allocate a new assistant bubble before appending. Apply the same pattern to `reasoning_id_` for symmetry. Acceptance: a turn with `assistant text -> tool call -> assistant text` produces three chronologically-ordered bubbles, not two with the tool wedged inside the first. **Done 2026-05-14.** Implemented the seal-and-restart pattern in `on_tool_pending`: finalise the open reasoning `<details>` (flush `reasoning_buffer_`, label "Thoughts", reset `reasoning_id_` + `current_reasoning_`), then either remove the assistant bubble outright when it has no content (placeholder-only or reasoning-only -- avoids leaving a phantom empty bubble below the tool), or do a final md4c render + drop `streaming-cursor` and reset `assistant_id_` + `current_response_` + `token_buffer_`. `on_token` and `on_reasoning_token` allocate a fresh `'msg-assistant streaming-cursor'` bubble whenever `assistant_id_ == 0`, so post-tool text opens a new bubble below. `on_turn_complete` guards its final assistant render against `assistant_id_ == 0` (turns that end immediately after a tool call). No new tests -- behaviour change is purely visual and verified manually against a multi-tool turn.
2. [x] **Search-in-chat (Ctrl+F).** Find text inside the current conversation without scrolling by hand. Reachable two ways: a Ctrl+F binding on the chat panel, and a small magnifying-glass button next to the existing footer chips. Both open a thin floating bar above the chat area with: a text input (focus on open), match counter (`3 / 17`), Prev / Next buttons (also bound to Shift+Enter / Enter and F3 / Shift+F3), a Case-sensitive toggle, and an Esc-to-close affordance. Implementation rides wxWebView's native find API ([wxWebView::Find](https://docs.wxwidgets.org/3.2/classwx_web_view.html) returns the match index; FindNext / FindPrevious step through them; FindClear drops highlights) -- no custom DOM-walking or JS injection needed. WebView2 on Windows handles the highlight rendering natively. Scope is the **current conversation only** -- searching across saved sessions is a different surface (would need a Sessions menu redesign, parked behind [S5.I](S5.I-tabs-and-sessions.md) which already restructures that area). Edge cases: open bar with no text -> no-op; matches outside the current scroll viewport -> auto-scroll to the active match; turn-in-progress streaming -> matches re-index on each `on_token` (wxWebView keeps its find state across content edits, but a fresh `Find()` call on every Nth token is the safe path). Acceptance: opens fast, finds + jumps in <50 ms on a multi-thousand-message conversation, Esc closes cleanly, no leaked highlights when the panel re-renders on next turn. Pure frontend work in [src/frontends/gui/chat_panel.cpp](../../src/frontends/gui/chat_panel.cpp); no agent-thread or core changes. (If wxWebView's Find turns out to be flaky against the WebView2 backend we're using -- it has historically been thin on Windows -- fall back to a JS-injected highlight pass driven from the same UI surface; the UI itself doesn't change.) **Done 2026-05-18.** Added a docked-at-top find bar (`find_bar_` wxPanel between webview and attach row in the vertical sizer -- pushing chat content down rather than overlaying, which keeps WebView2 focus model clean) with text input, `N / M` counter, Prev / Next, "Aa" case toggle, and a close X. Two entry points wired: a `Find` button in the chat footer next to the existing chips, and a new `View > Find in Conversation` menu item with the `Ctrl+F` accelerator -- LocusFrame's hook routes to the active chat tab's `toggle_find_bar()`. **Key implementation detour:** the original plan called for a `wxEVT_CHAR_HOOK` on `ChatPanel` (and a fallback `wxAcceleratorTable` on the panel) to catch Ctrl+F regardless of focused child; neither fired in practice because the chat input is a native Windows RichEdit (`wxTE_RICH2`) that swallows key messages before they reach the wx parent chain. Moving the accelerator to the menubar (same path Memory Bank `Ctrl+M` and Terminal `Ctrl+`` use) hands the keystroke to Win32 `TranslateAccelerator` before native controls see it, which is the only reliable Windows-wide hotkey path. F3 / Shift+F3 in the find input are still handled directly in `on_find_input_key` since the input owns focus once the bar is open. Spec named `FindNext` / `FindPrevious` / `FindClear` but wx 3.2 exposes only `wxWebView::Find(text, flags)` -- repeated calls advance, `Find("")` is the documented cancel path, `wxWEBVIEW_FIND_BACKWARDS` reverses direction, `wxWEBVIEW_FIND_WRAP` handles wrap. The bar tracks `find_total_` (from the fresh-query return value) and `find_index_` (manually advanced 1-based, wraps via modulo); case-toggle re-issues a fresh Find. Streaming re-index is **not** wired -- in practice the bar is opened to search the static post-turn transcript, and re-issuing Find every flush_timer tick would either snap the highlight to match 1 or noticeably flicker; chose simpler "stale until the user retypes" semantics. `on_session_reset` auto-closes the bar so a `clearChat()` doesn't leave a counter that references DOM that no longer exists. [ ] **`filter_output` tool: regex over arbitrary text.** Locus has no way to apply a regex to *previously-produced* tool output. `search` operates on the workspace FTS5 index (files on disk only); `run_command` returns its stdout/stderr as one blob the model has to eyeball or shell-pipe inside the same invocation. The Claude Code analog -- `cmake ... 2>&1 | tail -50` for context-budget control or `grep -i error build.log` after the fact -- has three workarounds today, all imperfect: (a) pipe inside the shell (`run_command "cmd 2>&1 | findstr error"`) -- works only when the OS shell supports it (cmd has `findstr`, POSIX has `grep`, neither is universal), (b) redirect to file + `read_file` -- two tool round-trips per filter, (c) dump the whole output into context and let the LLM filter -- burns tokens proportional to output size. New tool `filter_output(text: string, pattern: string, mode: "regex"|"substring" = "regex", case_sensitive: bool = false, max_matches: int = 50, before: int = 0, after: int = 0, invert: bool = false)` returning matched lines (with optional context) and a one-line summary (`"3 of 1247 lines matched"`). Pure-CPU + no FS access -> `approval_policy = auto_approve` + `category = "read"` in [src/tools/permission_presets.cpp](../../src/tools/permission_presets.cpp). Caller passes the text explicitly -- typically by referencing a prior tool_result, e.g. "filter the previous build output for 'error'." The model still has to put the output into the `text` arg, which is the obvious knock against this design (one round-trip's output gets re-sent as the next round-trip's input) -- but for outputs over a few KB the **net token win** is real because the LLM emits a short regex instead of restating the filtered conclusion. **Alternative considered, rejected**: a `last_tool_result` sigil the model could reference symbolically. Tempting but pushes state into the dispatcher (which tool_result is "last"?) and complicates the bytestream-stable history invariant. Pass-the-text is simpler. Acceptance: `filter_output(text=<25 KB build log>, pattern="^.*error.*$", after=2, max_matches=10)` returns the first 10 error blocks + 2 lines of trailing context + summary, in one tool call, with the model never having to see the full 25 KB. **Triggered 2026-05-18** -- user asked "can locus run a command and regex its output" while reviewing how Claude Code uses `Bash` + `Grep` together; today the Locus answer is "only if the shell already supports the pipe."
