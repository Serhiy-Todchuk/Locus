# S5.Z -- Miscellaneous Smaller Gaps

Items not worth their own stage but worth tracking. Mirrors [S4.V](../M4/S4.V-misc-gaps.md) for the M5 (Polish, UX & Performance) milestone. Expect this list to grow as polish/UX/perf candidates surface.

## Tasks

- [ ] **Bug**: assistant text bubble extends past tool calls in chat history. In [src/frontends/gui/chat_panel.cpp](../../src/frontends/gui/chat_panel.cpp), `assistant_id_` is created once per turn (`start_message`, line ~483) and only reset on `clear_chat()`. After a tool call fires, `on_tool_call()` adds a separate `msg-tool` bubble with a fresh `message_id_` but **never touches `assistant_id_` or `current_response_`** -- so when the LLM resumes streaming after the tool result, the new tokens get appended to `current_response_` and re-rendered into the original (pre-tool) assistant bubble, visually breaking the chronological order (old assistant text + tool bubble + *more text appearing inside the old bubble*). Fix is the seal-and-restart pattern: in `on_tool_call`, flush any remaining tokens, strip the `streaming-cursor` class from the current assistant bubble, set `assistant_id_ = 0`, clear `current_response_` + `token_buffer_`. In `on_token`, if `assistant_id_ == 0`, allocate a new assistant bubble before appending. Apply the same pattern to `reasoning_id_` for symmetry. Acceptance: a turn with `assistant text -> tool call -> assistant text` produces three chronologically-ordered bubbles, not two with the tool wedged inside the first.
- [ ] Tool-call history compaction. Today every tool call in a turn renders as its own `msg-tool` bubble (success and failure alike -- the asymmetry the user reported is perceived, not real: failures often have larger result tails that visually dominate). On turns with many tool calls (8+ search/read/edit chains) this floods the chat history. Consolidate either always ("`read_file × 2, edit_file × 1, run_command × 1`" chip per turn, click to expand) or above a threshold (show first 3 individually, fold the rest into "+ N more"). Threshold-fold is the better default -- preserves transparency for typical turns, scales for tool-heavy ones.
- [ ] Auto-restore last session on workspace open. Today every launch starts with a blank conversation; the user has to dig through Session -> Saved Sessions to resume yesterday's work. Most editors / IDEs / agent tools default to "pick up where you left off." Implementation: workspace-level setting `session.restore_last` (default `true`); on `LocusSession` ctor, after the agent thread starts, call `SessionManager::list_sessions()` and load the most recently modified one if the flag is set. Setting in Settings -> Workspace tab so users can opt out. Pairs with the next bullet -- auto-restore without a visible "start over" affordance is a trap.
- [ ] Visible "New Conversation" button in the chat area. Currently reset is only reachable via Session menu (Ctrl+R) or `/reset` / `/clear` slash commands -- discoverable only if the user opens the menu or knows the slash. Add a small button in the chat panel header (or above the input box) that calls the same `agent_.reset_conversation()` hook the menu uses. Optional confirm dialog when the current conversation is non-empty ("Start a new conversation? Current chat will be saved."). Becomes load-bearing once auto-restore lands.
- [ ] Tool approval panel: rework the always-docked AUI pane (`approval_panel_` in [src/frontends/gui/locus_frame.cpp](../../src/frontends/gui/locus_frame.cpp)) into a self-dismissing surface that demands attention only while a decision is needed. Today it lives in the bottom dock with `CloseButton(false)`; the user has no way to set it aside without choosing approve / modify / reject. Two viable directions to pick between during implementation:
  - **Modal dialog** (`wxDialog` derived from the existing panel content): blocks the rest of the UI while a tool call is pending. Strong attention pull, zero permanent screen real estate, but the user cannot scroll chat history or open another panel while deciding.
  - **Floating non-modal pane** (AUI `Float()` + auto-show on `tool_pending` + auto-hide on `tool_result` / `turn_complete`): keeps chat scrollable while the floating window asks for a decision; less coercive, more in line with the project's calm-UX leaning.
  Whichever is picked, the behaviour the user actually wants is: **the panel appears when needed, takes user input, and disappears** -- it should not occupy permanent dock space when no decision is pending. The agent-thread side already does the right thing (`tool_decision` callback unblocks the agent on each of the three responses); this is purely a frontend reshape. Acceptance: with no pending tool call, the chat panel uses the full window minus the activity sidebar; on `tool_pending` the approval surface appears centred-on-frame; on response it disappears entirely without leaving an empty dock slot.
