## Tasks

1. [x] **Bug**: assistant text bubble extends past tool calls in chat history. In [src/frontends/gui/chat_panel.cpp](../../src/frontends/gui/chat_panel.cpp), `assistant_id_` is created once per turn (`start_message`, line ~483) and only reset on `clear_chat()`. After a tool call fires, `on_tool_call()` adds a separate `msg-tool` bubble with a fresh `message_id_` but **never touches `assistant_id_` or `current_response_`** -- so when the LLM resumes streaming after the tool result, the new tokens get appended to `current_response_` and re-rendered into the original (pre-tool) assistant bubble, visually breaking the chronological order (old assistant text + tool bubble + *more text appearing inside the old bubble*). Fix is the seal-and-restart pattern: in `on_tool_call`, flush any remaining tokens, strip the `streaming-cursor` class from the current assistant bubble, set `assistant_id_ = 0`, clear `current_response_` + `token_buffer_`. In `on_token`, if `assistant_id_ == 0`, allocate a new assistant bubble before appending. Apply the same pattern to `reasoning_id_` for symmetry. Acceptance: a turn with `assistant text -> tool call -> assistant text` produces three chronologically-ordered bubbles, not two with the tool wedged inside the first. **Done 2026-05-14.** Implemented the seal-and-restart pattern in `on_tool_pending`: finalise the open reasoning `<details>` (flush `reasoning_buffer_`, label "Thoughts", reset `reasoning_id_` + `current_reasoning_`), then either remove the assistant bubble outright when it has no content (placeholder-only or reasoning-only -- avoids leaving a phantom empty bubble below the tool), or do a final md4c render + drop `streaming-cursor` and reset `assistant_id_` + `current_response_` + `token_buffer_`. `on_token` and `on_reasoning_token` allocate a fresh `'msg-assistant streaming-cursor'` bubble whenever `assistant_id_ == 0`, so post-tool text opens a new bubble below. `on_turn_complete` guards its final assistant render against `assistant_id_ == 0` (turns that end immediately after a tool call). No new tests -- behaviour change is purely visual and verified manually against a multi-tool turn.
2. [ ] Tool-call history compaction. Today every tool call in a turn renders as its own `msg-tool` bubble (success and failure alike -- the asymmetry the user reported is perceived, not real: failures often have larger result tails that visually dominate). On turns with many tool calls (8+ search/read/edit chains) this floods the chat history. Consolidate either always ("`read_file x 2, edit_file x 1, run_command x 1`" chip per turn, click to expand) or above a threshold (show first 3 individually, fold the rest into "+ N more"). Threshold-fold is the better default -- preserves transparency for typical turns, scales for tool-heavy ones.

5. [ ] **Search-in-chat (Ctrl+F).** Find text inside the current conversation without scrolling by hand. Reachable two ways: a Ctrl+F binding on the chat panel, and a small magnifying-glass button next to the existing footer chips. Both open a thin floating bar above the chat area with: a text input (focus on open), match counter (`3 / 17`), Prev / Next buttons (also bound to Shift+Enter / Enter and F3 / Shift+F3), a Case-sensitive toggle, and an Esc-to-close affordance. Implementation rides wxWebView's native find API ([wxWebView::Find](https://docs.wxwidgets.org/3.2/classwx_web_view.html) returns the match index; FindNext / FindPrevious step through them; FindClear drops highlights) -- no custom DOM-walking or JS injection needed. WebView2 on Windows handles the highlight rendering natively. Scope is the **current conversation only** -- searching across saved sessions is a different surface (would need a Sessions menu redesign, parked behind [S5.I](S5.I-tabs-and-sessions.md) which already restructures that area). Edge cases: open bar with no text -> no-op; matches outside the current scroll viewport -> auto-scroll to the active match; turn-in-progress streaming -> matches re-index on each `on_token` (wxWebView keeps its find state across content edits, but a fresh `Find()` call on every Nth token is the safe path). Acceptance: opens fast, finds + jumps in <50 ms on a multi-thousand-message conversation, Esc closes cleanly, no leaked highlights when the panel re-renders on next turn. Pure frontend work in [src/frontends/gui/chat_panel.cpp](../../src/frontends/gui/chat_panel.cpp); no agent-thread or core changes. (If wxWebView's Find turns out to be flaky against the WebView2 backend we're using -- it has historically been thin on Windows -- fall back to a JS-injected highlight pass driven from the same UI surface; the UI itself doesn't change.)

[ ] **Interactive stdin in the Terminal panel.** S5.B shipped the terminal as a *viewer* of agent-driven `run_command` / `run_command_bg` output -- each tab's `wxStyledTextCtrl` is `SetReadOnly(true)`, there's no input surface, and `BackgroundProcess` exposes a stdin pipe handle that nothing in the UI ever writes to. For commands that genuinely need user input (interactive REPLs, password prompts, `git rebase --interactive`, dev servers asking "rebuild?"), this is a dead-end -- the user has to kill the process and re-run it through a real shell. Scope: small single-line `wxTextCtrl` row docked below the STC on each bg-process tab, Enter submits the line + `\n` to the process stdin via `WriteFile` on the existing pipe handle, the submitted line is echoed into the STC's scrollback as plain text (mirrors what a terminal would do; many CLIs don't echo their own stdin). Sync `run_command` tab keeps no input row -- the dispatcher waits synchronously on exit, the user can't realistically inject anything mid-run. Edge cases worth thinking through: line buffering vs char-at-a-time (line buffering is the right default; matches how `Read-Host` / `input()` / shell prompts work), Ctrl+C forwarding (out of scope v1 -- the Kill bg context-menu item already covers the destructive case), bracketed-paste (out of scope v1 -- a chunk paste lands as one Enter-submitted line). New `BackgroundProcess::write_stdin(std::string_view)` on [src/tools/process_registry.h](../../src/tools/process_registry.h) (mutex-guarded `WriteFile` on the pipe handle the spawn path already keeps); new `TerminalPanel` input row per tab with the submit-on-Enter handler. Acceptance: spawn `python -i` via `run_command_bg`, type `1+1`, see `2` in the same tab; spawn a Node REPL, same thing. UI Automation `chat_round_trip.json`-style script optional. **Triggered 2026-05-14 -- user noticed `SetReadOnly(true)` made the panel non-interactive when they tried to drive a bg-spawned `python -m http.server` from inside the panel.**
