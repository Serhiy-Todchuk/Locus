# S5.R -- Per-Tab Terminal Panel

## Problem

[S5.I](S5.I-tabs-and-sessions.md) made `ProcessRegistry` per-tab (each tab's agent only sees the processes it spawned via `list_processes` / `read_process_output` / `stop_process` / `run_command_bg`) but kept the [S5.B](S5.B-terminal-panel.md) `TerminalPanel` workspace-wide -- every tab's `ProcessRegistry` was given the workspace's shared `ProcessSinkBroker` so all tabs' output funnels into one panel.

The result is asymmetric: the agent in tab A sees a clean per-tab process list, but the user sees tab A's `cmake --build`, tab B's `pytest`, and tab C's dev server interleaved in the same Terminal pane with no way to tell which tab spawned which. Switching tabs in the chat doesn't change what's in the terminal -- the mental model "tab = isolated conversation" breaks at the terminal boundary.

This stage scopes the terminal to the active tab: each tab owns its own terminal content (Run tab + per-bg-process tabs + scrollback + ANSI state), the AUI pane's visibility stays workspace-wide, content swaps on chat-tab activation, and the pane auto-shows only when the **active** tab spawns a command.

## Design

**Per-tab content, workspace-level visibility.** One `TerminalPanel` widget docked once in the AUI layout; its contents (the inner `wxAuiNotebook` of process tabs) are sourced from the active chat tab's terminal state. Switching chat tabs swaps the rendered notebook; the AUI pane's hidden/shown/floated/docked state is unaffected.

**No per-tab toggle button in `ChatPanel`.** The workspace-level **View > Terminal** menu item + Ctrl+` accelerator already exists and is sufficient. Adding a per-tab toggle clutters the chat surface for a workspace-level concern.

**Auto-show on first active-tab command, no auto-hide.** When the **active** tab's `ProcessRegistry` sees the first new process (sync or bg) and the pane is hidden, the pane auto-opens. Subsequent commands in the same tab do not re-open the pane (user may have deliberately closed it). The pane never auto-hides -- failed commands need post-mortem inspection, streaming bg output needs to stay watchable, short commands shouldn't flash open and shut. User closes the pane via View > Terminal / Ctrl+` when done.

**Background-tab commands do not pop the pane.** If tab B is active and tab A's agent spawns a command, the pane stays in whatever state the user left it. The existing tab-title `*` busy badge ([S5.I](S5.I-tabs-and-sessions.md)) already signals "this tab has activity"; the pane only updates when the user navigates to tab A. This keeps the rule "pane opened = I just ran something here" intuitive.

**Tab-switch preserves per-tab state.** Each chat tab carries its terminal state across switches: which Run tab exists, which bg-process tabs exist with their badges (`*` running / `[N]` exit / `[x:N]` killed), scrollback buffers, scroll positions, active sub-tab selection, ANSI parser state. Switching to tab B then back to tab A restores tab A's view exactly.

## Tasks

### Per-tab terminal state model

- [ ] Lift the per-process buffers, ANSI parser state, scrollback position, and active sub-tab index off the `TerminalPanel` widget into a new `TerminalPanelState` (struct or POD class) in [src/frontends/gui/terminal_panel.h/cpp](../../src/frontends/gui/terminal_panel.h). The widget becomes a view over a `TerminalPanelState*`.
- [ ] `LocusTab` ([src/core/locus_tab.h](../../src/core/locus_tab.h)) gains a `TerminalPanelState` member alongside its `ProcessRegistry`. State lives for the lifetime of the tab; destroyed when the tab closes (along with the registry tearing down processes).
- [ ] Remove the workspace-shared-broker wiring: `ProcessRegistry` for a tab no longer takes the workspace's broker as `external_broker`. Each tab's registry creates its own broker (the original constructor shape); each tab's `TerminalPanelState` subscribes to its own tab's broker as `IProcessSink`. Workspace-level broker stays on `Workspace` only if it has workspace-level processes (today there are none -- the workspace registry is symbolically clean, see [S5.I](S5.I-tabs-and-sessions.md)).

### Active-tab content swap

- [ ] `LocusFrame` ([src/frontends/gui/locus_frame.h](../../src/frontends/gui/locus_frame.h)) gains a `set_active_terminal_state(TerminalPanelState*)` call invoked from the existing tab-activation handler. `TerminalPanel::set_state(TerminalPanelState*)` rebuilds the inner `wxAuiNotebook` to match the new state (clear pages, re-add pages from state's process tabs, restore active sub-tab + scroll position).
- [ ] Content swap is **render-only** -- no buffer copy. The widget's `wxStyledTextCtrl` instances get their text re-loaded from the state's scrollback; on swap-away, scrollback in the state is already authoritative (subscribers wrote to it directly, the widget was just displaying it).
- [ ] Tab-close of a chat tab destroys the `TerminalPanelState` after the `ProcessRegistry` teardown. If the destroyed state was the active terminal state, `TerminalPanel` swaps to the new active tab's state (or to a placeholder "no commands have run in this tab yet" empty state for a fresh tab).

### Auto-show behavior

- [ ] `LocusFrame` subscribes to each tab's `ProcessSinkBroker` `on_bg_started` / `on_sync_started` events -- but for the **auto-show decision** only, not for content rendering (that's the state's job via its own subscription).
- [ ] Auto-show decision: if event came from the **currently active** tab AND the AUI pane is currently hidden, show the pane (and switch focus to the freshly created process tab as the new active sub-tab). If event came from a background tab, do nothing -- the `*` badge on the chat tab title is the signal.
- [ ] First-command-per-tab gating: the auto-show fires the first time the pane is hidden and the active tab spawns a process. If the user manually re-hides the pane after seeing the first command, the next command in the same tab does **not** re-pop the pane -- the user's explicit hide wins. Tracked by a per-tab "user has hidden the pane since the last auto-show" flag on the tab (resets when the user re-opens the pane manually).

### Background-tab signaling

- [ ] The S5.I `*` tab badge already covers "agent busy in this tab" -- reuse for "bg process running" as well? The S5.I implementation sets `*` from `set_tab_busy(tab_id)` based on agent turn state. Extend to also flip `*` when the tab has at least one running process in its `ProcessRegistry`; clear when both conditions are false. Single-glyph reuse keeps the tab strip uncluttered and matches the user's intuition ("something is happening in that tab").
- [ ] If real usage shows users wanting to distinguish "agent thinking" from "bg process running," split into two glyphs later. Out of scope for v1.

### Visibility persistence

- [ ] Pane visibility (shown/hidden/floated/docked) stays workspace-level, not per-tab. AUI perspective persistence is still parked (see [S5.B](S5.B-terminal-panel.md) -- defaulted hidden); when that lands as part of [S5.Z](S5.Z-misc-gaps.md) the perspective will capture the terminal pane state alongside the rest. No new perspective machinery in this stage.

### UI automation scripts

- [ ] New script [tests/ui_automation/scripts/terminal_per_tab.json](../../tests/ui_automation/scripts/terminal_per_tab.json): open two tabs, spawn a sync `run_command` in tab A (auto-show fires), switch to tab B (terminal content swaps to tab B's empty state), switch back to tab A (terminal content restores with the original Run tab + output). LLM-dependent (uses a real `run_command` round-trip) -- inherits the existing "fails loudly if LM Studio isn't reachable" pattern.
- [ ] Extend [tests/ui_automation/scripts/terminal_panel_run.json](../../tests/ui_automation/scripts/terminal_panel_run.json) (if it exists from S5.L) to assert the active-tab gating: spawn from background tab and assert pane stays hidden.

### Catch2 tests

- [ ] `TerminalPanelState` lifecycle: create, append chunks via the sink, swap views (no buffer loss), destroy with the owning tab.
- [ ] Per-tab broker isolation: register two `TerminalPanelState` instances each subscribed to its own broker; chunks emitted on broker A do not appear in state B.
- [ ] Active-tab auto-show predicate: pure function `should_auto_show(event_tab_id, active_tab_id, pane_hidden, user_hid_since_last_show)` returns true only when event is from active tab AND pane hidden AND user-hid flag is false.
- [ ] `LocusTab` teardown: closing a tab tears down its `ProcessRegistry` (killing processes via the existing Job Object) before its `TerminalPanelState` is freed; if it was the active terminal state, the swap to the new active tab's state lands before destruction.
- [ ] Per-tab `*` badge: badge is on when the tab has an active turn OR at least one running process; off when both are false; the existing turn-only assertion from S5.I still holds.

## Implementation notes

- The S5.B `IProcessSink` interface ([src/tools/process_sink.h](../../src/tools/process_sink.h)) is already the right abstraction -- per-tab `TerminalPanelState` implements `IProcessSink`, the tab's `ProcessSinkBroker` fans events to that one sink. No new interface, no broker fan-out across tabs.
- The S5.B widget's `wxAuiNotebook` of process tabs is internal; the per-tab state holds an in-memory model of "what tabs exist + their content," and the widget renders it. On swap, the widget destroys its current notebook pages and rebuilds from the new state -- O(small) since process tabs are typically <5 per tab.
- Scrollback in `wxStyledTextCtrl` is rebuilt on swap by re-issuing the per-line styled appends from the state's stored representation. Verify swap latency stays under 100 ms even with the full 10000-line scrollback cap; if measurable lag appears, cache the rendered STC document per state and swap document handles (`wxStyledTextCtrl::SetDocPointer`) instead of re-issuing styled writes -- this is the canonical wx pattern for fast text-view swaps.
- The state's ANSI parser ([src/frontends/gui/ansi_parser.h](../../src/frontends/gui/ansi_parser.h)) is already stateful across chunks and reentrant-safe per-instance; one parser per process-tab in the state, owned by the state, not the widget.
- Auto-show + user-hide tracking: the user-hid flag belongs on the tab (in the `LocusTab`), not on the state, because state lives as long as the tab does but the flag's semantics are workspace-pane-relative. Set when `LocusFrame::on_terminal_pane_hide()` fires and the active tab had auto-shown the pane; clear when the user re-opens the pane via the View menu.
- Sync `run_command` Run tab: per S5.B the Run tab is cleared on each new sync run (output retained until the next sync run). Per-tab: each tab's state has its own Run tab; clearing happens within that tab's state, not globally. This is the natural consequence of per-tab state and matches the user's mental model.
- The workspace-level `IWorkspaceServices::process_sink()` accessor stays as-is (the workspace's own broker exists for symbolic completeness even though no workspace-level processes exist today). Per-tab registries no longer take it; they construct their own brokers.

## Out of scope (v1)

- **AUI perspective persistence for the terminal pane.** Tracked under [S5.Z](S5.Z-misc-gaps.md) alongside the rest of the perspective-save work; the pane keeps the S5.B default of "always starts hidden" until that lands.
- **Per-tab terminal visibility state** (hidden in tab A, shown in tab B). Pane visibility is workspace-level. Considered and rejected -- would force the AUI pane to show/hide on every tab switch, which is jarring and contradicts AUI's docking model.
- **Distinct glyphs for "agent busy" vs. "bg process running"** in the tab title. Reuse the existing `*` badge for both. Add separation if real users report the conflation as confusing.
- **Cross-tab process search** ("find this output across all my tabs"). Per-tab isolation is the point of this stage.
- **Moving a bg process from tab A to tab B.** The process belongs to the tab that spawned it; users can `stop_process` and `run_command_bg` from the new tab if they want to relocate.
- **A dedicated "no commands have run in this tab yet" empty-state panel.** For v1 the terminal pane on a fresh tab just shows an empty `wxAuiNotebook` with no pages. A friendly empty state can be added if the blank pane confuses users.

## Manifest cost

**Zero new tools.** The agent's tool surface is unchanged. This stage is pure frontend restructuring + per-tab fan-out of an existing sink interface.
