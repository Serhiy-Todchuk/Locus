# S5.I -- Multi-Tab Conversations + Session Menu Redesign

**Status:** done 2026-05-16.

## Problem

A workspace runs one conversation at a time today. Users routinely want multiple parallel agent threads -- "research thread," "implementation thread," "scratch thread" -- without losing each one's context. The Claude Code VSCode extension does this with tabs and the UX is genuinely good. We don't have it.

This stage adds **tabbed conversations within the active workspace** and rebuilds the Session menu around a unified tab/session model. **Tab = session.** Every open tab is exactly one session on disk; every saved session is either open in a tab or sitting in the Saved Sessions submenu, never both. Each tab is an independent agent loop with its own history, compaction state, and checkpoint store. The workspace, index, LLM client, and tool registry are shared across tabs.

Note: this is **multiple sessions in one workspace**, not multiple workspaces in one window (the latter is a different idea, parked in the backlog).

## Tab semantics

| Concern | Per-tab | Shared across tabs |
|---|---|---|
| `AgentCore` (history, agent thread, system prompt, attached context) | yes | -- |
| `ConversationHistory` | yes | -- |
| `ContextBudget` + reserve enforcement ([S5.D](S5.D-reserve-and-token-xray.md)) | yes | -- |
| Compaction state + history archives ([S5.F](S5.F-compaction-v2.md)) | yes | -- |
| Checkpoint store ([S4.B](../M4/S4.B-checkpoint-undo.md)) | yes (separate session id per tab) | -- |
| `Workspace` + `Indexer` + `IndexQuery` + `FileWatcher` + `EmbeddingWorker` | -- | yes |
| `ILLMClient` (one endpoint, same model) | -- | yes |
| `ToolRegistry` | -- | yes |
| `ProcessRegistry` ([S4.I](../M4/S4.I-background-commands.md)) | yes (each tab owns its bg processes; killing a tab kills its tree) | -- |
| MCP servers (when [S4.G](../M4/S4.G-mcp.md) lands) | -- | yes (MCP servers are workspace-level subsystems) |

The mental model: **workspace = shared substrate; tab = an `AgentCore` instance**. `LocusSession` ([S3.G](../M3/S3.G-locus-session.md)) is currently 1:1 with workspace; this stage introduces a `LocusTab` layer so a session bundles workspace-shared subsystems plus N tabs.

## Tasks

### Tab infrastructure

- [x] New `LocusTab` class in `src/core/locus_tab.h/cpp`: owns one `AgentCore`, the tab's title, the tab's session id (assigned lazily on first user message -- see Persistence), the tab's `ProcessRegistry`. Constructor takes references to workspace-shared subsystems.
- [x] `LocusSession` refactored: holds `Workspace`, `ILLMClient`, `ToolRegistry` as shared resources + `std::vector<std::unique_ptr<LocusTab>> tabs_`. Methods: `add_tab()`, `close_tab(index)`, `delete_tab(index)`, `active_tab()`, `set_active_tab(index)`. Active-tab index persisted in workspace UI state. Closing the last tab automatically opens a fresh empty tab in its place (the workspace is never tab-less).
- [x] Per-tab `ProcessRegistry` ownership change: `Workspace`'s registry stays for workspace-level bg processes (none today, but symbolically clean); tab-spawned bg processes go on the tab's registry. On tab close: tab's registry tears down, killing its process tree.
- [x] `SessionManager` extension: `last_opened_at` timestamp stored in each session's JSON. Bumped on session creation, on workspace auto-restore, and on every Open from the Saved Sessions submenu or Manage Sessions dialog. Used by cleanup (see "Session cleanup" below).

### Frontend (wxWidgets)

- [x] `wxAuiNotebook` or equivalent at the top of the chat area. Each tab page contains the existing `ChatPanel` + activity events scoped to that tab.
- [x] Tab visual indicators:
  - Spinner / animated icon when the agent is mid-turn in that tab
  - Yellow / red dot when context approaches threshold ([S5.F](S5.F-compaction-v2.md) bands)
  - Awaiting-approval icon when the tab has a pending tool call (interacts with [S5.Z](S5.Z-misc-gaps.md) modal-style approval)
- [x] Tab interactions:
  - Click to switch
  - Middle-click or X button to **close** the tab. Empty tabs (never typed in) are silently discarded; non-empty tabs move to the Saved Sessions submenu. No "unsaved" confirm prompt -- every history mutation auto-saves to disk in real time, so there is nothing to lose by closing.
  - Right-click context menu: **Rename...**, **Delete session...** (closes the tab AND removes the session file with confirm), **Close**, **Close others**, **Close to the right**
  - Drag to reorder
- [x] Tab title source: auto-generated from first user message (truncated to ~30 chars), user-editable via right-click Rename. Persisted in session JSON. Empty tabs render as "New tab" until the first user message lands.
- [x] "+ New tab" button at the end of the tab strip; keyboard shortcut Ctrl+T.
- [x] Close-tab shortcut Ctrl+W. **Last tab can be closed too**: the close fires normally (move to Saved Sessions if non-empty, discard if empty), and a fresh empty tab opens in its place. The workspace is never tab-less.
- [x] Approval panel rework from [S5.Z](S5.Z-misc-gaps.md) becomes per-tab-aware: the modal/floating panel attaches to the tab whose agent raised the approval, and switching tabs while one is awaiting decision shows the awaiting-approval icon on the source tab.

There is no "Reset Conversation" action and no Ctrl+R shortcut. The user starts a fresh conversation by opening a new tab. This collapses one menu item, one shortcut, and one piece of state machinery (the in-place `reset_for_new_conversation` path on `LLMContext`) out of the UX surface -- the underlying call stays for programmatic use but is no longer wired to a menu.

### Session menu redesign

Top-level menu name: **Session**. Single hierarchy:

- **Session** menu:
  - **New Tab** (Ctrl+T) -- empty tab; no session file until the first user message
  - **Close Tab** (Ctrl+W) -- close active tab (empty -> discard; non-empty -> Saved Sessions)
  - **Rename Active Tab...** -- prompt for a new title
  - **Saved Sessions** submenu -- closed sessions only (currently-open ones are filtered out so each session appears in at most one place at any time). Each entry is itself a sub-submenu:
    - `Saved Sessions > <session title> > Open / Rename... / Delete...`
    - Submenu sort order: most-recently-`last_opened_at` first
  - **Clean Up Old Sessions...** -- opens the Manage Sessions dialog pre-filtered to cleanup candidates (sessions that match the auto-cleanup criteria, see "Session cleanup" below). Confirm dialog before any deletion happens.
  - **Manage Sessions...** -- opens the full dialog (list of all sessions + bulk actions). The user can browse, rename, delete, and trigger cleanup from one place.

The **Saved Sessions submenu** uses sub-submenus rather than inline-action rows because wx menu items don't support multiple actions per row. The Manage Sessions dialog is the discoverable surface for power use; the submenu is the quick-open path.

**Workspace switch** stays as it is; it now closes all tabs in the current workspace and opens the new workspace's restored tab set (interaction with [S5.Z](S5.Z-misc-gaps.md) auto-restore).

### Persistence (first-class part of this stage)

- [x] Saved-session JSON shape extended with optional `tab` metadata: `{title, last_opened_at, ...}`. Backward-compatible: old sessions without these fields open as a normal session with auto-generated title and `last_opened_at = file mtime` on first read.
- [x] **Lazy session-file creation**: a brand-new tab from "+ New tab" or Ctrl+T does NOT write a session JSON to disk. It lives in memory only with no session id. The session id is assigned and the file is written on the first user message of that tab. Empty tabs that the user never types in produce no disk artifact at all.
- [x] **Auto-save on every history mutation**: after the first user message lands, every subsequent change (new message, tool call, tool result, edit, delete) writes the session JSON through `SessionManager::save`. No "Save Tab as Session" action exists -- it's continuous and silent. Auto-save cost is negligible (small JSON, atomic temp-file rename, single workspace user).
- [x] Workspace UI state ([src/core/workspace.h](../../src/core/workspace.h) related) gains `open_tabs: [{session_id, title, active}]`. New workspace setting `session.restore_last` (default `true`).
- [x] **On workspace close**: every open non-empty tab is already on disk (auto-save handled this). The `open_tabs` UI state captures the list of open session ids + the active tab index. Empty tabs are not listed.
- [x] **On workspace open**: if `session.restore_last == true`, every entry in `open_tabs` is reopened in a tab (active flag restored). Each restored tab's `last_opened_at` is bumped to now. If `false`, a single empty tab opens. The "last single session" behaviour earlier discussed in [S5.Z](S5.Z-misc-gaps.md) is the degenerate case of this -- one tab in, one tab out.
- [x] Setting in Settings -> Workspace tab so users can opt out. The opt-out implies "every workspace open starts blank" -- explicit user choice, not a hidden default.

### Session cleanup

Without cleanup, `.locus/sessions/` grows unbounded. Two cleanup paths, both governed by the same criteria:

- **Auto-cleanup** runs at workspace open, after the index initial-scan completes. If any closed sessions match the cleanup criteria, the Manage Sessions dialog pops with those rows pre-selected; the user confirms / adjusts / cancels. Nothing is deleted without confirmation.
- **Manual cleanup**: **Session -> Clean Up Old Sessions...** opens the same dialog with the same pre-selection. The user can also browse and multi-select additional rows for one-off cleanup beyond the configured thresholds.

**Cleanup criteria** (a closed session is eligible when both clauses hold):

1. `last_opened_at` is older than `sessions.delete_after_days` ago (default 21)
2. By `last_opened_at` ordering, the session sits beyond the `sessions.keep_last_count` most-recently-opened closed sessions (default 10)

The first clause protects active reference sessions (you opened it recently, it stays). The second clause protects "small workspaces" from over-eager cleanup (you only have 8 closed sessions total -> nothing is eligible even if every one of them is 6 months old).

**The "pin" is being-open-in-a-tab.** Currently-open tabs are exempt from both auto-cleanup and manual cleanup (the criteria only see closed sessions). Users who want a session to be permanent keep it open in a tab. This collapses the entire pin concept into existing UX with zero new affordances.

**Hard delete, no trash.** Delete from Saved Sessions submenu, from the tab right-click "Delete session...", or from the Manage Sessions dialog removes the session's on-disk artifacts immediately. The single confirm dialog is the safety net -- accidental double-misclick is rare enough that a 30-day trash mechanism would be over-engineering.

**On-disk layout note.** A session is not a single JSON file. The live state lives at `<sessions_dir>/<session_id>.json`, but once [S5.F](S5.F-compaction-v2.md) compaction has run on the session, a sibling subdirectory `<sessions_dir>/<session_id>/` appears containing `history.before-compact-<N>.json` archive snapshots (see [src/agent/history_archive.h](../../src/agent/history_archive.h)). Cleanup and per-session Delete must therefore remove **both** the `<session_id>.json` file **and** the `<session_id>/` directory if it exists. The Manage Sessions dialog's per-row size column reports the sum across both. (Storage savings via delta-only archives are a separate backlog item; v1 keeps the shipped snapshot model.)

- [x] `sessions.auto_cleanup_enabled` (bool, default true), `sessions.keep_last_count` (int, default 10), `sessions.delete_after_days` (int, default 21) added to `WorkspaceConfig` under a `"sessions": {}` block in `.locus/config.json`.
- [x] Settings UI: new fields under Settings -> Workspace tab (or a Sessions tab if Workspace is already crowded). Three rows. Reasonable defaults; users who don't care never touch them.
- [x] **Manage Sessions dialog** lives in `src/frontends/gui/manage_sessions_dialog.h/cpp`. Columns: title, message count, `last_opened_at`, size (sum of `<id>.json` + any sibling `<id>/` archive directory), status (Open / Closed). Selection model: multi-select. Buttons: **Delete Selected...** (confirm dialog; removes both the session JSON and its archive subdirectory in one operation), **Open in New Tab**, **Rename...**, **Close** (dialog).
- [x] Manage Sessions dialog has two "entry modes": (a) opened via "Manage Sessions..." -- shows all sessions, no rows pre-selected; (b) opened via "Clean Up Old Sessions..." OR auto-cleanup-at-workspace-open -- pre-filters to candidates with rows pre-selected and a heading "These sessions match your auto-cleanup criteria. Confirm deletion or adjust selection."
- [x] Currently-open tabs appear in the dialog list with status = "Open" and are NOT selectable for deletion via this UI (use the tab right-click "Delete session..." instead, which closes the tab first). Avoids the awkward "delete a session that's currently running an agent turn" case.

## Implementation notes

- The wx tab widget choice is between `wxAuiNotebook` (already in our AUI dependency tree) and `wxNotebook` (simpler but doesn't dock). Prefer `wxAuiNotebook` since the rest of the layout is AUI-managed -- gives consistent floating / docking behavior and matches the project's anti-Electron stance ("native widgets, not framework chrome").
- LLM concurrency: a single LM Studio (or any local OpenAI-compatible) endpoint typically handles one inference at a time. Two tabs running simultaneously will queue at the server; the second tab's spinner shows "queued" rather than "running" until the first completes. Add a small inline status to the tab strip when this happens. No new threading -- the existing `AgentCore` thread per tab handles the wait naturally.
- Activity log: under [S5.G](S5.G-chat-activity-restructure.md), the activity panel becomes the system audit trail (indexing / embedding / file-watcher). Tab-scoped chat events (user_message / llm_response / tool_call / tool_result) live in their tab's chat. The activity panel stays workspace-shared (not per-tab) because system events are workspace-scoped, not conversation-scoped. Per-tab filtering ("show only events from this tab's agent") is a [S5.Z](S5.Z-misc-gaps.md) bullet for later if needed.
- Memory cost per tab: each `AgentCore` carries its history (a few MB at most for typical sessions), its budget tracker, its checkpoint store handle. Realistic upper bound: ~10 MB per tab. Open 10 tabs = ~100 MB. Acceptable; document the cap as a soft limit but don't hard-cap unless real users hit it.
- "Move to new tab from here" (right-click on a chat message in [S5.H](../backlog/S5.H-edit-delete-branch.md)'s original scope) was parked alongside the rest of S5.H. The simpler workflow -- "delete the last few messages and retype" using S5.G's per-message delete -- covers the realistic use case without an archive chain. Reactivate together with S5.H if a real workflow surfaces.
- Backlog item "Multiple workspaces open simultaneously (tabbed sessions)" is **a different concept** -- multiple workspaces in one window. This stage addresses multiple sessions in one workspace. The two could compose later (workspace tabs containing session sub-tabs), but neither blocks the other.
- Tab UI tooltips spell out the tab's full (untruncated) title. No "Unsaved" state -- auto-save makes that signal moot; if a session file exists, it is up to date.

## Out of scope (v1)

- **Multiple workspaces in one window** (parked in backlog).
- **Drag-drop messages between tabs.**
- **Tab groups / nesting.**
- **Per-tab LLM model override** (each tab targeting a different model). Useful for power users but couples to a multi-model story we don't have. Defer.
- **Cross-tab tool-result sharing** ("agent in tab B can see what tab A read"). Tabs are isolated by design.
- **Recent Tabs / Recently Closed Tabs submenu.** Auto-restore on workspace open re-opens whatever was open at last shutdown; everything else is in the Saved Sessions submenu by session title. The "recently closed" hot-list as a separate concept was reviewed 2026-05-16 and rejected -- it duplicates Saved Sessions without adding affordances.
- **Explicit pin / favorite toggle on sessions.** Considered and replaced by "currently open in a tab = effectively pinned." Pinning a session via dedicated UI would add a third UX state (open / closed-but-pinned / closed-and-collectable) that the current model collapses cleanly into two.
- **Trash / restore-from-deleted for sessions.** Considered and rejected. The single confirm dialog before delete is the safety net; a 30-day trash mechanism (like S4.R memory bank's `.deleted/`) was deemed over-engineering for an action that already requires explicit confirmation. Reactivate if real users report misclick-driven loss.
- **"Don't ask me again" checkbox on the auto-cleanup confirm dialog.** The dialog always pops; no silent-cleanup path exists. Users who want zero interruption set `sessions.auto_cleanup_enabled = false` and clean manually via Session -> Clean Up Old Sessions...
- **Per-session-message-count or per-session-size threshold for aggressive cleanup of small sessions.** Considered. Rejected because once a session has at least one user message, it has a derived title and a user-attached intent; the framework should not second-guess the user's sense of which sessions matter. Lazy session-file creation already handles the "empty tab never typed in" case.
- **`Reset Conversation` action / Ctrl+R.** Removed in favor of "open a new tab." The underlying `LLMContext::reset_for_new_conversation` call stays available for programmatic use but is no longer wired to a menu item or shortcut.

## Manifest cost

**Zero new tools.** Pure UX + lifecycle restructure. The agent in a tab sees the same tool catalog it always did; the agent does not know it lives in a tab.
