# S5.I -- Multi-Tab Conversations + Session Menu Redesign

## Problem

A workspace runs one conversation at a time today. Users routinely want multiple parallel agent threads -- "research thread," "implementation thread," "scratch thread" -- without losing each one's context. The Claude Code VSCode extension does this with tabs and the UX is genuinely good. We don't have it.

This stage adds **tabbed conversations within the active workspace** and rebuilds the Session menu around the tab model. Each tab is an independent agent loop with its own history, pin states, compaction state, and checkpoint store. The workspace, index, LLM client, and tool registry are shared across tabs.

Note: this is **multiple sessions in one workspace**, not multiple workspaces in one window (the latter is a different idea, parked in the backlog).

## Tab semantics

| Concern | Per-tab | Shared across tabs |
|---|---|---|
| `AgentCore` (history, agent thread, system prompt, attached context) | yes | -- |
| `ConversationHistory` + per-message pin states ([S5.E](S5.E-pin-and-tag.md)) | yes | -- |
| `ContextBudget` + reserve enforcement ([S5.D](S5.D-reserve-and-token-xray.md)) | yes | -- |
| Compaction state + history archives ([S5.F](S5.F-compaction-v2.md)) | yes | -- |
| Checkpoint store ([S4.B](../M4/S4.B-checkpoint-undo.md)) | yes (separate session id per tab) | -- |
| `Workspace` + `Indexer` + `IndexQuery` + `FileWatcher` + `EmbeddingWorker` | -- | yes |
| `ILLMClient` (one endpoint, same model) | -- | yes |
| `ToolRegistry` | -- | yes |
| `ProcessRegistry` ([S4.I](../M4/S4.I-background-commands.md)) | yes (each tab owns its bg processes; killing a tab kills its tree) | -- |
| MCP servers (when [S4.G](../M4/S4.G-mcp.md) lands) | -- | yes (MCP servers are workspace-level subsystems) |

The mental model: **workspace = shared substrate; tab = an `AgentCore` instance**. `LocusSession` ([S3.G](../M3/S3.G-locus-session.md)) is currently 1:1 with workspace; this stage introduces a `LocusTab` layer so a session bundles workspace-shared subsystems plus N tabs.

## Tasks

### Tab infrastructure

- [ ] New `LocusTab` class in `src/core/locus_tab.h/cpp`: owns one `AgentCore`, the tab's title, the tab's saved-session id (if loaded from disk), the tab's `ProcessRegistry`. Constructor takes references to workspace-shared subsystems.
- [ ] `LocusSession` refactored: holds `Workspace`, `ILLMClient`, `ToolRegistry` as shared resources + `std::vector<std::unique_ptr<LocusTab>> tabs_`. Methods: `add_tab()`, `close_tab(index)`, `active_tab()`, `set_active_tab(index)`. Active-tab index persisted in workspace UI state.
- [ ] Per-tab `ProcessRegistry` ownership change: `Workspace`'s registry stays for workspace-level bg processes (none today, but symbolically clean); tab-spawned bg processes go on the tab's registry. On tab close: tab's registry tears down, killing its process tree.

### Frontend (wxWidgets)

- [ ] `wxAuiNotebook` or equivalent at the top of the chat area. Each tab page contains the existing `ChatPanel` + activity events scoped to that tab.
- [ ] Tab visual indicators:
  - Spinner / animated icon when the agent is mid-turn in that tab
  - Yellow / red dot when context approaches threshold ([S5.F](S5.F-compaction-v2.md) bands)
  - Awaiting-approval icon when the tab has a pending tool call (interacts with [S5.Z](S5.Z-misc-gaps.md) modal-style approval)
- [ ] Tab interactions:
  - Click to switch
  - Middle-click or X button to close (with confirm if conversation has unsaved content)
  - Right-click context menu: Rename, Save as session, Close, Close others, Close to the right, Move to new tab from here (rewind-and-branch-into-new-tab; uses [S5.H](S5.H-edit-delete-branch.md) machinery)
  - Drag to reorder
- [ ] Tab title source: auto-generated from first user message (truncated to ~30 chars), user-editable via right-click rename. Persisted in saved session metadata.
- [ ] "+ New tab" button at the end of the tab strip; keyboard shortcut Ctrl+T.
- [ ] Close-tab shortcut Ctrl+W. Last tab cannot be closed (matches browser behavior); if user tries, the tab resets instead.
- [ ] Approval panel rework from [S5.Z](S5.Z-misc-gaps.md) becomes per-tab-aware: the modal/floating panel attaches to the tab whose agent raised the approval, and switching tabs while one is awaiting decision shows the awaiting-approval icon on the source tab.

### Session menu redesign

The current Session menu mixes "current conversation actions" (Reset, Save) with "session list" (Saved Sessions submenu) and a single global "load session" path. With tabs, this needs to split clearly:

- **File menu** (or Session, by convention):
  - **New Tab** (Ctrl+T) -- empty conversation in new tab
  - **New Tab from Session...** -- pick from saved sessions, opens in new tab
  - **Save Tab as Session** -- explicit save with title prompt
  - **Close Tab** (Ctrl+W)
  - **Reset Tab Conversation** (Ctrl+R) -- the existing "reset conversation" action, scoped to the active tab
  - **Recent Tabs** submenu -- last N closed tabs, click to reopen
- **Saved Sessions submenu** is now a discoverable list with two actions per entry: "Open in new tab" (default) and "Open in current tab" (replaces active tab's content -- confirm dialog if active tab has unsaved content). Add a "Manage sessions..." entry that opens a fuller dialog (rename, delete, export).
- **Workspace switch** stays as it is; it now closes all tabs in the current workspace and opens the new workspace's restored tab set (interaction with [S5.Z](S5.Z-misc-gaps.md) auto-restore).

### Persistence (first-class part of this stage)

- [ ] Saved-session JSON shape extended with optional `tab` metadata: `{title, active_tab_at_save, ...}`. Backward-compatible: old sessions without it open as a single-tab conversation with auto-generated title.
- [ ] Workspace UI state ([src/core/workspace.h](../../src/core/workspace.h) related) gains `open_tabs: [{session_id, title, active}]`. New workspace setting `session.restore_last` (default `true`).
- [ ] **On workspace close**: every open tab auto-saved (using existing `SessionManager::save`), all session ids written to `open_tabs` UI state. Unsaved tabs (never user-saved-with-name) get auto-generated session ids; saved tabs keep theirs.
- [ ] **On workspace open**: if `session.restore_last == true`, every entry in `open_tabs` is reopened (active flag restored). If `false`, a single empty tab opens. The "last single session" behaviour earlier discussed in [S5.Z](S5.Z-misc-gaps.md) is the degenerate case of this -- one tab in, one tab out.
- [ ] Setting in Settings -> Workspace tab so users can opt out. The opt-out implies "every workspace open starts blank" -- explicit user choice, not a hidden default.

### Catch2 tests

- [ ] `LocusTab` lifecycle: create, run a turn, close
- [ ] `LocusSession` multi-tab: add 3 tabs, switch active, close middle tab -- correct state
- [ ] Per-tab `AgentCore` isolation: a turn in tab A doesn't touch tab B's history
- [ ] `ProcessRegistry` per-tab teardown: bg process started in tab A killed when tab A closes; tab B's processes unaffected
- [ ] Saved-session backward compatibility: old single-conversation JSON loads as one tab
- [ ] Tab persistence round-trip: open 3 tabs, close workspace, reopen -- 3 tabs restored

## Implementation notes

- The wx tab widget choice is between `wxAuiNotebook` (already in our AUI dependency tree) and `wxNotebook` (simpler but doesn't dock). Prefer `wxAuiNotebook` since the rest of the layout is AUI-managed -- gives consistent floating / docking behavior and matches the project's anti-Electron stance ("native widgets, not framework chrome").
- LLM concurrency: a single LM Studio (or any local OpenAI-compatible) endpoint typically handles one inference at a time. Two tabs running simultaneously will queue at the server; the second tab's spinner shows "queued" rather than "running" until the first completes. Add a small inline status to the tab strip when this happens. No new threading -- the existing `AgentCore` thread per tab handles the wait naturally.
- Activity log: under [S5.G](S5.G-chat-activity-restructure.md), the activity panel becomes the system audit trail (indexing / embedding / file-watcher). Tab-scoped chat events (user_message / llm_response / tool_call / tool_result) live in their tab's chat. The activity panel stays workspace-shared (not per-tab) because system events are workspace-scoped, not conversation-scoped. Per-tab filtering ("show only events from this tab's agent") is a [S5.Z](S5.Z-misc-gaps.md) bullet for later if needed.
- Memory cost per tab: each `AgentCore` carries its history (a few MB at most for typical sessions), its budget tracker, its checkpoint store handle. Realistic upper bound: ~10 MB per tab. Open 10 tabs = ~100 MB. Acceptable; document the cap as a soft limit but don't hard-cap unless real users hit it.
- The "Move to new tab from here" right-click action on a chat message uses [S5.H](S5.H-edit-delete-branch.md)'s rewind+archive machinery: rewind the active tab's history to the message, archive the rewound lineage, then create a new tab seeded with the archived state up to the rewind point. Powerful but depends on S5.H landing first.
- Backlog item "Multiple workspaces open simultaneously (tabbed sessions)" is **a different concept** -- multiple workspaces in one window. This stage addresses multiple sessions in one workspace. The two could compose later (workspace tabs containing session sub-tabs), but neither blocks the other.
- Tab UI tooltips spell out the tab's full title and session-save state ("Unsaved", "Saved as 'foo' 2 minutes ago").

## Out of scope (v1)

- **Multiple workspaces in one window** (parked in backlog).
- **Drag-drop messages between tabs.**
- **Tab groups / nesting.**
- **Per-tab LLM model override** (each tab targeting a different model). Useful for power users but couples to a multi-model story we don't have. Defer.
- **Cross-tab tool-result sharing** ("agent in tab B can see what tab A read"). Tabs are isolated by design.

## Manifest cost

**Zero new tools.** Pure UX + lifecycle restructure. The agent in a tab sees the same tool catalog it always did; the agent does not know it lives in a tab.
