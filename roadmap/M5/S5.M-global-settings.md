# S5.M -- Global Settings Template + ~/.locus migration

**Theme**: UX / Config

**Status**: shipped 2026-05-14

## Goal

Introduce a per-user global config that templates new workspaces, and move all global resources (`mcp.json`, `prompts/`, `recent_workspaces.json`) from `%APPDATA%\Roaming\Locus` to `~/.locus/` so new installs match the convention used by `.aider`, `.claude`, `.vscode`.

## Motivation

Pre-S5.M, every setting lived per-workspace in `.locus/config.json`. Opening a fresh workspace meant re-doing the same LLM endpoint / sampler / capability decisions every time. Plus three subsystems each rolled their own platform-specific home-dir resolution (`%APPDATA%\Locus\mcp.json`, `%APPDATA%\Locus\prompts\`, `%APPDATA%\Locus\recent_workspaces.json`), all in slightly different ways on Unix.

## Design

**Snapshot-on-first-open template**. The Settings dialog still edits the local workspace config only. A single "Save as global defaults" button at the bottom snapshots the current dialog state into `~/.locus/config.json`. When a NEW workspace is opened (no `.locus/config.json` yet), `Workspace::load_config` reads `~/.locus/config.json` and copies the values into the local file; subsequent opens read the local file directly and never look at the global one again. No live-override semantics -- the workspace is fully self-describing after first open.

**Memory bank stays workspace-only.** A global memory layer would need a separate FTS5 + vec0 table, scoping rules in `search_memory`, separate budgets in the system prompt, and dim-swap migration for both stores. Not worth the surface area until there's a concrete need. User-level facts live in LOCUS.md or a global prompt template.

**MCP servers + prompt templates** keep the workspace-overrides-global merge they already had. Only the file location changed.

**Tool approvals split**: built-in tool keys (`read_file`, `edit_file`, ...) flow through the global template. MCP keys (`mcp:<server>:*`, `mcp:<server>:<tool>`) are stripped at save time -- they reference workspace-specific server names that don't make sense globally. Filter lives in `save_global_config`; load accepts whatever's in the file.

## Implementation

### New modules

- [src/core/global_paths.h/cpp](../../src/core/global_paths.h) -- single source of truth for the global dir. `global_dir()` returns `<home>/.locus` and creates it on first call. `config_path()` / `mcp_config_path()` / `prompts_dir()` / `recent_workspaces_path()` are direct children. Test-only env-var overrides (`LOCUS_GLOBAL_DIR`, `LOCUS_LEGACY_GLOBAL_DIR`) make the unit tests hermetic.
- [src/core/global_config.h/cpp](../../src/core/global_config.h) -- `using GlobalConfig = WorkspaceConfig` (same shape, different file). Two free functions: `load_global_config_or_defaults()` (never throws) and `save_global_config(GlobalConfig)` (returns bool, filters `mcp:*` keys).
- [src/core/workspace_config_json.h/cpp](../../src/core/workspace_config_json.h) -- extracted `workspace_config_from_json` / `workspace_config_to_json` from `workspace.cpp` so the global file can reuse the exact JSON shape without copy-paste. Workspace.cpp now delegates through these helpers; the legacy `config_from_json` / `config_to_json` static helpers are gone.

### Wiring

- `Workspace::load_config` ([src/core/workspace.cpp](../../src/core/workspace.cpp)): when `.locus/config.json` is missing or unparseable, seed from `load_global_config_or_defaults()` and save the result immediately. Logs `Seeded new workspace config at <path> from global defaults` (or `Created default config at <path>` if no global file exists).
- App-startup migration: `LocusApp::OnInit` ([src/frontends/gui/locus_app.cpp](../../src/frontends/gui/locus_app.cpp)) and the CLI `main` ([src/main.cpp](../../src/main.cpp)) both call `global_paths::migrate_legacy_global_dir_once()` before any global path is read. Per-resource: skip if target exists; else copy from legacy location. Idempotent.
- `mcp_config.cpp` / `prompt_templates.cpp` / `recent_workspaces.h`: path resolution shrunk from ~15 lines of `#ifdef _WIN32` / XDG / HOME branches to a single call into `global_paths::*`.
- `SettingsDialog` ([src/frontends/gui/settings_dialog.cpp](../../src/frontends/gui/settings_dialog.cpp)): bottom row is now `[Save as global defaults]  [spacer]  [OK] [Cancel]`. The Save button confirms via wxMessageBox, snapshots the dialog state via the new private `snapshot_dialog_state()` helper, and calls `save_global_config()`. Independent of the OK path -- declining OK does not affect the global save.
- `MenuController` ([src/frontends/gui/menu_controller.cpp](../../src/frontends/gui/menu_controller.cpp)): new **File > Open Global Config...** menu item. Creates the file with compiled defaults if missing, then `wxLaunchDefaultApplication(path)`.

### Migration scope

Each legacy resource is copied independently with `fs::copy_file` / `fs::copy(recursive)`:
- `%APPDATA%\Roaming\Locus\mcp.json`           -> `~/.locus/mcp.json`
- `%APPDATA%\Roaming\Locus\prompts\`           -> `~/.locus/prompts\`
- `%APPDATA%\Roaming\Locus\recent_workspaces.json` -> `~/.locus/recent_workspaces.json`

The legacy directory is left intact so the user can verify the copy and remove it manually. The autostart flag stays in the HKCU Run key (Windows registry; that's how the OS reads it at login).

## Tests

- 5 new `[s5.m][global-config]` Catch2 cases in [tests/test_global_config.cpp](../../tests/test_global_config.cpp) / 42 assertions: missing-file defaults, save->load round trip, `mcp:*` filter at save, migration copy + idempotency, first-open seed (with mid-test global mutation confirming the snapshot model).
- Full unit suite (430 cases, 1853 assertions) still green; no regressions.
- Live verification against WS1 confirmed the migration ran on first launch and was idempotent on the second.
- Manual test plan at [tests/manual/global-settings.md](../../tests/manual/global-settings.md): three broad tests (template flow, legacy migration, Open Global Config menu).

## Out of scope

- Global memory bank (deferred -- see Design section).
- Separate Global Settings dialog (deferred -- the "Save as defaults" button + Open-in-Editor menu cover the use cases without doubling the UI surface).
- Auto-deletion of the legacy `%APPDATA%\Roaming\Locus` folder (user keeps control; the manual test plan walks them through inspecting + deleting it).
- Per-user theme / locale settings (not yet present in WorkspaceConfig; will get the same treatment naturally when they land).
