# S5.B -- Live Terminal Panel

## Problem

Long-running commands give the user no live feedback. A 30 s `cmake --build` is 30 s of frozen UI followed by a wall of text dumped into the activity log; a `run_command_bg`-spawned dev server runs invisibly until the user thinks to ask the agent to read its output. Every comparable dev tool (VS Code, JetBrains, Visual Studio) integrates a terminal pane for exactly this reason. We currently have nothing.

This stage adds an in-app terminal panel that streams stdout / stderr from every command Locus runs (sync and background), with per-process tabs, ANSI color, and the standard show/hide ergonomics.

## Why an in-app wxWidgets panel, not AllocConsole

Considered and rejected: spawning a Windows console via `AllocConsole()` + `freopen("CONOUT$", ...)`. Reasons:

- **Windows only.** Cross-platform is on the M5 horizon; an AllocConsole shortcut becomes a tech-debt landmine the moment we target Mac or Linux.
- **Two-window UX.** The console floats outside the GUI's AUI layout. Closing the console kills the process tree; closing the GUI orphans the console. Both failure modes are confusing.
- **No per-process separation.** One global console for the whole app -- can't separate the build from the dev server from a `pytest` run.
- **Anti-Electron fit.** A second OS window weakens the "one integrated app" stance. Every comparable IDE went the integrated-panel route for this exact reason.

The wxWidgets panel preserves the cross-platform path, lives in the existing AUI dock so the user can dock/float/hide it like any other panel, and gets per-process tabs naturally. Performance concerns (high-frequency output, ANSI parsing) are well-understood and addressable; the engineering cost is real but bounded.

## Scope: user-facing only

This stage delivers **live output to the user**. It deliberately does not deliver streaming partial output back to the LLM. Reasons:

- Local 7-26 B models will not interrupt a running build mid-stream based on partial output -- the decision skill is not there yet.
- The agent's realistic value from a stream is "see the error tail on failure," which `run_command` already provides at exit and `run_command_bg` already provides via `read_process_output`.
- Pushing partial output into the agent's context burns tokens for negligible decision value -- exactly the tradeoff [S5.A](S5.A-capability-toggles.md) is trying to claw back.

If a future workload demonstrates real agent benefit from mid-command streaming (e.g. an agent that sensibly cancels a runaway test), revisit then. Until then, the LLM keeps seeing one tool result per command at completion.

## Tasks

- [ ] New panel `TerminalPanel` in [src/frontends/gui/](../../src/frontends/gui/), AUI-docked (default position: bottom; default state: hidden).
- [ ] View menu: "Terminal" toggle (Ctrl+`). Persists across sessions in workspace UI state.
- [ ] Tab per running process:
  - One tab is reserved for the most recent **sync** `run_command` invocation (cleared on the next sync run; output retained until then).
  - Each `run_command_bg`-spawned process gets its own tab, named after the command (truncated). Tab closes when the user kills the process or clicks the tab close button (which terminates the process, with confirmation).
  - Tab badges: idle / running spinner / exit-code (green for 0, red for non-zero).
- [ ] Output rendering: monospace, virtualised scrollback (cap retained lines per tab to a configurable max, default 10 000). `wxStyledTextCtrl` is the natural surface (already linked).
- [ ] ANSI escape parsing: support SGR (color, bold, dim) and basic cursor-erase sequences (`\x1b[K`, `\x1b[2J`). Skip cursor-position sequences -- a scrollback view doesn't honour absolute cursor moves cleanly.
- [ ] UI throttling: coalesce updates to ~30 fps. A spammy build (`-j 16` parallel compile) can produce hundreds of lines per second; rendering each immediately would peg the UI thread. Buffer chunks on the agent thread, flush on a `wxTimer`.
- [ ] Auto-scroll behavior: stick to bottom unless the user scrolls up (then pause auto-scroll until they hit a "scroll to bottom" button or scroll back down manually).
- [ ] Wire `ProcessRegistry` (S4.I) -- already streams stdout/stderr into a ring buffer; add a fan-out so each chunk also reaches the panel.
- [ ] Wire `RunCommandTool` -- currently captures output to a string. Refactor to stream chunks while still aggregating the full string for the agent's tool result. Existing capture-at-end semantics for the LLM are unchanged.
- [ ] Right-click menu in a tab: copy all, copy selection, clear, save to file, kill process (bg only).
- [ ] Search-in-tab: Ctrl+F opens an inline find bar -- standard wxStyledTextCtrl search.
- [ ] Status bar integration: small chip showing total active processes (clickable -> opens panel + focuses the most recent tab).
- [ ] Persistence: panel state (visible / hidden / floating / docked) round-trips with the rest of the AUI perspective string already saved.
- [ ] Catch2 tests: ANSI parser correctness (SGR sequences, partial-sequence handling across chunk boundaries), throttling behavior under high-frequency input, tab lifecycle.

## Implementation notes

- The existing `run_command_bg` ring buffer in [src/tools/process_registry.h](../../src/tools/process_registry.h) is already the right shape for fan-out -- one reader thread per process drains stdout+stderr into the buffer, and we just add the panel as a second consumer alongside the LLM-facing `read_process_output`. No new threading.
- The throttling buffer lives on the agent thread; the panel pulls coalesced chunks via `wxQueueEvent` -- same pattern `WxFrontend` already uses for streaming LLM tokens. No new cross-thread machinery.
- ANSI parsing: a simple state machine, ~150 lines. Don't pull in a full library; this is a known-bounded format.
- `wxStyledTextCtrl` styling: pre-define a style table for the 16 ANSI colors (and bright variants); SGR codes set the active style for the next inserted text. Reset on `\x1b[0m`.
- Cursor-erase semantics on a scrollback view: `\x1b[K` (erase to end of line) means "delete from current position to end of line" -- in our model that's "delete from caret to end of current line." Implement on the StyledTextCtrl via `LineEndDisplay` + `DeleteBack`. Most builds don't emit these often; cmake/ninja do for progress bars (`[ 42%] Built target ...`).
- A real TTY would let curses-style apps work properly. We deliberately do **not** ship a TTY -- this is a viewer, not a shell. Programs that detect non-TTY (e.g. `cmake` colour, `pytest` colour) usually have a `--color=always` flag if the user wants colour.
- Out of scope for v1: input back to the process (interactive prompts, `git push` password prompts). Document the limit in the panel's empty-state. Future work if demand surfaces.

## Manifest cost

**Zero new tools.** The agent's tool surface is unchanged; `run_command` and the four bg-process tools keep their existing schemas. This stage is pure frontend + a fan-out point in the existing process plumbing.
