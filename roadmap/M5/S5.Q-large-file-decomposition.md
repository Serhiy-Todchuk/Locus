# S5.Q -- Large File Decomposition (chat_panel & settings_dialog)

**Theme**: Refactor / Maintainability

**Status**: done 2026-05-15

## Goal

Break two of the largest M5-touched files into cohesive sub-units so future polish work has a smaller blast radius per change. `chat_panel.cpp` (81 KB) and `settings_dialog.cpp` (61 KB) are the two GUI files that have grown into "everything for this widget" rather than "this widget's coordination logic."

`agent_core.cpp` (49 KB) is intentionally excluded -- S5.J already pulled `LLMContext` out, and the remaining content is the agent thread / message queue / lifecycle surface that S5.J's design treats as an indivisible whole.

## Motivation

The S5.C / S5.B / S5.A / S5.M / S5.J wave landed in the wide GUI files because that's where the surface lived. The result:

- `chat_panel.cpp` now owns: HTML/CSS scaffold, md4c streaming, mode switcher, plan bubble rendering, `@`-mention popup wiring, slash popup wiring, mention parsing dispatch, generation-progress chip, auto-commit chip, LOCUS.md chip, plan chip, context meter footer, footer chip layout, generation cancellation, undo button, diff renderer integration, attached-context renderer, the wxWebView navigation hook for `locus://` URLs, the activity-log notebook tab integration, and the file-tree drag-drop receiver. Every M5 stage that touches "the chat" lands in this file.
- `settings_dialog.cpp` is five tabs (LLM, Index, Capabilities, Tool Approvals, MCP Servers) plus the per-tab build / load / save / validate / preset-apply paths and the new "Save as global defaults" flow, all in one class.

Reviewing a diff against either file means scanning a 1500+ line file for "what changed near my section." Several recent regressions (the assistant-bubble seal-and-restart bug fixed in S5.Z task 1, the sampler field plumbing in S4.V Task 7) cost more debugging time than they should have because the surface area made it hard to see what touched what.

## Design

Two parallel decompositions; either can ship independently.

### chat_panel.cpp

Split into one coordinator + four collaborator classes, all under `src/frontends/gui/chat/`:

- **`ChatPanel`** stays as the wxPanel + the public surface (`set_context_meter`, `set_attached_context`, `clear_chat`, the `IFrontend` callbacks the WxFrontend bridge fans out to). Owns the wxWebView + wxTextCtrl, holds references to the four collaborators, and is the only class allowed to post into the WebView. Target size: under 1000 lines.
- **`ChatStreamRenderer`** (new file, ~600 lines): all the streaming-bubble state (`assistant_id_`, `reasoning_id_`, `current_response_`, `token_buffer_`, `current_reasoning_`, the seal-and-restart machinery from S5.Z task 1, md4c rendering). `ChatPanel::on_token` becomes a one-liner forwarder.
- **`ChatFooterChips`** (new file, ~300 lines): context meter + plan chip + generation-progress chip + auto-commit chip + LOCUS.md chip + commit chip. Owns its own layout and state machine; `ChatPanel` calls `chips_->set_*` setters. Each chip is self-contained.
- **`ChatPopups`** (new file, ~400 lines): slash popup + mention popup wiring, key handling (Esc/Up/Down/Tab/Enter for both), at-cursor detection, accept/dismiss callbacks. `ChatPanel`'s `on_input_key` becomes a router that asks `popups_->handle_key(evt)` first.
- **`ChatLinkHandler`** (new file, ~150 lines): the `wxEVT_WEBVIEW_NAVIGATING` interceptor for `locus://plan-approve|reject/<id>`, future `locus://memory/<id>`, etc. Today this lives inline in `chat_panel.cpp` and grows on every URL-driven feature.

Key invariant: only `ChatPanel` calls into the wxWebView. Collaborators emit `wxString js;` fragments that `ChatPanel` evaluates. This keeps thread-safety reasoning local (the WxFrontend bridge already enforces UI-thread access; this just keeps the rule readable).

### settings_dialog.cpp

The current single `SettingsDialog` becomes a thin shell with a `wxNotebook` and one panel class per tab, all under `src/frontends/gui/settings/`:

- **`SettingsDialog`** (~250 lines): notebook + OK / Cancel / Save-as-global-defaults buttons. Per-tab panels are constructed in order, validated on OK by calling `panel->validate()` -> `panel->commit_to_config(WorkspaceConfig&)`.
- **`LlmSettingsPanel`** (~400 lines): endpoint, model, context, temperature, max_tokens, samplers (top_p / top_k / min_p / repeat_penalty), tool_format, preset row, "test connection" button.
- **`IndexSettingsPanel`** (~250 lines): exclude patterns, semantic search toggle, embedding model, reranker toggle, gitignore respect.
- **`CapabilitiesSettingsPanel`** (~150 lines): the five capability checkboxes + token-cost labels (S5.A surface, mostly unchanged).
- **`ToolApprovalsSettingsPanel`** (~250 lines): per-tool Ask/Auto/Deny dropdowns + the S4.V Task 4 friction-confirm for mutating tools.
- **`McpSettingsPanel`** (~250 lines): MCP server list, status display, restart button, trust toggle, Open mcp.json button.

Each panel implements a tiny common interface:

```cpp
class ISettingsPanel {
public:
    virtual ~ISettingsPanel() = default;
    virtual void load_from_config(const WorkspaceConfig&) = 0;
    virtual bool validate(wxString& out_error) const = 0;
    virtual void commit_to_config(WorkspaceConfig&) const = 0;
};
```

The "Save as global defaults" flow becomes: dialog asks each panel to `commit_to_config` against a scratch `WorkspaceConfig`, then calls `save_global_config` (S5.M unchanged).

## Implementation notes

- Refactor only -- zero behaviour change. Acceptance is the existing manual test plans for S5.A / S5.M / S5.C / capabilities / settings-tour all pass without edits.
- The S5.L `settings_tour.json` UIA script is the canary -- the `automation_id` strings under `src/frontends/gui/ui_names.h` must keep their current values so the script keeps clicking the same widgets. Each per-tab panel's `SetName` calls move with the widgets, not the dialog.
- File-rename churn is the main reviewer cost. Land each split as one commit (`refactor: extract ChatStreamRenderer from ChatPanel`, `refactor: split SettingsDialog into per-tab panels`) so a bisect on a future regression points at the right surface.
- Keep all five new files in `locus_core` (the existing build target). No new CMake target.

## Tests

No new tests. Existing coverage:

- 472 unit tests still pass byte-for-byte.
- `settings_tour.json` UIA script ([tests/ui_automation/scripts/settings_tour.json](../../tests/ui_automation/scripts/settings_tour.json)) walks all five tabs.
- Manual test plans referenced above.

If a manual test reveals a regression, that's the signal to add a unit test before fixing -- the refactor itself adds none.

## Out of scope

- `agent_core.cpp` further decomposition. S5.J's design treats the remaining content as the agent-loop runner / message-queue / lifecycle owner; further splitting risks scattering the threading invariants that `architecture/threading-model.md` documents.
- `locus_frame.cpp` (now ~1300 lines after S3.F + S5.B + S5.A pulled `MenuController` / `OpsStatusView` out). It's near the threshold but not over; revisit if S5.G / S5.I push it past 50 KB.
- Any logic changes inside the panels (sampler validation, MCP restart UX, etc). Those belong to whichever stage adds the feature, not to the refactor.

## Manual test plan

None new. The decomposition is complete when:

1. Full unit suite passes.
2. `locus_ui_tests.exe --all` passes (smoke + settings_tour + chat_round_trip).
3. The five tab-specific manual test plans pass without edits.
