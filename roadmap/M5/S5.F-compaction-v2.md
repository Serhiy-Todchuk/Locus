# S5.F -- Compaction v2 (Toolkit + Auto + History Archive) -- done 2026-05-16

## Problem

The current compaction story has structural gaps closed by this stage. From the cross-tool comparison (Pi / Claude Code / Cline / Roo Code) and the in-house design discussion:

1. Strategy A (LLM summary) is a non-functional radio button -- the headline gap.
2. Strategies are presented as mutually-exclusive radio buttons, but they should be **composable** -- a single compaction pass can run several layers.
3. No auto-compaction. Long unattended coding sessions overflow with no recovery.
4. No history archive. After a destructive compaction, the user (or the LLM, in a follow-up stage) cannot recover the original context. The dropped span is gone forever.
5. No custom-instruction prompt. Users can't say "preserve all file paths and test commands."
6. Threshold bands are hard-coded at 80% / 100% and operate against `context_limit` rather than the [S5.D](S5.D-reserve-and-token-xray.md) effective limit.

## Composable compaction layers (the design centerpiece)

Compaction is no longer "pick one strategy." It's a pipeline of layers, each independently toggleable, run in fixed order from least to most destructive. Each layer has its own knobs.

| # | Layer | Lossiness | Knobs | Default for unattended auto-compact |
|---|---|---|---|---|
| 1 | Drop redundant tool results (same tool + args called twice -> keep latest, replace earlier with one-line ref) | None | -- | ON |
| 2 | Strip large tool result bodies (above N tokens -> replace with header; "link to artifact" was conditional on the parked [S5.H](../backlog/S5.H-edit-delete-branch.md) archive chain -- header-only for v1) | Low (header retained) | `strip_threshold_tokens` (default 1000) | ON |
| 3 | Strip inline `<think>...</think>` blocks from older assistant `content` (no-op when reasoning rides the dedicated `reasoning_content` channel -- that channel never enters history; catches the leak case where the chat template doesn't strip server-side) | None -- `<think>` blocks are private scratch space, not conversational record | `older_than_turns` (default 3) | ON |
| 4 | _(parked)_ Drop low-priority tagged messages -- requires the parked [S5.E](../backlog/S5.E-pin-and-tag.md) pin/tag system | Per user tagging | -- | n/a (skipped) |
| 5 | Drop oldest default turn pairs (subject to auto-preservation heuristics -- see below) | Mechanical (loses content) | `keep_recent_turns` (minimum to retain) | OFF (escalation only) |
| 6 | LLM summary of dropped span (Strategy A, finally functional) | Lossy (depends on model) | `custom_instructions` (free-form Pi-style); `summary_max_tokens` cap | ON (when 1-3 don't free enough) |

Each layer runs only if the previous layers haven't freed enough room to bring usage under the target. The target is "below the soft threshold with the [S5.D](S5.D-reserve-and-token-xray.md) reserve still intact."

**Auto-preservation heuristics** (replace the parked [S5.E](../backlog/S5.E-pin-and-tag.md) explicit pin states for v1). Layer 5 and Layer 6 both skip messages matching any of:
- The system message (always preserved -- [S4.F](../M4/S4.F-kv-cache.md) invariant).
- The first user message of the session (the original goal / problem statement -- preserved regardless of length).
- The most recent `keep_recent_turns` turn pairs (typical default: 2-3).
- Any message that injected an `[Attached file: ...]` block (the user explicitly pinned that file).
- **Short user messages** -- any user-role message with content `<= preserve_short_user_msgs_max_tokens` (default 500). Rationale: typed prompts are typically 5-50 tokens and form the *trail of intent* through the conversation; preserving them lets the model still answer "what did I ask for 3 turns ago?" after compaction. The threshold lets bulky content pastes (multi-kilobyte stack traces / docs) get compacted normally. Disable by setting the threshold to 0.
- **Short tool calls** -- any tool-call message (the call args, not the result) with content `<= preserve_short_tool_calls_max_tokens` (default 500). Tool-call args (`read_file({path: "foo.cpp"})`) are short and high-signal; preserving them lets the model reconstruct *what was looked at* even after Layer 2 strips the result body. Tool results stay governed by Layer 2's threshold. Disable by setting the threshold to 0.

These are mechanical and need no per-message state. If real usage shows the heuristics miss real cases, reactivate [S5.E](../backlog/S5.E-pin-and-tag.md) and add Layer 4 back.

### Manual compaction UI

The existing `CompactionDialog` modal stays as the surface. Its **internals** are rebuilt: instead of the current radio-button-per-strategy + N-turns-slider, the body shows checkboxes per layer (each layer's knobs editable inline) plus a "Run" button. Live preview inside the dialog shows:
- Tokens before / after / freed (per layer breakdown)
- Which messages each enabled layer would touch
- A diff-like preview of what the LLM-summary message will replace (when Layer 6 enabled)

Modal because compaction is a discrete one-shot decision, not a continuous inspector -- side panels earn their permanent screen real estate by being something the user glances at all the time (Activity log, file tree). The compaction preview is something the user looks at for ~30 seconds twice an hour at most. That's modal-shaped work, and the existing `CompactionDialog` shell is already wired through `LocusFrame` -- only the inner sizer needs replacement.

### Auto-compaction (unattended sessions)

When auto-compact fires (see "Three-state progressive trigger" below), the cascade runs Layer 1 -> 2 -> 3 in order, stopping when usage drops under target. If 1-3 don't free enough, Layer 6 (LLM summary) fires automatically. Layer 5 (mechanical drop of oldest turns) is reserved for explicit user invocation -- the cascade prefers a faithful summary over silent loss. (Layer 4 is parked along with [S5.E](../backlog/S5.E-pin-and-tag.md).)

Auto-compaction surfaces a notification chip in the activity log: *"Auto-compact: 3 redundant tool results dropped (1.4k freed); summary of turns 5-12 inserted (3.1k freed)."* User can click to inspect the resulting state. Recovery is manual file inspection of the history archive (programmatic restore was the parked [S5.H](../backlog/S5.H-edit-delete-branch.md)).

## Three-state progressive trigger

| State | Trigger | Behaviour |
|---|---|---|
| Warn | usage > `compaction.warn_threshold` of effective_limit (default 70%) | Context meter turns yellow; small inline hint in chat ("compaction will trigger soon") |
| Auto-compact | usage > `compaction.auto_threshold` of effective_limit (default 85%) | Layers 1-4 + 6 cascade fires; user notified; no modal interruption |
| Hard refuse | usage + reserve > `context_limit` (the [S5.D](S5.D-reserve-and-token-xray.md) reserve breach) | Agent loop refuses new input; user must compact manually or pin/edit/delete in chat |

All three thresholds configurable per-workspace. `compaction.auto_enabled` (default `true`) gates the auto-compact band -- users who want strict manual control turn it off and the warn/refuse states still function.

## History archive (chained backups)

Before each compaction (auto or manual), the full pre-compaction history is snapshotted to:

```
.locus/sessions/<session_id>/history.before-compact-<N>.json
```

`<N>` is a monotonic counter per session. After three compactions, four files exist: the current live session JSON plus three archives.

The new compacted context inserts a footnote in the system-message preamble (or as the first user-side note depending on KV-cache discipline -- see [S4.F](../M4/S4.F-kv-cache.md)):

```
[Earlier context compacted; full pre-compaction history archived at
.locus/sessions/<id>/history.before-compact-2.json]
```

The footnote is for the **user**. The user can open the archive manually (the file is plain JSON on disk). Programmatic restore-from-archive was originally scoped as a [S5.H](../backlog/S5.H-edit-delete-branch.md) operation; that stage is parked, so v1 leaves recovery as manual file inspection. The LLM does not get a tool to read these archives in v1; that's a deliberate choice -- see "What this is NOT" below. A backlog item tracks lifting the LLM-side restriction conditionally.

GC: configurable `compaction.archive_keep_count` (default 5) caps the archive chain per session; older archives are deleted on the next compaction.

## What this is NOT (deliberate scope cuts)

- **No agent-side history-archive tool in v1.** The compacted footnote tells the user where the archive lives; the LLM cannot call a tool to read it. Adding such a tool risks the agent burning tokens re-reading content that was compacted for a reason. Backlog item tracks lifting this if a real workload demonstrates value (e.g., agent saying "I summarized X earlier; let me re-check the original").
- **No persistent per-message pin / tag states, and no transient per-run "keep this" override either.** Originally scoped as [S5.E](../backlog/S5.E-pin-and-tag.md); parked. The auto-preservation heuristics (system message, first user message, recent N turns, attached-context messages, short user messages, short tool calls) cover the load-bearing cases without committing the saved-session JSON schema or adding a per-message selection UI to the dialog. The dialog's per-layer checkboxes let users coarsely tune *what kinds of content* to compact (e.g., disable Layer 5 entirely to preserve all old turns at the cost of more Layer 6 work); finer-grained control would mean reactivating S5.E.
- **No per-message manual edit in this stage.** Edit was originally part of the parked [S5.H](../backlog/S5.H-edit-delete-branch.md). Per-message delete lands in [S5.G](S5.G-chat-activity-restructure.md) as a hover-reveal X button -- the only S5.H slice that pays rent. v1 of compaction is governed by the heuristics above, not by surgical message tools.
- **No model-aware threshold tuning.** Cline scales aggressiveness based on the loaded model's context size; we don't. The single set of thresholds applies everywhere; users with very small or very large contexts override.
- **No external LOCUS.md sectioning.** Backlog (per the design discussion).

## Implementation notes

- The compaction footnote placement is the most subtle [S4.F](../M4/S4.F-kv-cache.md) interaction. The system prompt must remain byte-stable across compactions to preserve KV cache. Rule: the system prompt **never changes** during compaction; the footnote (referencing the archive path) is injected as a synthetic *user-role* message at the top of the post-compaction history, where the volatile tail expects rewrites anyway.
- Layer 6 (LLM summary) costs an LLM call per compaction. On a 26B local model, that's tens of seconds of dead air. Auto-compact should fire it sparingly (only when 1-3 didn't free enough), and the UI should show a spinner + "compacting..." status during the call.
- Layer 6 prompt template lives in `src/agent/compaction_prompts.h` as a const string with `{conversation}`, `{custom_instructions}`, `{file_ops}` substitution slots. Keep it small (~400 tokens); the structured-template approach Pi uses is the design target. File-ops accumulation across compactions is a [S5.Z](S5.Z-misc-gaps.md) bullet (separate cumulative tracker; not in scope here).
- The rebuilt `CompactionDialog` body reuses Scintilla / chat-rendering primitives where possible -- no new widget systems. The diff-style preview can lean on the per-message machinery from [S5.D](S5.D-reserve-and-token-xray.md). Persistent pin states were originally scoped as [S5.E](../backlog/S5.E-pin-and-tag.md) but are parked; the auto-preservation heuristics above (short user messages, short tool calls, attached-context, recent N, first user, system) cover the v1 use cases without per-message state.
- Slash-command `/compact` integrates with the `SlashCommandDispatcher` ([S3.J](../M3/S3.J-slash-commands.md)) -- standard parser machinery; `instructions` parsed as the slash command's free-form arg.
- The existing `compact_context()` agent-core method becomes a thin shim during transition; remove once the new pipeline is wired in everywhere.

## Manifest cost

**Zero new agent-facing tools.** The agent does not gain tools to inspect / trigger compaction or to read archives (in v1). Compaction remains a Locus-internal mechanism the model is unaware of, except that summarized turns appear as a single assistant message in the history.
