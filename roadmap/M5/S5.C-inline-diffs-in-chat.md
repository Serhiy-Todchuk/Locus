# S5.C -- Inline Code Diffs in Chat

## Status: shipped

Implemented via `src/frontends/gui/diff_renderer.{h,cpp}` and wired into the
chat panel. Covered by `tests/test_chat_diff_rendering.cpp` plus the manual
plan at `tests/manual/inline-diffs-in-chat.md`.

## Problem

Today the chat history shows an `edit_file` / `write_file` / `delete_file` call as a small `msg-tool` bubble with the tool name and an optional preview, plus a collapsible `<details>Result</details>` block with the success message. The user can see *that* a file changed but not *what* changed without re-opening the file. The approval panel already renders a red/green unified diff (S4.A: `render_diff_view()` in [src/frontends/gui/tool_approval_panel.cpp](../../src/frontends/gui/tool_approval_panel.cpp)) -- but only at the approval moment, in a different widget. Once approved, the chat history loses that visual.

For a transparency-first agent, "what did the agent actually change?" is the most important post-hoc question. It should be visible at a glance in the chat, not require opening the file diff in another tool.

## Scope

Render an inline unified diff in the chat for each successful mutating tool call:
- `edit_file` -- per-edit hunks; old_string in red, new_string in green; multiple edits in one call render as multiple hunks under one file header.
- `write_file` -- full-content diff between previous file state (from the [S4.B](../M4/S4.B-checkpoint-undo.md) checkpoint snapshot) and new content. New file (no prior content) renders as all-green; replacing an existing file renders as a real before/after diff.
- `delete_file` -- one-line summary "deleted: `path` (N lines)" -- no diff body.

Diffs are collapsible, default-open. Cap rendered hunks at a configurable size; over-large changes show "(N additional hunks collapsed)" with click-to-expand.

## Implementation choice: dtl vs shell-out

| Path | Pros | Cons |
|---|---|---|
| **`dtl`** (header-only C++ diff template library, MIT) | In-process; deterministic; no shell-quoting; cross-platform; small (header-only); already mentioned in backlog as the line-level-diff candidate | New dependency (vcpkg has it: `dtl`) |
| **Shell out to `diff -u` / `git diff --no-index`** | No new dependency | `diff` is POSIX, missing on Windows by default; `git diff` requires git on PATH; subprocess overhead per render; shell quoting on file paths; cross-platform behavior diverges |

**Pick `dtl`.** It matches the project's stack (vcpkg + static CRT + cross-platform target), adds <2 MB to binary size, and avoids the shell entirely. Add via vcpkg or `FetchContent`.

For `edit_file` specifically, **dtl is not needed** -- the tool's args already include `old_string` (the matched content) and `new_string` (the replacement) per edit. Rendering each edit's chunk as a hunk needs no diff algorithm; it's already a diff. dtl only earns its keep on `write_file` overwrites where we have to compare full file content against the [S4.B](../M4/S4.B-checkpoint-undo.md) checkpoint snapshot.

## Implementation notes

- The same `render_diff_html()` helper could later replace the wxStyledTextCtrl-based `render_diff_view()` in the approval panel ([S4.A](../M4/S4.A-diff-editing.md)) -- one diff rendering path, two surfaces. Out of scope for this stage; the approval panel's existing implementation works and uses a different widget. Note for a future cleanup pass.
- Markdown re-rendering interaction: the current chat-panel render path runs every tool message's content through `markdown_to_html()` (md4c). Diff HTML must NOT go through md4c (it would mangle the line spans). The `render_diff_html()` output should bypass md4c and inject directly via a dedicated path -- mirrors how `<details>Result</details>` is currently injected.
- KV cache invariance: this is purely frontend rendering. No system-prompt change, no LLM-facing addition. The agent's tool-result message is unchanged.
- Streaming consideration: the diff is rendered once on `on_tool_result`, after the tool has executed. No mid-call streaming -- diff requires both before and after states.
- Theme support: red/green colours must respect dark/light mode. Use the same theme-variable approach the chat panel already uses (`--add-bg`, `--del-bg` CSS variables driven by `prefers-color-scheme` or the explicit theme setting).

## Manifest cost

**Zero new tools.** Pure frontend rendering enhancement on top of the existing `edit_file` / `write_file` / `delete_file` tool calls. The agent's tool surface is unchanged.
