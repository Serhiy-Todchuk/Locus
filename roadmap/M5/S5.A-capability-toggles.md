# S5.A -- Workspace Capability Toggles

## Problem

Locus's tool manifest currently ships every built-in tool to the LLM regardless of whether the workspace will ever use it. The design target for coding workloads is **32k context** (Gemma 4 E4B at 8k is a verified minimum, not the typical case). Even at 32k, the manifest tax is real: background-process tools alone cost ~700 prompt tokens of schema and are useless on a doc workspace; code-aware search modes cost another ~250 and are useless on a wiki archive. The Pi comparison highlighted this -- their default-on tool set is ~4 tools; ours is 13. The wins compound on smaller contexts (where every percent matters) without being negligible at 32k.

The infrastructure to gate this already exists. [S3.L](../M3/S3.L-tool-catalog-hygiene.md) added `available()` / `visible_in_mode()` hooks on `ITool` exactly for conditional manifest exposure. What's missing is a per-workspace policy that drives those hooks and a UX that lets the user opt in/out of capability buckets.

This stage repurposes the existing first-open semantic-search prompt as a multi-checkbox modal covering the buckets that actually move the needle, and mirrors those checkboxes in Settings for later changes.

## Capability buckets and defaults

| Bucket | Tools / state gated | ~Prompt tokens saved when OFF | Default |
|---|---|---|---|
| Background processes | `run_command_bg`, `read_process_output`, `stop_process`, `list_processes` | ~700 | **OFF** |
| Semantic search | `search` modes `semantic`/`hybrid` (description); embedder model load | ~80 + RAM/disk | **ON** |
| Code-aware search | `search` modes `symbols`/`ast` (description); `get_file_outline` | ~250 | **ON** |
| Memory bank ([S4.R](../M4/S4.R-memory-bank.md)) | `add_memory`, `search_memory` tools; in-context memory slot in system prompt; `/memorize` slash command | ~300 + up to `memory.in_context_budget_tokens` (default 500) | **ON** |
| Web retrieval (M6 [S6.1](../M6/S6.1-web-retrieval.md)) | web fetch/search tools when they ship | ~300 | **OFF** |

Static defaults -- no auto-detection. The user makes the choice on first open and can revisit in Settings.

Worst case (a doc-style workspace with everything off except read/edit/write/search-text/run_command): ~1300-1800 prompt tokens reclaimed vs an all-on code workspace. At 32k that's 4-6% of the budget back; at 8k it's 16-22%. The reclaimed budget is most valuable when stacked with the [S5.D](S5.D-reserve-and-token-xray.md) reserve enforcement -- every prompt token saved is one less token competing with the response-headroom guarantee.

## Tasks

- [x] `WorkspaceConfig` (in [src/core/workspace.h](../../src/core/workspace.h)) gains a `capabilities` sub-struct holding one bool per bucket. Persisted to `.locus/config.json` under `capabilities: {...}`. Defaults match the table above.
- [x] First-open modal: replace the current semantic-search-only prompt with a "Workspace capabilities" dialog showing all five buckets as checkboxes, each with the default applied, a one-line description, and the prompt-token estimate next to it (e.g. "Background processes (~700 tokens)"). Single OK / Cancel; OK writes config + opens the workspace; Cancel still opens with defaults (no nag).
- [x] Settings dialog ([src/frontends/gui/settings_dialog.cpp](../../src/frontends/gui/settings_dialog.cpp)) gains a "Capabilities" tab with the same five checkboxes. Saving applies changes immediately -- the manifest is rebuilt on the next turn.
- [x] Per-bucket `available()` wiring on the affected tools:
  - `RunCommandBgTool` / `ReadProcessOutputTool` / `StopProcessTool` / `ListProcessesTool` return `false` when `capabilities.background_processes` is off.
  - `GetFileOutlineTool` returns `false` when `capabilities.code_aware_search` is off.
  - `AddMemoryTool` / `SearchMemoryTool` ([S4.R](../M4/S4.R-memory-bank.md)) return `false` when `capabilities.memory_bank` is off; the in-context memory slot is also suppressed in the system-prompt assembly when off.
  - `SearchTool` keeps shipping (text/regex are always available) but its `description()` prunes the comma-separated mode list to drop `symbols`/`ast` when `code_aware_search` is off and `semantic`/`hybrid` when `semantic_search` is off. The token saving comes from the description, not from withholding the whole tool.
- [x] Per-bucket tooltip in both UIs spelling out exactly which tools are gated and the current token cost. No mystery checkboxes.
- [x] Failure-mode UX: if the agent emits a tool call for a currently-disabled tool, surface a one-line activity-log event ("`run_command_bg` is disabled in this workspace -- enable in Settings -> Capabilities") instead of the generic "unknown tool" error. Keeps the user oriented when the agent has stale assumptions about the tool surface.
- [x] Manifest-size telemetry: extend the existing per-turn manifest-size log (S3.L threshold warning) so the log line reports both the raw byte count and the bucket combination active. Cheap diagnostic when debugging "why is this turn slow".
- [x] Migration: existing workspaces without a `capabilities` block in `.locus/config.json` get the default values silently injected on first load. No re-prompt for users who already accepted the old semantic-search-only dialog -- their existing semantic preference carries through.
- [x] Catch2 tests in `tests/test_capability_toggles.cpp`:
  - default values match the table above
  - each bucket toggling actually changes the tool manifest the LLM sees (assert via `ToolRegistry::available_tools()` or the schema builder)
  - `SearchTool::description()` text varies correctly with the two search-related buckets
  - config round-trip: write, reload, values preserved
  - migration: workspace without `capabilities` block loads with defaults

## Implementation notes

- This is wiring, not new mechanism. The `available()` hook from [S3.L](../M3/S3.L-tool-catalog-hygiene.md) is the gating primitive; this stage just adds a `WorkspaceConfig`-backed predicate behind it. No new tool-execution paths, no new prompt-injection points.
- The "search modes pruned from description" approach matters for the *text* inside the manifest. The schema for `SearchTool` keeps the `mode` parameter as a string (no enum constraint), so the LLM can still send `mode=semantic` -- it just won't be advertised. If it does send a hidden mode, the runtime returns the existing "Error: semantic search is not enabled" message. This is intentional: schema-level enum changes between turns would invalidate KV cache (see [S4.F](../M4/S4.F-kv-cache.md)).
- The token estimates next to each checkbox are static numbers documented in code (one constant per bucket) -- not live-computed. Keeps the dialog snappy and the numbers honest enough for a budget choice. Re-measure once a milestone if tool descriptions drift.
- Settings changes take effect on the next turn, not immediately. A live system-prompt swap mid-turn would torch the cache and is the wrong moment to apply policy.
- Future buckets to consider (not in scope for v1): MCP-as-a-bucket (when [S4.G](../M4/S4.G-mcp.md) ships), plan mode availability, memory bank. Add them to the same dialog as they land. Five is the soft cap before dialog fatigue sets in -- if we exceed five, the dialog should grow an "Advanced..." section rather than a longer top-level list.
