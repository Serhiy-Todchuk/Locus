# S5.G -- Chat / Activity Restructure

## Problem

Chat and activity log have overlapping but inconsistent responsibilities. Tool calls appear in both (chat as `msg-tool` bubbles, activity as `tool_call`/`tool_result` events). The system prompt is invisible to the user. Indexing and embedding events live in spdlog at trace level only -- the activity log shows nothing useful about what the index is doing right now. Per-message inspection / token-cost controls (introduced in [S5.D](S5.D-reserve-and-token-xray.md)) need a clean home; today's split is the wrong place to put them.

The cleanest division of responsibility:

- **Chat** = everything the LLM sees. System prompt (collapsible bubble at the top), user messages, assistant text + reasoning, tool calls with arg / result rendering, tool results.
- **Activity log** = everything the LLM does NOT see. Indexing progress, embedding queue progress, file-watcher events, errors, workspace lifecycle, per-event timestamps, an audit cross-reference to chat-message ids.

No duplication. Chat is the model's view; activity log is the system's audit trail.

## Tasks

### Chat side: become the LLM's view

- [ ] **System prompt as a collapsible bubble** at the top of the chat -- styled like the existing reasoning ("Thinking...") block. Closed by default; expanding shows the full prompt with the per-section breakdown chip from [S5.D](S5.D-reserve-and-token-xray.md). Updates on `refresh_system_prompt` (e.g., attach/detach) -- the bubble re-renders, but the [S4.F](../M4/S4.F-kv-cache.md) byte-stability invariant remains owned by `AgentCore`, not the chat panel.
- [ ] **Tool-call bubbles improved incrementally** (deliberate scope cut -- not a full Claude-Code-style rework):
  - Args displayed as collapsed JSON tree (single-line summary at top, click to expand). Reuse Scintilla / md4c primitives if practical.
  - File-mutating tools (`edit_file` / `write_file` / `delete_file`) inherit the inline diff from [S5.C](S5.C-inline-diffs-in-chat.md).
  - Tool-result `<details>` block already exists; add a token-count chip alongside.
  - Failed tool calls visually distinguished (red border / icon) -- already partially done.
- [ ] **Token chip per bubble** (from [S5.D](S5.D-reserve-and-token-xray.md)) integrated into every bubble shape (system prompt / user / assistant / tool call / tool result).
- [ ] _Persistent pin button per bubble was originally scoped here (sourcing from [S5.E](../backlog/S5.E-pin-and-tag.md)); both are parked. [S5.F](S5.F-compaction-v2.md)'s auto-preservation heuristics + CompactionView per-run "keep this" overrides cover the v1 use cases._
- [ ] **Reasoning blob handling** -- existing collapsible "Thoughts" block stays. Token chip applies. [S5.F](S5.F-compaction-v2.md) Layer 3 may drop reasoning during compaction; the chat shows a placeholder bubble in that case ("[reasoning dropped by compaction]") rather than vanishing the blob entirely.
- [ ] **Bubble id round-trip** -- every chat bubble carries the `ChatMessage` id so the activity log can cross-reference.
- [ ] **Per-message delete** -- hover over any chat bubble reveals a small `x` button in the corner. Click prompts a single yes/no confirm ("Delete this message? Subsequent turns will no longer see it.") and drops the message from `ConversationHistory`. No archive infrastructure -- if the user wants to back up several turns they delete the last few in sequence. System-prompt bubble has no delete affordance (owned by `AgentCore`, byte-stable for [S4.F](../M4/S4.F-kv-cache.md)). This is the load-bearing slice of the parked [S5.H](../backlog/S5.H-edit-delete-branch.md) -- covers redaction of secrets / PII landed in tool results, dropping a fat stale tool result the compaction heuristics didn't catch, and trimming wrong-direction reasoning. Edit / rewind / branch / restore-from-archive stayed in S5.H and are parked for now.

### Activity log side: become the system audit trail

- [ ] **Add the missing event sources** to the activity log:
  - Indexing events: file added / removed / re-indexed / batch complete. Already exist as spdlog trace; promote to `IFrontend::on_activity` with a new `index_event` kind.
  - Embedding queue progress: chunks queued / completed / errored. Currently invisible. Add a periodic progress event from `EmbeddingWorker` (every N chunks or M seconds, whichever first).
  - File-watcher activity: external file changes (already partially tracked via [S4.T](../M4/S4.T-file-change-awareness.md)) surfaced as activity events alongside the volatile-tail injection.
- [ ] **Per-event timestamp** column in the activity panel (already in `ActivityEvent`; surface in the renderer).
- [ ] **Cross-reference** -- chat-related events (user_message / llm_response / tool_call / tool_result) carry the bubble id. Clicking an activity row scrolls / highlights the matching chat bubble.
- [ ] **Filtering** -- activity panel gains a small filter row: "all / errors / indexing / chat-related." Default "all"; users with chatty sessions can filter to just errors.
- [ ] **No more chat-event duplication** -- once chat shows everything the LLM sees, the activity panel stops rendering full bubbles for tool calls. It keeps a one-line audit row per event, with a click-through to the chat bubble.

### Wiring

- [ ] `IFrontend::on_activity` retains all existing event kinds; new `embedding_progress` and `index_event` kinds added.
- [ ] `EmbeddingWorker` posts progress through the existing core-callback path (same channel `Indexer` already uses).
- [ ] Chat panel's `addMsg` JS helper takes an extra `id` argument; activity rows store the same id; click handler navigates between them.
- [ ] Catch2 tests:
  - system-prompt bubble renders the right token-cost breakdown
  - bubble id round-trips through save / load
  - activity log receives `embedding_progress` and `index_event`
  - cross-reference: clicking an activity row produces a `scroll-to` script for the right chat element

## Out of scope (deliberately, per design discussion)

- **Full Claude-Code-style tool-bubble polish** (clickable file paths, syntax-highlighted args trees, advanced collapse states). [S5.C](S5.C-inline-diffs-in-chat.md) does the load-bearing diff work; the rest is incremental polish that can land in [S5.Z](S5.Z-misc-gaps.md) bullets if friction surfaces in real use.
- **Activity panel as a context inspector** (the alternative path I floated and we rejected). The inspector lives in the chat now; activity panel stays an audit log.
- **Removing the activity panel entirely.** It still has a job -- non-LLM events. We just move the LLM-visible events out of it.

## Implementation notes

- The system-prompt bubble is the most novel piece. Visually it should feel like part of the chat (no special background), open inline (not a separate dialog), and show per-section token costs as a small inline chip cluster (Base: 800 / LOCUS.md: 600 / Tools: 1300 / Metadata: 100). This is the single best surface for users to understand "where my budget went."
- KV-cache discipline ([S4.F](../M4/S4.F-kv-cache.md)) is *not* affected by this stage. The system prompt stays byte-stable; the bubble just *renders* it. Re-rendering doesn't trigger a refresh of `base_system_prompt_`.
- The activity panel's filter UI is a small wxChoice + a checkbox row. No new wx widgets.
- Migration: existing saved sessions don't have bubble ids. On load, generate ids from message indexes; old activity logs lose cross-reference but render correctly otherwise.
- The "no more chat-event duplication" cleanup is the main risk -- making sure no production code path depends on the activity panel having full tool-bubble content. Audit `ActivityPanel` and any clients that read its events.

## Manifest cost

**Zero new tools.** Frontend reshape only.
