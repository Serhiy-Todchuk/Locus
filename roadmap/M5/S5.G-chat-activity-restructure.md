## Problem

Three concrete gaps to close (the original "chat and activity log have overlapping responsibilities" framing was reviewed and dropped on 2026-05-16 -- see "Out of scope" for the rationale):

1. **The system prompt is invisible to the user.** No surface shows "what the LLM actually starts with each turn" or where the context budget is going (base / LOCUS.md / tool manifest / metadata). The token gauge in the footer is a single aggregate number.
2. **Per-message inspection / control has no home in chat.** [S5.D](S5.D-reserve-and-token-xray.md) ships per-message token estimates, but the chat bubble has no slot to render them. The chat also has no way to drop a single message without nuking the whole session (the load-bearing slice of the parked [S5.H](../backlog/S5.H-edit-delete-branch.md)).
3. **Some events users care about are invisible in the activity panel.** Most notably embedding queue progress (currently no signal between "indexing kicked off" and "everything's ready"), and per-event timestamps (data already in `ActivityEvent`, just not rendered).

## Tasks

### Chat side

- [x] **System prompt as a collapsible bubble** at the top of the chat -- styled like the existing reasoning ("Thinking...") block. Closed by default; expanding shows the full prompt with the per-section breakdown chip from [S5.D](S5.D-reserve-and-token-xray.md) (see Implementation notes for the chip-cluster layout). Re-rendered on session reset; the [S4.F](../M4/S4.F-kv-cache.md) byte-stability invariant stays owned by `AgentCore`. _Shipped via new `addSystemPromptMsg` JS helper + `.msg-system-prompt` CSS + `ChatPanel::set_system_prompt_bubble`; dom_id 0 reserved for this slot._
- [x] **Token chip per bubble** -- user (client-side estimate of the typed text), assistant (server-reported completion), tool result (client-side estimate of the display text). System-prompt bubble has its own per-section breakdown instead. Reasoning bubbles + intermediate-assistant tool-call bubbles deliberately have no chip (not history-deletable; not user-relevant).
- [x] **Per-message delete** -- hover over a user or final-assistant bubble reveals a small `x` (CSS `:hover` + opacity transition). Click goes through `locus://delete-message/<history_id>` intercepted in `ChatLinkHandler::handle_url`; `wxMessageDialog` confirm; on accept, `AgentCore::delete_message(history_id)` queues onto the agent thread, `LLMContext::delete_message` drops via `ConversationHistory::delete_by_id` (which refuses the leading system message) and broadcasts `on_history_message_deleted`; chat panel removes the DOM bubble. System-prompt / tool-result / assistant-with-tool-calls bubbles deliberately don't get the X (the `deletable` bit on `on_history_message_added` controls injection). Stable per-message id is `ChatMessage::history_id` assigned by `ConversationHistory::add` -- in-memory only (not part of the LLM wire format); `from_json` re-walks on session load so saved sessions get fresh monotonic ids regardless of pre-load gaps.

## Out of scope (deliberately)

- **Removing chat-event duplication from the activity panel.** The conceptual goal of "chat = LLM's view, activity = system's view, no overlap" doesn't survive the workflow: the activity panel is a timestamped + scannable + click-to-detail debug surface, and chat events (user / llm_response / tool_call / tool_result) are exactly the events that benefit most from that projection. The list-view row is not visual duplication of the chat bubble -- it's a different projection (one-line summary + timestamp + click-to-detail pane). Reviewed 2026-05-16 and rejected; revisit only if real usage shows the panel becoming unscannable.
- **Bubble-id round-trip + activity-row -> chat-bubble cross-reference.** Depended on the dedup above. Without dedup, the activity row already contains the event detail in its click-to-detail pane; jumping to chat would be a tertiary navigation aid users haven't asked for.
- **Activity-log filter row** (all / errors / indexing / chat-related). Once the unmotivated dedup is dropped, the panel's existing list-view scannability is fine. A single "errors only" toggle can land in [S5.Z](S5.Z-misc-gaps.md) if real sessions become noisy.
- **Indexing events as activity rows** (file added / removed / re-indexed / batch complete). `index_event` kind already exists for batch lifecycle; per-file events were proposed but the value is trace-level. Stays in spdlog under `-verbose`.
- **File-watcher activity as activity rows.** [S4.T](../M4/S4.T-file-change-awareness.md) already surfaces external changes by prepending them to the next user message (`[Files changed since last turn: ...]`). A second channel duplicates the signal.
- **"Reasoning dropped by compaction" placeholder bubble.** Reasoning shown in chat comes from the live `on_reasoning_token` stream -- a DOM artifact, not stored in `ChatMessage`. It vanishes on session reload regardless of compaction. [S5.F](S5.F-compaction-v2.md) Layer 3 only operates on inline `<think>...</think>` blocks in `content` (a different path); a placeholder would solve a problem the data model never creates.
- **Tool-call bubble polish** (collapsed JSON tree for args, clickable file paths, advanced collapse states). [S5.C](S5.C-inline-diffs-in-chat.md) does the load-bearing diff work; the rest is incremental polish that can land in [S5.Z](S5.Z-misc-gaps.md) if friction surfaces in real use.
- **Activity panel as a context inspector** (the alternative path floated and rejected). The inspector lives in the chat now; activity panel stays an audit log.

## Implementation notes

- The system-prompt bubble is the most novel piece. Visually it should feel like part of the chat (no special background), open inline (not a separate dialog), and show per-section token costs as a small inline chip cluster -- e.g. `Base: 800 / LOCUS.md: 600 / Tools: 1300 / Memory: 200 / Metadata: 100`. The per-section counts already exist on `SystemPromptAssembly` from [S5.J](S5.J-llm-context-refactor.md); the bubble just renders them. This is the single best surface for users to understand "where my budget went."
- KV-cache discipline ([S4.F](../M4/S4.F-kv-cache.md)) is *not* affected by this stage. The system prompt stays byte-stable; the bubble just *renders* it. Re-rendering doesn't trigger a refresh of the system prompt itself.
- Per-message delete is implemented as: hover state on the bubble's outer container reveals the `x` (CSS `:hover` + `opacity` transition for a soft reveal -- not a jump-in); click goes through a `locus://delete-message/<id>` URL intercepted in `on_webview_navigating` (same pattern as the plan-approve / plan-reject URLs); confirm dialog uses `wxMessageDialog`; on accept, `AgentCore::delete_message(id)` removes the message from `ConversationHistory` (owner-thread-only via the existing `ConversationOwnerScope` discipline) and the chat re-renders the affected region.
- Token chips reuse the existing `.token-chip` CSS class shipped in [S5.D](S5.D-reserve-and-token-xray.md); only the per-bubble-type wiring is new.
