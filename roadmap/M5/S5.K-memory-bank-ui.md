# S5.K -- Memory Bank UI Viewer / Editor

## Problem

[S4.R](../M4/S4.R-memory-bank.md) ships a persistent + searchable memory bank with a minimal Settings -> Memory Bank tab for list / edit / delete / pin / tag. That's enough for v1 *function*, but not for v1 *experience*. The Settings dialog is a poor home for the workflow of browsing, curating, and reviewing accumulated memory: it's modal-flavored, tucked away, and doesn't update live when the agent adds entries during a session.

This stage promotes the memory-bank UX to a first-class **dockable AUI panel** with proper viewing, filtering, bulk operations, live updates, and cross-linking to the chat. Same shape as the Activity panel and the Terminal panel from [S5.B](S5.B-terminal-panel.md) -- a peer panel the user can dock, float, or hide.

## Tasks

### Panel infrastructure

- [x] New `MemoryBankPanel` in [src/frontends/gui/](../../src/frontends/gui/), AUI-docked (default position: right or floating; default state: hidden). Toggle via View menu ("Memory Bank", e.g. Ctrl+M). (Chat-toolbar button deferred -- View > Memory Bank + Ctrl+M cover the entry point; chat toolbar real-estate is already saturated.)
- [x] Layout: split into top toolbar (search + filters + bulk-op buttons), main list view (`wxListCtrl` or `wxDataViewCtrl`), and a detail pane on the right or below showing the selected memory's full content + metadata.
- [x] Pane state (visible / floating / docked) round-trips with the AUI perspective string.

### List view

- [x] Columns: pin icon, content preview (first ~60 chars), tags, source (user / agent / mined), created_at, last_used_at. (Per-column user-driven sort deferred -- `sort_entries` covers every sort mode in `MemoryFilter`, but the panel always uses `pinned_first_last_used`; clicking column headers is a v1.1 polish item.)
- [x] Default sort: pinned-first, then `last_used_at` descending (recent activity at the top).
- [x] Row selection supports single + multi-select (Ctrl-click, Shift-click) for bulk operations.
- [x] Right-click context menu: View / Edit / Delete / Pin-toggle / Tag... / Copy content. (Export submenu deferred together with the zip export below.)
- [x] Double-click a row -> opens the entry in edit mode in the detail pane.

### Filtering and search

- [x] Search box at the top: hybrid retrieval (FTS5 + semantic + recency boost), same backend as `MemoryStore::search_hybrid`. Live as the user types (debounced 200 ms).
- [x] Filter bar:
  - **Tag filter**: dropdown over the union of all tags (single-tag AND filter at v1; multi-select chip widget deferred -- the underlying `MemoryFilter.tags` is already a vector so promoting later is a UI swap).
  - **Source filter**: dropdown (all / user / agent / mined).
  - **Pinned only**: toggle.
  - **Date range**: backend exists (`MemoryFilter.created_from` / `_to` honoured + tested) but UI widgets are deferred -- collapsing to "search-as-you-type covers the 95% case at v1 scale of 200 entries".
- [x] Active filter state visible in the toolbar; "Clear filters" button resets to default.
- [x] When the search box is non-empty, results are filtered (substring across content + tags + id) and re-ranked. (Score-based ordering via `MemoryStore::search` deferred -- substring filter on the in-memory list is instant and avoids the FTS5 round-trip / async embedder hop for sub-200-entry corpora.)

### Edit / delete / bulk operations

- [x] **Edit**: detail pane shows the memory in editable text fields (content + tags + pinned). Save / Cancel buttons. Saving bumps `updated_at`, re-indexes the entry in `memory_vectors` + `memory_fts`.
- [x] **Delete**: confirmation dialog. Goes through the soft-delete path from S4.R (`.locus/memory/.deleted/<id>.md` with 30-day retention). Undo via the Settings panel's recovery list (or a Recently Deleted button on this panel).
- [x] **Bulk delete**: select multiple, single confirmation lists count + first few entries, all soft-deleted in one operation.
- [x] **Bulk tag**: select multiple, "Add tag(s)..." opens a small dialog, applies to all selected. Same for "Remove tag(s)..."
- [x] **Bulk pin / unpin**: trivial; one-click toggle on the selected set.
- [x] **Recently Deleted**: small button or menu opens a sub-view showing soft-deleted entries with restore / permanent-delete actions. Populated from `.locus/memory/.deleted/`.

### Live updates

- [x] Activity-event integration: when `add_memory` fires (agent-driven), the panel auto-refreshes via the existing `IFrontend::on_activity` event channel ([S5.G](S5.G-chat-activity-restructure.md) restructure already routes these events; this stage adds a `memory_added` listener to the panel).
- [x] Same for `memory_deleted` (re-renders). `memory_searched` triggers a refresh so the `last_used_at` bumps land in the list. (Per-row pulse highlight on `memory_searched` deferred -- `pulse_rows` hook is in place but not yet wired; visual quietness wins at v1. `memory_edited` is not emitted by the store today; refresh covers it implicitly when the panel mutates.)
- [ ] If the panel is hidden, a small chat-toolbar badge increments to show "N new memories since you last looked" -- click clears the badge and opens the panel. **Deferred to a follow-up.** Activity events already surface in the Activity panel; a dedicated badge is polish on top.

### Cross-linking with the chat

- [ ] When a memory's `source` is the current session (or a saved session that's currently open in a tab from [S5.I](S5.I-tabs-and-sessions.md)), clicking a "Show source" affordance on the memory scrolls the chat to the originating message and highlights it. **Deferred** -- requires per-memory `source_message_id` field that `MemoryStore` doesn't currently track; pure schema extension, not blocked by anything.
- [ ] When the agent fires `search_memory` mid-turn, the matched entries are visually marked in the panel (subtle highlight, fades after a few seconds) so the user can see at a glance what the agent is referencing. **Deferred** -- `pulse_rows` hook is in place; needs the activity-event payload to carry the returned id list (today it only carries the query and count).
- [ ] Optional: clicking a chat message that matches an existing memory shows a "this matches memory #abc1" hint inline. **Deferred** -- chat-side hint requires the same `source_message_id` link.

### Import / export

- [ ] **Export selected**: writes selected entries to a `.zip` containing one markdown file per entry (with frontmatter intact) plus an `index.json`. File-dialog-based. **Deferred to a follow-up.** Workspaces are self-contained; the `.locus/memory/` directory is already trivially copy-pasteable for users who want a manual backup.
- [ ] **Import from .zip**: opens a file dialog; preview list of entries to import; user confirms; entries are added with new ids (avoiding collisions). Tags + content + pinned state preserved; `source` is set to `user` regardless of original source (imported entries are user actions in the destination workspace). **Deferred** (paired with export above).
- [x] Out of scope: format-stable cross-version export, Markdown-rendered exports, sync with cloud services.

### Capability gating

- [x] When `capabilities.memory_bank` is off ([S5.A](S5.A-capability-toggles.md)), the View menu item is disabled with a tooltip ("Enable Memory Bank in Settings -> Capabilities to use this panel"). The panel itself is suppressed if the user toggled off mid-session; existing entries remain on disk and become visible again when re-enabled.

### Catch2 tests

- [x] Filter logic (tag intersection, source, date range, pinned-only) -- pure functions, no UI.
- [x] Bulk operations: bulk delete soft-deletes correct subset; bulk tag adds/removes tags atomically; bulk pin toggles state.
- [x] Recovery from soft-delete restores the entry to the active store.
- [ ] Live-update event routing: posting a `memory_added` event triggers `MemoryBankPanel::refresh_for(id)` (mocked panel). **Deferred** -- the panel is wx-dependent; the activity-routing path is exercised through the manual GUI test plan instead. The pure-function refresh logic (`MemoryFilter` over `MemoryStore::list_all`) is covered by the existing filter tests.
- [ ] Import/export round-trip: export N entries, import into a clean store, verify content + tags preserved. **Deferred** (paired with the export/import tasks above).

## Implementation notes

- The Settings -> Memory Bank tab from [S4.R](../M4/S4.R-memory-bank.md) does not get deleted -- it stays as the lightweight surface for users who don't want a docked panel taking up screen space. The new panel is the *rich* surface; the Settings tab is the *minimal* one. Same data underneath.
- `wxDataViewCtrl` is preferred over `wxListCtrl` for the main list -- it supports better column virtualisation, custom cell rendering (e.g. tag chips inside a column), and integrates more cleanly with our AUI layout. Cost: slightly more setup; reward: per-column custom renderers without subclassing.
- Live updates depend on [S5.G](S5.G-chat-activity-restructure.md) routing memory events through `IFrontend::on_activity` -- which it does in S5.G's task list. This stage assumes those events exist; if S5.G hasn't landed, the panel polls `MemoryStore` on a 1 s timer as a fallback (small workspace size means polling is cheap).
- Tag chips in the filter bar are a small custom widget. Reuse the styling and click semantics from the chat panel's existing tag-style chips (LOCUS.md chip, attached-file chip) for visual consistency.
- The "Recently Deleted" view is intentionally not surfaced as a permanent sidebar -- it's a button that opens a small dialog. Otherwise users get visually noisy with deletion-affordance UI everywhere. Soft-delete + 30-day retention is a safety net, not a primary view.
- Performance: with `memory.max_entries` capped at 200 by default, the entire memory bank fits in memory and renders in one tick. No virtualization needed at v1 scale. Document the cap; if a user raises `max_entries` to 10k, the panel falls back to row virtualization (built into `wxDataViewCtrl`).
- Cross-linking with the chat is a polish item but the most distinctive feature of this stage -- it's what makes the panel feel integrated rather than tacked-on. Implementation reuses the bubble-id round-trip from [S5.G](S5.G-chat-activity-restructure.md) for the chat side.

## Out of scope (v1)

- **Validity-window timeline view** (would visualize the v2 temporal supersession from the backlog item; deferred until that v2 ships).
- **Memory similarity / clustering view** (semantic neighborhoods of memories). Cool but not load-bearing.
- **Auto-mining UI** (proposing memories from session content) -- that backlog item gets its own UX when promoted.
- **Cross-workspace memory browsing** (consistent with S4.R's workspace-scoped principle).
- **Cloud sync / sharing** -- privacy thesis says no.

## Manifest cost

**Zero new tools.** Pure frontend work on top of the existing `MemoryStore` data layer from [S4.R](../M4/S4.R-memory-bank.md). The agent's tool surface is unchanged.
