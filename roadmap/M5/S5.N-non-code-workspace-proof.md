# S5.N -- Non-Code Workspace Proof

## Problem

Locus claims to be more than a coding agent: it should also work as a local
assistant for document folders, offline references, notes, and wiki-style corpora.
That claim is central to the product positioning, but the strongest shipped proof
is still concentrated around WS1 (the Locus codebase). Without a polished WS2/WS3
demo, "any data, not just code" reads like aspiration instead of product evidence.

This stage turns the non-code workspace story into a concrete M5 deliverable:
repeatable demo workspaces, scripted/manual QA, and documentation that shows where
Locus is already useful and where it is still rough.

## Scope

- WS2: Wikipedia/Kiwix or an equivalent large offline reference corpus.
- WS3: private-document-style folder using PDFs, DOCX/XLSX, Markdown notes, and
  mixed text files.
- No new indexing subsystem unless a blocker is discovered. This is a proof and
  polish stage built on the existing M2/M4 engine.
- No M6 web retrieval, remote access, or ZIM-reader implementation. If native ZIM
  support is not ready, use an extracted or representative corpus and document the
  gap honestly.

## Outcomes -- bugs surfaced by the acceptance run

The non-code workspace probe surfaced six real product issues that pure-code
testing hadn't. Five fixed in this stage, one deferred to backlog:

1. **FTS5 query crash on punctuation in production `search_text`.** Natural
   queries like "HTTP/3" or "TLS 1.3" raised `fts5: syntax error near "/"` /
   `near "."` and returned zero results silently. Fixed in
   [src/index/index_query.cpp](../../src/index/index_query.cpp) -- new
   `sanitise_fts_query` helper strips a broad punctuation set before
   binding. The eval harness's `sanitise_for_fts` was already doing this
   for `/`; the production fix expands the set and ports it.
2. **`max_file_size_kb=1024` default silently dropped rfc9114.pdf** (HTTP/3,
   1158 KB). Per-workspace mitigation: [build_ws3.ps1](../../scripts/build_ws3.ps1)
   auto-writes `4096` into the workspace config; raising the project-wide
   default is in [roadmap/backlog/README.md](../backlog/README.md).
3. **`LLMConfig::context_limit = 8192` default silently capped 32k models
   to 8k.** The `<= 0` auto-detect path never fired because the seed value
   was 8192, not 0. Triggered premature auto-compaction. Fixed in
   [src/llm/llm_client.h](../../src/llm/llm_client.h) (default -> 0 sentinel).
4. **Search snippets were raw file bytes**, not extracted text. On code
   workspaces this is invisible (source == extracted form); on WS2 it
   dumped raw HTML / CSS into the LLM context (~1200 tokens per call on
   a single Wikipedia lookup). Fixed in
   [src/index/index_query.cpp](../../src/index/index_query.cpp):
   `search_text` now uses FTS5's `snippet()` over the extracted column;
   semantic snippet cap tightened 500 -> 250; default `max_results`
   lowered for text/semantic/hybrid modes.
5. **Tool descriptions misled the LLM** about which formats are searchable.
   The model claimed inability to read XLSX (Locus does extract it), and
   dismissed `search` as poor for binary formats (it isn't). Fixed in
   [src/tools/file_tools.h](../../src/tools/file_tools.h) +
   [src/tools/search_tools.cpp](../../src/tools/search_tools.cpp): the
   `read_file` and unified `search` descriptions now name PDF / DOCX /
   XLSX explicitly and steer the LLM to the right tool.
6. **Per-message delete X didn't attach on the same run** the bubble was
   created. `ChatStreamRenderer` zeroed `assistant_id_` inside the seal
   path; `on_history_message_added` then read 0 and bailed. Fixed via
   `last_sealed_assistant_id_` in
   [src/frontends/gui/chat/chat_stream_renderer.h](../../src/frontends/gui/chat/chat_stream_renderer.h);
   [chat_panel.cpp](../../src/frontends/gui/chat_panel.cpp) falls back
   to it when current is 0.

The acceptance run also surfaced a real but unfixable retrieval ceiling:
**bge-small (the small profile) cannot bridge wide synonymy gaps** on a
5000-article corpus -- "species change over time" doesn't surface the
Darwin article. Documented as a measured trade-off in
[test-workspaces.md](../../test-workspaces.md), [README.md](../../README.md)
small-vs-recommended-profile section, and
[tests/retrieval_eval/README.md](../../tests/retrieval_eval/README.md).
Not a Locus bug -- the larger bge-m3 closes the gap (per
[results_ws3.md](../../tests/retrieval_eval/results_ws3.md) hitting
MRR ~0.88).

## Acceptance

- A new contributor can set up WS2 or WS3 from the docs and reproduce the demo.
- At least 8 of the 10 acceptance prompts return useful, sourced answers without
  manually pasting file contents.
- Failures are diagnosable from the UI/logs: bad retrieval, extractor miss, model
  reasoning failure, or unsupported file type.
- The README contains one real non-code example, not just positioning copy.

## Why M5

This belongs before M6 because M6 multiplies frontend surface area. If the non-code
story is weak in the local desktop app, a browser frontend or remote client will
only make the weakness more visible. Prove the workflow once, then carry it forward.
