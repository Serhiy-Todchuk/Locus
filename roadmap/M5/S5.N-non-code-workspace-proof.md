# S5.N -- Non-Code Workspace Proof

## Problem

Locus claims to be more than a coding agent: it should also work as a local
assistant for document folders, offline references, notes, and wiki-style corpora.
That claim is central to the product positioning, but the strongest shipped proof
is still concentrated around WS1 (the Locus codebase). Without a polished WS2/WS3
demo, "any data, not just code" reads like aspiration instead of product evidence.

This stage turns the non-code workspace story into a concrete M5 deliverable:
repeatable demo workspaces, scripted/manual QA, and documentation that shows where
Locus is already useful and where it is still rough.

## Scope

- WS2: Wikipedia/Kiwix or an equivalent large offline reference corpus.
- WS3: private-document-style folder using PDFs, DOCX/XLSX, Markdown notes, and
  mixed text files.
- No new indexing subsystem unless a blocker is discovered. This is a proof and
  polish stage built on the existing M2/M4 engine.
- No M6 web retrieval, remote access, or ZIM-reader implementation. If native ZIM
  support is not ready, use an extracted or representative corpus and document the
  gap honestly.

## Tasks

- [ ] Define two repeatable demo datasets:
  - WS2 large reference corpus with enough content to stress search/retrieval.
  - WS3 document-library corpus with PDFs, DOCX/XLSX, Markdown, and plain text.
- [ ] Add a short `test-workspaces.md` section for the exact setup, expected disk
      footprint, and how to rebuild each corpus.
- [ ] Write 10 acceptance prompts covering:
  - factual lookup with cited snippets,
  - cross-document synthesis,
  - "find the source" navigation,
  - outline extraction for PDFs/DOCX/Markdown,
  - semantic vs keyword retrieval comparison,
  - failure behavior when the answer is not in the corpus.
- [ ] Add a manual test plan under `tests/manual/non-code-workspaces.md`.
- [ ] Add retrieval-eval seed queries for WS2/WS3 where expected source files are
      known and stable.
- [ ] Capture performance notes: first index time, embedding time, DB sizes, query
      latency, and reranker cost on the recommended and small model profiles.
- [ ] Update README/Quickstart with one concise non-code workflow example once the
      demo passes.
- [ ] Record honest gaps in backlog or the M6 stage docs instead of letting the
      demo imply unsupported behavior.

## Acceptance

- A new contributor can set up WS2 or WS3 from the docs and reproduce the demo.
- At least 8 of the 10 acceptance prompts return useful, sourced answers without
  manually pasting file contents.
- Failures are diagnosable from the UI/logs: bad retrieval, extractor miss, model
  reasoning failure, or unsupported file type.
- The README contains one real non-code example, not just positioning copy.

## Why M5

This belongs before M6 because M6 multiplies frontend surface area. If the non-code
story is weak in the local desktop app, a browser frontend or remote client will
only make the weakness more visible. Prove the workflow once, then carry it forward.
