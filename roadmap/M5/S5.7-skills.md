# S5.7 -- Skills (On-Demand Capability Packs)

## Problem

[S4.X -- Prompt Templates](../M4/S4.X-prompt-templates.md) gives the user `/<name>` text expansion. [S4.G -- MCP](../M4/S4.G-mcp.md) gives the user external tools. Neither covers the case where a user wants to opt the workspace into a *workflow*: "for this session, behave like a Rust debugging assistant -- preferred tools, system-prompt addendum, a few canned prompts."

Pi calls these *skills* and they are the on-demand layer above prompt templates. `/skill:debug-rust` activates a bundle; `/skill:end` deactivates it. The bundle composes a system-prompt fragment, a curated tool subset (from built-ins + MCP), and optionally a set of prompt templates -- all without permanently touching `LOCUS.md`.

Lands in M5 because (a) skills want a distribution story, which leans on M5's connected/remote infrastructure, and (b) MCP and prompt templates need to ship first; skills compose them.

## Tasks

- [ ] On-disk layout:
  - Project-local: `.locus/skills/<name>/`
  - Global: `%APPDATA%/Locus/skills/<name>/` (Windows), `~/.locus/skills/<name>/` (Unix)
  - Each skill is a directory containing `skill.md` (manifest + system-prompt fragment) and optional `prompts/` subdirectory (skill-scoped templates).
- [ ] `skill.md` frontmatter:
  ```
  ---
  name: debug-rust
  description: Rust debugging workflow -- prefer ast/regex search, suggest gdb commands
  tools_allow: [read_file, edit_file, search, run_command]
  tools_deny: [write_file, delete_file]
  mcp_servers: [rust-analyzer]   # optional MCP servers to require
  prompts: [bisect, repro]        # optional prompt-template names this skill exposes
  ---

  (Markdown body below = system-prompt addendum injected when active)
  ```
- [ ] `SkillRegistry` in `src/agent/`: scans both dirs, validates manifests, exposes `list()`, `activate(name)`, `deactivate()`, `active()`.
- [ ] `/skill:<name>` slash command activates; `/skill:end` (or `/skill:none`) deactivates.
- [ ] On activation:
  - System-prompt addendum injected on the **user message**, not the system prompt -- preserves KV cache invariant from [S4.F](../M4/S4.F-kv-cache.md). Format: `[Skill: debug-rust]\n<addendum>\n\n<user message>`.
  - Tool catalog filtered through `tools_allow` / `tools_deny`. Implemented at the `ToolDispatcher` layer; the manifest sent to the LLM still shrinks accordingly. (Hooks into [S3.L](../M3/S3.L-tool-catalog-hygiene.md) `available()` / `visible_in_mode()`.)
  - Required MCP servers verified to be running; if not, activation fails with a clear error and a one-click "start now" if the user has the server configured.
  - Skill-scoped prompt templates (`prompts/` subdir) registered in [S4.X's](../M4/S4.X-prompt-templates.md) `PromptTemplateRegistry` under a `skill:<name>:` namespace so they don't leak across deactivation.
- [ ] Active skill is persisted in session state (saved sessions reload with the same skill active); cleared on workspace switch.
- [ ] UI: chat-footer chip shows the active skill (`🎯 debug-rust`) with a click-to-deactivate. Chat input placeholder updates to mention the active skill.
- [ ] Distribution story:
  - Project-local install: `git clone <repo> .locus/skills/<name>` -- no Locus-specific tooling needed.
  - `locus skill install <git-url|zip-url|local-path>` CLI command for ergonomics. No automatic dependency resolution; skills must be self-contained.
  - **No npm-style transitive deps.** A skill is one directory, no install scripts, no postinstall hooks. This is a deliberate divergence from Pi's `pi install npm:...` model -- our trust boundary is "review one directory" rather than "review a transitive npm tree".
  - Public skill registry (web frontend, M5) lists curated + community skills with source links. Locus refuses to install skills from unverified URLs without an `--unsafe` flag. **Sandboxing is still not provided** -- the Markdown body can instruct the model to do anything, same as Pi -- so the trust model is "read the manifest + the addendum body before activating."
- [ ] `/skill:list` and `/skill:show <name>` for discovery without activating.
- [ ] Catch2 tests:
  - manifest parsing: well-formed, missing fields, unknown fields
  - activation flow: addendum injection, tool-catalog filtering, MCP precondition check
  - deactivation: state cleanup, prompt-template namespace eviction
  - session round-trip: save with active skill, reload, skill still active

## Implementation notes

- Skills compose existing M4 surfaces. Do not invent new tool-execution paths or new prompt-injection points -- they go through `ToolDispatcher` and the existing user-message channel respectively. This stage is wiring, not new mechanism.
- The `tools_allow` / `tools_deny` lists support `mcp:server:*` glob patterns so a skill can grant access to a whole MCP server's tool surface without enumerating each tool. Built-in tools are always referenced by name (no globs) so the trust review is explicit.
- Prompt-template namespacing inside a skill (`skill:<name>:tplname`) means activating two skills with the same template name doesn't collide. The unscoped slash form falls back to the active skill's namespace before global / project lookups -- typing `/bisect` while `debug-rust` is active resolves to `skill:debug-rust:bisect`.
- We deliberately do NOT mirror Pi's "skill auto-load by the model." Activation is always a user action -- matches the user-as-architect thesis and avoids the failure mode where a model decides to load a skill that hides tools it just decided it needs.
- The CLI `locus skill install` lives in the connected M5 frontend, not the workspace daemon -- the daemon never speaks HTTP to skill repositories. Frontend downloads, verifies, copies into `.locus/skills/`, then asks the daemon to rescan.

## Open questions

- Should a skill be able to declare its own MCP servers (config to spawn) or only require already-configured servers? Current draft is the latter; the former is more powerful but blurs the trust boundary.
- Cross-session vs within-session activation: should skills be sticky across workspace reopens by default, or always start clean? Leaning toward "remember the most recent active skill but require explicit confirmation on workspace open" -- discuss before implementation.
