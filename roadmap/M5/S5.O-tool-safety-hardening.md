# S5.O -- Tool & Filesystem Safety Hardening

**Theme**: Safety / Robustness

**Status**: planned

## Goal

Close three latent safety holes in the tool layer that an external review surfaced: workspace path containment, tool-approval call-id matching, and atomic `write_file`. Each is small in isolation; bundling them into one stage keeps the touched surface (shared tool helpers + dispatcher) reviewable in a single pass.

## Motivation

The tool dispatcher and the shared file helpers are Locus's main agent safety boundary -- they decide what the LLM is allowed to read, write, and execute. Three issues were flagged:

1. **Path prefix containment**. [src/tools/shared.cpp:22-25](../../src/tools/shared.cpp) currently does a raw byte-prefix compare to verify a resolved path stays inside the workspace. With workspace `D:\foo`, the resolved path `D:\foo-bar\evil.txt` passes the check (the prefix matches; there is no separator boundary check). Under Windows, the compare also misses case differences (`D:\foo` vs `d:\Foo`). `fs::canonical` smooths slash direction but not case. This is the single check `read_file` / `write_file` / `edit_file` / `delete_file` / `list_directory` / `get_file_outline` / the checkpoint snapshot path / the auto-commit path all rely on -- one wrong helper, every consumer is wrong.

2. **Approval decision not keyed by call_id**. [src/agent/tool_dispatcher.cpp:32-39](../../src/agent/tool_dispatcher.cpp) stores a single `pending_decision_` slot updated by `submit_decision(decision, args)`. Today the agent loop dispatches tool calls strictly serially on the agent thread, so no current code path can mismatch. The latent risk: a stale UI event (double-click after the dialog already sent its decision, late delivery, future parallel-tool-call work, future "re-approve" surface) wakes the next pending call with a decision the user made for the previous one. The fix turns "today this is fine because tools are serial" into a type-level guarantee that survives future refactors.

3. **`write_file` not atomic**. [src/tools/file_tools.cpp:131-136](../../src/tools/file_tools.cpp) opens the destination with `std::ios::trunc` and writes bytes directly. `write_atomic()` (same file, line 234) already exists and is used by `edit_file`. Process kill / power loss between truncate and the final flush leaves a half-written or zero-byte file. Checkpoints back up the *previous* content, but the user authored the new content too -- both copies should survive a crash, not just one.

## Design

Three independent fixes, one stage. Each lands as its own commit so a regression bisect is precise.

### 1. Path containment

Replace the raw prefix check in `resolve_path` with a path-component-aware containment check.

```cpp
// New helper in shared.h:
bool is_inside(const fs::path& candidate, const fs::path& root);
```

Algorithm:
- `lexically_normal()` both `candidate` and `root`.
- Walk `root`'s components against `candidate`'s. Containment requires `candidate.size() >= root.size()` AND every `root[i]` equals `candidate[i]`.
- On Windows, compare components case-insensitively (`std::filesystem::path::compare` already does this for `path == path`, but component-wise comparison needs `_wcsicmp` or `CompareStringOrdinal` with `NORM_IGNORECASE`). Use `fs::equivalent` only when both paths exist; otherwise the component walk.
- Equal paths (workspace root itself) are inside.

`resolve_path` keeps its existing canonical / weakly_canonical resolution and just calls `is_inside` for the final check. Same return contract (empty path on rejection) so callers don't change.

### 2. Approval decision keyed by call_id

Change the dispatcher's pending-decision slot from a single optional to a small map keyed by call id.

```cpp
struct PendingDecision {
    ToolDecision   decision;
    nlohmann::json modified_args;
};
std::unordered_map<std::string, PendingDecision> pending_decisions_;
```

`submit_decision` grows a `call_id` parameter and writes `pending_decisions_[call_id] = {...}`. The condvar predicate inside `dispatch` checks `pending_decisions_.contains(call.id) || cancel_flag_.load()` and erases the entry on consumption. `IFrontend::tool_decision` (the path the GUI / CLI use today) gains the `call_id` argument and forwards it through; in the GUI it's already known via `ToolApprovalDialog::run`'s `call.id`.

The `wake()` no-op stays for cancellation. Stale decisions (call id no longer pending because the call already completed or was cancelled) are dropped with a `spdlog::warn` -- silently ignoring them keeps the failure mode "old click does nothing" instead of "old click approves a different call."

### 3. `write_file` atomicity

In `WriteFileTool::execute`, replace the inline `ofstream` block with a call to the existing `write_atomic(full, content)`. Hoist the helper out of the unnamed namespace into `tools/shared.{h,cpp}` so both files use it (already a candidate for sharing -- delete_file's eventual atomic-trash work in S5.Z will want it too).

Edge: `write_atomic` writes `<path>.locus-tmp` then renames over the target. If the target is on a different volume than the temp, `fs::rename` fails and the existing fallback path (`copy_file` + remove tmp) kicks in. Both branches are already tested for `edit_file`; coverage carries over.

## Tests

New `tests/test_tool_safety.cpp` tagged `[s5.o]`:

1. **Containment** -- 8+ cases. Workspace `D:\foo`, candidates `D:\foo\bar.txt` (in), `D:\foo-bar\bar.txt` (out, was a false positive pre-fix), `D:\Foo\bar.txt` (in on Windows, case-insensitive), `D:\foo\..\bar.txt` (out), `D:\foo` (in, equals root), `D:\foob` (out), `\\server\share\foo\bar.txt` with workspace `\\server\share\foo` (in, UNC), forward-slash and back-slash mix.
2. **Approval keying** -- two pending dispatches in flight (use a fake registry / fake frontend that schedules submit_decision after a short sleep), verify decision A consumed by call A, decision B by call B; mismatch (submit_decision for call_id "X" while only call_id "Y" is waiting) leaves Y waiting, X drops with a warn-log.
3. **write_file atomicity** -- existing file, kill the process mid-write by replacing `write_atomic` with a fault-injecting variant that throws after the temp file is written but before the rename. Original file content must still be readable (byte-identical to before the call). New file: temp file removed; target absent. Same shape as the existing `edit_file` atomicity test.

Add an integration tag `[integration][safety]` exercising the containment fix against a real workspace -- ask the LLM to read `<workspace>/../sibling.txt` (rejected) and read `<workspace>-evil/x.txt` (rejected, the previously-broken case).

## Out of scope

- Cross-platform stub fill-in for `run_command` / `run_command_bg` -- tracked in S5.Z task list.
- Approval-dialog UX changes (the modal shape S5.Z task 6 already shipped is fine).
