# S5.S -- Permission Presets

**Theme**: Safety / UX

**Status**: planned

## Problem

Per-tool approval policy already exists ([src/tools/tool.h](../../src/tools/tool.h) `ToolApprovalPolicy::{ask, auto_approve, deny}`; per-workspace overrides in [WorkspaceConfig::tool_approval_policies](../../src/core/workspace.h); resolver in [tools::resolve_approval_policy](../../src/tools/shared.h) called from [ToolDispatcher](../../src/agent/tool_dispatcher.cpp)). The settings UI is a 15-row tool-by-tool table in [tool_approvals_settings_panel.cpp](../../src/frontends/gui/settings/tool_approvals_settings_panel.cpp). Two friction points:

1. **First-open paralysis** -- a new user staring at 15 rows has no guidance on safe defaults. The current default ("everything `ask`") is correct but verbose; there's no way to express "I want read-only" or "trust file edits, gate shell" without clicking 15 cells.
2. **No fast lever for autonomous runs** -- combined with [S5.F auto-compaction](S5.F-compaction-v2.md), a user could in principle leave Locus running on a long task. Doing so today means flipping every mutation tool to `auto`, then flipping them back when done. The mismatch between "I want a quick autonomous sprint" and "tune 8 cells, then tune them back" is the gap this stage closes.

Permission Presets layer named snapshots over the existing override map, plus a runtime-only override mirroring Claude Code's bottom-bar selector for the AFK-autonomous case.

## Design

Four named presets + a `Custom` sentinel. Each preset is a **snapshot** -- selecting one writes a full per-tool map into `tool_approval_policies` (no implicit category gating, no new core mechanism). The user can still tweak any cell afterward; that flips the detected preset to `Custom`. This is purely a UX layer over what already exists.

### Preset table

| Tool | Read-only | Ask before edits | Allow edits | Allow all |
|---|---|---|---|---|
| **Reads (no side effects)** | | | | |
| `read_file` / `search` / `list_directory` / `get_file_outline` / `search_memory` / `read_process_output` / `list_processes` | auto | auto | auto | auto |
| **File mutations** | | | | |
| `write_file` / `edit_file` | deny | ask | auto | auto |
| `delete_file` | deny | ask | **ask** | auto |
| **Memory & process control** | | | | |
| `add_memory` / `stop_process` | deny | ask | auto | auto |
| **Shell** | | | | |
| `run_command` / `run_command_bg` | deny | ask | **ask** | auto |
| **Interactive** | | | | |
| `ask_user` | ask | ask | ask | ask |
| **Plan mode** | | | | |
| `propose_plan` / `mark_step_done` | auto | auto | auto | auto |
| **External** | | | | |
| `mcp:*` (per-server `mcp:<name>:*` wildcard) | ask | ask | ask | **ask** |

Asymmetric cells -- read down the columns:

- `delete_file` stays `ask` at "Allow edits" -- file content is checkpointable, file existence after a `delete_file` followed by external state changes is not. Loud by design.
- `run_command*` stays `ask` at "Allow edits" -- "edits" is files, not the shell. Shell auto-approves only at "Allow all".
- `mcp:*` stays `ask` even at "Allow all" -- MCP servers are external code over stdio. Elevating them requires an explicit Custom tweak (per-server wildcard already handled in [mcp_settings_panel.cpp](../../src/frontends/gui/settings/mcp_settings_panel.cpp)). "Allow all" warning copy specifically calls this out.
- `ask_user` stays `ask` everywhere -- the agent asking a human is orthogonal to approval gating. AFK users flip it to `deny` via Custom.
- Read-only uses `deny`, not "hide tool" -- schema stays byte-stable (KV-cache invariant, see [S4.F](../M4/S4.F-kv-cache.md)); the model gets a clear "denied by Read-only preset" result rather than a confusing tool absence.

### Custom detection

```cpp
// shared.h (or new presets.h)
enum class PermissionPreset { ReadOnly, AskBeforeEdits, AllowEdits, AllowAll, Custom };

PermissionPreset detect_preset(
    const std::unordered_map<std::string, ToolApprovalPolicy>& overrides,
    const IToolRegistry& registry);
```

Walks each registered tool, resolves its effective policy (override OR `ITool::approval_policy()` default), and compares against each named preset's signature. Returns the first exact match; falls through to `Custom`. The signatures live in one table in the same header so the preset table above and the detector stay in sync.

`mcp:*` wildcard entries are checked against the preset's MCP cell (`ask` for all four named presets). Per-server `mcp:<name>:*` entries that diverge from the wildcard flip detection to `Custom`.

### Settings UI

Above the existing per-tool table in [tool_approvals_settings_panel.cpp](../../src/frontends/gui/settings/tool_approvals_settings_panel.cpp), add a "Preset:" `wxChoice` with the five entries. Selecting a named preset:

- Pops a warning modal for "Allow edits" (info, first time per workspace) and "Allow all" (orange, every time).
- Writes the preset's full snapshot into the panel's working copy (the existing baseline-diff machinery already handles "did anything change since last save").
- Per-tool dropdowns below stay editable; tweaking any cell flips the preset chip to `Custom`.

Custom (auto-detected) is selectable from the dropdown only as a label -- choosing it from the menu is a no-op (no signature to apply). It's the state the panel sits in when the per-tool table doesn't match any named preset.

### Runtime override (chat-footer combobox)

Mirrors Claude Code's selector. Lives in the chat-panel footer next to the context meter (built in [chat_panel.cpp](../../src/frontends/gui/chat_panel.cpp)). Per-tab, in-memory only. Closing the tab discards the override; reopening reads from the saved settings.

**Resolution order** (highest wins): runtime override -> `WorkspaceConfig::tool_approval_policies` -> `ITool::approval_policy()` default. Plumbed through `ToolDispatcher` via a new `set_runtime_override(std::optional<PermissionPreset>)` setter; the dispatcher composes a one-shot override map at the start of each `dispatch()` call rather than carrying it as live state (cheap, no mutation race).

**Settings-save interaction**: saving the settings dialog clears every tab's runtime override (the new saved policy is now baseline -- explicit user action). Activity-log line noting the reset.

**Allow all from the runtime selector**: same orange modal as settings, plus a "Don't ask again this session" checkbox. Per-tab session scope -- closing the tab forgets it.

### Bottom-bar chip styling

| Effective state | Chip | Tooltip first line |
|---|---|---|
| Read-only | gray | "Read-only: mutations denied" |
| Ask before edits | gray | "Asks for every mutation" |
| Allow edits | yellow | "Files auto; shell asks" |
| Allow all | orange | "Shell auto; MCP still asks" |
| Custom (no MCP elevation) | blue | "Custom (closest: <name>)" + list of diverging cells |
| Custom + any MCP at `auto` | **red** | "ELEVATED -- MCP servers auto-approved" + list of elevated servers |
| Any preset + runtime override active | same color + dotted border | "Runtime override (settings says <name>)" |

## Implementation notes

- **Snapshot, not category gating.** The preset is a map snapshot written into the existing `tool_approval_policies`, not a separate "current preset" field. Two upsides: (a) zero migration -- existing workspaces detect-as `Custom` (or as a preset if the user happens to be on default `ask` everywhere, which matches "Ask before edits"); (b) no resolution layering at runtime beyond the existing override map.
- **Schema is byte-stable.** Presets only flip approval policy, never `available()` or `visible_in_mode()`. The per-turn manifest doesn't change shape when the preset changes -- preserves the [byte-stable system prompt](../../CLAUDE.md#key-invariants) invariant.
- **Runtime override lives on the dispatcher, not the chat panel.** The panel is a view; the dispatcher is what actually resolves policy when a tool call fires. Keeping the override one level below the UI means session-tabbed callers (CLI, future remote frontends) inherit the same mechanism without re-implementing it.
- **No "fifth preset for headless".** Discussed and rejected -- `ask_user` stays `ask` in named presets; AFK users flip it to `deny` via Custom. Adding a fifth named preset just for that case is preset bloat for one user.
- **Autonomous-run loop**: "Allow all" + [S5.F](S5.F-compaction-v2.md) auto-compaction is the documented combination for an unattended sprint. Caveats spelled out in the manual test plan: (a) `ask_user` still pauses the loop; (b) coherence across many compactions is the limiting factor, not approval gating; (c) MCP servers stay gated by default so a hostile MCP can't ride the autonomous mode.
- **Don't auto-elevate `mcp:*`.** Even if a user picks "Allow all", MCP wildcard stays `ask`. Elevating MCP is a deliberate per-server (or `mcp:*=auto` wildcard) action via the existing MCP panel. Warning copy on the "Allow all" modal: _"Shell commands inside the workspace will run without approval. MCP tools still ask -- elevate per server under Settings -> MCP."_

## Out of scope

- Time-limited auto-elevation ("Allow all for the next 30 minutes, then revert"). Useful but separate -- defer until real usage shows the manual flip-back is the friction point.
- Per-tab differing presets via the runtime selector are supported (each tab owns its own dispatcher) but no UI badge on the tab itself; the chat footer is the indicator. Tab-level chip is a polish item.
- Slash command (`/preset allow-all`, `/preset reset`) -- the bottom-bar combobox is the surface. Slash command can come later if it's actually faster than a click.
