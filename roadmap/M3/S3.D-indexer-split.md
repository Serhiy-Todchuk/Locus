# S3.D ÔÇö Split `Indexer` Ôťö

## Problem

[src/indexer.cpp](../../src/indexer.cpp) is 700 LOC / 20 methods ÔÇö roughly as big as `AgentCore`. It mixes: Tree-sitter parser lifecycle, languageÔćĺgrammar map, per-language `SymbolRule` tables, symbol-name recursion, file I/O, FTS/symbols/headings/chunks SQL persistence via 12 raw `sqlite3_stmt*` pointers, chunking dispatch, incremental updates, extension-based language detection.

The `ExtractorRegistry` pattern already proved the model works: adding docx/xlsx/pdf did **not** touch `indexer.cpp`. Apply the same to the Tree-sitter side.

M4 pressure points:
- **S4.E (LSP)** will want the language-grammar map externally accessible for diagnostics + the extractor-registry shape.
- **S4.L (git-native)** wants a `.gitignore` hook into indexing.
- **S4.M (ast_search)** wants to *reuse* the languageÔćĺgrammar map.
- **S4.J (better embeddings + reranker)** wants to re-embed with a different model; Indexer and Embedder are currently coupled through `on_chunks_created`.

## Implementation notes

- DB schema unchanged.
- `TreeSitterRegistry` is the shared seam for S4.M (`ast_search`); the parser is owned here and ts_parser_set_language is called per-extract by `Indexer::extract_symbols`. ast_search will need its own parser instance (TSParser is not safe under concurrent use), but the language map is now reusable.
- Per-language extractors stay tiny (~10 LOC each) by going through `make_rule_based_symbol_extractor(rules)`. A future language with non-rule-based logic (e.g. a custom Rust impl-block walker) can implement `ISymbolExtractor` directly without touching the shared infrastructure.
- The spec called for "static registrars (mirror `ExtractorRegistry`)" ÔÇö `ExtractorRegistry` is itself populated by an explicit `register_extractor()` call from `Workspace`, not static init, so the symbol extractors follow the same explicit pattern (`register_builtin_symbol_extractors()` from the `Indexer` ctor). Avoids static-lib initializer-elision.
