# S3.G — `LocusSession` bundle

## Problem

[src/gui/locus_app.cpp](../../src/gui/locus_app.cpp):232 is a 120-line manual DI graph: it tears down workspace → llm → tools → agent in reverse order, reconstructs them in forward order, inlines the initial-index-drain dance, re-inits logging, rebuilds the frame. Every future lifecycle concern (checkpoint store, MCP registry, process registry, LSP client) has to slot into this procedure by hand, at the right moment, or something leaks.

[src/main.cpp](../../src/main.cpp) does the same wiring for the CLI. Two copies today; a Crow frontend (M5 S5.3) will be the third.

## Target

```cpp
// src/core/locus_session.h
class LocusSession {
public:
    LocusSession(const fs::path& ws_path, const LLMConfig& llm_cfg);
    ~LocusSession();

    Workspace&       workspace() { return *workspace_; }
    AgentCore&       agent()     { return *agent_; }
    IToolRegistry&   tools()     { return *tools_; }
    ILLMClient&      llm()       { return *llm_; }

private:
    std::unique_ptr<Workspace>     workspace_;
    std::unique_ptr<ILLMClient>    llm_;
    std::unique_ptr<IToolRegistry> tools_;
    std::unique_ptr<AgentCore>     agent_;
    // Later M4 subsystems (CheckpointStore, LspClient, ProcessRegistry, ...)
    // slot in here, and tear down happens automatically in reverse declaration order.
};
```

`LocusApp::open_workspace` becomes:

```cpp
frame_->detach();
session_.reset();                            // clean teardown of everything in order
session_ = std::make_unique<LocusSession>(ws_path, cfg);
frame_->attach(session_.get());
```

`main.cpp` becomes:

```cpp
LocusSession session(ws_path, cfg);
CliFrontend cli(session.agent());
session.agent().register_frontend(&cli);
// ... REPL ...
```
