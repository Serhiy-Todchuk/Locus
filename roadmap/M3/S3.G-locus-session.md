# S3.G — `LocusSession` bundle

## Problem

[src/gui/locus_app.cpp](../../src/gui/locus_app.cpp):232 is a 120-line manual DI graph: it tears down workspace → llm → tools → agent in reverse order, reconstructs them in forward order, inlines the initial-index-drain dance, re-inits logging, rebuilds the frame. Every future lifecycle concern (checkpoint store, MCP registry, process registry, LSP client) has to slot into this procedure by hand, at the right moment, or something leaks.

[src/main.cpp](../../src/main.cpp) does the same wiring for the CLI. Two copies today; a Crow frontend (M5 S5.3) will be the third.

## Implementation notes

- Single source of LLM-config layering ([locus_session.cpp](../../src/core/locus_session.cpp)). Caller passes a seed `LLMConfig`; `LocusSession` then merges (in order, low→high priority): `LLMConfig` defaults → `.locus/config.json` → seed → live `query_model_info`. The CLI's `--model` flag still wins over a stale value in `config.json`; a workspace switch from the GUI menu picks up that workspace's saved endpoint without any explicit user input. Behaviour change worth flagging: the old `main.cpp` did **not** merge `.locus/config.json`, so the CLI now respects it like the GUI always has.
- Lifecycle: ctor starts the agent thread last; `~LocusSession` calls `agent_->stop()` before any `unique_ptr` runs out, so the agent thread joins while every subsystem it touches (Workspace, LLM, ToolRegistry) is still alive. After `stop()` returns, member destruction unwinds in reverse declaration order — same teardown sequence as the old hand-wired path, just guaranteed by the language instead of by hand-ordered `.reset()` calls.
- `LocusApp::spawn_session(ws_path, seed_cfg)` is the one spot that constructs the session + the frame; `OnInit` and `open_workspace` both call it. Workspace-switch teardown is now `frame_->Destroy(); session_.reset();` — the rest is deleted.
- `main.cpp` shrunk by ~70 lines; `LocusApp` shrunk by ~120 lines. Wiring duplication eliminated.
- Future M4 lifecycle subsystems (LSP client, MCP registry, process registry) slot in as additional `unique_ptr` members on `LocusSession`. No frontend code changes.
