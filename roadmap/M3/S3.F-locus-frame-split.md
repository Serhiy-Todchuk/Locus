# S3.F — Split `LocusFrame`; move file-watcher pump to Core ✔

## Problem

Two separate but co-located issues.

### 1. `LocusFrame` is 900 LOC / 30 methods

[src/gui/locus_frame.cpp](../../src/gui/locus_frame.cpp) owns: menu bar creation, recent-workspaces submenu, sessions submenu, AUI layout, tray wiring, **13 agent event handlers**, file-watcher timer/batching, index stats refresh, ops-status composition, dialog launching. Every M4 UI touch (plan mode toolbar, checkpoint-undo button, MCP server panel, process panel, verify-status chip, metrics tab) will grow it further.

### 2. File-watcher pump is duplicated 3 times

- [src/gui/locus_frame.cpp](../../src/gui/locus_frame.cpp):602 runs `wxTimer` every 500ms, drains watcher, batches, flushes.
- [src/main.cpp](../../src/main.cpp):232 sleeps 300ms, drains once at startup.
- [src/gui/locus_app.cpp](../../src/gui/locus_app.cpp):282 does the same dance on workspace switch.

When the Crow frontend lands (M5 S5.3), there will be a fourth copy. This logic belongs in `Workspace` (or a dedicated `WatcherPump`), not every frontend.

## Target

### Part A — Move watcher pump to Core

```
src/core/watcher_pump.{h,cpp}
  class WatcherPump {
      // Owns its own thread or a pluggable tick-source.
      // Drains FileWatcher queue, batches by quiet period + hard cap,
      // calls Indexer::process_events, emits progress through a callback.
  };
```

`Workspace` owns one `WatcherPump`. Frontends no longer touch the watcher directly. `IFrontend::on_indexing_progress` keeps working.

### Part B — Slim `LocusFrame`

```
src/gui/
  locus_frame.{h,cpp}          window shell + AUI + agent-event routing (~300 LOC)
  menu_controller.{h,cpp}      menu bar, recent workspaces submenu, sessions submenu
  ops_status_view.{h,cpp}      composes indexing/embedding progress for the status pane
```

## Implementation notes

- `Indexer::process_events()` got an internal `std::mutex` to serialize concurrent calls — pump thread vs tests that call directly.
- `WatcherPump::flush_locked()` releases its own mutex while `indexer_.process_events()` runs, so external `flush_now()` callers don't block on long batches.
- `Workspace` destructor explicitly resets `watcher_pump_` first to ensure it stops + joins before `FileWatcher` and `Indexer` destruct.
- Index-stats refresh in GUI is now driven by `on_agent_indexing_progress(done >= total)` since the pump runs off-UI-thread.

## Notes

- Watcher pump's tick source: keep it simple — a `std::thread` with a condvar-sleep is fine; a wx-timer-only version was GUI-coupled. The GUI just subscribes to `on_indexing_progress`.
- Be careful of watcher lifetime on workspace switch — the pump must stop + join before `Workspace` destructs its `FileWatcher`.
- Part A and Part B are independent; ship either first.
