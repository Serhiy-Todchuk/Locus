# S3.I — Threading-model documentation + invariants

## Problem

Four threads exist today: UI (wx main), agent (AgentCore), embedding worker, file-watcher (efsw internal). They share: `AgentCore::history_`, `ConversationHistory`, `FrontendRegistry`, two SQLite connections, activity ring buffer. Each has ad-hoc synchronization — `std::mutex` sprinkled where someone noticed a race.

M4 multiplies threads:
- **S4.E (LSP)**: JSON-RPC reader thread per language server.
- **S4.G (MCP)**: stdio reader thread per MCP server.
- **S4.H (parallel tool calls)**: thread pool for read-only tools.
- **S4.I (background commands)**: one thread per long-running process.
- **S4.U (subagents)**: child `AgentCore` threads.

Today's "happens to work" assumptions will break silently when subagents write to conversation history from a different thread.

## Target

A single authoritative doc, plus a small number of codified invariants.

### Document: `architecture/threading-model.md`

Sections:
1. **Thread inventory** — table: thread name, owner, what it reads, what it writes, how started/stopped.
2. **Shared resources** — for each: what protects it (mutex, atomic, single-writer invariant).
3. **Invariants** — cross-cutting rules that must hold. E.g.:
   - *The agent thread is the only writer to `ConversationHistory`.*
   - *SQLite main connection is owned by `AgentCore` thread and the indexer thread; never both simultaneously — WAL lets readers proceed.*
   - *Frontend callbacks fire on the agent thread; frontends must post to their own UI thread if they need to touch widgets.*
4. **How to add a new thread** — checklist: which invariants to preserve, which shared resources it may touch.

### Code changes (minimal)

- Document-only where the code is already correct.
- Where it isn't: add `assert()` / `spdlog::trace` fences that catch future violations (e.g., a `thread_id` check in `ConversationHistory::add()`).
