# S3.I ‚Äî Threading-model documentation + invariants

## Problem

Four threads exist today: UI (wx main), agent (AgentCore), embedding worker, file-watcher (efsw internal). They share: `AgentCore::history_`, `ConversationHistory`, `FrontendRegistry`, two SQLite connections, activity ring buffer. Each has ad-hoc synchronization ‚Äî `std::mutex` sprinkled where someone noticed a race.

M4 multiplies threads:
- **S4.E (LSP)**: JSON-RPC reader thread per language server.
- **S4.G (MCP)**: stdio reader thread per MCP server.
- **S4.H (parallel tool calls)**: thread pool for read-only tools.
- **S4.I (background commands)**: one thread per long-running process.
- **S4.U (subagents)**: child `AgentCore` threads.

Today's "happens to work" assumptions will break silently when subagents write to conversation history from a different thread.

## Target

A single authoritative doc, plus a small number of codified invariants.

### Document: `architecture/threading-model.md`

Sections:
1. **Thread inventory** ‚Äî table: thread name, owner, what it reads, what it writes, how started/stopped.
2. **Shared resources** ‚Äî for each: what protects it (mutex, atomic, single-writer invariant).
3. **Invariants** ‚Äî cross-cutting rules that must hold. E.g.:
   - *The agent thread is the only writer to `ConversationHistory`.*
   - *SQLite main connection is owned by `AgentCore` thread and the indexer thread; never both simultaneously ‚Äî WAL lets readers proceed.*
   - *Frontend callbacks fire on the agent thread; frontends must post to their own UI thread if they need to touch widgets.*
4. **How to add a new thread** ‚Äî checklist: which invariants to preserve, which shared resources it may touch.

### Code changes (minimal)

- Document-only where the code is already correct.
- Where it isn't: add `assert()` / `spdlog::trace` fences that catch future violations (e.g., a `thread_id` check in `ConversationHistory::add()`).

## Implementation notes

- Ownership is enforced via `ConversationHistory::owner_thread_id_` (an `std::atomic<std::thread::id>`). Default value `std::thread::id{}` means "no owner" ‚Äî any thread may mutate, matching how the M0‚ÄìM2 code has always used the class in tests and in inter-turn operations.
- `AgentCore::agent_thread_func` wraps each `process_message` call in a `ConversationOwnerScope` RAII guard. Owner is set on turn entry and cleared on turn exit; between turns the fence is dormant, so CLI REPL `/reset` / `/compact` and the GUI equivalents continue to mutate `history_` directly from the caller thread while the agent thread is idle on `queue_cv_`. That caveat is documented in `threading-model.md` ¬ß5 so future callers who fire during an active turn know to route through the agent thread.
- Custom copy/move constructors and assignment operators on `ConversationHistory` were required once the atomic member made the defaulted ones ill-formed. The assignment overloads call `assert_owner_thread("operator=...")` so `history_ = sessions_.load(...)` during an active turn still trips the fence.
- Assertions fire in `add`, `clear`, `replace_system_prompt`, `drop_tool_results`, `drop_oldest_turns`, and both assignment operators ‚Äî the complete set of mutating entry points.
