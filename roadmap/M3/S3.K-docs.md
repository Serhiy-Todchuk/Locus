# S3.K — Architecture docs + ADR trail

## Problem

[architecture/](../../architecture/) today has `overview.md`, `tool-protocol.md`, `workspace-index.md`, `tech-stack.md`, `web-retrieval.md`. These describe **subsystems**. They don't describe the two things a new contributor (human or AI) most needs:

1. **The agent loop itself** — how a user message becomes a tool call becomes another LLM round, including approval, compaction, session save, error recovery. Today this is implicit in [src/agent_core.cpp](../../src/agent_core.cpp); after S3.A it will be split across 5 files. Without a doc, the next person reads all 5 files to learn the loop.
2. **The threading model** — covered separately in [S3.I](S3.I-threading-model.md).

Beyond that, the project has accumulated ~30 significant decisions (split index db, llama.cpp over ONNX, wxWebView over CEF, two-step semantic query, FrontendRegistry pattern, `.locus/` config dir, etc.) recorded only in commit messages, [CLAUDE.md](../../CLAUDE.md) tables, and the user's memory. New decisions get re-litigated; old decisions get accidentally reversed.

## Target

### New docs

- `architecture/agent-loop.md` — the canonical sequence diagram and prose:
  - User input → `AgentCore::send_user_message` → history append → LLM stream → tool-call extraction → approval gate → tool dispatch → result append → loop until LLM returns text-only → idle. Covers compaction triggers, session autosave, error / cancel paths.
  - Updated to reflect post-S3.A collaborator split (agent_loop, tool_dispatcher, etc.).
- `architecture/threading-model.md` — see [S3.I](S3.I-threading-model.md). Listed here so it shows up in the doc index together.
- `architecture/decisions/` — lightweight ADR trail. One file per decision: `NNNN-short-title.md`. Each is ~1 page: Context / Decision / Consequences / Alternatives considered.

### ADR seeds (ship with the folder, ~6 to start)

- `0001-workspace-bound-single-folder.md` — one app instance per folder
- `0002-split-index-and-vectors-db.md` — why `.locus/index.db` + `.locus/vectors.db`
- `0003-llama-cpp-over-onnx.md` — embedder choice
- `0004-wxwidgets-and-wxwebview.md` — desktop UI stack
- `0005-frontend-registry-pattern.md` — multi-frontend over single-frontend
- `0006-tools-not-context-stuffing.md` — the architectural axiom

Future M3/M4/M5 decisions get a new ADR rather than a buried commit message.

## Notes

- ADRs are not change requests. They record **decisions that have already been made**. Past decisions get an ADR retroactively only when worth the effort (the 6 above). Future decisions should get an ADR at the time of the decision.
- Don't build tooling around ADRs (no template generator, no linter). The whole point is they're cheap to write.
- This stage is the natural place to also retire any architecture doc that has gone stale during M3 — sweep the folder for outdated paragraphs while writing the new ones.
