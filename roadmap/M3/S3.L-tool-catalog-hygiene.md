# S3.L — Tool-catalog hygiene ✔

## Problem

M4 adds ~20 built-in tools to the 12 we have today (S4.A/P/M/L/E/I/U/R/D); S4.G (MCP) lets users attach unbounded external tools at runtime. End state: 30–50 tools exposed at once.

On local LLMs in the 13–30B range, a large tool catalog degrades three things:

- **Prompt bloat.** Each tool schema is ~150–250 tokens. 32 × 200 ≈ 6–7K tokens of system prompt before the user types a word — ~20% of a 32K context.
- **Selection accuracy.** Small models fumble when several tools overlap. The search family is the landmine: `search_text` / `search_symbols` / `search_semantic` / `search_hybrid` + M4's `grep` (S4.P) + `ast_search` (S4.M) = six search-ish tools.
- **Schema-conformance errors.** More tools with richer parameter shapes → more malformed tool calls (S4.N is the partial counter).

S4.Q (weak/strong split) and S4.F (KV cache discipline) mitigate from the LLM side; this stage fixes the catalog itself.

## Target

Three independent changes, shippable one at a time.

### 1. Consolidate the search family

Single `search` tool with a `mode` parameter:

```
search(query, mode: text|symbols|semantic|hybrid|regex|ast, max_results?, kind?, language?)
```

`SearchTextTool` / `SearchSymbolsTool` / `SearchSemanticTool` / `SearchHybridTool` disappear from the public catalog — one `SearchTool` face dispatches to the existing implementations in `src/tools/search_tools.cpp`. S4.P (`grep`) and S4.M (`ast_search`) ship as `mode=regex` / `mode=ast` instead of new top-level tools.

Cuts ~1K tokens from the manifest, removes the worst selection ambiguity.

### 2. Context-gated tool exposure

`ITool` gains optional predicates; `ToolRegistry::build_schema_json(ws, mode)` filters by them when building the per-turn manifest:

- `available(IWorkspaceServices&)` — default `true`. Overridden by e.g. git tools (check for `.git/`), LSP tools (check for attached server), process-inspection tools (check registry non-empty).
- `visible_in_mode(ToolMode)` — default `{agent}`. `ToolMode { agent, plan, subagent }`. Execute-side tools (`write_file`, `edit_file`, `run_command`, `delete_file`) hide in plan mode; subagents receive a role-specific subset.

Existing tools don't change behavior — the defaults preserve today's visibility.

### 3. Token-cost guardrail

`AgentCore` logs the manifest size (tokens) on each turn at `info` level. Warns when it crosses a threshold (default 4K tokens, configurable in `.locus/config.json`). Makes regressions visible; doesn't block them.

## Notes

- Land **before** the first M4 stage that adds new tools (realistically S4.A). Every subsequent M4 tool is then authored against the new hooks instead of retrofitted.
- The `available()` hook is what lets S4.L/S4.E/S4.I ship tools without inflating the default manifest — they show up only when their substrate (git repo, LSP server, bg process) exists.
- The deferred/on-demand schema pattern (Claude Code's ToolSearch) is intentionally **not** here. Worth considering at 40+ tools; premature at this size.
- S4.V is the natural fallback location if any of the three changes slips past M3.
