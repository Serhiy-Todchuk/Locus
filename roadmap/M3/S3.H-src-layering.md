# S3.H — `src/` layering

## Problem

`src/` today is flat-ish: core classes, tool implementations, LLM client, index query, agent core, workspace at the root; GUI in `src/gui/`; frontends in `src/frontends/`; extractors in `src/extractors/`. The inconsistency ("why is `indexer.cpp` at the root but `extractor_registry.cpp` in a subfolder?") makes it hard for a collaborator — human or AI — to guess where a new file belongs.

S3.A, S3.B, S3.D, S3.E each introduce their own subfolder. This stage makes it consistent across the codebase.

## Target layout

```
src/
  core/         workspace, database, file_watcher, watcher_pump, frontend registry,
                locus_session, workspace_services, activity_event
  agent/        agent_core, agent_loop, tool_dispatcher, activity_log,
                context_budget, conversation, session_store, system_prompt,
                slash_commands
  llm/          llm_client, openai_transport, sse_parser, stream_decoder(s),
                llm_router, token_counter
  index/        indexer, index_query, chunker, glob_match, tree_sitter_registry,
                symbol_extractor(s), prepared_statements
  tools/        file_tools, search_tools, index_tools, process_tools,
                interactive_tools, shared, tool_registry
  extractors/   (exists)
  frontends/
    cli/        cli_frontend
    gui/        (existing src/gui moved here)
    crow/       (future, M5 S5.3)
  util/         small utilities
  main.cpp      stays at src root (CLI entry)
```
